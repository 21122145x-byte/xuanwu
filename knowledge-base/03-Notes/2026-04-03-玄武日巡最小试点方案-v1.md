# 第一部分：我的判断

## 一、总体判断

这套设想的方向是对的，
而且是真有机会活起来的。

它真正有价值的地方，不是“用了低成本模型”，
而是：

> **把巡检执行、状态监控、风险判断三层真正拆开了。**

当前我最认可的结构是：
- **skill**：定义怎么巡
- **cron + Minimax**：按时去巡
- **heartbeat**：盯有没有正常巡
- **玄武主 Agent**：最后判断、收口、升级

这条职责线如果守住，
后面才不会滑成：
- 主 Agent 每天亲手跑所有重复检查
- 低成本模型被误当成最终裁决者

---

## 二、我最认同的三点

### 1. 四层拆分是对的
这份架构最值钱的地方就是：
- skill 不负责定时
- cron 不负责最终判断
- heartbeat 不替代巡检
- 主 Agent 不再亲手做所有重复劳动

这条层级关系现在是成立的。

### 2. heartbeat 不替代巡检
heartbeat 只看：
- cron 有没有按时跑
- 结果文件有没有生成
- 是否连续失败
- 是否出现沉默失效

这个边界很重要。
heartbeat 该像监控器，不该像第二执行器。

### 3. Minimax 放在执行层是合理的
如果任务是：
- 结构化检查
- 规则比对
- 路径存在性检查
- 固定模板输出

那把它挂在 Minimax 这种低成本模型上，是合理的。

它适合做：
> **执行层 + 初筛层**

不适合做：
> **高风险制度裁决层**

---

## 三、现在最需要防的 3 个问题

### 1. 不要让 skill 长成“大而全巡检总表”
玄武日巡 skill 只保留：
- 高频
- 固定
- 结构化
- 可模板化
的检查项。

凡是：
- 低频
- 重判断
- 长上下文
- 易歧义

都不要塞进每天自动巡里。

---

### 2. 不要让 cron 变成隐形 orchestrator
cron 只负责：
> **按时触发这一轮 skill**

不要偷偷承担：
- 流程编排
- 多步治理
- 自动修复
- 多环节串联

---

### 3. 模板要兼顾机器可读
当前巡检模板适合人看，
但如果以后想让玄武主 Agent 稳定接结果，建议再增加机器可读字段，例如：
- `status`: ok / warn / block
- `severity`: P0 / P1 / P2
- `needs_review`: true / false
- `evidence_paths`: []
- `suggested_action_type`: observe / retry / escalate / ignore

一句话：
> **模板下一步要从“人能看”升级到“人能看 + 机器能接”。**

# 第二部分：玄武日巡最小试点方案 v1

## 四、试点目标

本轮试点不追求完整自动化，
只验证一件事：

> **这套“skill + cron + Minimax + heartbeat + 主 Agent 收口”的结构，能不能稳定跑通一条最小主巡检线。**

一句话：

> **先验证结构，再考虑扩规模。**

---

## 五、试点范围（只做最小版）

本轮试点只做：
- **1 条主巡检线**
- **1 个 daily-inspection skill**
- **1 条主巡检 cron**
- **1 条 heartbeat 状态监控规则**
- **1 份结构化巡检结果模板**

本轮不做：
- 多条 cron 并发
- 多子 skill 联动
- heartbeat 上模型做深巡检
- 自动升级 / 自动修复
- 周复盘 / 月复盘自动化

---

## 六、试点职责拆分

### 1. skill（巡检逻辑层）
只负责：
- 定义巡检对象
- 定义检查项
- 定义通过 / 异常 / 升级规则
- 定义输出模板

不负责：
- 定时触发
- 最终裁决
- 自动修复

### 2. cron（执行层）
只负责：
- 每天固定时间触发一次主巡检
- 调用 skill 跑一轮
- 落结果文件

不负责：
- 多步编排
- 风险裁决
- 长期追踪

### 3. heartbeat（监控层）
只负责：
- 看 cron 是否按时跑
- 看结果文件是否生成
- 看是否连续失败
- 看是否出现沉默失效

不负责：
- 代替巡检
- 代替主 Agent 做结论
- 自己下高风险判断

### 4. 玄武主 Agent（判断层）
只负责：
- 审阅结果
- 判断是否升级
- 判断异常是否成立
- 形成正式治理结论

不负责：
- 每天亲手跑所有重复检查

---

## 七、试点建议检查项（仅保留固定高频项）

建议试点只保留以下几类：

### A. 运行态
- gateway / node 是否在线
- channel / account 是否明显异常

### B. 结果生成链
- cron 是否按时触发
- 巡检结果文件是否生成

### C. 明显错误簇
- gateway.err.log 是否有新增 ERROR / Exception
- 是否有 read-without-path / ENOENT / timeout 这类高频固定问题

### D. 状态一致性（轻量版）
- status / cron / session 是否出现明显不一致

### E. 升级标记
- 是否存在需要主 Agent 复核的问题

本轮不要把：
- autorecall 深诊断
- topic judge 深诊断
- 长链治理判断

都塞进最小试点。

---

## 八、试点节奏

### 主巡检 cron
- **每天 09:30** 跑 1 次

### heartbeat 监控
- 只负责检查：
  - 这轮主巡是否跑了
  - 结果有没有生成
  - 是否连续失败

### 主 Agent 收口
- 玄武主 Agent 在巡检结果出来后，再做：
  - 是否升级
  - 是否补诊断卡
  - 是否进入问题池

## 十、试点通过标准

本轮试点只要满足以下 4 条，就算通过：

1. cron 能按时触发 1 次主巡检
2. Minimax 能稳定产出 1 份结构化巡检结果
3. heartbeat 能发现“没跑 / 没出结果 / 连续失败”
4. 玄武主 Agent 能基于结果稳定判断哪些要升级复核

---

## 十一、试点不通过标准

出现以下任一情况，当前结构应视为未跑通：
- cron 经常不触发
- 结果文件不稳定生成
- 输出模板过散，主 Agent 无法稳定接结果
- heartbeat 无法识别沉默失效
- Minimax 因 prompt 过重导致结果明显漂移

---

## 十二、试点跑通后的下一步

只有最小试点跑通后，再考虑：
1. 扩更多检查项
2. 增加轻量复查 cron
3. 接问题池刷新
4. 接周复盘 / 月复盘
5. 评估是否需要更复杂的多 cron 结构

---

## 十三、一句话收口

> **玄武日巡最小试点的目标，不是一次性做完整自动化，而是先用一条主巡检线验证：执行层、监控层、判断层是否真的能稳定拆开并接起来。**
