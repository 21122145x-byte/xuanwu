# 给粉雪的巡查改造建议 v1

- 日期：2026-04-03
- 对象：粉雪
- 目的：把粉雪的巡查动作收口到“协作链是否堵塞”的岗位主责，不与玄武当前的系统治理日巡 skill 重叠。

---

## 一、结论先行

粉雪的巡查，不应继续往“系统深度巡检”方向长。

建议直接收口为：

> **看协作链有没有堵。**

也就是：
- 任务有没有卡在已分发未回收
- handoff 有没有超时未闭环
- 多 Agent 协作有没有无人承接 / 责任悬空 / 长时间停滞
- 发现堵点后，只做轻量提醒与升级，不做深度诊断

一句话：

> **粉雪负责暴露协作堵点，不负责承担系统治理。**

---

## 二、为什么要改

当前团队里，玄武已经有成型的系统治理巡检 skill：
- `xuanwu-daily-inspection`

玄武负责的是：
- gateway / node / channel / account 健康
- status / heartbeat / cron / session 一致性
- memory / autorecall / topic judge
- timeout / aborted / 参数缺失 / ENOENT
- token / cache / context / concurrency 风险

Anna负责的是：
- 审计
- 裁决
- 规则收口

在这个格局下，粉雪再去扩系统巡查，会出现三种问题：
1. 与玄武重叠
2. 把调度位做重，失去轻量协作敏感度
3. 明明是协作堵塞，却被伪装成“系统巡检问题”

所以现在最合理的改造方向不是“让粉雪更像玄武”，而是：

> **让粉雪更像粉雪。**

---

## 三、粉雪应保留的巡查动作

### 1. 协作状态检查
保留，而且应升为第一优先。

固定检查：
- 是否有任务卡在“已分发但未回收”
- 是否有 handoff 超过 2 小时未闭环
- 是否有任务分配后无人承接
- 是否有多 Agent 任务停滞超过 4 小时

这就是粉雪最该盯的主轴。

---

### 2. 承诺履约检查
保留。

固定检查：
- 我刚刚让谁去做什么
- 对方现在有没有按承诺推进
- 我自己承诺要推进的事情有没有卡住

这也是协作位的核心动作。

---

### 3. Cron 连续失败检查
保留，但只保留“协作升级”层。

意思是：
- 粉雪可以继续看 cron 有没有连续失败
- 但发现失败后，不负责深度系统诊断
- 她负责的是：
  - 记录
  - 提醒责任位
  - 若持续失败则上报 Shaun / 转玄武

---

### 4. 关键服务异常提醒
保留“提醒”，不保留“深查”。

例如：
- gateway.err.log 有明显 ERROR/Exception
- Discord / Feishu 有明显报错

粉雪可以做：
- 发现
- 轻量记录
- 升级给玄武 / Shaun

粉雪不应做：
- 根因诊断
- 系统治理建议
- 长链修复判断

---

## 四、粉雪应删除或下放的动作

以下内容不要继续长在粉雪的巡查逻辑里：

### 1. 系统深度巡检
例如：
- gateway / node / channel / account 全量健康检查
- status / heartbeat / cron / session 一致性深查
- runtime failure cluster 分型
- topic judge / autorecall / prependContext 污染诊断
- token / cache / context / concurrency 风险分析

这些已经属于玄武。

---

### 2. 复杂根因诊断
例如：
- 为什么 timeout
- 为什么 ENOENT
- 为什么 tool calling 空参数
- 为什么 session 路由漂移

这些也应交给玄武 / Anna，不是粉雪 heartbeat 位处理。

---

### 3. 高风险修复动作
粉雪不应因为 heartbeat 发现异常就去做：
- 高风险系统改动
- 配置修复
- 巡检链路大重构

她的边界应该是：
- 发现堵点
- 升级责任
- 推动闭环

---

## 五、改造后的 HEARTBEAT 核心动作（建议版）

建议粉雪的 HEARTBEAT 只保留 4 个动作：

### 动作 1：协作链堵塞检查
- 已分发未回收
- handoff 超时未闭环
- 多 Agent 协作任务长时间无人承接 / 无推进

### 动作 2：承诺履约检查
- 对外承诺有没有兑现
- 上轮要求的回收动作有没有落地

### 动作 3：连续失败提醒
- cron 是否连续失败 ≥ 3
- 关键通道是否持续报错
- 若持续失败，升级责任位

### 动作 4：升级流转
根据问题类型直接分流：
- 协作堵塞 → 粉雪自己推动闭环
- 系统治理 / runtime / config / memory / routing → 升级玄武
- 审计裁决 / 规则收口 / 是否放行 → 升级 Anna
- 影响全局节奏或需要拍板 → 升级 Shaun

---

## 六、最小改写建议

把粉雪 HEARTBEAT 里的口径压成下面这一句就够了：

> **HEARTBEAT 对粉雪的作用，不是做系统深巡，而是固定检查：协作链有没有堵。**

然后动作就围绕：
- 有没有卡单
- 有没有超时 handoff
- 有没有无人承接
- 有没有持续失败需要升级

---

## 七、和玄武 / Anna 的边界

### 粉雪
- 看协作链堵没堵
- 推进闭环
- 做提醒与升级

- 看系统链坏没坏
- 做系统治理巡检
- 产出诊断卡 / 治理建议

### Anna
- 做审计
- 做裁决
- 做规则收口

这三者不要混。
