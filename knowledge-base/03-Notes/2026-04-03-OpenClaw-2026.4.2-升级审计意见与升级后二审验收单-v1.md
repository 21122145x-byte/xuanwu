## 三、证据

### 1. 2026.4.2 中与当前环境相关的正向价值
本次 release 包含多项对当前使用场景有直接价值的修复：

#### A. Agents / subagents
- 修复 subagent / gateway calls 与 pairing / scope-upgrade 相关问题
- 对 `sessions_spawn` 稳定性有直接正收益

#### B. Providers / transport / routing / headers
- 一整组 provider transport、routing、headers 收口修复
- 这类修复通常直击链路层稳定性，而不是表面功能堆料

#### C. Gateway / exec loopback
- 修复 `2026.3.31` 之后本地 exec / node client 可能出现的 pairing-required 类问题

### 2. 2026.4.2 中的风险项
本次 release 含 breaking changes：

#### A. xAI `x_search` 配置路径迁移
旧路径迁移到 plugin-owned path。

#### B. Firecrawl `web_fetch` 配置路径迁移
旧路径迁移到 plugin-owned path。

官方给出迁移方式：
- `openclaw doctor --fix`

这意味着：
- 若当前环境依赖这两类旧配置，直接升级可能产生配置断点。

### 3. issue 类型二审结果
当前最相关风险簇的代表问题包括：

#### Discord / DM / routing
- Discord DMs can fork into duplicate channel sessions
- Discord DM messages routed to channel instead of user session
- DMs routed to main agent do not collapse correctly
- slash commands ignore `session.dmScope` routing
- Discord DMs inbound messages silently dropped

#### runtime / subagents / sessions_spawn
- `sessions_spawn(runtime="subagent")` model override 不生效
- subagents 不加载目标 workspace context
- `sessions_spawn` 因 schema 参数暴露异常失败
- ACP / subagent runtime 行为边界问题

这些问题说明：
- 当前环境关注点确实落在项目已知高风险带上
- 因此不能升级后不验

## 四、根因判断

本次是否值得升级，不能只看：
- issue 总量高不高
- 仓库 bug 多不多

真正关键的是：
- 新版本改动是否命中我们当前痛点
- 新版本是否在修当前高风险链路，而不是引入无关扰动

本次判断结果是：

> **2026.4.2 的改动，确实直接命中当前环境最容易踩的链路层问题。**

因此，审计口径不是“因为 issue 多所以别升”，而是：

> **值得升，但要验。**

## 五、风险

### 风险 1：Discord 私聊路由仍处于已知高风险区
即使升级，DM routing 仍是必须优先验收的生死线。

### 风险 2：subagent / `sessions_spawn` 仍需实测
虽然 4.2 有修复，但这类问题历史上属于高风险簇，不能只看 release 文案放行。

若在用：
- `x_search`
- Firecrawl `web_fetch`

升级后若不做迁移检查，容易出现功能断点。

## 七、升级后二审验收单（只验最关键 5 项）

## 1. Discord 私聊路由
### 要验什么
- Discord 私聊消息是否进入正确 session
- 不误路由到 channel session
- 不出现 duplicate / fork / silent drop

### 通过标准
- 私聊发一条普通消息
- 回复正常
- session 归属正确
- 没有重复 session / 分叉 session / 消息丢失

### 不通过现象
- DM 被路由错
- DM 不进主会话
- 入站消息丢失
- 一条消息分裂成多个 session

---

## 2. subagent / `sessions_spawn`
### 要验什么
- subagent 是否能正常拉起
- 不出现 pairing required / scope-upgrade 类错误
- 子 agent 能正常返回结果

### 通过标准
- 成功 spawn 一个最小任务
- 子 agent 正常执行
- 能回传结果
- 无 `close(1008)` / `pairing required`

### 不通过现象
- spawn 失败
- 子 agent 启不来
- 回传中断
- pairing / 权限报错

---

## 3. 基础工具调用链
### 要验什么
至少覆盖：
- `read`
- `exec`
- 一个网络类工具（如 `web_search` 或 `web_fetch`）

### 通过标准
- 三类工具都能正常调用
- 无明显 schema 异常
- 无空参数异常
- 无无故超时

### 不通过现象
- 工具能说话但不能干活
- 参数组装错
- 执行无记录
- 网络工具异常退化

---

## 4. provider / transport 稳定性
### 要验什么
- 升级后基础回复与工具请求链是否稳定
- 不出现明显代理 / header / transport 回归

### 通过标准
- 普通对话正常
- 至少一次工具调用正常
- 无连续 transport / proxy / websocket 异常

### 不通过现象
- `fetch failed`
- `ENETUNREACH`
- websocket / stream 异常
- provider 请求异常增多

---

## 5. 配置迁移检查
### 要验什么
检查 4.2 breaking changes 是否已正确迁移，重点看：
- `x_search`
- Firecrawl `web_fetch`

### 通过标准
- 已执行：`openclaw doctor --fix`
- 若当前在用上述功能：
  - 配置仍有效
  - 调用仍可用

### 不通过现象
- 升级后相关配置失效
- `web_fetch` / `x_search` 行为异常
- 旧配置路径失效但未迁移

---

## 八、最终判定规则

### 可放行
满足以下条件时，可放行升级结果：
- 第 1 项和第 2 项 **必须通过**
- 第 3、4 项无阻塞级异常
- 第 5 项若涉及现用配置，也必须通过

### 有条件放行
满足以下条件时，可有条件放行：
- 第 1、2 项通过
- 其余仅存在轻微可绕过问题
- 不影响当前主链路使用

### 不放行 / 回退
出现以下任一情况，直接不放行：
- Discord 私聊路由异常
- `sessions_spawn` / subagent 异常
- 工具调用链明显失稳
- breaking change 导致现用配置失效
