# 卡死现象与端点问题高概率主因判断 v1

- 日期：2026-04-04
- 记录人：Anna
- 性质：阶段性判断文档

---

## 一、现象

近期多 Agent 出现了几类相似问题：
- 会话长时间不回复
- 看起来像“卡住”，但没有及时显式报错
- 某些 subagent 形成 running 残留，像僵尸子任务
- 人工一介入、重新触发或重新发消息后，系统又似乎恢复
- 某些主会话则直接落入 `failed`，并出现连续 `502 Bad Gateway`

这些现象出现在：
- 富贵主会话
- 玄武侧 subagent
- Anna 自身部分 cron / 汇报链
- 团队其他 Agent 的长链执行场景中也有类似影子

---

## 二、关键证据

### 1. 富贵主会话不是“纯卡住”，而是明确 failed
富贵主会话曾出现：
- 连续 `502 Bad Gateway`
- 0 input / 0 output
- 最终状态直接落为 `failed`

这说明：
- 并非所有“没回复”都属于单纯挂起
- 至少有一部分是链路层明确失败

---

### 2. system-guardian 的 subagent 存在真实 running 残留
`memos-ui-reset-verify` 这个 subagent：
- 状态为 `running`
- 历史停在 `openclaw gateway restart` 之后
- restart 返回了 tool result，但后续没有继续推进

这说明：
- 某些子任务不是正常 done 后残留展示
- 而是真实停在链路中途，形成僵尸子任务

---

### 3. 切换端点后，系统突然顺畅
最关键的新证据是：
- 在切换端点之后，原本卡死 / 不顺的现象立刻缓解
- 新 turn / 新轮次恢复明显

这条证据意味着：
- 问题不是单纯因为任务本身天然跑不动
- 也不是所有问题都能归因于 prompt 或任务太长
- 至少在当前这批现象里，端点切换对恢复有显著影响

---

## 三、根因判断

当前最合理的判断是：

> **“卡死”现象背后，端点 / provider / 路由链问题，已经从怀疑项升级为高概率主因。**

更准确一点说：
- 某些 turn 很可能死在坏端点上
- 某些子任务卡在链路切换点或上游响应异常处
- 后续看起来“恢复”，不一定是旧 turn 活过来，而可能是新 turn 在更健康的端点上重新跑起来了

---

## 四、为什么不能只把问题归因于“任务太长”

任务太长、上下文太厚，确实会增加脆弱性。
但当前这批现象如果只解释成“任务太长”，证据不够。

因为这次我们已经看到：
- 切端点后，系统顺了
- 这说明链路条件变化本身就能显著影响结果

所以当前更合理的排序应该是：

1. **端点 / provider / 网关链问题**
2. turn 挂起 / session 状态失真
3. 长链任务放大了问题，但不一定是主因

也就是说：
> 长链任务是放大器，端点问题更像触发源。 

---

## 五、对系统设计的启发

这件事进一步证明：

### 1. 长链任务不该压在单个主 turn 里
因为一旦端点挂了：
- 整轮容易死
- 旧 turn 死后还容易被新 turn 恢复的假象掩盖
- 不容易做精细 kill

### 2. spawn / 短链化仍然是对的
因为链条越短：
- 故障半径越小
- 更容易重派
- 更容易局部恢复
- 不会整条任务一起被坏端点拖死

### 3. 以后看到“卡死”，优先先查链路
不要默认先怪：
- Agent 不努力
- Prompt 不对
- 任务不会做

而应优先问：
> 这是不是端点 / provider / 路由链先出了问题？

---

## 六、当前结论边界

### 当前可以成立的结论
- 端点 / provider / 路由链问题，已是近期“卡死 / 假恢复 / 僵尸子任务”现象的高概率主因
- 长链任务会放大这种问题，但当前不应被视为唯一主因

### 当前仍不能过度声称的结论
- 不能说所有卡死都只有端点一个原因
- 不能说任务形态完全无关
- 不能说已经完成最终根因修复

所以当前最准确的口径是：

> **端点问题已经从怀疑项收敛为高概率主因，但仍需继续积累重复样本与链路日志做后续收口。**

---

## 七、后续排查优先顺序建议

以后再遇到类似“卡死 / 假恢复 / 僵尸子任务”，优先顺序建议改为：

1. **先查端点 / provider / 网关链**
2. 再查是不是 turn 挂起 / session 状态失真
3. 最后再查是不是任务本身过长、过厚、不可拆

---

## 八、一句话收口

> **当前阶段可以明确记录：近期多 Agent 的“卡死”现象，端点 / provider / 路由链问题已经从怀疑项升级为高概率主因；长链任务更多是在放大故障，而不再适合作为第一解释。**
