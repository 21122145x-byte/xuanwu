## 二、当前已成立的部分

### 1. 硬边界已经写清
当前执行卡已经明确写死：
- `observe-only`
- `no canonical truth takeover`
- `no blocking by default`
- `no host-structure rewrite`

这四条边界如果不松，这张卡就不会轻易长歪。

### 2. 宽度已经明显收住
当前只保留了 3 个监控点：
- 风险分类监控点
- subagent 并发 / 限额监控点
- stream / dropped event / heartbeat 观测点

没有再把：
- run / thread 并线
- 完整 message bus 对象化迁移
- store / checkpointer 对齐

这些 Phase 2 内容混进来。

### 3. 每个监控点已具备执行卡最小形状
每个点现在基本都写到了：
- 目标
- 当前不做
- 当前要做
- 可借来源
- 落点建议
- 最小观测字段
- 验收方式
- 升级条件

这说明它已经不是研究摘要，而是具备了执行卡应有的结构。

### 4. 结尾建议已主动收缩
文档最后没有写成“三个点一起上”，而是收成：

> **只先放 1 个风险分类点 + 1 个 stream 健康点。**

这一点我认定为积极信号，说明这轮收窄不是形式动作，而是真正在控制 Phase 1 的宽度。

---

## 三、当前仍需守住的边界

### 1. 当前不建议 3 个点一起直接做
虽然执行卡列了 3 个点，但当前不宜把它理解成：
> **Anna 已同意 3 个监控点一起进入实施。**

更稳的口径应是：
> **第一轮只做 2 个点：风险分类点 + stream 健康点。**

原因：
- 这两类更偏纯观测
- 对主调度链扰动更小
- 更容易先建立 observe-only 纪律

### 2. 风险分类点里的 `warn` 不应被偷偷产品化为软阻断
当前方向是 classify / warn / pass / flag，这本身没问题。

但进入实施时必须防止：
> **warning 被默默当成事实上的阻断信号。**

因此第一轮如实施，warning 只应进入：
- 记录面
- 状态面
- heartbeat / 审计面

不应默认进入用户可见强提示，更不应干扰调度决策。

### 3. stream 健康点需要先压字段定义
当前方向成立，但在实施前，至少要先统一：
- 什么算 `dropped_event`
- 什么算 `heartbeat_gap_detected`
- 什么算 `cleanup_completed`

否则后面容易出现字段有了、口径却不稳的问题。

## 五、下一步建议

### 建议 1：再压一层，形成“第一轮最小实施卡”
建议当前执行卡再往下压一层，只保留：

把 `subagent 并发 / 限额监控点` 保留在本卡中，但不进入第一轮实际接入。

### 建议 2：实施前补一张最小字段定义卡
在真正接入前，建议至少先定义：
- `risk_level`
- `risk_reason`
- `dropped_event_count`
- `heartbeat_gap_detected`
- `cleanup_completed`

这一步是为了防止后续观测字段口径漂移。

---

### 建议 3：第一轮仍坚持 observe-only 验证
第一轮目标不是治理接管，而是：
- 验证字段是否能稳定出现
- 验证状态面是否可读
- 验证是否完全不扰动现有主链
