## 1. 回滚目标

回滚不是“把升级取消掉”这么模糊，
而是：

> **把 OpenClaw 恢复到升级前最后一个已确认可用的稳定状态，并恢复关键主链路可用性。**

本次回滚优先目标：
- 恢复 Discord 私聊路由可用
- 恢复 `sessions_spawn` / subagent 可用
- 恢复基础工具调用链可用
- 避免因 4.2 breaking changes 导致配置链断裂继续扩散

---

## 2. 触发回滚的阻塞条件

出现以下任一情况，直接触发回滚，不继续硬顶：

### A. Discord 私聊路由失败
包括但不限于：
- DM 被错误路由到 channel session
- duplicate / fork session
- inbound silently dropped
- 主私聊链路不可用

### B. `sessions_spawn` / subagent 失败
包括但不限于：
- spawn 失败
- `pairing required`
- `close(1008)`
- 子 agent 无法正常回传

### C. 基础工具链阻塞级失稳
包括但不限于：
- `read` / `exec` 明显失稳
- 工具参数组装异常明显恶化
- 常用网络工具不可用

### D. breaking change 命中现用配置并导致功能断裂
包括但不限于：
- `x_search` 失效
- Firecrawl `web_fetch` 失效
- 配置迁移失败后主链路受影响

### E. 升级后出现明显比升级前更差的系统性回归
包括但不限于：
- fallback timeout 明显恶化
- slow listener 明显恶化
- skill path 越界问题明显恶化并影响正常链路

---

## 3. 不触发回滚的情况

以下情况默认不直接回滚，而是先记录：
- 轻微可绕过问题
- 非主链路的小功能异常
- 与当前使用面无关的边缘回归
- 仅日志变吵，但核心功能仍可用

原则：
- **阻塞主链路才回滚**
- **非阻塞问题先记录，不抢回滚**

---

## 4. 最小回滚顺序

### 第一步：冻结现场
先记录：
- 当前版本状态
- 当前异常现象
- 当前报错文本
- 当前受影响链路

至少保留：
- `openclaw update status`
- 关键报错原文
- 哪一项验收失败

### 第二步：停止继续扩动作
- 不继续新增测试项
- 不继续叠加修补动作
- 避免把回归和补救混在一起

### 第三步：回退到升级前稳定版本
目标版本：
- **升级前最后一个已确认稳定且可工作的本机版本**
- 若本机升级前可见更新为 `2026.4.2`，则通常回退目标应为升级前稳定版本（例如当时实际使用版本）

执行原则：
- 按当前安装方式 `pnpm` 的可回退方式执行
- 不在未确认目标版本号前盲目 downgrade
- 若需要，优先查本机包管理记录 / 全局安装记录 / lock 信息

### 第四步：必要时重新执行修复性迁移动作
若升级中改动过配置：
- 检查是否需要恢复配置文件
- 检查 `doctor --fix` 是否引入了路径迁移
- 确保配置回到升级前可用状态

### 第五步：做最小复验
回滚后至少复验：
1. Discord 私聊路由
2. `sessions_spawn` / subagent
3. `read`
4. `exec`
5. 一项网络类工具

---

## 5. 回滚后通过标准

回滚后只有满足以下条件，才算“回滚成功”：
- 主私聊链路恢复可用
- `sessions_spawn` / subagent 恢复可用
- 基础工具链恢复到升级前基线水平
- 无新增阻塞级配置断点

如果只是“版本退回去了”，但关键链路没恢复：
- 不算成功回滚
- 需继续作为故障事件处理
