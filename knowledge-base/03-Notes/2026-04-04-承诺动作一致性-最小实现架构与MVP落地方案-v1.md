# 承诺-动作一致性：最小实现架构与 MVP 落地方案 v1

- 日期：2026-04-04
- 性质：实现方案文档
- 上游规则：`/Users/xiaojingbo/.openclaw/workspace/knowledge-base/03-Notes/承诺-动作一致性系统约束_v1_2026-04-04.md`

---

## 一、目标

这份文档只解决一个问题：

> **系统如何真正具备“自动拦截 + 自动记账 + 后续可审计”的最小能力。**

不追求一次做成完整治理系统，
只追求先把下面这条最小闭环做出来：

- 识别承诺语句
- 判断当前回合有没有真实动作
- 没有动作时触发拦截或记账
- 后续动作出现后能自动结清
- heartbeat / 审计能看见未兑现项

---

## 二、最小实现架构图

```text
[用户输入]
   ↓
[Assistant 生成回复草稿]
   ↓
[Promise Detector 承诺语句检测]
   ↓
[Action Evidence Checker 动作证据校验]
   ├─ 有动作证据 → 正常放行回复
   └─ 无动作证据 → Promise Guard 触发
                          ├─ 路径A：直接拦截回复
                          └─ 路径B：允许发出，但创建 pending_action
                                            ↓
                              [Pending Action Store]
                                            ↓
                 [Tool/Event Resolver 工具事件 / 结果事件回填]
                                            ↓
                ├─ fulfilled（已兑现）
                ├─ failed（已失败）
                ├─ abandoned（已放弃）
                └─ overdue（超时未兑现）
                                            ↓
                         [Heartbeat / Audit Scanner]
                                            ↓
                           输出 OK / WARN / BLOCK
```

---

## 三、模块说明

### 1. Promise Detector
负责：
- 检测 assistant 草稿中是否出现强执行承诺语句

只识别这类表达：
- 我现在去做
- 我立刻执行
- 我去跑
- 我来触发
- 我现在验证
- 我去检查
- 我直接开始

原则：
> **只抓强承诺，不抓普通计划表达。**

---

### 2. Action Evidence Checker
负责：
- 判断当前回合是否已经有“真实动作证据”

最小证据定义：
- 本回合存在 tool call
- 本回合已给出完成结果
- 本回合明确写明未执行及原因
- 本回合明确写明失败及原因

原则：
> **承诺语句不是问题，承诺后无动作证据才是问题。**

---

### 3. Promise Guard
负责：
- 当命中承诺语句且没有动作证据时，决定怎么处理

最小支持两种模式：

#### 模式 A：拦截
- 阻止该回复直接发出
- 系统提示：
  - 检测到承诺语句，但当前回合未进入执行。请先执行再回复，或改写为未执行说明。

#### 模式 B：记账放行
- 允许回复发出
- 同时创建一笔 `pending_action`
- 供后续 heartbeat / 审计追踪

建议：
> **先做记账放行，再逐步增加硬拦截。**

因为这样更稳，不容易先把正常协作误伤掉。

---

### 4. Pending Action Store
负责：
- 保存“说了要做但还未结清”的动作记录

实现可选：
- MVP 阶段：本地 JSONL / SQLite
- 后续阶段：统一任务表 / 事件表

原则：
> **承诺不能只停在语言里，必须成为一笔系统里的待兑现动作。**

---

### 5. Tool/Event Resolver
负责：
- 监听后续真实动作事件
- 自动把 pending_action 改状态

可接入事件：
- tool call 发起
- tool call 成功 / 失败
- assistant 结果输出
- assistant 明确放弃执行

---

### 6. Heartbeat / Audit Scanner
负责：
- 定期扫描 pending_action
- 输出一致性状态
- 供系统审计和复盘使用

输出建议：
- OK：无异常
- WARN：延迟兑现
- BLOCK：承诺未兑现

---

## 四、字段级 schema 草案

### 1. pending_action

```json
{
  "pending_action_id": "pa_20260404_001",
  "agent_id": "fugui",
  "session_key": "agent:fugui:main",
  "message_id": "1489856288471908423",
  "created_at": "2026-04-04T13:16:00+08:00",
  "promise_text": "我现在去手动跑一次",
  "promise_type": "strong_execution_promise",
  "expected_action_type": "tool_call_or_exec",
  "status": "pending",
  "fulfilled_at": null,
  "failure_reason": null,
  "abandoned_reason": null,
  "evidence": []
}
```

#### 字段说明
- `pending_action_id`：唯一 ID
- `agent_id`：哪个 agent 说的
- `session_key`：所属会话
- `message_id`：承诺所在消息
- `created_at`：承诺创建时间
- `promise_text`：原始承诺语句
- `promise_type`：承诺类型，MVP 先只保留 `strong_execution_promise`
- `expected_action_type`：预期动作类型
- `status`：`pending | fulfilled | failed | abandoned | overdue`
- `fulfilled_at`：兑现时间
- `failure_reason`：失败原因
- `abandoned_reason`：放弃原因
- `evidence`：后续动作证据列表

---

### 2. evidence item

```json
{
  "type": "tool_call",
  "tool_name": "exec",
  "created_at": "2026-04-04T13:16:08+08:00",
  "result": "started",
  "ref": "toolcall_abc123"
}
```

#### 最小 evidence.type 建议
- `tool_call`
- `tool_result_success`
- `tool_result_error`
- `assistant_result`
- `assistant_failure`
- `assistant_abandon`

---

### 3. heartbeat summary（可选汇总结构）

```json
{
  "check_name": "promise_action_consistency",
  "checked_at": "2026-04-04T14:00:00+08:00",
  "status": "WARN",
  "pending_count": 1,
  "overdue_count": 1,
  "recent_examples": [
    {
      "agent_id": "fugui",
      "promise_text": "我现在去手动跑一次",
      "status": "overdue"
    }
  ]
}
```

## 五、状态流转草案

```text
pending
  ├─(检测到真实动作并完成)────────→ fulfilled
  ├─(检测到真实动作但执行失败)────→ failed
  ├─(明确放弃 / 改方案)──────────→ abandoned
  └─(超过时限仍无动作)──────────→ overdue
```

### 最小时限建议
MVP 不做复杂 SLA，先用简单规则：
- 同一回合结束仍无动作 → 可直接记 `pending`
- 进入下一轮仍无动作 / 超过固定分钟数 → 记 `overdue`

---

## 六、MVP 落地顺序

## MVP-1：承诺语句检测 + 记账
### 目标
先把“说了没做”变成系统里可见的一笔记录。

### 要做的事
1. 在 assistant 回复发送前，检测是否命中强承诺语句
2. 若命中，则创建 `pending_action`
3. 先不硬拦截，只记账

### 为什么先做这个
- 风险最小
- 最容易上线
- 先解决“完全无留痕”问题

### MVP-1 通过标准
- 命中承诺语句时，系统能稳定创建 `pending_action`
- 不命中时，不乱建记录

---

## MVP-2：动作事件回填 + 自动结清
### 目标
让 pending_action 不是死账。

### 要做的事
1. 接入工具调用事件
2. 接入工具结果事件
3. 接入 assistant 明确失败 / 放弃表达
4. 自动更新：
   - `fulfilled`
   - `failed`
   - `abandoned`

### 为什么第二步做这个
因为只有先能记账，
后面才谈得上自动结清。

### MVP-2 通过标准
- 至少能用 tool call / tool result 自动结清大多数执行型任务
- 不需要人工回填才知道任务做没做

---

## MVP-3：heartbeat 扫描 + 异常暴露
### 目标
让承诺未兑现从“系统知道”升级到“系统会报”。

### 要做的事
1. heartbeat 定时扫描 pending_action
2. 输出：
   - `OK`
   - `WARN`
   - `BLOCK`
3. 将 `overdue` 项暴露给审计 / 复盘

### 为什么第三步再做
因为前两步没接好时，heartbeat 只会报一堆假异常。

### MVP-3 通过标准
- overdue 项能被 heartbeat 稳定看见
- 能按 agent / session 追踪未兑现承诺

---

## MVP-4（可选）：硬拦截模式
### 目标
在规则稳定后，再把部分场景升级为真正拦截。

### 要做的事
- 当用户明确要求“立刻 / 直接 / 做完再回”
- 且 assistant 只给了承诺语句、没有真实动作
- 直接拦截回复

### 为什么放最后
因为硬拦截最容易误伤，
必须建立在前面检测规则已经足够稳的基础上。

---

## 七、推荐实现策略

### 1. 先轻后重
先记账，再结清，再 heartbeat，再硬拦。
不要一上来就全拦。

### 2. 先抓强承诺，不抓模糊计划句
先把误伤概率压低。

### 3. 先接工具事件，不急着做复杂 NLP
当前目标不是语言学完美识别，
而是先把最痛的“说了没做”抓出来。

---

## 八、已知风险

### 1. 误报风险
有些计划型句子可能被误识别成承诺。

### 2. 漏报风险
有些变体表达可能绕开关键词检测。

### 3. 结清不完整风险
如果工具事件流没接好，`pending_action` 可能积压。

所以 MVP 阶段一定要接受：
> **先不求完美识别，先求能抓住高价值违约。**

---

## 九、一句话收口

> **承诺-动作一致性的最小实现，不是先做一个大而全规则引擎，而是先补一条最小闭环：承诺检测 → pending_action 记账 → 动作事件结清 → heartbeat 暴露，再根据真实误报/漏报逐步收紧。**
