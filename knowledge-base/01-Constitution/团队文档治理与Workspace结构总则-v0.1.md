## 一、总则定位

本总则用于回答：

1. 团队 workspace 与各 Agent workspace 的目录法位如何统一解释
2. 什么是正式件、过程件、证据件、运行态文件、归档件
3. 根目录应长期保留什么，不应长期保留什么
4. 正式知识、运行实施、项目专属交付三者的边界如何区分
5. 文件何时升格、何时归位、何时归档
6. 正式文档最小文档链应包含什么

本总则不承担：
- 某个具体 workspace 的一次性清理实施细节
- 某个 Agent 的本地实例参数
- 某类目录的动作级执行步骤

## 二、统一治理原则

### 2.1 稳定入口原则
- 正式规则必须有稳定入口
- 版本件、历史件、审计件、术语过渡件不得替代正式入口

### 2.2 文档层级原则
- 团队统一治理规则进团队 workspace
- Agent workspace 不承载团队长期上位法，只承载本地实例入口、参数化实现与运行面文件

### 2.3 反膨胀原则
- 不以“说明如何整理”的名义持续制造新噪音
- 同一类规则只能有一个团队统一上位口径

### 2.4 最小必要原则
- 不为未来可能性提前堆过多目录、术语、流程与说明件
- 有更简单可解释路径时，优先采用更简单路径

### 2.5 正式知识 / 运行实施 / 项目专属分区原则
- 正式知识 → `knowledge_base` / `knowledge-base`
- 运行实施主线 → `runtime/` 或等价运行专区
- 项目专属交付与过程产物 → 项目区 / `projects/`

## 三、Workspace 标准结构法位

以下目录为团队统一最小法位说明；具体 workspace 可有差异，但不得违背其承担对象。

### 3.1 根目录
- 放什么：bootstrap/reference、顶层导航、极少数顶层状态入口
- 不放什么：长期并存的项目中间件、提案、审计稿、状态件、过程稿
- 是否承担正式入口法位：是，但仅限顶层导航与少数全局入口
- 默认维护者：workspace 主责任 Agent
- 检索默认：纳入，但应保持低噪音

### 3.2 `docs/`
- 放什么：面向说明、文档型材料、非知识库但需结构化说明的文档
- 不放什么：运行态状态件、即时过程件、长期归档件
- 是否承担正式入口法位：可承担，但需稳定入口
- 默认维护者：对应文档主责方
- 检索默认：纳入

### 3.3 `projects/` / 项目区
- 放什么：项目专属 Plan、program、交付、阶段件、项目过程产物
- 不放什么：团队统一上位法、长期稳定通用能力规则
- 是否承担正式入口法位：承担项目级正式入口，不承担团队长期上位法入口
- 默认维护者：项目主责方
- 检索默认：纳入

### 3.4 `knowledge_base/` / `knowledge-base/`
- 放什么：正式知识、来源卡、研究底稿、专题入口、索引、长期可复用知识
- 不放什么：仅服务当前运行的状态件、纯过程施工件、一次性中间稿
- 是否承担正式入口法位：是，承担正式知识入口
- 默认维护者：知识主责 Agent
- 检索默认：纳入

### 3.5 `runtime/`
- 放什么：运行系统实施件、runtime pipeline、system/harness 接线与运行专区文件
- 不放什么：被误当作知识入口的研究正式件
- 是否承担正式入口法位：是，但法位属于运行实施专区，不属于知识库
- 默认维护者：运行系统主责方
- 检索默认：纳入

### 3.6 `skills/`
- 放什么：稳定可调用能力、参数化 skill、references、templates、assets
- 不放什么：项目专属过程稿、一次性聊天延伸说明件
- 是否承担正式入口法位：是，承担能力层入口
- 默认维护者：能力主责方
- 检索默认：纳入

### 3.7 `handover/`
- 放什么：正式交接件、交接记录、回执型流转件
- 不放什么：长期规则与知识母件
- 是否承担正式入口法位：否，默认不作为长期上位入口
- 默认维护者：交接主责方
- 检索默认：按需纳入，默认低于正式主链入口

### 3.8 `proposals/`
- 放什么：建议书、提案、待吸收输入件
- 不放什么：已生效正式规则母件
- 是否承担正式入口法位：否
- 默认维护者：提出方或吸收主责方
- 检索默认：按需纳入，不作为正式规则默认入口

### 3.9 `memory/`
- 放什么：daily memory、承诺、阻塞、临时记忆辅助件
- 不放什么：正式知识母件、当前有效团队规则
- 是否承担正式入口法位：否
- 默认维护者：各 Agent 自维护
- 检索默认：按需纳入，不默认当正式规则入口

### 3.10 `tmp/` / `archive/` / `.backup/`
- 放什么：临时件、归档件、备份件
- 不放什么：当前正式入口母件
- 是否承担正式入口法位：否
- 默认维护者：对应 workspace 主责方
- 检索默认：默认排除或降权

## 四、文档类型规则

团队统一定义五类：

### 4.1 正式件
- 稳定名
- 稳定入口
- 文档链完整
- 承担当前有效规则、知识或项目主交付

### 4.2 过程件
- 允许日期 / 版本号
- 保留演进轨迹
- 用于方案形成、阶段推进、判断沉淀
- 不自动等于正式母件

### 4.3 证据件
- 用于证明做过什么、发生过什么、如何验证
- 不替代母件
- 不独立定义当前规则

### 4.4 运行态文件
- 服务当前运行、状态、台账、checkpoint、临时承接
- 不默认视为知识入口
- 不参与长期规则裁决

### 4.5 归档件
- 用于保存历史事实
- 不参与当前裁决
- 必须和当前有效入口分开

## 五、根目录清洁规则

### 5.1 根目录长期允许保留
- bootstrap/reference 文件
- workspace 顶层 README / INDEX / 导航入口
- 少量全局状态入口
- 对整个 workspace 有稳定法位的少数文档

### 5.2 根目录不得长期混放
- 项目中间件
- 提案件
- 审计稿
- 状态件
- 过程稿
- 临时回执件
- 备份件

### 5.3 默认口径
> 根目录只保留 bootstrap/reference、顶层导航、极少数顶层状态入口；其余文档应归位。

## 六、正式知识与运行实施的边界

### 6.1 正式知识
进入 `knowledge_base` / `knowledge-base`

适用对象：
- 来源卡
- 研究底稿
- 专题页
- 索引页
- 长期可复用知识资产

### 6.2 运行实施主线
保留在 `runtime/` 或等价运行专区

适用对象：
- runtime pipeline
- harness 接线
- system 实施件
- 运行承载结构说明

### 6.3 项目专属交付与过程产物
保留在项目区 / `projects/`

适用对象：
- 项目级 Plan
- 任务级 program
- 阶段记录
- 项目过程交付与项目专属治理件

## 七、升格 / 归位 / 归档机制

### 7.1 何时升格为正式件
满足以下多数条件时，过程件应升格为正式件：
- 结构已稳定
- 承担对象明确
- 后续将被反复引用
- 已有稳定入口需求
- 已不再只是阶段性讨论痕迹

### 7.2 升格后至少执行的动作
- 确认正式放置位置
- 确认稳定命名
- 补最小文档链
- 回挂到上位入口 / 索引 / 导航
- 标明当前状态

### 7.3 何时应归档
- 已被替代
- 已退出当前裁决
- 只保留历史事实与回看价值
- 不应继续占当前工作面

### 7.4 被替代旧件处理
- 明确标状态
- 明确去向
- 不静默与新件并存为双入口

## 八、正式文档治理规则

### 8.1 本总则不只管文件夹，也管正式文档

本总则当前不仅回答 workspace 目录法位，也回答：
- 什么样的文档可以作为正式母件存在
- 正式文档最少应具备哪些链路字段
- 文档何时升格、何时吸收、何时归档
- 文档如何避免双入口、断链与漂移

也就是说：
> **目录治理只解决“放哪”，文档治理还必须解决“它在链路里是谁、挂在哪、由谁维护、是否仍有效”。**

### 8.2 正式文档最小治理要求

任何正式文档，不应只满足“放进了正确文件夹”；还至少应满足：
1. 有稳定承担对象
2. 有稳定入口或稳定挂载点
3. 有最小文档链
4. 有责任归属
5. 有状态说明
6. 必要时有适用范围说明

若仅完成存放、未完成挂链与责任闭合，不视为真正治理完成。

### 8.3 正式文档链制度（团队硬口径）

团队正式文档最少应补齐以下字段：
- **上位文档**：该文档挂在哪个更高层入口/母件之下
- **同位文档**：与其同层、需要一起理解或并行使用的关联件
- **下位文档**：它继续分发到哪些执行件、记录件、资产件、模板件
- **主任务 Agent**：当前谁主责维护该文档
- **所属任务 / 协调位**：该文档属于哪条任务线，由谁负责跨 agent 协调
- **当前状态**：draft / active / deprecated / archived 等
- **适用范围**：该文档适用于哪些对象、哪些 workspace、哪些任务类型

### 8.4 最小链路判断标准

- 没有上位挂载点的正式文档，不算真正进入主链
- 没有主任务 Agent 与所属任务的正式文档，不算责任闭合
- 没有状态说明的正式文档，容易造成旧件与现行件并存
- 没有适用范围的正式文档，容易被跨场景误用

### 8.5 双入口与漂移处理规则

若出现以下情况，必须治理：
1. 同一承担对象出现两个正式入口
2. 旧件已被替代但未标状态
3. 历史件继续被当成现行入口引用
4. 文档名稳定，但链路已漂移到别处

默认处理原则：
- 明确母件
- 旧件标状态
- 写清去向
- 不允许静默并存为双入口

### 8.6 文档治理与目录治理的关系

- 目录治理负责：文件应放在哪一层
- 文档治理负责：该文档在链路里承担什么法位

因此：
- 文件放对目录，不等于文档治理完成
- 文档链写全，但目录法位错误，也不算治理完成

必须同时满足：
> **目录法位正确 + 文档链路成立**

### 8.7 本总则对各 workspace 的要求

各 workspace 在落地本总则时，不仅要做目录清洁，还要同步检查：
- 是否存在无上位挂载点的正式文档
- 是否存在旧件未标状态
- 是否存在“过程件在事实层面充当正式母件”
- 是否存在正式文档缺主责/缺协调位/缺适用范围

若命中上述问题，应纳入归位实施清单，而不能只做文件搬移。

## 九、文档链制度（团队最小模板）

正式文档最少建议包含：
- 上位文档
- 同位文档
- 下位文档
- 主任务 Agent
- 所属任务 / 协调位
- 当前状态
- 适用范围

说明：
- 文档链制度服务于可追溯与可导航
- 不要求所有文档一开始写成重模板
- 但正式母件不应长期无链存在

## 十、实施顺序

### Phase 1：先立总则
先完成：
- 统一 workspace 分区法位
- 文档类型规则
- 根目录清洁规则
- 正式 / 过程 / 归档边界

### Phase 2：再出归位实施清单
在总则成立后，再按总则做：
- 各 workspace 根目录清理
- `knowledge_base` / `projects` / `runtime` 边界调整
- README / INDEX 补齐
- 检索降噪规则落地

### Phase 3：再抽象为 skill 模板
当规则稳定后，再沉淀：
- 通用 skill 模板
- workspace 参数表
- 各 Agent 本地实例化说明

---

## 十一、一句话结论

> 团队文档治理不再由各 Agent 各自立法，而由团队统一总则先定义 workspace 结构法位、文档类型规则、根目录清洁规则与升格归档机制；各 workspace 只做参数化实例化与实施落地。
