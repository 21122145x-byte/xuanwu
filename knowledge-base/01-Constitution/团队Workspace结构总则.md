## 一、总则定位

本总则主轴是统一团队 workspace 与各 Agent workspace 的**结构法位**。
文档治理相关内容仅作为补充约束存在，用于避免少数正式入口件失链、漂移或双入口。

本总则用于回答：

1. 团队 workspace 与各 Agent workspace 的目录法位如何统一解释
2. 什么是正式件、过程件、证据件、运行态文件、归档件
3. 根目录应长期保留什么，不应长期保留什么
4. 正式知识、运行实施、项目专属交付三者的边界如何区分
5. 文件何时升格、何时归位、何时归档
6. 在少数正式入口件上，最小文档治理补充应包含什么

本总则不承担：
- 某个具体 workspace 的一次性清理实施细节
- 某个 Agent 的本地实例参数
- 某类目录的动作级执行步骤

## 二、统一治理原则

### 2.1 团队文档标准强制执行原则
- 团队统一文档标准（如文档链、正式件口径、治理层级口径）必须严格执行
- Agent 不得以“本地习惯”“本地轻量化”为由改写团队统一文档标准
- 若本地执行面与团队文档标准冲突，以上位团队标准为准

### 2.2 workspace 管理参数化原则
- workspace 管理标准采用“团队总则 + 本地参数化 skill”模型
- 团队治理母件保持统一留在团队 workspace
- 各 Agent 只本地化自己的参数面、目录名差异、root keep 白名单与运行边界，不再平行复制一张本地治理母件

### 2.3 稳定入口原则
- 正式规则必须有稳定入口
- 版本件、历史件、审计件、术语过渡件不得替代正式入口

### 2.4 文档层级原则
- 团队统一治理规则进团队 workspace
- Agent workspace 不承载团队长期上位法，只承载参数化实现、运行面文件与必要的本地执行材料

### 2.5 反膨胀原则
- 不以“说明如何整理”的名义持续制造新噪音
- 同一类规则通常只应保留一个团队统一上位口径；如出现并列口径，应尽快收敛

### 2.6 最小必要原则
- 不为未来可能性提前堆过多目录、术语、流程与说明件
- 有更简单可解释路径时，优先采用更简单路径

### 2.7 治理服务执行原则
- 总则用于降低混乱、提升可解释性，不用于抢占当前真实任务的前台
- 当前明确任务已在推进时，优先完成执行、验证、交付，再补挂载、归位与治理包装
- 目录归位、文档链补齐、治理议题化，默认服务执行，不应反客为主

### 2.8 正式知识 / 运行实施 / 项目专属分区原则
- 正式知识 → `knowledge_base` / `knowledge-base`
- 运行实施主线 → `runtime/` 或等价运行专区
- 项目专属交付与过程产物 → 项目区 / `projects/`

## 三、Workspace 标准结构法位

以下目录为团队统一最小法位说明；具体 workspace 可有差异，但不得违背其承担对象。

### 3.1 根目录
- 放什么：bootstrap/reference、顶层导航、极少数顶层状态入口
- 不放什么：长期并存的项目中间件、提案、审计稿、状态件、过程稿
- 是否承担正式入口法位：是，但仅限顶层导航与少数全局入口
- 默认维护者：workspace 主责任 Agent
- 检索默认：纳入，但应保持低噪音

### 3.2 `docs/`
- 放什么：面向说明、支持性文档、非知识库但需结构化阅读的说明材料
- 不放什么：运行态状态件、队列/日志/checkpoint、项目阶段记录、团队统一上位母件
- 是否承担正式入口法位：一般不承担团队长期上位入口；仅在该 workspace 明确指定时，才可承担局部稳定入口
- 默认维护者：对应文档主责方
- 检索默认：纳入

### 3.3 `projects/` / 项目区
- 放什么：项目专属 Plan、program、交付、阶段件、项目过程产物
- 不放什么：团队统一上位法、长期稳定通用能力规则
- 是否承担正式入口法位：承担项目级正式入口，不承担团队长期上位法入口
- 默认维护者：项目主责方
- 检索默认：纳入

### 3.4 `knowledge_base/` / `knowledge-base/`
- 放什么：正式知识、来源卡、研究底稿、专题入口、索引、长期可复用知识
- 不放什么：仅服务当前运行的状态件、纯过程施工件、一次性中间稿
- 是否承担正式入口法位：是，承担正式知识入口
- 默认维护者：知识主责 Agent
- 检索默认：纳入

### 3.5 `runtime/`
- 放什么：运行系统实施件、runtime pipeline、system/harness 接线与运行专区文件
- 不放什么：被误当作知识入口的研究正式件
- 是否承担正式入口法位：是，但法位属于运行实施专区，不属于知识库
- 默认维护者：运行系统主责方
- 检索默认：纳入

### 3.6 `skills/`
- 放什么：稳定可调用能力、参数化 skill、references、templates、assets
- 不放什么：项目专属过程稿、一次性聊天延伸说明件
- 是否承担正式入口法位：是，承担能力层入口
- 默认维护者：能力主责方
- 检索默认：纳入

### 3.7 `handover/`
- 放什么：正式交接件、交接记录、回执型流转件
- 不放什么：长期规则与知识母件
- 是否承担正式入口法位：否，默认不作为长期上位入口
- 默认维护者：交接主责方
- 检索默认：按需纳入，默认低于正式主链入口

### 3.8 `proposals/`
- 放什么：建议书、提案、待吸收输入件
- 不放什么：已生效正式规则母件
- 是否承担正式入口法位：否
- 默认维护者：提出方或吸收主责方
- 检索默认：按需纳入，不作为正式规则默认入口

### 3.9 `memory/`
- 放什么：daily memory、承诺、阻塞、临时记忆辅助件
- 不放什么：正式知识母件、当前有效团队规则
- 是否承担正式入口法位：否
- 默认维护者：各 Agent 自维护
- 检索默认：按需纳入，不默认当正式规则入口

### 3.10 `tmp/` / `archive/` / `.backup/`
- 放什么：临时件、归档件、备份件
- 不放什么：当前正式入口母件
- 是否承担正式入口法位：否
- 默认维护者：对应 workspace 主责方
- 检索默认：默认排除或降权

## 四、文档类型规则

团队统一定义五类：

### 4.1 正式件
- 稳定名
- 稳定入口
- 文档链完整
- 承担当前有效规则、知识或项目主交付

### 4.2 过程件
- 允许日期 / 版本号
- 保留演进轨迹
- 用于方案形成、阶段推进、判断沉淀
- 不自动等于正式母件

### 4.3 证据件
- 用于证明做过什么、发生过什么、如何验证
- 不替代母件
- 不独立定义当前规则

### 4.4 运行态文件
- 服务当前运行、状态、台账、checkpoint、临时承接
- 不默认视为知识入口
- 不参与长期规则裁决

### 4.5 归档件
- 用于保存历史事实
- 不参与当前裁决
- 必须和当前有效入口分开

## 五、根目录清洁规则

### 5.1 根目录长期允许保留
- bootstrap/reference 文件
- workspace 顶层 README / INDEX / 导航入口
- 少量全局状态入口
- 对整个 workspace 有稳定法位的少数文档

### 5.2 根目录不得长期混放
- 项目中间件
- 提案件
- 审计稿
- 状态件
- 过程稿
- 临时回执件
- 备份件

### 5.3 默认口径
> 根目录只保留 bootstrap/reference、顶层导航、极少数顶层状态入口；其余文档应归位。

## 七、升格 / 归位 / 归档机制

### 7.1 何时升格为正式件
满足以下多数条件时，过程件应升格为正式件：
- 结构已稳定
- 承担对象明确
- 后续将被反复引用
- 已有稳定入口需求
- 已不再只是阶段性讨论痕迹

### 7.2 升格后至少执行的动作
- 确认正式放置位置
- 确认稳定命名
- 补最小文档链
- 回挂到上位入口 / 索引 / 导航
- 标明当前状态

### 7.3 何时应归档
- 已被替代
- 已退出当前裁决
- 只保留历史事实与回看价值
- 不应继续占当前工作面

### 7.4 被替代旧件处理
- 明确标状态
- 明确去向
- 不静默与新件并存为双入口

## 八、正式入口件的最小文档治理补充

### 8.1 本章定位

本章不是把总则主轴从 workspace 结构法位改写成一套重文档治理制度。

本章只补充回答一件事：
- 当少数文档已经承担**正式入口 / 母件法位**时，最低限度还需要补什么，才能避免双入口、断链与漂移

也就是说：
- **主轴仍是目录法位与 workspace 结构边界**
- 文档治理只在少数正式入口件上补最低必要约束
- 不把所有正式文档、过程件、阶段件都抬升到同等治理强度

### 8.2 触发条件

只有当文档同时命中以下特征中的多数时，才进入本章治理范围：
1. 它正在承担正式入口或母件法位
2. 它会被反复引用，作为后续判断或导航依据
3. 若不补最小链路，已经容易形成双入口、误引或漂移

未命中时：
- 仍以目录归位正确为主
- 不额外要求重链路化
- 不因治理补充而抬高执行成本

### 8.3 最小补充项

对命中本章触发条件的文档，默认只补三项：
- **上位文档**：它挂在哪个更高层入口/母件之下
- **主任务 Agent**：当前谁主责维护
- **当前状态**：draft / active / deprecated / archived 等

仅在确有需要时，再补：
- 同位文档
- 下位文档
- 所属任务 / 协调位
- 适用范围

这里的“确有需要”主要指：跨 workspace 复用、跨任务扩散，或已出现误用 / 失链 / 双入口风险。

默认口径：
- 先保证不迷路、不双入口、不失责
- 不追求把每一份正式文档都补成全字段母件模板

### 8.4 最小判断标准

命中本章范围的文档，优先只看三件事：
- 有没有上位挂载点
- 有没有主任务 Agent
- 有没有当前状态

若这三项成立，通常不必继续上升治理强度；
只有在跨 workspace 复用、跨任务扩散、或已出现误用风险时，才补更多字段。

### 8.5 本章真正要防的风险

本章主要只防四类问题：
1. 同一承担对象出现两个正式入口
2. 旧件已被替代但未标状态
3. 历史件继续被当成现行入口引用
4. 文档名稳定，但链路已漂移到别处

默认处理原则：
- 明确母件
- 旧件标状态
- 写清去向
- 不允许静默并存为双入口

### 8.6 与目录治理的主次关系

- 目录治理负责：文件应放在哪一层
- 本章补充负责：少数正式入口件如何避免失链与漂移

主次顺序是：
1. 先判目录法位是否正确
2. 只有文档已承担正式入口法位时，才补本章要求

因此：
- 目录法位是主轴
- 本章是补充，不是另一条并列大主线

### 8.7 对各 workspace 的最低要求

各 workspace 落地本总则时：
- 默认先做目录法位检查与归位
- 只对少数正式入口件追加本章检查

最小检查项：
- 是否存在无上位挂载点的正式入口件
- 是否存在旧件未标状态
- 是否存在过程件在事实层面充当正式入口件

若未命中上述问题，不必为一般文档额外扩充治理动作。

## 九、正式入口件最小模板（可选补充）

对承担正式入口 / 母件法位的文档，最少可补：
- 上位文档
- 主任务 Agent
- 当前状态

其余字段按需要补齐：
- 同位文档
- 下位文档
- 所属任务 / 协调位
- 适用范围

说明：
- 这是可选补充模板，不是把全体文档模板化
- 只服务少数正式入口件的可追溯与可导航
- 不要求一般文档一开始写成重模板

## 十、实施顺序

### Phase 1：先立总则
先完成：
- 统一 workspace 分区法位
- 文档类型规则
- 根目录清洁规则
- 正式 / 过程 / 归档边界

### Phase 2：再出归位实施清单
在总则成立后，再按总则做：
- 各 workspace 根目录清理
- `knowledge_base` / `projects` / `runtime` 边界调整
- README / INDEX 补齐
- 检索降噪规则落地

### Phase 3：再抽象为 skill 模板
当规则稳定后，再沉淀：
- 通用 skill 模板
- workspace 参数表
- 各 Agent 本地实例化说明

补充口径：
- skill references 层内容不得与本总则正文产生实质矛盾
- 若出现偏离，以本总则正文为准，skill 引用层须同步修正

---

## 十一、一句话结论

> 团队先统一 workspace 结构法位，再在少数正式入口件上补最小文档治理约束；各 workspace 只做参数化实例化与实施落地，不再各自扩写一套上位规则。
