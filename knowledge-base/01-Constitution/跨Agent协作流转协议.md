# 跨Agent协作流转协议 v1

- 日期：2026-04-01
- 状态：v1 / 草案
- 适用范围：团队在 Discord 协作面上的跨 Agent 任务流转
- 目标：让跨 Agent 协作从“靠猜和顺手提醒”变成“可触发、可观测、可回传、可打回、可熔断”的制度链

## 二、触发条件

当出现以下情形时，应进入正式跨 Agent 协作流转：
- 某任务需要另一个 Agent 正式接手
- 某产物需要另一个 Agent 正式审计 / 校验
- 某任务需要协调人分派并跟踪状态
- 某执行结果需要进入验收 / 打回链路

凡进入上述情形，不应再依赖自然聊天或模糊提醒，而应进入协议化流转。

---

## 三、角色定义

### 1. 协调人
负责：
- 发起任务
- 定向唤醒目标 Agent
- 指定任务边界与交付要求
- 跟踪状态
- 收到结果后决定汇总 / 打回 / 转交验收

### 2. 执行人
负责：
- 接收被分派任务
- 明确回应是否收到
- 按要求执行
- 完成后按协议回传结果

### 3. 验收人 / 审计人
负责：
- 对回传产物进行验收、审计或打回
- 给出放行、驳回、待补证据或待重投结论

同一 Agent 在不同任务里可承担不同角色，但在同一条协作链的单一节点上，应明确当前角色。

---

## 四、正式触发规则

### 1. 定向唤醒必须使用 `<@Discord数字ID>`
原因：
- `@名字` 可能只是文本高亮，不稳定
- role mention 不能精确唤醒目标 Agent session
- `sessions_send` 能传消息，但不能稳定唤醒 idle session
- `<@Discord数字ID>` 才是当前协作环境下最稳定的定向触达方式

### 2. 定向触发与回执必须放在消息末尾
- 正式交接内容应先完整输出，再在消息末尾放置目标 Agent 的 `<@Discord数字ID>`。
- 不得在消息开头先 @ 再补交接内容。
- 若平台存在分段式输出风险，默认以“末尾 @”作为正式触发动作。
- 被触达后的【收到】回执，也必须在完整内容后，于消息末尾 @ 原发起人。
- 【任务完成】与【内容打回】若需要继续流转，也应在消息末尾 @ 下一责任人。

### 3. 被 `<@自己ID>` 触达后，默认进入协议链判断
被触达的 Agent 必须判断：
- 我当前是执行人？
- 我当前是验收人？
- 我当前只是被抄送的观察方？

### 3. 消息结构：`【协作分派】` + 正文 + 末尾 @目标 Agent
Discord 消息为分段展示，接受方可能只读到前半段。  
因此：**`<@目标ID>` 必须放在消息末尾**，不得放在开头或中部。

## 五、固定格式

### 1. 【协作分派】
由协调人发出。

建议结构：
- 目标 Agent：`<@ID>`
- 任务目的
- 产物位置 / 输入材料
- 重点要求
- 期望回传格式
- 是否需要验收链

### 2. 【收到】
由执行人发出。

作用：
- 明确表示已被唤醒并接单
- 让协调人与其他 Agent 可观测当前状态

最小格式：
- 已收到
- 当前角色判断
- 预计何时给出首个结果节点

### 3. 【任务完成】
由执行人发出。

最小格式建议包含：
- Where
- What
- How to verify
- Known Issues
- Next Steps

### 4. 【内容打回】
由协调人或验收人发出。

最小格式建议包含：
- 打回原因
- 需要补什么
- 最小修复路径
- 是否需要重新提交

---

## 六、交接内容规则

- 跨 Agent 正式交接，必须使用 **5 字段 handoff + 文档链接 / 文件路径**。
- 交接频道不用于大段自由对话。
- 同一内容不得在交接频道重复发送 3 次以上。
- 若需长讨论，应转入讨论频道或在其他上下文完成后，再回交接频道提交结构化结果。

---

## 七、状态可观测规则

跨 Agent 协作不得靠猜测推进。

必须至少能看到以下状态之一：
- 【协作分派】已发出
- 【收到】已确认
- 【任务完成】已回传
- 【内容打回】已执行

如果没有这些显式状态，不应默认认为任务已进入稳定流转。

---

## 八、硬规则

- 跨 Agent 正式协作，必须使用 `<@Discord数字ID>` 定向触发。
- 被 @ 必回，不得沉默失联。
- 不得只 @ 名字或 role mention 代替正式触发。
- 不得依赖 `sessions_send` 作为 Discord 协作面的默认唤醒手段。
- 任务回传必须有可检查的交付结构，不得只回一句“弄好了”。
- 不合格产物可明确打回，不得用模糊态度代替结论。

---

## 九、超时与推进

- 若被定向触发后长时间未出现【收到】，由协调人先检查是否触达失败。
- 若执行中长时间未出现节点反馈，由协调人跟进。
- 若验收 / 审计长时间未收口，由粉雪作为默认推进位介入催收。
- 若任务已影响正式交付或团队关键决策，必要时直接升级给 BOSS。

---

## 十、熔断与升级规则

- 被打回后的再次提交，必须附本轮修改说明。
- 若未针对前次审计 / 验收意见做出实质修改，不得重新进入正式审计链。
- 连续多轮因无实质修改被打回时，应触发熔断，不再原地循环，由粉雪介入判断：返工、重拆、转讨论或暂停。
- 若该循环已影响正式交付、关键决策或暴露结构性失效，应升级给 BOSS。

---

## 十一、系统加载与默认适用规则

- 团队级跨 Agent 协作默认受本协议约束。
- 各 Agent 的 `AGENTS.md` 应引用本协议，作为跨 Agent 正式交接的默认行为规则。
- 在 Discord 协作面被 `<@Discord数字ID>` 定向触发时，默认进入本协议判断。
- 若任务属于正式交接、审计提交、协作分派或回传，不得脱离本协议自由发挥。

---

## 十二、与验收协议的关系

本协议解决的是：
> 任务怎么正式传过去、被接住、被回传。

它不单独定义完整的质量验收标准。

质量是否合格、是否放行、是否待补证据，应由后续验收 / 审计协议进一步细化。

也就是说：
- 本协议解决“流转链”
- 验收协议解决“合格链”

---

## 十三、一句话收口

> 跨 Agent 协作不该再靠顺手提醒和默认默契推进；在 Discord + OpenClaw + mention activation 的环境里，`<@Discord数字ID>`、固定格式、显式状态、打回机制与熔断升级，才构成一条真正可运行的制度性交接链。
