# Anna Spawn 试跑说明 v1：memos 幽灵上下文排查

## 一句话判断

这次不是普通的“参考执行”，
而是一次正式的机制联测：

> 用一个真实排查任务，同时验证 `spawn 决策 + Subtask Card + 回传链 + 上游验收` 是否成立。

当前试跑对象：
- **任务主题**：memos 幽灵上下文排查
- **执行方式**：Spawned Subagent Execution
- **执行单元**：Anna 的 spawn 执行单元
- **发起人 / Issued By**：Shaun

这意味着：
这次不是把 Anna 当成“默认子 agent”，
而是用 Anna 的执行单元做一次正式 spawn 试跑，验证机制本身。

---

## 一、这次试跑到底在测什么

本轮不是只测任务结果，
而是同时测下面 5 件事：

### 1）spawn 决策是否站得住
重点看：
- 该任务是否已经过意图确认
- 是否边界足够清楚
- 是否适合从主判断链中剥离出去
- 是否值得通过独立会话执行

### 2）Subtask Card 能不能支撑独立执行
重点看：
- 拿到卡后是否仍需大量追问
- 边界是否会跑偏
- 输入是否足够开工
- 输出格式是否可被执行

### 3）回传链是否成立
重点看：
- 是否能按约定结构回传
- 是否能明确区分任务结果与机制反馈
- 是否有 completion signal

### 4）上游是否能直接验收
重点看：
- 上游能否根据回传直接判断下一步
- 是否还需要大量二次翻译

### 5）整体摩擦成本是否可接受
重点看：
- spawn 是否太重
- 卡片是否太厚
- 对这类排查任务而言，这套机制到底是护栏还是负担

---

## 二、这次为什么值得 spawn

本次任务满足以下条件：

### 1）目标明确
任务不是开放式脑暴，而是：
**排查 memos 幽灵上下文的可能来源、触发链和污染路径。**

### 2）边界可定义
本轮重点是：
- 现象识别
- 证据整理
- 可疑链路排查
- 当前最可能路径判断

明确不包含：
- 直接修复
- 改系统配置
- 全面系统重构

### 3）输入可给定
可提供：
- 对话记录
- 日志与现象记录
- 当前已知判断
- 本轮任务边界

### 4）输出可定义
输出不是泛泛而谈，
而是一份可验收的结构化排查报告。

### 5）结果可验收
上游可以明确判断：
- 是否区分了现象 / 证据 / 推测 / 结论
- 是否给出至少一条最可能污染路径
- 是否能据此决定下一步是继续排查还是起修复任务

因此，这次任务具备正式 spawn 的基本条件。

---

## 三、这次试跑的执行纪律

本次试跑必须遵守三条纪律：

### 纪律 1：不越界
排查就是排查，
不要因为看到了可能的修法，就自动把任务扩成修复。

### 纪律 2：不脑补
如果缺日志、缺配置、缺复现路径，
必须明确标出缺口，而不是自行补全想象。

### 纪律 3：不把推测写成结论
输出中必须区分：
- 已确认事实
- 高概率推测
- 待确认项

## 四、这次正式 Spawn 用的 Subtask Card

```md
# Subtask Card v1

## 1. Identity
- Subtask ID: ST-20260328-MEMOS-001
- Parent Task ID: T-20260328-MEMOS-GHOST
- Issued By: Shaun
- Assigned To: Anna Spawn Unit
- Execution Mode: Spawned Subagent Execution
- Current Status: NEW

## 2. Purpose
- Goal:
  排查 memos 幽灵上下文问题的可能来源、触发链和当前污染路径。
- Why This Was Delegated:
  该任务属于验证与排查型工作，边界明确、输入可给定、输出可定义、结果可验收，适合作为一次正式 spawn + Subtask Card 联测任务。
- Spawn Justification:
  本任务执行链较长，且适合从主判断链中剥离出来；同时需要验证 spawn + 卡片 + 回传链是否成立，因此采用 spawn 执行。

## 3. Scope
- In Scope:
  1. 识别“幽灵上下文”具体表现
  2. 排查可能的注入链路、触发条件、污染来源
  3. 区分现象、推测、证据
  4. 输出当前最可能的问题路径与未确认点
- Out of Scope:
  1. 不直接修改系统配置
  2. 不直接做修复动作
  3. 不扩大到无关模块的全面系统重构
- Success Boundary:
  给出一版结构清楚的排查结果，能让上游判断下一步该修哪里，即视为完成。

## 4. Inputs
- Input Materials:
  1. 当前与 memos 幽灵上下文相关的对话记录
  2. 相关日志、现象记录、已有判断（如有）
  3. 本轮任务目标：先排查，不先修复
- Assumed Context:
  1. 当前重点是识别污染链，而不是立刻修复
  2. 本轮还要同步测试 spawn 与 Subtask Card 机制本身
  3. 回传需要同时包含任务结果与机制反馈
- Missing Information:
  若缺少关键日志、配置、复现路径，请明确列出，不要自行脑补

## 5. Execution
- Action Required:
  对 memos 幽灵上下文问题做结构化排查，并输出阶段性诊断结果。
- Suggested Method:
  先定义问题现象，再列可疑来源，再按证据强弱排序，最后给出最可能链路与未决问题。
- Constraints:
  结论必须区分“已证实 / 高概率 / 待确认”。
- Non-Negotiables:
  1. 不得把推测写成结论
  2. 不得越界直接修系统
  3. 不得省略证据依据

## 6. Output
- Expected Output:
  一份排查报告初稿 + 一段机制反馈
- Output Format:
  Markdown
- Required Sections:
  1. 问题现象
  2. 已确认事实
  3. 可疑链路
  4. 证据与依据
  5. 当前最可能结论
  6. 未确认点
  7. 下一步建议
  8. 机制反馈（字段有用性 / 冗余感 / 卡点）

## 7. Acceptance
- Acceptance Criteria:
  1. 能看出“现象 / 证据 / 推测 / 结论”的区分
  2. 能定位到至少一条最可能污染路径
  3. 输出可直接供上游决定下一步动作
  4. 能反馈本次 spawn + 卡片链路的真实使用感受
- Rejection Conditions:
  1. 把猜测当结论
  2. 只有散点观察，没有结构化结果
  3. 越界直接进入修复方案
  4. 只交任务结果，不交机制反馈
- Self-Check Before Return:
  1. 是否区分了事实与推测？
  2. 是否已经写出未确认点？
  3. 是否仍然停留在排查边界内？
  4. 是否补上了机制反馈？

## 8. Return
- Return To: Shaun
- Return Format:
  1. 结果摘要
  2. 排查正文
  3. 机制反馈
  4. 风险 / 缺口 / 还需要的输入
- Completion Signal:
  Subtask Complete: Yes / No

## 9. Failure / Escalation
- If Blocked:
  明确说明卡在哪个输入缺口或证据缺口。
- If Scope Is Unclear:
  直接回传需要确认的问题，不要自行扩任务。
- If Input Is Insufficient:
  列出缺失的具体日志、配置或上下文。
- If Better Alternative Appears:
  可提出，但不得擅自把“排查任务”改成“修复任务”。

## 10. Notes
- Risks:
  该问题可能涉及多条注入链，初次排查未必能一次性锁死。
- Special Notes:
  本轮目标是结构化排查 + 机制联测，不追求一步到位修复。
- Next Suggested Step:
  若排查结果成立，再单独起修复任务卡；若机制反馈显示卡片过重，再回头压缩字段。
```

## 五、回传时必须额外反馈的机制项

本轮与普通任务最大的区别在于：
除了任务结果，还必须返回机制反馈。

建议至少反馈这 5 件事：

1. 哪些字段最有用
2. 哪些字段显得冗余
3. 哪些字段写了但执行时仍然不够清楚
4. spawn + 卡片是否比普通口头委派更稳
5. 如果压简，最先删哪几个字段

## 六、这次试跑完成后怎么判断成败

### 如果以下条件成立，可认为试跑通过：
- spawn 流程没有造成明显额外混乱
- Subtask Card 能支撑独立执行
- Anna 能稳定停在排查边界内
- 回传结构清楚，可直接供上游验收
- 机制反馈能支持后续模板优化

### 如果以下情况明显出现，可认为试跑不通过：
- spawn 带来的摩擦成本显著高于收益
- 卡片字段过厚，导致执行负担过重
- 仍然频繁需要大量口头补充
- 回传结构不稳定，无法直接验收
- 执行边界依旧容易失真
