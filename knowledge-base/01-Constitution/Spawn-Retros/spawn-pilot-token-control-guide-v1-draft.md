# Spawn Pilot Token 可控指南 v1（初案）

## 一句话判断

pilot 不是不能烧，
但不能靠无上限燃烧来换机制验证。

这份指南的目标不是一味省 token，
而是：

> **在保住试跑价值的前提下，让 spawn 的模型选择、拆分方式、上下文厚度和回传长度都变得可控。**

当前阶段，这不是正式定稿，
而是一份可供后续异源测试前使用的制度草案。

---

## 一、先立住三条总原则

### 原则 1：任务评级 ≠ 子任务难度
- **任务评级** 决定：流程厚度、卡片版本、验收护栏
- **子任务难度** 决定：spawn 模型档位

也就是说：
- 一个 P3 主任务里，完全可以同时包含 D1 / D2 / D3 子任务
- 不能因为主任务高阶，就让所有子任务都默认上高配模型

---

### 原则 2：同一个主任务内允许混合模型配置
这条必须明确：

> **主任务可以是高风险主任务，但其中的搜索、浏览、归档子任务依然可以交给低成本模型。**

例如：
- 样本搜索 / 归档 → 低档模型
- 常规代码定位 / 标准分析 → 中档或低档起步
- 根因归因 / 结构判断 → 高档模型

不要把“高风险主任务”错误理解成“整条链全高配”。

---

### 原则 3：同源不是强制要求，可控才是要求
spawn 不必强绑定主任务的：
- 同一端点
- 同一 provider
- 同一 API key

真正该绑定的是：
- 任务定义
- 回传协议
- 权限边界
- 验收标准

因此：
> **同源是可选策略，不是默认铁律；同规才是必须的。**

---

## 二、pilot 值不值得烧：先过四个判断

在决定一个 pilot 是否值得较高 token 成本之前，先问：

### 1）这个任务是不是在测机制，而不只是测结果？
如果只是普通结果交付，不必轻易上高成本 pilot 模式。

### 2）这个任务能不能带来制度化价值？
例如：
- 能反推出模板调整
- 能帮助确定版本切换规则
- 能帮助确定任务评级规则
- 能帮助确定模型分配原则

### 3）这次烧的成本，能不能换来以后更稳定的低成本运行？
如果只是一次性消费，不值得重烧。

### 4）是否已有足够边界，避免这次 token 主要烧在混乱上？
如果意图不清、边界不清、回传不清，先别上高配 pilot。

---

## 三、子任务难度分档（用于选模型，不用于选卡片）

### D1｜低能力需求子任务
特征：
- 搜索
- 浏览
- 样本归档
- 提取字段
- 结构化整理
- 固定格式摘要
- 明确边界内的信息收集

默认模型策略：
- 优先低成本模型
- 例如：`gpt5.1`

原则：
能用低档模型做对的，不上高配。

---

### D2｜中能力需求子任务
特征：
- 需要一定筛选与组织
- 轻度结构设计
- 标准对比分析
- 常规代码定位
- 一般性排查
- 中等复杂度但不属于高推理高归因

默认模型策略：
- 优先“先低后高”
- 可先用低档试跑，不足时再升
- 例如：先 `gpt5.1`，不够再升 `gpt5.4`

原则：
不是一上来顶配，而是先匹配，再升级。

---

### D3｜高能力需求子任务
特征：
- 多证据链整合
- 根因归因
- 高不确定推理
- 架构判断
- 制度设计
- 高返工成本判断任务

默认模型策略：
- 优先高档模型
- 例如：`gpt5.4`

原则：
高难度任务别省错地方。

---

## 四、模型选择规则（初案）

### 规则 1：先按子任务难度选，不按主任务标题选
也就是说：
- 不是“这个主任务重要，所以所有 spawn 都上高配”
- 而是“这个子任务到底需要多强能力”

---

### 规则 2：中难度任务默认先低后高
只有出现以下信号，再升档：
- 输出明显不稳定
- 逻辑链断裂
- 返工 2 次以上
- 上游验收 repeatedly 不通过
- 不能稳定维持边界

---

### 规则 3：高难度任务可直接高配，但数量必须收
如果某段任务已明确属于 D3：
- 可以直接高配
- 但不要把多个高难度判断同时平铺并行乱烧

原则：
少开，但开对。

---

## 五、spawn 数量控制规则

### 1）低难度任务可以多并行
例如：
- 搜索多个来源
- 浏览多份文档
- 样本归档并行

前提：
- 每个子任务都足够薄
- 低档模型即可支撑

---

### 2）高难度任务宁可少开几个
原因：
- 高推理任务通常不是线性增益
- 多个高配并行很容易迅速烧穿 token
- 上游整合成本也会显著增加

---

### 3）pilot 任务里，优先把真正高耗的那一段单独挑出来
例如：
- 搜索 / 归档 / 浏览 → 分给低档
- 归因 / 结构判断 → 保留给高档

这叫“高低混编”，不是“一锅高烧”。

---

## 六、上下文厚度控制

上下文过厚，是 token 爆炸最常见的原因之一。

### 控制原则
- 给子任务只发“够开工”的上下文，不发“所有历史”
- 区分：必须输入 vs 可选背景
- 能摘要就不要丢整段原文
- 对已经稳定的共同前提，用 `Assumed Context` 代替重复贴长上下文

### 警戒信号
如果一个子任务卡里：
- 解释性文字远多于动作定义
- 输入材料远多于实际需要
- 大量重复主任务背景

说明上下文已经过厚。

---

## 七、回传长度控制

token 不只烧在输入，也烧在回传。

### 控制原则
回传至少分层：
1. 结果摘要
2. 正文
3. 风险 / 缺口 / 未确认点

但不要默认把每次回传都写成长报告。

### 什么时候必须长
- 正式 pilot
- 高风险排查
- 需要制度化复盘

### 什么时候应压短
- 常规标准任务
- 低风险重复任务
- 只需要一个可验收结果时

---

## 八、同源 / 异源策略（初案）

### 同源适合的场景
建议保持主任务与 spawn 同端点 / 同 provider / 同 key，当：
- 正在做机制基准测试
- 不想引入额外变量
- 输出风格需要强一致性
- 工具/权限绑定较强

---

### 异源适合的场景
建议允许异源，当：
- 需要成本分层
- 需要能力分层
- 需要限流隔离
- 需要预算隔离
- 低难度子任务适合便宜模型，高难度子任务适合强模型

---

### 异源前提
不论同源还是异源，必须保证：
- 回传协议一致
- 任务边界一致
- 验收标准一致
- 权限边界一致

一句话：
> **可以异源，但不能异规。**

---

## 九、资源层失败不是机制失败

这条必须单独立出来。

当出现以下现象时，不要急着先判协议问题：
- 4 秒退出
- 0 token
- 0 output
- 同批多个 subagent 一起失败
- 某 provider 当天已接近额度上限
- 同一 provider 的高阶模型集体异常

### 必须优先补查的资源项
- 当前 spawn 使用的 provider
- 当前模型档位
- 当前 API key 是否还有额度
- 是否触发 rate limit / quota exhaustion
- 是否存在 provider 级 403 / 429 / silent fail

### 一条关键规则
> **子任务失败样本，不能默认都归因为机制问题；必须先区分是任务层失败、机制层失败，还是资源层失败。**

### 对当前经验的直接修正
Anna 那轮 `4 秒 / 0 token / 0 output` 的失败样本，不能再只解释为 spawn 初始化或任务设计问题。
由于当日 `mdlbus-anna / gpt-5.4` 通道额度已耗尽，该样本至少同时具有明确的资源层干扰变量。

这意味着：
- 它不是“纯机制失败样本”
- 而是“机制层 + 资源层混合失败样本”

因此，后续异源测试的价值被进一步抬高：
> **异源不只是成本优化手段，也是资源容灾手段。**

---

## 十、如何避免再次出现“15 倍 token 级别”的 pilot 燃烧

### 1）不要整条链全高配
高风险主任务不等于全链高配。

### 2）不要整条链都走完整版上下文
高风险任务可以用完整版卡片，
但子任务上下文仍应按需裁切。

### 3）不要把机制试跑和内容试跑无限叠加
一次 pilot 最多优先验证 1~2 个机制点，
不要所有变量同时打开。

### 4）不要让回传默认长报告化
除非任务确实需要制度复盘。

### 5）把“升档”变成条件动作，而不是默认习惯
模型升档、卡片升档、spawn 增量，都应由失败信号触发。

---

## 十一、当前版本的结论

如果把这份初案压成一句话，就是：

> **Spawn token 可控，不靠一刀切地省，而靠任务评级、子任务难度、模型混编、上下文裁切、回传分层一起控制。**

再补一句：

> **主任务决定流程厚度，子任务决定模型档位；可同源，也可异源，但必须同规。**

---

## 十二、这份初案还待后续异源测试验证的地方

后续建议重点验证：
- 同源与异源在回传一致性上的差异
- 低档模型在 D1 / D2 任务上的稳定阈值
- 不同 provider / key 的成本差异是否值得制度化利用
- 哪些任务看似 D2，实则常常隐性升到 D3

这几个点不测清，正式定稿还会偏早。
