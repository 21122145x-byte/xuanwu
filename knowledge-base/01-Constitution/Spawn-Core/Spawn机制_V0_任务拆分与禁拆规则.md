# Spawn机制 V0｜任务拆分与禁拆规则

- 版本：V0
- 状态：骨架版 / 待首轮试跑校正
- 维护者：Anna
- 目的：明确 Spawn 机制下哪些任务适合拆给子任务，哪些内容必须保留在主控位，避免拆分后边界失守。

---

## 一、文档目的

本文件解决的是 Spawn 机制里最容易出事的一步：

> **任务到底怎么拆，拆到哪里为止，哪些绝不能下放。**

如果只讲“多 Agent 协作”而不写清边界，最终很容易回到：
- 子任务越权
- 主控失位
- 结论漂移
- 回收困难

所以 V0 必须先立边界，再谈效率。

---

## 二、V0 拆分原则

### 原则 1
**拆的是局部闭环，不是最终判断权。**

### 原则 2
**子任务负责局部事实与局部结论，主控负责跨任务统一口径与最终裁决。**

### 原则 3
**凡是“越界一次就会污染最终结论”的内容，优先列入禁拆区。**

### 原则 4
**先求可回收、可验收，再求并行与速度。**

---

## 三、可拆区（适合下放给子任务）

以下内容在 V0 阶段通常适合拆出为子任务：

### 1. 局部样本处理
- 单批样本抽取
- 单批样本分类
- 局部标签整理
- 局部异常标记

### 2. 局部统计工作
- 某一阶段的数字核对
- 局部统计冲突排查
- 单一表格或单一口径对账

### 3. 局部事实核验
- 单一路径核验
- 单一数据源检查
- 单一因果链证据补查
- 单个怀疑点的真伪排除

### 4. 局部文档产物
- 单阶段完成报告
- 单问题返工建议单
- 单项核对单
- 单项异常说明

### 5. 结构化返工任务
- 针对已定位问题的局部修正
- 针对口径偏差的独立返工
- 针对统计错误的独立纠偏

这些任务的共同点是：
- 边界相对清晰
- 验收相对明确
- 即使局部做错，也容易定位责任与返修

---

## 四、禁拆区（必须保留给主控位）

以下内容在 V0 阶段默认不下放：

### 1. 最终裁决类
- 最终放行 / 返工 / 暂停结论
- 是否升级为更高优先级问题
- 是否触发对外动作

### 2. 跨任务统一类
- 跨子任务口径统一
- 跨阶段结论合并
- 不同子任务之间的冲突裁定

### 3. 根因排序类
- 最终根因排序
- 因果链权重判断
- 哪个问题是“表层症状”还是“底层根因”的最终决定

### 4. 风险表述类
- 最终风险等级表达
- 对团队 / 系统的最终影响描述
- 是否采用“已证实 / 高置信 / 中置信 / 待核实”等最终措辞

### 5. 对外表述类
- 对 BOSS 或团队的最终结论版输出
- 任何会被当成正式结论传播的最终文本

一句话说：

> **凡是决定“这件事最后怎么被定义”的工作，默认都留在主控位。**

---

## 五、主控位职责（V0）

主控位在 Spawn 机制中的职责，不是多做一遍，而是守住整个机制的结构边界。

### 主控位至少负责：
- 判断是否启动 Spawn
- 决定拆分方式
- 定义禁拆区
- 编写 / 审核子任务卡
- 派发子任务
- 回收子任务结果
- 统一跨任务口径
- 给出最终结论
- 判断是否放行 / 返工 / 暂停

### 主控位不应做：
- 用 Spawn 之名把责任全部外包
- 在没有边界定义的情况下胡乱拆包
- 让子任务顺手替自己下最终判断
- 回收后又把所有工作完整重做一遍

---

## 六、子任务位职责（V0）

子任务位的职责是：

> **在明确边界内完成一个局部闭环，并把结果以可回收格式交回主控。**

### 子任务位至少应做到：
- 只处理任务卡内指定问题
- 只使用被授权的输入材料
- 产出明确输出物
- 写清楚局部结论、证据、未决点
- 不越界扩写最终结论

### 子任务位不应做：
- 顺手修改其他子任务的结论
- 把局部发现直接上升为全局根因
- 越界给最终放行 / 暂停建议
- 擅自统一跨阶段口径

---

## 七、推荐拆分方式（V0）

V0 阶段优先采用以下三种拆法：

### 方案 A｜按阶段拆
适用于：
- 每一阶段边界清楚
- 阶段间衔接明确
- 主控可以逐段验收

示例：
- A / B / C / D 各自局部闭环
- 最终 E 由主控收口

### 方案 B｜按问题类型拆
适用于：
- 一个任务里混有不同类型问题
- 每类问题适合独立核验

示例：
- 子任务 1：口径返工
- 子任务 2：统计对账
- 子任务 3：数据源核验

### 方案 C｜按证据链拆
适用于：
- 有多个可独立验证的证据链
- 最终需要主控统一根因排序

示例：
- 子任务 A：样本证据
- 子任务 B：日志证据
- 子任务 C：底层数据路径证据

---

## 八、不推荐拆分方式（V0）

### 1. 按输出格式乱拆
例如：
- 一个 Agent 写表格
- 一个 Agent 写报告
- 一个 Agent 下结论

这类拆法表面分工，实际上会造成信息断裂。

### 2. 把最终判断也一起拆下去
这是最危险的做法。
一旦多个子任务都在偷偷下最终结论，主控就会失位。

### 3. 把高度耦合的连续思考强行切碎
如果问题本身边界还没立住，拆得越细，漂得越快。

---

## 九、边界违规示例（V0）

### 示例 1：子任务越权
- 任务卡只要求核对统计数字
- 子任务却在结果中写“建议立即暂停系统”

判定：**越界**
处理：主控退回或降级其相关表述

### 示例 2：主控失位
- 子任务已各自完成局部结论
- 主控不做统一口径，直接拼接成最终汇报

判定：**主控缺位**
处理：必须补做收口层

### 示例 3：禁拆区被错误下放
- 子任务负责底层数据核验
- 却直接定义某空库为最终根因

判定：**禁拆区污染**
处理：仅保留事实层结果，移除根因定性

## 十、拆分检查清单（主控用）

派发前，主控建议自检：

- 这个子任务是不是一个局部闭环？
- 这个子任务有没有明确输入和明确输出？
- 这个子任务的验收标准是否可判断？
- 这个子任务是否碰到了最终判断权？
- 子任务之间是否会互相踩边界？
- 主控最后是否仍保留统一裁决权？

若任一项回答不清楚，先别派。

## 十二、一句话收口

> **Spawn 机制的核心，不是“多几个 Agent 一起干”，而是把局部执行权和最终判断权切开，保证任务能拆、能收、还能成立。**
