# 方案断点整理｜子任务卡样例（富贵）v1

- 日期：2026-04-02
- 用途：给富贵在“项目方案重写”场景中，作为第一批最稳 spawn 的正式子任务卡样例
- 推荐搭配：`subagent_AGENTS_执行者版.md`
- 适用场景：正式方案还在主链重写中，但希望先把断点、重复段、证据缺口整理出来，减轻前处理负担

---

## 一、Task Goal

对当前项目方案版本做一次**断点整理**，目标不是直接重写终稿，而是把后续重写最需要先处理的问题找出来并结构化回传。

你本轮负责的是：
- 找出当前版本中的**重复段**
- 找出**逻辑断裂段**
- 找出**需要补证据 / 补数据 / 补定义**的段
- 找出**可保留的强段落**

你本轮**不负责**：
- 直接改写整份方案
- 统一整份方案口径
- 替主 agent 做最终结构重组
- 对未明确的商业判断自行补完

---

## 二、Input Materials

至少包含：
- 当前方案版本文件路径
- 如有必要，补充旧版方案 / 提纲 / 参考材料路径
- 如有明确项目目标，也可附一句本轮重写目标

### 当前主版本（由主 agent 填）
- `【在这里填当前方案文件路径】`

### 可选补充输入
- 旧版方案：`【可选】`
- 提纲 / 框架稿：`【可选】`
- 参考材料：`【可选】`

---

## 三、Output Required

请输出一份**结构化断点清单**，至少包含以下四类：

### 1. 重复段
- 哪些段落表达重复
- 重复位置在哪里
- 哪段更值得保留

### 2. 逻辑断裂段
- 哪些地方前后衔接断了
- 断裂原因是什么（跳步 / 定义缺失 / 论据不足 / 结构错位）

### 3. 证据缺口段
- 哪些段落在讲判断，但没有足够依据
- 哪些地方需要补数据 / 案例 / 定义 / 来源

### 4. 可保留强段落
- 哪些段落现在已经相对稳
- 为什么稳（表达清楚 / 结构完整 / 判断边界诚实 / 证据相对够）

---

## 四、Return Format

建议按以下结构回传：

### A. 总体判断
- 当前方案最大的问题是什么
- 本轮最优先该先改哪一类问题

### B. 重复段清单
- 文件路径 / 标题 / 段落位置
- 问题说明
- 建议保留哪段

### C. 逻辑断裂清单
- 文件路径 / 标题 / 段落位置
- 断裂类型
- 影响是什么

### D. 证据缺口清单
- 文件路径 / 标题 / 段落位置
- 缺什么
- 如继续沿用，会带来什么风险

### E. 可保留强段落
- 文件路径 / 标题 / 段落位置
- 为什么建议保留

### F. 下一步建议
- 如果主 agent 要开始重写，建议优先从哪三处下手

---

## 五、Acceptance Standard

这次子任务算完成，至少要满足：
- 不是泛泛而谈，而是挂到具体文件 / 位置
- 四类清单（重复 / 断裂 / 证据缺口 / 强段）至少都覆盖到
- 能让主 agent 一眼知道下一步先改哪里
- 不把“发现问题”直接越权写成“替主链完成重写”

---

## 六、Red Lines

- 不要直接改写终稿正文
- 不要给出整份方案的最终定稿建议
- 不要把推测写成对项目的确定判断
- 不要因为想显得完整，就把没证据的地方写满

---

## 七、One-Line Summary

> 你这次不是来替富贵重写方案的；你是来把这份方案最该先处理的断点清楚地找出来，并带着证据回来的。
