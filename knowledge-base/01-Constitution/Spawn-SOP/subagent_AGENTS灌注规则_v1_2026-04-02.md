# subagent AGENTS 灌注规则 v1

- 日期：2026-04-02
- 经手：Anna
- 用途：规定旺财、富贵等主 agent 在日常 spawn 时，何时给 subagent 灌执行者版 / 评估者版 AGENTS 模板，何时叠加子任务卡，何时只用轻量版。

---

## 一、结论先行

spawn 时不要再靠临时口头补“你注意一下边界”。

默认规则应改成：

> **先判任务类型，再判模板厚度，再决定是否叠加子任务卡。**

也就是说，spawn 前至少做三件事：
1. 这是执行型任务，还是评估型任务？
2. 这是完整版场景，还是轻量版就够？
3. 这次需不需要额外叠加子任务卡？

---

## 二、默认灌注规则

### 1. 默认灌“执行者版”的任务
以下任务，默认灌 **执行者版 AGENTS**：
- 文件整理 / 结构重写
- 前处理子任务
- 资料归纳 / 分类 / 归档
- 项目方案拆段重写
- 固定格式产物生产
- 已有明确边界的研究 / 归纳 / 梳理任务

一句话：
> **凡是主要目标是“做出东西”，而不是“判断别人做得行不行”，默认灌执行者版。**

---

### 2. 默认灌“评估者版”的任务
以下任务，默认灌 **评估者版 AGENTS**：
- 审计 / 验收 / 打回
- 验证是否达标
- 判断证据链是否闭合
- 评估方案是否站得住
- 检查回传是否满足口径
- 对现有产物做通过 / 驳回 / 待补证据判断

一句话：
> **凡是主要目标是“判断是否成立”，而不是“补做内容”，默认灌评估者版。**

---

## 三、什么时候要同时叠加子任务卡

以下情况，**AGENTS 模板 + 子任务卡** 必须同时上：

### 1. 任务边界不天然清楚
例如：
- 重写项目方案中的某一段
- 研究一堆输入材料后只输出某个结构结论
- 对现有方案做局部重构，不允许越界扩写

原因：
- AGENTS 只能管“你是谁、怎么做”
- 子任务卡必须补“这次具体做哪一段”

---

### 2. 输出需要正式验收
例如：
- 要回主 agent 再进入二审
- 要被 Anna / BOSS / 上游主链验收
- 输出不只是参考，而要进入正式流程

原因：
- 子任务卡需要把交付物、回传格式、验收口径钉死

---

### 3. 输入材料较多，容易脑补或漂移
例如：
- 多文档归纳
- 多文件改写
- 需要引用若干已有方案 / 规则 / 历史版本

原因：
- AGENTS 解决的是行为纪律
- 子任务卡解决的是本次输入与输出边界

---

## 四、什么时候轻量版就够，不必上完整版

以下情况，可只灌**轻量版约束**，不必上完整版：
- 任务明确、边界短、输入少
- 输出只需一个小结论 / 小片段 / 小修订
- 不进入正式验收链
- 即使失败，代价也低
- 主 agent 可以快速复核与兜底

典型例子：
- 检查一个文档里的单个字段
- 生成一段简短摘要
- 对一个已明确段落做小幅润色
- 单次资料分类，不涉及正式结论

### 轻量版最低应保留的内容
即使轻量版，也至少要保留：
- 角色（执行者 / 评估者）
- 不准越界
- 不准把假设写成事实
- 回传时要说清：做了什么、没做成什么

---

## 五、什么时候必须上完整版

以下情况，必须用完整版（评估者版 / 执行者版）并尽量叠加子任务卡：
- 输出会进入正式决策链
- 任务容易越界、漂移、脑补
- 任务失败成本较高
- 主 agent 不会立即亲自重做
- 需要稳定回传格式
- 需要明确停机条件和升级条件

典型例子：
- 富贵重写项目方案中的正式章节
- 旺财做长文结构整理并准备交付上游
- 对方案 / SOP / 主任务卡做正式审计
- 为下一轮制度化 / 验收链准备正式输入

---

## 六、给旺财 / 富贵的默认建议

### 旺财
旺财的日常 spawn 里，默认更常见的是：
- 长文结构整理
- 方案压缩
- 分段重写
- 审计前的结构预处理

因此：
- **生产 / 改写 / 整理类** → 默认灌执行者版
- **审计 / 校对 / 站得住性判断类** → 默认灌评估者版
- 只要涉及正式交付段落，默认叠加子任务卡

### 富贵
富贵的日常 spawn 里，默认更常见的是：
- 资料归纳
- 市场材料扫描
- 项目方案重写的局部执行
- 热点前处理 / 信息整理

因此：
- **资料整理 / 前处理 / 重写执行类** → 默认灌执行者版
- **判断某份材料 / 结论是否成立** → 默认灌评估者版
- 只要涉及项目方案正文改写，默认叠加子任务卡

---

## 七、最小操作规则（给主 agent）

spawn 前，主 agent 至少问自己四个问题：

1. 这次 subagent 是来“做”，还是来“判”？
2. 这次任务边界有没有清楚写在子任务卡里？
3. 这次失败的代价高不高？
4. 这次输出要不要进入正式验收链？

### 快速决策表
- **做 + 低风险 + 边界短** → 轻量执行者版
- **做 + 正式交付 + 边界复杂** → 执行者版 + 子任务卡
- **判 + 低风险快速复核** → 轻量评估者版
- **判 + 正式审计 / 验收** → 评估者版 + 子任务卡

---

## 八、当前不建议做的事

- 不建议每次 spawn 都灌一份超长完整版
- 不建议让 AGENTS 模板替代子任务卡
- 不建议在没有正式交付需求时上过厚模板
- 不建议让 subagent 同时承担“执行 + 最终裁决”双角色

---

## 九、一句话收口

> **先判“做”还是“判”，再判轻重，再决定是否叠加子任务卡。spawn 灌注规则不该靠临场发挥，而应成为固定动作。**
