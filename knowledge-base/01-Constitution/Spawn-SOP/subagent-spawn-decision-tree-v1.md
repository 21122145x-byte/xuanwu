# Subagent Spawn 决策树 v1

## 一句话判断

spawn subagent 不是默认动作，
也不是“有任务就拆出去”的习惯反应。

它本质上是一个决策关口：

> 这段工作，是否已经被证明值得从主判断链里剥离出去，交给一个独立执行工位来跑？

所以真正的流程不是：
有任务 → 起 subagent

而是：

> 有任务 → 先确认意图 → 判断是否该亲自做 → 判断能否拆成独立子任务 → 判断拆出去值不值 → 值得才 spawn → 回传后验收整合。

---

## 一、总流程骨架

### Step 1：收到任务
主 agent 接收到任务请求。

### Step 2：先做 L0 意图确认
确认：
- 用户到底要什么
- 当前理解是否对
- 需求是否仍然模糊

如果还没经过意图确认，不进入 spawn 决策。

### Step 3：判断是否该亲自做
先判断这件事的核心价值是否在主 agent 的专业判断、结构判断、创意判断或表达判断上。

### Step 4：如果不该亲自做，再判断能不能拆
不是所有任务都能拆成合格子任务。

### Step 5：如果能拆，再判断是否值得拆
能拆不等于值得拆。需要额外判断拆分收益。

### Step 6：定义子任务边界
把子任务写成可独立执行的工作单元。

### Step 7：spawn subagent
调用会话生成工具，拉起独立子会话。

### Step 8：等待回传
子会话执行后回传结果。

### Step 9：主 agent 验收结果
判断是否合格、是否需要整合、是否返工。

### Step 10：闭环 / 再流转
根据结果决定：
- 直接闭环
- 继续整合
- 返工
- 再拆分下一段任务

---

## 二、第一关：该不该我自己做？

这一关比“能不能 spawn”更重要。

### 判断问题 1：这件事的核心价值是什么？
如果核心价值在：
- 判断
- 创意
- 结构
- 表达
- 方向取舍

那么通常更适合主 agent 亲自下场。

### 判断问题 2：这件事是不是足够小？
如果任务小到：
- 一句话建议
- 一段改稿
- 一次快速比较
- 一个小判断

那么直接做通常更快。

### 判断问题 3：拆出去会不会更慢？
spawn 并非零成本，它意味着：
- 需要定义任务
- 需要独立理解
- 需要等待执行
- 需要后续验收

如果这些成本高于直接做，就不值得拆。

### 判断问题 4：我是否有验收能力？
如果主 agent 自己都无法判断什么叫“做对了”，就不该先 spawn。

---

## 三、第二关：能不能拆成独立子任务？

就算不该亲自做，也不代表立刻能拆。

一个合格子任务，至少要满足以下四个条件：

### 条件 1：边界清楚
必须能清楚说明：
- 只做哪一段
- 不要做哪一段

如果边界不清，subagent 极易自动扩张任务范围。

### 条件 2：输入足够
必须明确：
- 给它哪些材料
- 这些材料是否足够开工
- 当前缺什么信息

如果输入不足，subagent 只是在替主 agent 猜。

### 条件 3：输出可定义
必须能说明它最终应回传：
- 一个摘要
- 一版文档
- 一个结论
- 一组候选方案
- 一个执行结果

如果连输出长什么样都说不清，任务还没成熟。

### 条件 4：结果可验收
主 agent 必须能判断：
- 完成没有
- 跑偏没有
- 能不能进入下一步

否则就不该拆。

---

## 四、第三关：拆出去值不值？

能拆，不代表值得拆。

以下是五个典型的“值得拆”信号：

### 信号 1：需要并行
例如：
- 多方向并行探索
- 多来源调研
- 多版初稿同时生成

### 信号 2：属于长链执行
子任务执行链较长，但中间不需要主 agent 不断介入。

### 信号 3：偏标准化
例如：
- 批量整理
- 结构化提取
- 模板化处理
- 标准化输出

### 信号 4：会阻塞主判断链
如果主 agent 自己去做，会被执行细节拖住，影响更高价值的判断与整合。

### 信号 5：需要会话隔离
例如：
- 不想污染主线程上下文
- 希望子问题单独聚焦
- 想把执行过程和主判断链隔离开

如果以上信号都不强，通常不必拆。

---

## 五、真正进入 spawn 前，要定义什么？

只有前三关都过了，才进入子任务定义阶段。

此时必须至少定义清楚：
- 任务目标（Goal）
- 任务边界（Scope）
- 输入材料（Inputs）
- 输出格式（Output）
- 限制条件（Constraints）
- 验收标准（Acceptance）
- 回传路径（Return Path）

如果这些写不清，spawn 结果通常不稳。

---

## 六、spawn 之后，主 agent 还要做什么？

spawn 不等于把事情扔出去不管。

主 agent 仍然承担三项责任：

### 1）保持主线程任务感知
主 agent 必须持续知道：
- 子任务在跑什么
- 为什么拆出去
- 回来后怎么接

### 2）必要时进行介入
如果出现：
- 卡住
- 跑偏
- 边界不清
- 需要补材料

主 agent 需要补充信息或调整任务。

### 3）准备验收口径
子结果回来后，主 agent 必须有能力判断：
- 是否合格
- 是否要整合
- 是否要返工

---

## 七、回传之后的四种结果

### A：合格，直接闭环
- 满足原目标
- 没跑偏
- 可以直接进入下一步或交付

### B：合格，但需要整合
- 单个子结果没问题
- 但还需要与其他结果合并

### C：部分合格，需要补一刀
- 方向对
- 但还差一部分
- 可由主 agent 补，或再次委派修正

### D：不合格，返工或放弃
- 边界跑偏
- 输出不符合要求
- 不值得继续投入

所以必须明确：

> subagent 完成的是子任务执行，不是总任务闭环。

总任务的闭环责任始终在主 agent。

## 八、可直接使用的决策树

### Q1：任务是否已经过 L0 意图确认？
- 否 → 不 spawn，先确认意图
- 是 → 进入 Q2

### Q2：这件事的核心价值是否在主 agent 的专业判断？
- 是 → 主 agent 亲自做
- 否 → 进入 Q3

### Q3：这件事是否足够小，直接做更快？
- 是 → 直接做
- 否 → 进入 Q4

### Q4：能否拆成边界清楚、输入足够、输出可定义、结果可验收的子任务？
- 否 → 不 spawn，继续澄清或改结构
- 是 → 进入 Q5

### Q5：拆出去是否真的更值？
判断依据：
- 是否可并行
- 是否长链
- 是否标准化
- 是否阻塞主线程
- 是否需要会话隔离

- 否 → 主 agent 直接做
- 是 → 进入 Q6

### Q6：定义子任务卡
写清：
- Goal
- Scope
- Inputs
- Output
- Constraints
- Acceptance
- Return Path

### Q8：接收结果并验收

## 九、最常见的三种误判

### 误判 1：意图还没确认，就急着 spawn
这会把误解外包。

### 误判 2：边界没定义清楚，就急着 spawn
这会让 subagent 自动脑补任务。

### 误判 3：没有验收标准，却先 spawn
这会让主 agent 丢失控制权。

---

## 十、最后两句

> spawn subagent 不是默认动作，而是任务被证明值得拆分后的动作。

> 真正的 spawn 决策，不发生在执行层，而发生在“我要不要把这段工作从主判断链里剥离出去”这一层。
