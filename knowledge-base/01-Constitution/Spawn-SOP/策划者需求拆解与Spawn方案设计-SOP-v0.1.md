# 策划者需求拆解与 Spawn 方案设计 SOP v0.1

**日期：** 2026-03-30  
**状态：** 初版可用  
**用途：** 规定当任务被判断为 P2 及以上后，策划者应如何判断是否值得 Spawn、如何拆解任务、如何配置模型与成本。

---

## 一、策划者的职责，不只是“会拆任务”

> **策划者的职责，是判断什么时候值得拆、怎么拆、哪里不能拆，以及这次拆解会带来多少时间、subagent、token、协调和返工成本。**

Spawn 不是默认执行方式，而是一种：
- 控制上下文负载
- 增加脚手架和回传成本
- 换取质量稳定性与可控性的机制选择

因此，策划者不是“拆得越多越好”，而是要对拆解后的收益与成本负责。

---

## 二、进入条件

本 SOP 只适用于：
- **P2 / P3 / P3+** 任务
- 已经完成需求对齐
- 已确认不适合直接按 P0 / P1 轻流程执行

---

## 三、Step 1｜先判断：值不值得 Spawn

不是先问“怎么拆”，而是先问：

> **这个任务是否值得进入 Spawn 设计？**

### 四个门槛判断

#### 门槛 1｜上下文负载门槛
单 agent 直接做，是否会因为：
- 上下文过长
- 任务过耦合
- 目标过多
- 推理链过深
而明显失稳？

如果不会，优先别 Spawn。

---

#### 门槛 2｜可分拆性门槛
这个任务能否切成若干：
- 边界清楚
- 目标明确
- 完成后可回传
- 子任务彼此关系可管理
的执行单元？

如果不能切，就不该硬拆。

---

#### 门槛 3｜可验收性门槛
每个子任务完成后，能否根据明确标准判断：
- 做对没
- 做完没
- 是否需要返工

没有验收标准的 Spawn，后面会退化成垃圾转运链。

---

#### 门槛 4｜协调收益是否大于协调成本
增加以下开销后：
- handoff
- subagent
- 回传
- 评估
- 中途重组

质量提升是否值得？

如果不值得，别 Spawn。

---

## 四、Step 2｜判断任务类型

策划者需要先判断任务属于哪类结构：

### A 类｜低耦合、可并行
- 子任务独立
- 可并行跑
- 相互影响小

**适合：** 多 agent 平行 Spawn

---

### B 类｜中耦合、可拆但需顺序协调
- 可以拆
- 但顺序与回传关系重要
- 需要明确关键路径

**适合：** 有限 Spawn / 串并结合

---

### C 类｜高耦合、主链必须保持统一心智
- 统一判断、统一审美、统一架构要求高
- 局部拆分容易损失整体一致性

**适合：** 单 agent 主控，按需 Spawn 辅助子任务

---

### D 类｜高风险 / 对抗 / 长时程 / 真实世界任务
- 对抗输入
- 真实资源
- 高权限
- 长时程 autonomy
- 需要外层 guardrail

**适合：** P3+ 逻辑单列处理

---

## 五、Step 3｜明确“哪里不能拆”

这一步必须显式写出来。

策划者需要写：

### 不可拆部分
例如：
- 最终整合
- 核心判断
- 审美一致性
- 统一架构决定
- 关键策略裁决

> **Spawn 的关键不是“能拆多少”，而是“知道哪里不能拆”。**

---

## 六、Step 4｜设计拆解方案

每个子任务至少写清：

1. **目标**
2. **输入**
3. **输出**
4. **验收标准**
5. **回传字段**
6. **是否需要中途澄清**
7. **是否可独立完成**

如果某子任务连这 7 项都写不清，就说明还没到可稳定 Spawn 的程度。

---

## 七、Step 5｜模型配置与成本设计（必填）

这是策划者必须承担的新职责。

> **任务等级 ≠ 模型等级。高等级主任务，可以包含大量低成本子任务。**

### 模型分层原则

#### 高模
适合：
- 高难判断
- 高耦合整合
- 复杂策略设计
- 关键写作 / 核心审美判断
- 高不确定分析

#### 中模
适合：
- 常规分析
- 一般结构化整理
- 中等复杂度研究
- 普通手工推理

#### 低成本模型
适合：
- 搜索
- 抓取
- 格式清洗
- 基础分类
- 大批量初筛
- 机械重复前处理

---

## 八、Step 6｜策划者必须显式报出的成本项

### 1. 预估任务等级
- P0 / P1 / P2 / P3 / P3+

### 2. 建议执行模式
- 单 agent
- 限定 Spawn
- 完整 Spawn

### 3. 预估周期
至少写：
- 最快闭环时间
- 保守闭环时间
- 关键路径

### 4. 预计 subagent 数
至少写：
- 固定需要几个
- 哪些并行
- 哪些按需生成

### 5. 成本说明
至少写：
- token 成本（低 / 中 / 高，或区间估计）
- 协调成本（handoff、回传、评估次数）
- 返工风险（哪个节点返工最贵）
- 为什么这些成本值得

---

## 九、建议新增字段：Spawn 方案成本评估

策划者方案模板中建议新增以下字段：

### Spawn 方案成本评估（必填）
1. **预估任务等级**
2. **建议执行模式**
3. **预估周期（最快 / 保守）**
4. **预计 subagent 数（固定 / 按需）**
5. **模型配置说明**
6. **token / 协调 / 返工成本说明**
7. **为什么这次 Spawn 值得做**

---

## 十、收口

> **Spawn 方案不是“拆任务建议”，而是一次正式的执行设计。策划者必须同时对拆解逻辑、上下文负载控制、模型配置、时间周期、subagent 数量和协调成本负责。**
