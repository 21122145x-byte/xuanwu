## 一句话流程图

```
BOSS 提出需求
    ↓
策划者对齐需求 → BOSS 放行需求文档
    ↓
策划者出方案 → Anna 评估 → 放行
    ↓
转到调度者（进入 Spawn 执行链）
    ↓
┌─────────────────────────────────────────┐
│  P1 / P2 级任务 Spawn 完整流程          │
│                                         │
│  ① 调度者生成评估者 subagent           │
│     → 灌注评估者专用 agents.md          │
│                                         │
│  ② 调度者出任务卡                      │
│     → 评估者 subagent 评估 → 放行      │
│                                         │
│  ③ 调度者出子任务卡                    │
│     → 评估者 subagent 评估 → 放行      │
│                                         │
│  ④ 调度者生成执行 subagent             │
│     → 执行 subagent 返回结果            │
│                                         │
│  ⑤ 调度者回收整理结果，落档            │

│                                         │

└─────────────────────────────────────────┘
```

## 各环节说明

### 环节 1｜BOSS 提出需求

发起者：BOSS

无固定格式，可以是任何形式的原始需求输入。

---

### 环节 2｜策划者对齐需求

执行者：策划者（粉雪或对应角色）

任务：理解需求、澄清模糊点、确认边界、和 BOSS 对齐理解

输出：对齐纪要或修订后的需求文档草案

---

### 环节 3｜BOSS 放行需求文档

决策者：BOSS

动作：明确批准进入方案阶段

---

### 环节 4｜策划者出方案

执行者：策划者

任务：把需求转化为可执行方案

输出：方案文档

---

### 环节 5｜Anna 评估方案

执行者：Anna（作为前置评估者角色）

任务：独立评估方案是否成立、风险是否可控、方向是否正确

输出：评估结论（放行 / 打回 / 条件放行）

---

### 环节 6｜放行，转到调度者

触发条件：Anna 给出放行结论，提供项目专用评估者 agents.md

动线：从方案审核切换到 Spawn 执行链

---

### 环节 7｜Spawn 执行链（P1/P2 级）

#### 7.1 生成评估者 subagent

调度者生成专用评估者 subagent，并向其灌注**评估者专用 agents.md**（含角色定义、校准标准、判断边界，Anna提供）

#### 7.2 评估任务卡

调度者出具正式任务卡（Intent Card 或主任务卡）
→ 评估者 subagent 独立判断：是否放行

#### 7.3 评估子任务卡

调度者出具子任务卡（Subtask Card）
→ 评估者 subagent 独立判断：是否放行

#### 7.4 生成执行 subagent

调度者根据子任务卡，生成对应的执行 subagent

#### 7.5 执行 subagent 返回结果

执行 subagent 完成工作，结果返回调度者

#### 7.6 调度者整理结果落档

调度者回收、整理执行结果，形成正式产物文档

#### 7.7 评估者评估落档结果

评估者 subagent 对最终落档内容进行独立评估

#### 7.8 调度者复盘落档

调度者完成复盘文档

#### 7.9 发给 BOSS

将完整产物和复盘发给 BOSS

#### 7.10 任务闭环

BOSS 确认收到，流程正式关闭

## 待确认节点

以下节点在当前版本里没有明确，需要补充：

### 节点 A：P1/P2 的判断标准是什么 ✅ 已定

- 第一周期提前做模糊框架设定
- 现场定级：策划者提出建议 → BOSS 拍板 → Anna 复核

### 节点 B：评估者 subagent 的判断被记录在哪里

每次放行/打回，是否有显式记录？记录在哪里？

### 节点 C：调度者和评估者意见冲突怎么办 ✅ 已定（修正）

**评估者打回，调度者无权跳过，只能修改后重新提交评估。**

评估者具有独立否决权，调度者不得 override。但打回不是终点——调度者必须根据评估意见修改产物，重新进入评估流程，直到评估者放行为止。

### 节点 D：执行 subagent 失败怎么办

执行 subagent 超时、退出、报错，调度者如何处理？

### 节点 E：P3 级任务走什么路径

不是所有任务都走完整的 Spawn 链。P3 级任务是否走简化路径？简化路径是什么？

### 节点 F：评估者 agents.md 的来源 ✅ 已定

**Anna 负责预先维护，评估者 agents.md 是固定文档，非动态生成。**
