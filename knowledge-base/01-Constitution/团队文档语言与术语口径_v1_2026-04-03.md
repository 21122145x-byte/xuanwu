# 团队文档语言与术语口径 v1

- 日期：2026-04-03
- 状态：v1
- 适用范围：团队知识库、`AGENTS.md`、`SOUL.md`、`USER.md`、`SKILL.md`、`cron`、`HEARTBEAT.md`、README、执行卡、审计意见、汇报文档

---

## 一、总原则

团队文档默认采用以下写法：

> **逻辑、规则、判断用中文；专属名词、技术对象、稳定标识用英文。**

这是当前团队最稳的口径，因为文档同时服务：
1. 人类成员阅读
2. Agent 读取与执行
3. 系统检索、链接、索引与路径引用

如果全部中文：
- 技术对象容易漂移
- 同一概念会出现多种说法
- 搜索与引用不稳定

如果全部英文：
- 规则类文档阅读成本高
- 细微判断难以准确表达
- 协作语境不够贴近团队实际

因此采用折中口径：
- **叙述语言：中文**
- **系统对象名：英文原名保留**

---

## 二、默认写法规则

## 1. 中文负责表达逻辑
以下内容默认优先用中文：
- 规则
- 判断条件
- 风险说明
- 建议动作
- 升级原因
- 边界说明
- 汇报口径
- 成败标准
- 协作关系

## 2. 英文保留稳定对象名
以下内容默认不硬翻译，保留英文：
- `cron`
- `heartbeat`
- `skill`
- `session`
- `sessionKey`
- `topic`
- `index`
- `note`
- `source`
- `README`
- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `OpenClaw`
- `Claude Code`
- `LMAP`
- `Spawn`
- `SOP`
- `Checklist`
- `Template`
- `Archive`
- `Research`
- `Notes`
- `Resources`

这些在团队系统中已经是：
- 固定术语
- 固定对象
- 固定层名
- 固定文件名

翻译后反而会增加歧义。

---

## 三、推荐写法风格

推荐统一使用：

> **中文句子 + 英文对象名**

例如：
- 若 `Research/` 有新增，但 `INDEX.md` 未更新，则记为索引断点。
- 当 `cron` 连续失败 2 次且无人工说明时，升级审阅。
- 新项目优先补 `README.md`，再补详细结构。
- `heartbeat` 只负责发现断点，不负责直接搬文件。

这是当前最稳定、最适合团队协作与系统检索的写法。

---

## 四、不同文档类型的建议

## 1. `SKILL.md`
- 用途、触发条件、步骤、边界 → 中文
- 固定技术对象、路径、接口、命令对象名 → 英文保留

示例：
- 当任务涉及 `cron` 创建、`heartbeat` 检查、`sessionKey` 路由时，优先先核对当前配置。

---

## 2. `cron`
- 任务说明、汇报内容、动作逻辑 → 中文
- job 名称可采用中英混合

示例：
- `Curie-任务B研究状态汇报`
- `fenxue-context-sync`
- `anna-memory-integrity-check`

原则：
- 业务语义可中文
- 技术职能与稳定标识保留英文更利于日志与检索

---

## 3. `HEARTBEAT.md`
- 规则、判断逻辑、动作分级 → 中文
- 检查对象路径、文件名、术语对象名 → 英文保留

示例：
- 检查 `knowledge_base/03-Notes/` 是否有新的 Claude Code 研究底稿。
- 若 `INDEX.md` 未接住新增 topic，则提醒处理。

---

## 4. README / 索引 / 执行卡
- 层级用途、判断逻辑、维护说明 → 中文
- 文件名、目录名、对象名、稳定术语 → 英文保留

---

## 五、首次出现术语时的写法

对于新术语或可能有歧义的术语，建议首次出现时采用：

> 中文解释 + 英文原词（括号）

例如：
- 任务主题页（topic）
- 索引入口（index）
- 心跳检查（heartbeat）
- 子任务拆分（Spawn）

后文如已无歧义，可直接使用英文原词。

---

## 六、不推荐的写法

### 1. 把稳定对象名硬翻成中文
例如：
- 把 `heartbeat` 写成“心跳器”
- 把 `sessionKey` 写成“会话键”
- 把 `README` 写成“读我文件”

### 2. 一份文档内同一对象多种叫法并存
例如同一页里混用：
- topic / 专题页 / 主题页 / topic 页

### 3. 为了显得“正式”而把本来清楚的英文对象名换成绕的中文描述

## 七、一句话收口

团队文档的目标，不是追求形式统一得漂亮，而是：

> **让规则更清楚、对象更稳定、检索更可靠、协作更少歧义。**

因此当前团队统一采用：

> **逻辑、规则、判断用中文；专属名词、技术对象、稳定路径名用英文。**
