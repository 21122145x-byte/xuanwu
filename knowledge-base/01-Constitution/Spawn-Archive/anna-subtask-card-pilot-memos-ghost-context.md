# Anna 试跑说明：用 Subtask Card 执行「memos 幽灵上下文排查」

## 目的

这份文档不是在重新定义任务本身，
而是给 Anna 一次试跑用的参考说明：

> 用 `Subtask Card v1` 这套方法，执行一次真实的排查型子任务，观察这张卡是否真的能支撑独立执行、边界控制、结果回传与后续验收。

当前试跑对象：
- **任务主题**：memos 幽灵上下文排查
- **执行角色**：Anna
- **试跑重点**：不是先看修没修好，而是先看 `Subtask Card` 能不能让这类任务跑得更稳

---

## 这次试跑要验证什么

这次试跑重点验证 4 件事：

### 1）任务边界是否能被卡片钉住
重点看：
- Anna 会不会把“排查任务”自动扩成“修复任务”
- 是否能停留在“现象识别、链路排查、证据归纳”边界内

### 2）输入不足时，卡片是否能支撑“停下来并上报缺口”
重点看：
- 是否会主动标出 `Missing Information`
- 是否会在信息不足时暂停脑补，而不是继续硬推

### 3）回传结构是否清楚
重点看：
- 回传是否区分摘要、正文、风险 / 缺口
- 是否能让上游快速看懂现象、证据、推测、结论之间的关系

### 4）上游是否能依据卡片直接做验收
重点看：
- 结果是否足够清楚，能让上游决定下一步
- 是否需要大量二次翻译才能继续流转

---

## 推荐执行方式

建议 Anna 在执行时，按下面顺序走：

1. 先读这次任务的 Subtask Card
2. 明确任务是“排查”，不是“直接修复”
3. 如果输入不足，先列缺失项
4. 进入排查
5. 输出结构化结果
6. 按指定回传格式返回

这次试跑最重要的一条纪律：

> **不要因为你看到了可能的修法，就提前越过排查边界直接进入修复。**

本轮目标是：
**先把问题链路看清，再决定要不要起下一张修复任务卡。**

## 给 Anna 的试跑 Subtask Card（参考执行）

```md
# Subtask Card v1

## 1. Identity
- Subtask ID: ST-20260328-MEMOS-001
- Parent Task ID: T-20260328-MEMOS-GHOST
- Requested By: Shaun
- Assigned To: Anna
- Current Status: NEW

## 2. Purpose
- Goal:
  排查 memos 幽灵上下文问题的可能来源、触发链和当前污染路径。
- Why This Was Delegated:
  该任务属于验证与排查型工作，需要较强的资料核查、问题定位与留痕能力，适合由 Anna 主导执行。

## 3. Scope
- In Scope:
  1. 识别“幽灵上下文”具体表现
  2. 排查可能的注入链路、触发条件、污染来源
  3. 区分现象、推测、证据
  4. 输出当前最可能的问题路径与未确认点
- Out of Scope:
  1. 不直接修改系统配置
  2. 不直接做修复动作
  3. 不扩大到无关模块的全面系统重构
- Success Boundary:
  给出一版结构清楚的排查结果，能让上游判断下一步该修哪里，即视为完成。

## 4. Inputs
- Input Materials:
  1. 当前与 memos 幽灵上下文相关的对话记录
  2. 相关日志、现象记录、已有判断（如有）
  3. 任务目标：先排查，不先修复
- Assumed Context:
  1. 当前重点是识别污染链，而不是立刻修复
  2. 输出需要可复盘、可继续流转
- Missing Information:
  若缺少关键日志、配置、复现路径，请明确列出，不要自行脑补

## 5. Execution
- Action Required:
  对 memos 幽灵上下文问题做结构化排查，并输出阶段性诊断结果。
- Suggested Method:
  先定义问题现象，再列可疑来源，再按证据强弱排序，最后给出最可能链路与未决问题。
- Constraints:
  结论必须区分“已证实 / 高概率 / 待确认”。
- Non-Negotiables:
  1. 不得把推测写成结论
  2. 不得越界直接修系统
  3. 不得省略证据依据

## 6. Output
- Expected Output:
  一份排查报告初稿
- Output Format:
  Markdown
- Required Sections:
  1. 问题现象
  2. 已确认事实
  3. 可疑链路
  4. 证据与依据
  5. 当前最可能结论
  6. 未确认点
  7. 下一步建议

## 7. Acceptance
- Acceptance Criteria:
  1. 能看出“现象 / 证据 / 推测 / 结论”的区分
  2. 能定位到至少一条最可能污染路径
  3. 输出可直接供上游决定下一步动作
- Rejection Conditions:
  1. 把猜测当结论
  2. 只有散点观察，没有结构化结果
  3. 越界直接进入修复方案
- Self-Check Before Return:
  1. 是否区分了事实与推测？
  2. 是否已经写出未确认点？
  3. 是否仍然停留在排查边界内？

## 8. Return
- Return To: Shaun
- Return Format:
  1. 结果摘要
  2. 排查正文
  3. 风险 / 缺口 / 还需要的输入
- Completion Signal:
  Subtask Complete: Yes / No

## 9. Failure / Escalation
- If Blocked:
  明确说明卡在哪个输入缺口或证据缺口。
- If Scope Is Unclear:
  直接回传需要确认的问题，不要自行扩任务。
- If Input Is Insufficient:
  列出缺失的具体日志、配置或上下文。
- If Better Alternative Appears:
  可提出，但不得擅自把“排查任务”改成“修复任务”。

## 10. Notes
- Risks:
  该问题可能涉及多条注入链，初次排查未必能一次性锁死。
- Special Notes:
  本轮目标是结构化排查，不追求一步到位修复。
- Next Suggested Step:
  若排查结果成立，再单独起修复任务卡。
```
