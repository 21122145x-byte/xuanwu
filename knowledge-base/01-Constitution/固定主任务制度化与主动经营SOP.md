# 固定主任务制度化与主动经营SOP v1.1

- 日期：2026-04-01
- 状态：v1.1 / 团队级元 SOP
- 上位引用：`/Users/xiaojingbo/.openclaw/workspace/knowledge-base/01-Constitution/团队协作宪法与治理总纲.md`
- 适用范围：团队全体 Agent 的固定主任务

---

## 一、触发点

在联邦式多 Agent 协作结构下，固定主任务若仅停留在 AGENTS 角色描述层，常出现以下问题：
- 主任务能说清，但不能稳定复用
- 环境信息散落在聊天里，重启即丢
- 周期性动作无人准时触发
- 任务状态对粉雪与 BOSS 不可见
- 主任务缺乏持续经营与复盘机制

因此，需要一份团队级 SOP，将固定主任务推进为：
> 可 skill 化、可提醒、可汇报、可复盘、可持续经营的能力单元。

---

## 三、定义

本 SOP 用于回答：

> 一个 Agent 的固定主任务，如何从“角色描述”推进成“可持续经营的能力单元”。

它不是某个具体主任务的执行手册，
而是所有固定主任务共用的制度化流程。

---

## 四、制度化目标

固定主任务完成制度化后，应满足：
- 可复用
- 可维护
- 可升级
- 可汇报
- 可巡检
- 可复盘

若一个主任务仍只依赖临时记忆、聊天上下文或个体经验，则不视为完成制度化。

---

## 五、执行责任与推进位

- 各 Agent 对自己的固定主任务负直接落位责任。
- 粉雪作为默认推进位，负责制度化进度对账、统一巡检、催收与跨 Agent 收口。
- BOSS 保留固定主任务优先级调整权、资源倾斜权与最终裁决权。
- 推进位负责推进，不替代主责 Agent 完成其本应自行落位的部分。

---

## 六、标准流程（8 步）

### 1. 明确主任务定义
必须明确：
- 主任务名称
- 主任务边界
- 默认主责 Agent
- 不负责什么
- 与谁存在高频接口

目标：先把主任务归属、边界与协作关系钉死，避免后续漂移。

### 2. 明确最小交付单元
必须明确：
- 该主任务的标准产物是什么
- 最小可交付单元是什么
- 周期性交付是什么
- 临时响应交付是什么

目标：让主任务不只是一种身份，而是能稳定落成产物。

### 3. 推进主任务 skill 化
必须明确：
- 哪部分已经 skill 化
- 哪部分仍停留在人工经验中
- 哪些 skill 属于主任务核心能力
- 哪些 skill 仍需补全

最小验收标准（示例）：
- 已有明确 skill 名称
- 已有可读说明或 SKILL.md
- 已能说明适用场景、输入、输出与边界

目标：让主任务从“会的人记得怎么做”推进到“团队可复用的稳定能力单元”。

### 4. 同步主任务环境到 `TOOLS.md`
必须同步：
- 环境信息
- 依赖条件
- 本地约束
- 工具路径
- 常用命令
- 数据源 / 文件目录 / 节点信息

格式示例：
```md
### 主任务：市场跟踪
- 数据源：东方财富 / 同花顺 / 巨潮资讯
- 输出目录：knowledge-base/01-Projects/.../
- 定时检查：工作日 08:30 / 15:30
- 约束：非确认数据不得写入正式研报
```

目标：让主任务的运行条件不再散落于聊天和记忆中。

### 5. 建立主任务主动计划机制
必须明确：
- 长期目标
- 当前周期目标
- 当前状态
- 下一步动作
- 风险 / 卡点
- 是否需要升级协同或进入 Spawn

目标：让主任务进入“主动经营”状态，而不是长期停留在被动接单模式。

### 6. 配置主任务提醒机制（`cron` + `heartbeat`）
固定主任务不得只依赖临时记忆或人工想起。

- 对于需要准时触发、周期固定、可独立执行的动作，默认优先使用 `cron`。
- 对于需要轻量巡检、状态确认、失败补漏、知识沉淀与计划提醒的动作，默认通过 `heartbeat` 承担。
- 若固定主任务涉及跨 Agent 正式交接、审计提交或回传，默认同时受《跨Agent协作流转协议.md》约束。
- 每个主任务都必须明确：
  - 哪些动作由 `cron` 触发
  - 哪些状态由 `heartbeat` 检查
  - 失败后如何升级
  - 何时同步粉雪与 BOSS

### 7. 向粉雪与 BOSS 做周期汇报与对账
必须同步：
- 当前在推进什么
- 已完成什么
- 下一步做什么
- 卡在哪
- 是否需要支持 / 升级

目标：让主任务的存在、状态和变化对协同系统可见。

### 8. 周级 / 月级巡检并更新 `skill` / `TOOLS.md` / `SOP`
- 每周末，由粉雪对全队固定主任务做一次周级巡检，重点检查：
  - skill 是否过时
  - `TOOLS.md` 是否缺失关键环境信息
  - `cron` / `heartbeat` 是否失效
  - 汇报是否漏发
- 每月底，由粉雪做一次月级巡检，重点判断：
  - 主任务是否仍有效
  - 是否需要降频
  - 是否需要合并
  - 是否应从固定主任务退回临时任务

目标：防止主任务制度化后长期失修或无限堆积。

## 七、主任务提醒与维护机制

### 1. `cron` 负责“该开始了”
每个固定主任务都应回答：
- 哪些动作必须定时触发？
- 频率是什么？
- 触发后产出什么？
- 失败后怎么处理？
- 是否需要主动推送给粉雪 / BOSS / 飞书群？

### 2. `heartbeat` 负责“有没有漂”
每个固定主任务都应回答：
- `heartbeat` 要检查什么？
- 多久检查一次？
- 检查到什么状态要提醒？
- 检查到什么状态只记日志不打扰？
- 连续失败多少次要升级？

### 3. 小母句
- `cron` 保证主任务不会忘。
- `heartbeat` 保证主任务不会漂。

---

## 八、主任务提醒机制分级

- **高周期性主任务** → `cron` 为主，`heartbeat` 为辅
- **高不确定性主任务** → `heartbeat` 为主，`cron` 为辅
- **高风险主任务** → `cron` + `heartbeat` 同时上
- **低频主任务** → 可只保留 `heartbeat` 巡检，不必强上 `cron`

目标：避免制度机械化，按任务性质配置提醒强度。

---

## 九、异常升级链

- 连续漏跑 / 漏汇报达到预设阈值 → 默认先报粉雪。
- 高风险、高影响或已影响正式输出的失败 → 直接报 BOSS，并同步粉雪。
- 轻微一次性异常 → 先记日志，由 `heartbeat` 跟进，不默认升级打扰。
- 结构性失效（如 skill 失真、提醒机制长期失效、主任务边界长期混乱） → 升级为规则 / SOP / skill 修订项。

## 十二、边界说明

- 本 SOP 是团队级元 SOP，不替代各 Agent 各自主任务的专属 skill / SOP。
- 主任务的执行细节，仍应在各自 lane 的 skill、SOP 与 TOOLS.md 中细化。
- 若一个主任务尚未完成 skill 化、提醒配置与周期汇报，则视为仅完成“职责声明”，未完成“制度化落位”。

---

## 十三、一句话收口

> 固定主任务不是写在 AGENTS 里就算成立，只有当它能被持续计划、定时提醒、周期汇报、稳定复盘，才算真正进入团队运行系统。
