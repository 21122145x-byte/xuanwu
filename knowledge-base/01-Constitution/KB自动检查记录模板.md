## 前置说明：个人知识库的管理与维护原则

> 知识库不是文件夹，不是有空再整理的草稿堆。
> 它是一个需要持续维护的「可发现、可导航、可追溯」的研究资产。

### 每个 Agent 成员应遵守的底线

**1. 进来就要归档，不要只存在聊天里**
- 任何研究结论、判断、来源，入库时就要想清楚：放进哪个层级（来源 / 专题 / 笔记）？
- 只在聊天里出现的结论 = 知识还在蒸发

**2. 进了知识库的东西，必须能被人找到**
- 有文件就要有入口
- 有专题就要有导航页
- 有来源就要有来源卡
- 进了 Inbox 的临时件，必须写清楚为什么暂时不能归档

**3. 归档要有质量判断，不要把未验证当已验证**
- 来源卡要写清楚「可以直接当结论用」还是「只做参考」
- 研究笔记要标明「已验证事实」还是「推论 / 待审」
- 审计前件不要混进正式归档层

**4. 不要重复归档，也不要让重要研究变成孤岛**
- 一个结论只在一个地方是主引用，其他地方用链接
- 发现孤岛文件是维护者的责任，发现就要补入口或标注

**5. 定期跑维护检查，把知识库当成工作的一部分**
- 建议每个维护周期跑一次：`python3 scripts/kb_maintenance.py --write`
- 看到 orphan 报告不要忽视，要判断：是暂时不用，还是忘了挂入口

### 快速入口（各 Agent 成员应替换为各自 KB 路径）

| 工具 | 说明 |
|------|------|
| 知识库维护 Skill | 各 Agent 应维护自己的 KB 维护 Skill，并持续更新 |
| 维护脚本 | 各 Agent 应建立自己的 KB 维护脚本，建议支持 `--write` 快照输出 |
| 最近维护报告 | 各 Agent 放在自己 workspace 的 `logs/` 下，周期性刷新 |
| KB 主索引 | 各 Agent 放在自己 `knowledge_base/99-Index/INDEX.md`，作为 KB 唯一总入口 |
