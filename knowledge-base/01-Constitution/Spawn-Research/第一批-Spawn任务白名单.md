# 第一批 Spawn 任务白名单 v0.1

## 说明

本清单回答的是：
> **当前哪些任务允许优先接入 Spawn 机制，哪些任务暂不接入，哪些任务需要在评估后才能接入。**

来源依据：
- 第一阶段多轮试跑样本
- Anna / 粉雪 pilot 结论
- 第一轮外部校准结论
- 多轮审计对主战场判断的确认

---

## 适用前提

本清单默认在以下条件下生效：
- Resource Gate 已通过（模型可用、quota 未耗、tool chain 完整）
- 任务边界已由调度者明确定义
- done 的签收标准已由调度者或评估者确认
- 异源执行已获明确授权

不满足上述前提的任务，即使在白名单范围内，也不得直接开跑。

---

## Tier 1｜允许优先接入（可立即 Spawn）

以下类型任务，满足适用前提即可 Spawn，无需额外审批。

---

### 1.1 RCA / 事故调查类

**任务特征：**
- 有明确事故对象或事件
- 需要从多源提取证据
- 结论需要有证据链支撑
- 完成后有明确签收标准

**Spawn 方式：**
- 主任务卡 + 结构化回传
- 优先单层拆分，不超过 3 个子任务
- 执行者必须显式回报 Blocker

**不适用的边界：**
- 事故根因尚不明确、可能需要大幅方向调整的初期调查（先在主会话处理）
- 需要临时引入未在任务卡中定义的新信息源（需重新过 Delegation Gate）

---

### 1.2 验证 / 排查类

**任务特征：**
- 有明确的验证目标（文档/声明/数据/代码）
- 有明确的对照基准
- 结论是「是否通过验证」或「差距在哪里」
- 结果可被客观证据复核

**Spawn 方式：**
- 轻量版子任务卡
- 单层拆分优先，不超过 3 个子任务
- 回传必须包含证据摘要与明确结论

**不适用的边界：**
- 需要多轮交叉验证且每轮可能改变验证方向的复杂排查（先在主会话处理）
- 验证结论可能影响上游核心判断的（需先过 Evaluation Gate）

---

### 1.3 审计类

**任务特征：**
- 有明确的审计范围（SOP / 文档 / 流程 / 系统）
- 有明确的审计标准
- 结论是「合规 / 不合规 / 部分合规 + 具体差距」
- 结果需要有审计轨迹可查

**Spawn 方式：**
- 标准版子任务卡
- 可按审计维度拆分为 2-4 个子任务并行
- 每个子任务必须结构化回传

**不适用的边界：**
- 审计范围尚不清晰、需要审计者自行定义标准的（先在主会话处理）
- 审计结果可能导致重大制度变更的（需高风险任务特别审批路径）

---

### 1.4 结构化调研类

**任务特征：**
- 有明确调研问题和边界
- 多个信息源可并行提取
- 结论是「有来源支撑的明确判断」，不是资料堆砌
- 每个子维度有独立的 done 定义

**Spawn 方式：**
- 标准版子任务卡
- 按信息源或维度拆分为 2-4 个子任务并行
- 最终由调度者或指定评估者统一合并结论

**不适用的边界：**
- 调研结论可能推翻上游已确认方向的（需先在主会话做意图对齐）
- 需要临时扩展调研范围超越原任务边界的（需重新过 Delegation Gate）

---

### 1.5 标准化前处理子任务

**任务特征：**
- 处于固定生产链中的标准化环节
- 输入 / 输出格式已明确
- 任务边界清晰、不涉及跨域判断
- 不依赖最终整合层的创意判断

**Spawn 方式：**
- 轻量版子任务卡
- 优先单层拆分，不超过 3 个子任务
- 可建立短路径快速验收

**不适用的边界：**
- 前处理结果会直接影响最终产出的质量判断的（需评估者参与验收）
- 输入格式本身不稳定、可能随时调整的（先在主会话处理输入标准化）

---

## Tier 2｜需评估后接入（满足条件才可 Spawn）

以下类型任务，需要在过完 Resource Gate 之后，额外过 Delegation Gate 明确任务边界，方可接入。

---

### 2.1 跨域复杂排查

**特征：**
- 排查范围跨越多个知识域
- 每轮排查结论可能影响下一轮方向
- 不适合完全并行处理

**接入条件：**
- 任务边界已由调度者明确定义
- 每轮子任务完成后，由调度者确认是否继续下一轮
- 不允许执行者自行扩大排查范围

---

### 2.2 证据链构建任务

**特征：**
- 需要从多个证据点推演结论
- 证据权重需要人工判断
- 结论存在多解性

**接入条件：**
- 调度者已明确定义「哪些是合法证据来源」
- 评估者已参与 done 定义
- 执行者不得自行引入未在上游确认的证据类型

---

### 2.3 多轮调研汇总结类

**特征：**
- 多个子调研已完成，需要合并成统一结论
- 结论需要对内有对外一致性
- 涉及判断权衡

**接入条件：**
- 各子调研结论均已过 Return Gate
- 调度者已明确合并规则
- 评估者已确认合并后的 done 标准

---

## Tier 3｜暂不接入（当前阶段不推荐 Spawn）

以下类型任务，在候选版阶段不推荐 Spawn。原因并非原则上不能 Spawn，而是当前阶段配套件（评估者校准样例 / 最小字段集）尚未补齐，接入风险较高。

---

### 3.1 创作主链核心判断类

**典型任务：**
- 故事核心创意决策
- 剧本主线走向判断
- 统一叙事风格决策

**暂不接入原因：**
- 这类任务高度依赖统一创意人格和整体上下文
- spawn 引入的上下文断裂会直接导致判断失真
- 当前阶段评估者样例库尚未建立，难以校准这类判断

---

### 3.2 高耦合最终整合写作类

**典型任务：**
- 需要将多个子模块整合为统一最终产出的写作
- 风格 / 语调 / 叙事一致性要求高的长文
- 需要在整合过程中持续保持全局上下文的任务

**暂不接入原因：**
- 整合层对全局一致性要求极高
- 多 agent 各自产出会带来不可忽视的风格断裂
- 这类任务在当前阶段更适合在主会话统一处理

---

### 3.3 需要统一审美判断的任务

**典型任务：**
- 视觉风格决策
- 内容调性判断
- 受众匹配度的主观评估

**暂不接入原因：**
- 审美判断当前阶段无法被结构化验证
- 评估者缺乏足够的校准样例
- 容易产生「看起来完成了，但实际不符合预期」的隐性失败

---

### 3.4 依赖单一上下文连续性的任务

**典型任务：**
- 上下文链断裂会导致判断质量大幅下降的任务
- 需要跨长文本维持引用一致性的任务

**暂不接入原因：**
- 当前 spawn 的上下文传递机制尚未积累足够样例
- 上下文断裂是高风险失败模式
- 这类任务应等上下文传递机制成熟后再评估

---

## 快速判断清单

拿到一个任务时，按以下顺序快速判断：

1. **Resource Gate 通过？** → 否 → 不得 Spawn
2. **任务边界已由调度者明确定义？** → 否 → 先在主会话处理
3. **属于 Tier 1 类型？** → 是 → 可立即 Spawn
4. **属于 Tier 2 类型？** → 是 → 过 Delegation Gate 后 Spawn
5. **属于 Tier 3 类型？** → 是 → 当前阶段不推荐 Spawn，先在主会话处理

## 最后一刀

> **这份白名单的最大价值，不是「列出能跑的任务」，而是「让调度者在任务入口就做好分流」：能跑的稳住，不能跑的别硬放，条件满足才接入。**
