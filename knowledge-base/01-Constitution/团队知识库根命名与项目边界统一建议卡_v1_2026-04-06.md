# 团队知识库根命名与项目边界统一建议卡 v1

- 日期：2026-04-06
- 状态：v1 / 建议卡
- 适用范围：团队全体 Agent workspace
- 上位关系：挂载于《团队协作宪法与治理总纲 v1》的知识治理条目之下

---

## 一、问题背景

当前团队内已经暴露出两类结构性问题：

### 1. 本地知识库根命名不统一
部分 agent workspace 同时存在：
- `knowledge-base`
- `knowledge_base`

这会导致：
- skill / 脚本默认目标不稳定
- agent 容易把材料放错根
- 周期性维护、回挂、健康检查出现假阴性
- 团队难以形成统一维护口径

同时需要注意：
- 团队总知识库当前稳定根名是 `knowledge-base`
- 本建议卡讨论的“统一”，主要针对 **各 agent 本地 workspace 内的知识库根**，而不是要求反向重命名团队总知识库

### 2. 项目文件夹与知识库边界不清
部分 agent 的正式项目产物、过程产物、可复用知识、工作中间件混在项目目录或临时目录中，导致：
- 项目目录越来越厚
- 可复用知识被埋在项目上下文里
- agent 自己的知识库长期空转或失真
- 周期性知识维护无法真正建立

---

## 二、统一建议

## 1. 双层命名口径

### 团队层
团队共享总知识库固定保留：

> `knowledge-base`

即：
- `/Users/xiaojingbo/.openclaw/workspace/knowledge-base/`

该根名既然已经稳定运行，就不应为了表面统一而反向制造迁移成本与引用断裂。

### agent 本地层
各 agent 自己 workspace 内的正式知识库根，建议统一使用：

> `knowledge_base`

### 统一理由
- 与当前多个 agent workspace 的实际状态更一致
- 对脚本、周期维护、跨 workspace 规则更容易统一匹配
- 减少 `knowledge-base` / `knowledge_base` 在本地 workspace 内并存造成的歧义
- 保留“团队总知识库”和“agent 本地知识库”两层之间的可辨识差异

### 统一规则
- 团队共享总知识库继续保留 `knowledge-base`
- 每个 agent workspace 只保留一个正式本地知识库根
- 若历史上存在 `knowledge-base` 与 `knowledge_base` 并存，后续逐步收敛到 `knowledge_base`
- 迁移期间允许临时兼容，但不得长期双轨并存而无说明

## 三、知识库 vs 项目文件夹边界

## 1. 知识库负责什么
知识库用于沉淀：
- 可复用判断
- 方法卡 / 流程卡
- 稳定来源卡
- 阶段研究结论
- 跨项目可继续使用的知识
- 周期维护、导航、回挂后的正式知识对象

## 2. 项目文件夹负责什么
项目文件夹用于承载：
- 正式交付文件
- 项目过程产物
- 当前项目上下文
- 仅对该项目成立的中间结果
- 面向该项目的最终版本文件

## 3. 统一边界口径
> 正式交付进项目文件夹；可复用知识回到个人知识库。

这条口径适用于：
- 富贵的市场产物与报告链
- 旺财的长文 / 表达 / 创意产物
- Curie / Anna / 玄武的研究、治理与机制文档

---

## 四、当前三类典型问题判断

## 1. 旺财
当前问题：
- 实际使用的本地知识库根疑似偏向 `knowledge-base`
- 但 workspace 内存在 `knowledge-base` / `knowledge_base` 并存风险

建议：
- 在旺财本地 workspace 内，明确唯一正式根为 `knowledge_base`
- 团队共享总知识库的 `knowledge-base` 不受此条影响
- 其余命名视为待迁移或兼容残留
- 在迁移完成前，应在维护说明中显式标注正式本地根

## 2. 富贵
当前问题：
- 主要产物更集中在团队 workspace 的项目文件夹中
- `knowledge_base` 目前未真正承担知识沉淀功能

建议：
- 不再让项目文件夹充当知识库替代品
- 将可复用经验、判断、方法、稳定来源回沉到个人知识库
- 项目文件夹保留正式交付与项目上下文

## 3. 玄武
当前问题：
- 已出现知识区雏形，但导航中心与正式维护面尚不完整

建议：
- 优先补齐导航中心与知识分层
- 使其尽快进入可维护、可周期检查的正式状态

---

## 五、统一后的执行口径

后续团队知识治理默认采用以下规则：

1. 团队共享总知识库继续使用 `knowledge-base`
2. 各 agent workspace 的正式本地知识库根统一为 `knowledge_base`
3. 项目文件夹不替代知识库
4. docs / raw / inbox 只作为中间整理区，不默认等于正式知识层
5. 只要材料具有长期复用价值，就应回到知识库
6. 周期性维护、回挂、导航修补、健康检查均以正式目标知识库根为目标
7. 迁移与统一过程优先在 observe / suggest 下确认，再进入 maintain

---

## 六、与 wiki-maintenance 的关系

本建议卡为 `wiki-maintenance + periodic-maintenance profile` 提供团队级结构前提：
- 若 agent 本地正式知识库根不统一，skill 的 target 识别会长期不稳
- 若项目文件夹与知识库边界不清，周期维护会被误导为“整理项目目录”而不是“维护知识库”

因此：
> 本建议卡不是 skill 的替代品，而是 skill 能稳定跨 workspace 落地的前置统一条件。

---

## 七、一句话收口

> 团队要真正建立稳定的知识治理，不只是让 agent 会维护知识库，还要先区分清楚“团队总知识库叫什么”“agent 本地知识库叫什么”，并守住“项目文件夹不替代知识库”这条底层边界。
