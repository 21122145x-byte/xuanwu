# Comic-Studio-Ai 深度拆解提纲-v0.1

## 一、拆解目的

本提纲用于深度拆解 `Comic-Studio-Ai`，
判断它是否适合作为 SaltFire 路线 C 的参考原型。

重点不是看它“能不能生成漫画”，
而是看它：

1. Agent 分工是否真实成立
2. 理解层与生成层是否真正解耦
3. 哪些结构适合 SaltFire 直接借鉴
4. 哪些部分对 SaltFire 不够，必须重做

---

## 二、核心拆解问题

### 2.1 Agent 分工是否是真分工

要看：
- 各 Agent 输入输出是否不同
- 是否存在明确职责边界
- 交接是否有结构，而不是只是换名字的 prompt 串联

重点问题：
- Researcher Agent 到底做了什么
- Script Director 是否真的负责节奏控制
- Panel Artist 是否只是生图工具包装
- Bubble Agent 是否只做排版，还是参与叙事组织

---

### 2.2 理解层是否真实存在

这是最关键的一层。

要看：
- Researcher Agent 是否真的做资料理解 / 背景提取 / 术语整理
- 还是只是在外部搜索后喂信息
- 是否能形成结构化理解结果，而不是一段摘要

重点问题：
- 它有没有“理解世界”的能力，还是只有“提取信息”的能力
- 这种理解层是否足够支撑 SaltFire 这种高术语密度项目

---

### 2.3 Director 层是否能压结构

要看：
- Script Director 是否能把理解结果压成切口
- 是否有阅读结构 / 分镜节奏设计
- 是否存在开头钩子、中段推进、尾部留钩
- 还是仅仅输出一串 panel 描述

重点问题：
- 它到底是“导演”，还是“镜头列表生成器”

---

### 2.4 Visual 层是否有真正控制力

要看：
- 是否有风格锁定机制
- 是否有角色一致性方案
- 是否可插入 LoRA / ControlNet / reference image
- 是否支持中途返修与局部调整

重点问题：
- 这条线更像“demo”，还是更像可持续工作流

---

### 2.5 人类在系统中还剩多少决策权

要看：
- 中途是否可人工修改切口
- 是否可人工修改结构与视觉路线
- 是否可以人工筛选候选结果
- 还是只能“提交 → 等成品”

重点问题：
- 这套系统适不适合 SaltFire 这种必须保留强判断力的项目

---

## 三、拆解维度

建议按以下 6 个维度打分或判断：

1. **Agent 架构清晰度**
2. **理解层真实性**
3. **结构控制能力**
4. **视觉控制能力**
5. **人工介入友好度**
6. **对 SaltFire 的迁移适配性**

可使用：
- 高
- 中
- 低

或：
- 可直接借用
- 可改造借用
- 不建议借用

---

## 四、输出要求

深拆 `Comic-Studio-Ai` 后，至少产出以下内容：

### 4.1 一页判断摘要
回答：
- 这项目值不值得作为路线 C 的第一参考原型
- 值得借的是什么
- 不值得借的是什么

### 4.2 结构拆解表
建议字段：
- 模块 / Agent 名称
- 输入
- 输出
- 职责
- 对 SaltFire 有什么用
- 问题 / 风险

### 4.3 SaltFire 迁移建议
明确写：
- 哪些结构可直接拿来
- 哪些必须改造成 C-lite
- 哪些对 SaltFire 不适用

---

## 五、最终要回答的核心结论

拆完以后，必须能回答这句话：

> **Comic-Studio-Ai 到底是一个值得借鉴的多 Agent 原型，还是一个不够深的生成 demo？**

如果值得借鉴，
还要继续回答：

> **它最适合进入 SaltFire 路线 C 的哪一层：理解层、结构层，还是生成层？**

---

## 六、一句话收口

这份拆解提纲的目标，不是理解 `Comic-Studio-Ai` 本身，
而是借它来判断：

**SaltFire 的路线 C，究竟该怎么搭，哪些可以借，哪些必须自己重建。**
