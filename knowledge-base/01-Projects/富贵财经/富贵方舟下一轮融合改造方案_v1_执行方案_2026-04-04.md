- 目标：把“下一轮融合改造方案 v1”拆成可执行、可验证、可逐单元收口的最小执行方案。

---

## 一、执行总原则

### 1. 当前不是重写产品，而是增量改造当前版本
所以执行方式必须是：
- 先补字段
- 再补 render
- 再补留痕
- 再补池层次
- 最后出半自动样稿

不允许一上来大改所有模块。

### 2. 每一步都必须是最小单元
每个单元都必须满足：
- 有明确输入
- 有明确输出
- 有明确完成标准
- 能独立验证

### 3. 每一步都要能停
任何单元做完后，即使后续未继续，也要留下：
- 可读代码 / 文档
- 可验证产物
- 当前状态说明

### 4. 当前阶段不做的事
- 不重写打分体系
- 不扩量化因子
- 不接午盘主链
- 不接长图 / 视觉主链
- 不并入完整版主 writer
- 不把本轮执行说成“已正式主链接入”

---

## 二、执行路径总览

本轮执行拆成 6 个最小单元：

1. **E1：字段口径补齐**
2. **E2：render 可见字段接线**
3. **E3：移出留痕接线**
4. **E4：研究附层最小接线**
5. **E5：核心长期池 / 动态短期池口径落地**
6. **E6：半自动完整版样稿落地与结构检查**

顺序不得倒。

---

## 三、最小单元执行单

## E1：字段口径补齐

### 目标
先把本轮融合改造需要的新字段和旧字段口径补齐，不先碰全文渲染。

### 必做项
1. 明确并补齐以下字段定义：
   - `change_type`
   - `delta_note`
   - `thesis_note`
   - `risk_note`
   - `watch_point`
   - `validate_trigger`
   - `falsify_trigger`
   - `pool_type`（如需要承接长期池 / 短期池）
   - `removal_type`（承接移出类别）
2. 明确 `track_days` 口径：
   - 交易日累计
   - 新入池当天 = 第 1 天
3. 明确 `change_type` 枚举：
   - `新入池`
   - `连续留池`
   - `升级`
   - `降级`
   - `移出`
4. 明确 `validation_state` 枚举：
   - `待验证`
   - `已验证`
   - `已证伪`

### 输入
- 当前 `models.py`
- 当前 `schema.py`
- 当前主方案 v1

### 输出
- 一版收紧后的字段定义
- 一版最新 schema / model 口径说明

### 完成标准
- 相关字段均有明确含义
- 不再存在“写在方案里但代码口径没有承接”的悬空字段

### 验证方式
- 核对 schema / model 是否覆盖全部新增字段
- 检查是否存在重复概念或互相覆盖的字段命名

---

## E2：render 可见字段接线

### 目标
把真正需要用户看到的字段接进 render，不先补研究长说明。

### 必做项
1. 在 render 中新增展示：
   - `track_days`
   - `validation_state`
   - `change_type`
   - `delta_note`
   - `watch_point`
2. 明确展示顺序：
   - 为什么在这里
   - 已跟踪第几天
   - 当前验证状态
   - 当前变化类型
   - 较昨日变化
   - 关键观测点
3. 保证 `change_type` 与 `validation_state` 分开展示，不能混写成一句模糊话

### 输入
- E1 后的字段定义
- 当前 `renderer.py`

### 输出
- 一版新的 render 文本格式
- 一次本地渲染产物

### 完成标准
- 用户能一眼看懂：这只票盯了多久、今天怎么变、当前是否已验证

### 验证方式
- 手动触发一次方舟输出
- 检查 render 文件是否已显式出现新增字段内容

---

## E3：移出留痕接线

### 目标
让“谁离船、为什么离船”进入用户可见层，不再只在系统内部消失。

### 必做项
1. 在 daily / render 中增加“移出摘要”区
2. 每个移出对象至少展示：
   - 标的名
   - `change_type=移出`
   - `exit_reason`
   - `removal_type`
3. 明确 `removal_type` 最小枚举：
   - `证伪移出`
   - `替换移出`
   - `周期到期移出`

### 输入
- E1 的字段定义
- 当前 state / daily 生成逻辑

### 输出
- 一版支持移出留痕的 daily / render 产物

### 完成标准
- 被移出对象不会在当日输出中无痕消失

### 验证方式
- 构造至少一个带移出对象的最小样例
- 检查 render / daily 是否正确展示移出摘要

---

## E4：研究附层最小接线

### 目标
只补最小研究附层，不恢复老版本重日报。

### 必做项
1. 给重点对象接入以下最小研究字段：
   - `thesis_note`
   - `risk_note`
   - `watch_point`
2. 写死研究附层触发条件：
   - 上船位
   - 远望位中最核心 1~2 个
   - 升级 / 降级 / 移出的重点对象
   - 发生验证 / 证伪的关键对象
3. 写死研究附层上限：
   - 单标的不超过 4 行 / 120 字

### 输入
- E1 字段定义
- E2 新 render

### 输出
- 一版带最小研究附层的 render 产物

### 完成标准
- 能明显看出“为什么盯它”，但没有回潮成重日报

### 验证方式
- 检查重点标的是否补出研究说明
- 检查是否有边缘对象被不必要展开

---

## E5：核心长期池 / 动态短期池口径落地

### 目标
把“养船感”补回来，但先从口径和最小字段落地，不先大改评分逻辑。

### 必做项
1. 在状态层引入最小池子分类：
   - `核心长期池`
   - `动态短期池`
   - `移出池`
2. 明确什么对象优先进入核心长期池：
   - 连续跟踪
   - 具备长期观察价值
   - 不因单日涨跌就轻易移出
3. render 中让用户可感知对象属于哪类池

### 输入
- E1 字段体系
- 当前 state_store / runner 输出逻辑

### 输出
- 一版带池子分类的状态输出
- 一版带池子分类提示的 render 产物

### 完成标准
- 用户能感知这不是当天临时点票，而是有长期盯盘对象层次

### 验证方式
- 手动触发一次方舟
- 检查 render / daily 中是否能分辨长期池与短期池

---

## E6：半自动完整版样稿落地与结构检查

### 目标
把融合后的方舟真实挂入完整版正文一次，并检查是否自然。

### 必做项
1. 生成一版新的方舟正文片段
2. 以半自动方式挂入完整版手工稿一次
3. 检查三件事：
   - 是否与结构判断重复
   - 是否与全天节奏复盘打架
   - 是否仍然保持轻，不变回重日报

### 输入
- E2 ~ E5 的最新 render 产物
- 完整版手工稿 / 半自动稿

### 输出
- 一版“带融合后方舟板块”的完整版样稿
- 一版结构检查结论

### 完成标准
- 方舟板块自然嵌入完整版
- 有研究温度，但不过重
- 可交下一轮审阅

### 验证方式
- 人工通读整版样稿
- 对照主方案与补充纪律逐条检查

---

## 四、建议执行顺序（更细一级）

### 第 1 轮：先只做结构层
- E1 字段口径补齐
- E2 render 可见字段接线

目标：
> 先让“变化 / 验证 / 跟踪”真的被用户看见。

### 第 2 轮：再补追踪感
- E3 移出留痕接线
- E4 研究附层最小接线

目标：
> 先让“养船感”和“为什么盯它”长出来。

### 第 3 轮：最后补对象层次
- E5 核心长期池 / 动态短期池落地
- E6 半自动完整版样稿落地

目标：
> 先让它像一个真的被持续照看的系统，再验证它进正文是否自然。

---

## 五、每轮收口口径

### 第一轮收口
> 已完成用户可见字段接线，方舟从“跑表”推进到“能看见状态变化”。

### 第二轮收口
> 已完成移出留痕和最小研究附层，方舟开始具备“养船感”和解释力。

### 第三轮收口
> 已完成池层次与半自动正文样稿验证，方舟进入“可继续审正文板块自然度”的阶段。

---

## 六、当前一句话结论

> 富贵方舟下一轮融合改造方案 v1 的执行，不应大步并改，而应按 6 个最小单元推进：先补字段，再补 render，再补移出留痕，再补研究附层，再补池层次，最后做半自动完整版样稿验证。只有这样，当前版本才有机会在不返工、不回潮成重日报的前提下，真正从“会跑的系统”长成“像真的在养船的系统”。
