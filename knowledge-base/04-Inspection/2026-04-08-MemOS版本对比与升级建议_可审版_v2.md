## 三、版本差异判断（重收口后）

### 当前版本对比
| 项目 | 当前运行 | npm 最新稳定 | GitHub monorepo |
|------|----------|--------------|-----------------|
| 版本 | `1.0.3` | `1.0.8` | `v2.0.12` |
| 状态 | 当前真实运行中 | 可升级目标 | 研究参考，不直接等于 npm 稳定版 |

### 当前最相关的工程差异
#### 1. context engine 接法
- `v1.0.3`：`before_agent_start` hook
- `v1.0.8`：`registerContextEngine` + `assemble()`

当前判断：
> **`registerContextEngine` 是更稳的官方接线方式。**

#### 2. native binding 覆盖
- `v1.0.8` 对 native binding 的预编译覆盖更全

当前判断：
> **升级后可降低新系统/新环境下手动 rebuild 的概率。**

#### 3. 去重与上下文装配
- `v1.0.8` 在 context engine dedup 与 assemble 层更完整

当前判断：
> **升级存在工程收益，但不自动等于 recall 污染问题自然消失。**

## 五、升级风险重评估

### 1. 不升级的风险
- 后续 OpenClaw 若继续收紧 context engine / injection 路径，`before_agent_start` 旧接法存在继续老化风险
- 旧 native binding 长期维护收益更差
- context 装配与 dedup 层将继续停留在旧实现

### 2. 升级的风险
- 需要重启 gateway
- 需要验证 memory 数据是否正常可读
- 需要验证升级后 auto-recall 仍可触发
- 需要验证升级后是否引入新的 recall / inject 差异

### 3. 当前风险结论
> **升级风险可控，但不是零成本动作；只能按受控升级做，不能按“随手升一下”处理。**

## 六、当前建议升级路径

### 当前建议
**条件通过，可进入受控升级。**

建议动作顺序：
1. 使用稳定版升级路径，将插件从 `1.0.3` 升到 `1.0.8`
2. 重启 gateway
3. 做升级后最小验证
4. 再决定是否维持为默认 memory slot

### 当前不建议
- 直接跳到 monorepo `v2.0.12` 自行追主线
- 在未做验证前把升级写成“修复完成”
- 把升级本身当成 recall 治理问题的结论性修法

## 七、升级后必做验证

### 1. 记忆完整性验证（必做）
至少核：
- 现有 `memos.db` 是否可正常读取
- 升级后数据结构是否可继续访问
- 无明显 schema 读写异常

### 2. auto-recall 最小行为验证（必做）
至少核：
- 升级后是否仍有 `auto-recall` 触发日志
- 是否仍能正确返回 `prependContext` / 对应新 context engine 输出
- 无直接报错 / strip 异常

### 3. recall 边界复验（必做）
至少核：
- cross-agent 幽灵上下文是否仍出现
- 过期 session / task 误召回是否仍出现
- 命中后是否仍错误注入

### 4. slot 绑定与配置复验（必做）
至少核：
- `plugins.slots.memory` 未漂移
- `entries.enabled` 未失效
- `memorySearch.enabled=false` 未被破坏
