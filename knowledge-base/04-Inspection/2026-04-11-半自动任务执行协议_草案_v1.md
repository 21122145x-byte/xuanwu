## 一、协议目标

本协议面向以下类型任务：
- 后台执行时间较长
- 由 worker/subagent 先产生产物
- 由主 Agent 进行验收与收口
- 产物需要进入下一环（如 Anna 审计、外部回传、项目归档）

目标不是让任务“自动跑就行”，而是保证：

> **半自动任务即使在前台消息干扰、会话切换、announce 跳过、worker 完成后无人接手等情况下，也能稳定完成阶段流转与闭环。**

---

## 二、当前已暴露的机制缺口

基于 Curie memory 主线任务的实际表现，当前半自动机制至少存在以下缺口：

### 缺口 1：产物完成 ≠ 已被接住
worker 可能已完成主文，但结果没有稳定进入下一环，外部体感会误以为“卡住了”。

### 缺口 2：heartbeat 只能保活，不能保闭环
heartbeat 能发现链路还活着，却不能强制推动：
- review-ready → 送审
- 送审 → 承接
- 接受 → 回传

### 缺口 3：前台对话与后台流转未彻底解耦
用户在 Discord 中途发言，可能会：
- 抢占当前会话控制权
- 打断原有 announce / 回传链
- 导致后台明明已推进，前台却像没动

### 缺口 4：worker 职责与工具边界不匹配
某些 worker 不具备跨 session / 跨 Agent 发送结果的工具能力，却被设计成承担“交下一环”的职责，造成机制性断点。

---

## 三、核心原则

### 原则 1：后台执行不以“有人记得接”为前提
每一环都必须显式记录状态，而不是靠记忆、默契或聊天上下文维持。

### 原则 2：worker 只负责产出，不负责跨环交接
worker 的职责是：
- 写稿
- 修订
- 打包结果

worker 默认不负责：
- 送审
- 对外 announce
- 处理前台确认消息

### 原则 3：主 Agent 对阶段流转负责
主 Agent 必须承担：
- 验收 worker 产物
- 做 accept / revise 判断
- 送下一环
- 决定是否对外回传

### 原则 4：审计位只处理审计，不兜底状态识别
Anna 负责审计与裁决，但不应承担“自己去猜当前任务到了哪一环”的成本。

### 原则 5：heartbeat 负责发现停滞，不负责代替业务判断
heartbeat 可以提醒、记录、升级，但不替代主链做 accept / revise / send-to-anna 判断。

---

## 四、角色边界

### 1. Worker / Subagent
负责：
- 产出主文与 supporting artifacts
- 明确写出当前结果包位置
- 把状态推进到 `review-ready` 或 `revision-needed`

不负责：
- 直接送 Anna
- 对外 announce
- 响应前台新消息

### 2. 主 Agent
负责：
- 接住 worker 结果
- 检查质量
- 决定送审、返修、回传
- 维护主链状态

### 3. 审计 Agent（如 Anna）
负责：
- 审计承接
- 放行 / 打回
- 输出审计意见

### 4. Heartbeat 子链
负责：
- 盯停滞
- 提醒与升级
- 暴露“活着但没流转”的节点

---

## 五、标准阶段门

半自动任务必须显式处于以下状态之一：

- `drafting`
- `revision-needed`
- `review-ready`
- `sent-to-audit`
- `audit-in-progress`
- `accepted`
- `returned`
- `announced`
- `closed`

禁止使用“差不多好了”“已经在做了”这类非状态表达充当流转依据。

---

## 六、每次状态更新的强制字段

每次阶段流转必须至少写出：

- `status`
- `owner`
- `next_owner`
- `next_action`
- `updated_at`
- `artifact_where`
- `known_issues`

建议模板：

```md
status: review-ready
owner: curie
next_owner: anna
next_action: audit
updated_at: 2026-04-11 01:30 Asia/Shanghai
artifact_where:
- /path/mainline-v0.2.md
- /path/supporting-card-1.md
known_issues:
- section 2 still needs audit on scope boundary
```

---

## 七、标准流转

### 流转 1：worker 完成
worker 完成后必须：
1. 产出主文与 supporting artifacts
2. 明确写 `status=review-ready`
3. 明确 `next_owner=主 Agent`
4. 停止继续对外扩散

### 流转 2：主 Agent 验收
主 Agent 在 `review-ready` 上只允许两种正式动作：
- `accept` → 推到 `sent-to-audit`
- `revise` → 推到 `revision-needed`

### 流转 3：审计承接
审计位接到任务后必须显式进入：
- `audit-in-progress`
- 或 `returned`

### 流转 4：对外回传
只有当状态达到：
- `accepted`
- 或需要人工知情的 `returned`

才进入 `announced`。

默认不允许 worker 自行对外公告。

---

## 八、Heartbeat 升级规则

heartbeat 不只检查“活着没”，还检查“流转了没”。

### 规则 1：review-ready 停滞
- `review-ready > 20 分钟` 未变化 → 提醒主 Agent

### 规则 2：sent-to-audit 停滞
- `sent-to-audit > 30 分钟` 未承接 → 提醒审计位
- 再超 30 分钟 → 升级粉雪

### 规则 3：accepted 未回传
- `accepted > 15 分钟` 未进入 `announced` → 标记为回传链异常

### 规则 4：returned 未重分配
- `returned > 20 分钟` 未进入新一轮 `revision-needed` 或 `review-ready` → 提醒主 Agent

---

## 九、前台消息与后台执行的解耦规则

### 原则
后台主链以阶段状态为准，不以前台 Discord 对话节奏为准。

### 规则
1. 若后台已进入更后阶段，前台旧问题不得把主链拉回早期阶段
2. 前台确认只允许发生在显式的 `awaiting-human-confirmation` 节点
3. 用户新消息到来，不自动推翻后台已完成的状态流转
4. announce 被跳过或打断，不等于主链未执行；必须以状态门判断真实进度

## 十、最小修复动作（当前建议）

### 动作 1：统一阶段门模板
所有半自动任务统一使用同一套状态字段模板，避免各自用不同表达。

### 动作 2：送审职责固定由主 Agent 承担
不再让工具受限的 worker 负责跨环 handoff。

### 动作 3：heartbeat 增加“停滞升级”而不只“保活记录”
让“看起来活着但实际停住”能被及时暴露。

### 动作 4：前后台解耦制度化
当 webUI 已显示后台推进时，前台旧问题不应再反向打断主链。

---

## 十一、适用判断

当任务满足以下 3 条以上时，应优先使用本协议：
- 持续时长 > 20 分钟
- 存在 worker / subagent
- 需要主 Agent 验收
- 需要送审或跨 Agent handoff
- 需要对外回传 / 公告
- 用户可能在执行期间继续发消息

---

## 十二、一句话收口

> **半自动任务的稳定，不在于“有人还在线”，而在于“每一环都有阶段门、责任位、下一动作，并且停滞会被升级”。**
