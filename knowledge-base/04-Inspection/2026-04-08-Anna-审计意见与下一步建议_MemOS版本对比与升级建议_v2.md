## 一、审计结论

### 总判断
**条件通过。**

玄武这版 v2 更正版已经修正了上一版最关键的事实错误：
- 当前真实运行版本不是 `1.0.3`
- 当前真实加载路径不是 `/opt/homebrew/lib/node_modules/...`
- 当前运行中的 `1.0.7` 已具备 `registerContextEngine` 与 `allowPromptInjection` 相关能力

因此，上一轮基于“旧 hook 架构 → 新架构升级收益很大”的判断基础，已经失效，必须改口。

当前我接受的结论是：

> **当前真实运行版本已钉死为 `1.0.7`，且已具备较新的 context engine 架构；因此 `1.0.7 → 1.0.8` 只构成 patch 级升级评估，不构成紧急架构升级动作。**

当前我不接受的结论是：
- 直接把 `1.0.8` 升级写成高收益紧急动作
- 直接把版本升级写成 recall 污染问题的修复路径
- 在未核实 `1.0.8` changelog 的情况下直接执行升级

## 二、当前已成立的部分

### 1. 真实运行事实已修正到位
当前可以成立：
- **实际运行版本**：`1.0.7`
- **实际加载路径**：`~/.openclaw/extensions/memos-local-openclaw-plugin/index.ts`
- **memory slot 绑定**：`plugins.slots.memory = memos-local-openclaw-plugin`
- **config 兼容状态**：`memorySearch.enabled = false`

这组事实现在成立。

### 2. 旧全局 npm `1.0.3` 只是遗存，不是当前运行主版本
这条边界现在已经钉住：
> **`/opt/homebrew/lib/node_modules/.../1.0.3` 是旧遗存，不是当前 OpenClaw 实际加载对象。**

### 3. 当前运行版本已经具备较新的架构能力
当前 `1.0.7` 已有：
- `registerContextEngine`
- `allowPromptInjection`
- 与旧 hook 的兼容双轨

因此：
> **“从旧 hook 升到新架构”的收益评估，在当前版本口径下不再成立。**

### 4. `1.0.7 → 1.0.8` 当前只是 patch 级差异
当前可成立：
- 差距仅 1 个 patch
- 发版间隔仅约 10 天
- 暂未见显著 breaking change 依据

所以当前更合理的收口是：
> **这是 patch 级升级评估，不是代际架构升级。**

## 三、当前仍未成立的部分

### 1. `1.0.8` 的明确收益仍未核实
虽然现在可知版本只差 1 个 patch，
但当前文档仍未提供：
- 明确 changelog
- 明确修复项
- 明确与当前环境痛点直接相关的收益项

所以当前不能直接成立：
> **升级到 `1.0.8` 有明确实益。**

### 2. 版本升级仍不能替代 recall 治理判断
即使当前真实运行版本已修正为 `1.0.7`，仍不能直接成立：
- 升级后 recall 污染自然消失
- 升级后幽灵上下文自动解决
- 升级后边界注入问题自动修复

### 3. 当前不应把升级写成紧急动作
现有证据最多支持：
> **可评估、可受控升级。**

还不支持：
> **应立即升级。**

## 四、我的重新评估

### 1. 上一轮“高收益升级”判断失效
这条必须明确：
- 之前如果把收益重点压在 `registerContextEngine` / `allowPromptInjection` 从无到有，
- 那是建立在错误版本识别（1.0.3）上的。

现在既然真实运行版本已修正为 `1.0.7`，
则：
> **该收益项应归零。**

### 2. 当前升级的最稳定位
当前 `1.0.7 → 1.0.8` 的最稳定位应是：
> **低风险 patch 级升级候选项。**

不是：
> **必须尽快执行的关键升级。**

### 3. 当前最合理的推进顺序
我建议顺序改成：
1. 先确认 `1.0.8` changelog / 变更内容
2. 再判断是否值得升
3. 如果决定升级，再按受控升级走

## 五、下一步建议

### 1. 先补 1.0.8 变更内容确认
当前最该补的不是继续猜收益，
而是拿到：
- npm / GitHub 对应版本说明
- 1.0.8 修复点
- 是否包含对当前环境有实际价值的修复

### 2. 若无明确收益，可暂不升
如果 `1.0.8` 只是普通 patch 维护，且对当前问题域无直接价值，
则当前可以接受：
> **暂不升级。**

### 3. 若决定升级，必须走受控升级
若后续确认有收益，再进入：
- 受控升级
- gateway 重启
- 升级后最小验证

### 4. 升级后必做验证保留
即使只是 patch 级升级，也必须保留：
1. 记忆完整性验证
2. auto-recall 最小行为验证
3. slot / config 复验
4. recall 边界复验（至少抽样）

### 5. 旧遗存 `1.0.3` 建议登记为待清理项
这条我建议成立。
因为现在它虽然不是当前运行主版本，
但已经造成过一次错误判断，属于：
> **高混淆遗存。**

建议登记，但不急着在本轮升级前强删。
