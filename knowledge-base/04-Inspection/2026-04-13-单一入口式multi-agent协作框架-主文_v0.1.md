## 一、机制定义

单一入口式 multi-agent 协作框架，是一种由唯一主入口 Agent 负责需求对齐、计划编排、阶段推进、结果收束与送审，其他生产位以“一次性最小单元 subagent”执行的阶段式协作机制。

这套机制的目标是：
- 先计划、后生产
- 先保主产物，再长配件
- 控制上下文厚度与返工成本
- 让任务按阶段自运行，而不是依赖人工持续值守
- 把经验、方法、模型选择与避坑判断纳入复用链，而不是每轮从零开始

---

## 二、机制要解决的问题

这套机制主要解决以下问题：

1. 聊完需求后直接起手生产，缺少整体计划与阶段结构
2. 多 agent 长上下文轮转，导致主目标失焦、上下文膨胀与空轮转增加
3. 主文长期缺位，配件先于主线膨胀
4. 生产、编排、审计混位，缺少稳定主收束位
5. 中段容易断层，阶段边界模糊，任务容易虎头蛇尾
6. 长任务无法自运行，过度依赖人工盯盘
7. 缺少成本意识、预算意识与模型选择意识
8. 沉淀的方法无法稳定复用，踩过的坑持续重复发生

---

## 三、角色结构

这套框架当前锁定 5 个角色：
- 主入口 Agent
- 一次性生产 subagent
- 审计位
- runtime keeper
- 人类拍板位

### 1. 主入口 Agent
主入口 Agent 是唯一入口与唯一主收束位，负责：
- 需求对齐
- 读取可复用的 skill、经验、注册层入口与历史方法
- 计划编排与阶段拆分
- 锁定当前阶段唯一主产物
- 单元派发、结果回收与并入
- 阶段送审
- 阶段推进、停损与复盘
- 成本与预算控制，包括模型选择

主入口 Agent 不负责：
- 长时间亲自承担大量生产单元
- 吞并审计、拍板与系统治理职责

### 2. 一次性生产 subagent
subagent 是施工单元，只负责被派发的最小单元：
- 一个最小单元
- 一个 prompt
- 一个产物
- 在当前机制下，默认完成即删除

subagent 不负责：
- 自行扩题
- 自行改阶段目标
- 自行决定进入下一阶段
- 自行承担主线收束

### 3. 审计位
审计位是阶段门正式裁决位，负责：
- 判断当前阶段是否 accepted / revision-needed / blocked
- 判断是否偷宽、越界、证据不足或结构失真

审计位不负责持续插手施工过程，也不接管主入口编排职责。

### 4. runtime keeper
runtime keeper 负责保持运行态稳定，重点关注：
- 卡断识别
- timeout 恢复提醒
- 假推进识别
- 阶段悬空提醒
- 停发提醒与异常分流

runtime keeper 不负责：
- 改任务目标
- 代替生产位完成最小单元
- 代主入口重编排
- 判断阶段是否成立
- 代审计位做放行裁决
- 代人类拍板位做资源与边界决策

### 5. 人类拍板位
人类拍板位负责：
- 边界外拍板
- 优先级取舍
- 预算与资源判断
- 任务目标修订

一句话关系：

> 主入口管结构，subagent 管施工，runtime keeper 管不断线与恢复，审计位管放行，人类拍板位管边界外决策。

## 四、起手顺序

需求对齐后，默认不直接起手生产，而是先进入计划与编排，再决定是否派发最小单元。

统一起手顺序为：

1. 需求对齐
   - 压清任务目标、不做什么、当前阶段目标、成功标准、时间要求、关键约束
2. 读取可复用能力
   - 先检查 skill、经验、注册层入口、历史方法、低成本默认路径
3. 锁定主入口 Agent
   - 在当前机制下，未显式交接前，默认不更换主入口
4. 锁定当前阶段唯一主产物
   - 主产物不清，不许开工
5. 做阶段拆分与计划编排
   - 明确阶段目标、送审门、放行条件、退回条件、停损点、预算策略
6. 判断是否需要 spawn
   - 能直接做清楚的，不为协作而协作
7. 定义最小单元与模型策略
   - 明确 task goal、scope、deliverable、done condition、constraints、return target、suggested model / cost tier
8. 正式进入生产

---

## 五、阶段推进机制

任务进入正式生产后，默认按阶段制推进，而不是按聊天轮次推进。

每阶段至少应明确：
- 阶段目标
- 当前阶段唯一主产物
- 允许的最小单元范围
- 送审门
- 放行条件与退回条件
- timeout / 卡断恢复口径
- 停损点

阶段内标准推进链为：

1. 主入口 Agent 锁定本阶段唯一主产物
2. 主入口 Agent 派发最小单元
3. subagent 返回结果
4. 主入口 Agent 回收并判断是否能并入主产物
5. runtime keeper 检查是否出现 timeout、卡断、假推进或主产物停长
6. 达到阶段门后送审
7. 审计位给出 accepted / revision-needed / blocked
8. 主入口 Agent 决定进入下一阶段、回本阶段重编排或暂停升级

阶段推进的硬规则是：
- 对正式生产阶段而言，不能服务主产物的回收，不应作为主要推进依据
- 阶段成熟后应送审，不应无限补细节
- 恢复优先于硬拖，停损优先于假推进

---

## 六、最小单元原则

最小单元的统一口径是：

> 在当前机制下，最小单元默认按“一个最小单元 → 一个 prompt → 一个 subagent → 一个可回收产物”组织。

最小单元必须满足：
- 目标单一
- 产物单一
- 边界清楚
- 可独立回收
- 失败成本可控

每个最小单元至少应具备字段：
- `task goal`
- `scope`
- `deliverable`
- `done condition`
- `constraints`
- `return target`
- `suggested model / cost tier`

补充口径：
- prompt 是施工指令，不是把不确定性整体下放
- subagent 是一次性工作单，不是并行常驻主脑
- 模型选择跟着单元难度、风险与返工成本走，不跟着习惯走
- 反复出现且结构稳定的最小单元，应进一步考虑 skill 化与注册层复用

---

## 七、主产物优先原则

任何阶段都必须先保主产物，再谈配件；不能让配件增长替代主线推进。

主产物是当前阶段主目标的主承载位。它应同时满足：
- 承载当前阶段主要目标
- 成为后续阶段继续生长的主承载对象
- 成为审计位判断的主要对象
- 直接代表主线有没有推进

主入口 Agent 在每阶段开始前，必须先锁定唯一主产物，并明确：
- 长到什么程度可送审
- 哪些补件可以服务它
- 哪些内容当前阶段不先做

对正式生产阶段而言，有效推进主线主要包括：
- 主产物有实质增长
- 新内容已并入主产物
- 主产物的关键阻塞被消除
- 主产物更接近送审状态

补充说明：
- 探针验证、排障、反例排除等工作，可计入辅助推进
- 但不应长期替代主线增长

补充原则：
- 配件可以长，但前提是它真的在帮当前主产物过关，而不是在替未来阶段提前铺陈，反客为主。
- 审计看的是主产物是否成立，而不是附件是否热闹。
- 预算优先保证主线，不为外围热闹买单。

---

## 八、断层点检查

断层是指协作链在关键接口上失去连续性，导致任务虽然看似还在动，但已经无法稳定地派发、回收、并入、送审或推进下一阶段。

当前重点检查 7 类断层：

1. 需求对齐 → 计划编排
2. 计划编排 → 单元派发
3. 单元派发 → 结果回收
4. 结果回收 → 并入主产物
5. 阶段形成 → 送审进入
6. 审计结论 → 恢复推进
7. 阶段完成 → 复盘沉淀

统一断层预警信号包括：
- 需求已清，但没有计划
- 计划已出，但迟迟不能派单
- 单元已派，但没有正式回收
- 回收有了，但主产物不长
- 阶段已成熟，但没人送审
- 审计已结束，但没有进入下一动作
- 任务结束了，但没有复盘沉淀

断层后的默认处理顺序：
1. 先标记断层点
2. 判断断在哪一层
3. 由对应角色先接链
4. 再决定恢复原阶段、回本阶段重编排，或停损并升级

一句话规则：

> 断层后先接链，不先假装继续推进。

---

## 九、停损点与防虎头蛇尾机制

停损点，是指当协作继续按当前方式推进，新增成本已明显高于新增收益时，系统应停止硬拖、停止假推进，转入重编排、恢复或升级判断的正式节点。

停损判断的典型触发信号包括：
- 主产物连续不增长
- 小单元错误开始牵连整阶段返工
- 回收成本显著高于继续生产收益
- 审计连续指出同类结构问题
- runtime keeper 连续发出卡断 / timeout / 假推进提醒
- 预算消耗开始偏离主线目标
- 任务目标已变化，但计划未重编排

停损提出与裁定机制：
- 主入口 Agent 可以提出停损
- runtime keeper 可以发出停损提醒
- 审计位可以提出停损建议
- 人类拍板位可以做最终停损裁定

停损后的默认路径：
- 回本阶段重编排
- 缩小粒度后恢复推进
- 暂停并升级
- 阶段性收口后转复盘

防虎头蛇尾的关键不是催快，而是每个阶段都必须明确收束。阶段结束时必须三选一：
- 进入下一阶段
- 回本阶段重编排
- 阶段性收口并升级 / 复盘
