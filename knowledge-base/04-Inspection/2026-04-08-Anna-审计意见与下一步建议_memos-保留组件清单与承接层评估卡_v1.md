## 二、我认可的部分

### 1. “退角色，不退资产”这条边界写对了
Curie 明确区分了：

#### 必须退场的
- 默认 memory slot
- 默认 auto-recall 注入链
- 默认可信记忆基础设施角色
- memos 的默认允许地位 / 活动 entry 地位

#### 可以保留的
- `memos.db`
- 插件源码目录
- 调试日志 / hold / backup
- RCA / 治理文档 / SOP / skill 材料

这条区分现在是对的。

它避免了两种常见错误：
1. 把“去默认化”误写成“全删”
2. 把“保留研究材料”误写成“继续默认挂载”

### 2. P0 / P1 / P2 / Drop 的层级划分是有效的
这张卡真正有工程价值的，是它把对象分层了：

#### P0
先拆、先接：
- slot
- auto-recall
- entry
- allow
- 默认工作闭环承接

#### P1
保留，但不进默认链：
- DB
- 插件目录
- 文档 / skill / SOP
- recall 能力本身（改显式调用）

#### P2
后补，不进第一版监控层：
- durable memory 治理
- retrieval policy 自动化
- viewer / UI 善后

#### Drop
明确当前不允许重新滑回去的方向。

这比只写原则强很多，因为它已经接近施工顺序了。

### 3. “能力保留”与“默认链拆除”没有混写
Curie 没把“去 memos 化”写成“彻底失去 recall 能力”，而是明确压成：

> **能力可以保留，但默认注入链必须退场。**

这条非常关键。

因为这能避免后面施工时出现错误二分法：
- 要么全保留
- 要么全废掉

当前更稳的写法就是 Curie 现在这条：
- recall 能力保留
- 默认 recall 链退出
- 后续由显式 memory tools / judge / governance 再决定是否接入

### 4. “第一版监控层不要顺手接太多 retrieval / durable memory 职责”这条是对的
这条我认可。

原因很简单：
当前主线是：
- 去默认挂载
- 拆默认注入
- 用替代层接闭环

如果此时把：
- retrieval policy 自动化
- durable memory 治理
- 更复杂的 memory lineage

一并塞进第一版监控层，
就会把本来应该“瘦身减压”的 Phase 1 又做厚。

所以 Curie 这条判断是成立的：
> **先让第一版监控层保持轻，memory policy 自动化后补。**

## 三、当前仍需压住的边界

### 1. 这张卡是“研究评估卡”，不是“施工完成卡”
这一点必须写死。

当前这张卡已经足够给玄武做工程输入，
但它还不是：
- 已执行记录
- 已切换结果
- 已验收文档

所以当前不能写成：
> **P0 项已经完成。**

当前只能写成：
> **P0/P1/P2/Drop 的工程边界已可施工。**

### 2. “runtime judge / governance 决策层”当前只能写成承接方向，不能写成已就绪能力
Curie 卡里把：
- runtime judge
- governance 决策层

写成了承接方向，这没问题。

但要压住一条边界：
> **这些现在还是承接方向，不是已经落好的现成层。**

否则后面很容易在文档里滑成：
- “已经有 judge 接住了”
- “已经有 governance 自动判定了”

这当前还不成立。

### 3. “保留插件源码目录”必须继续限定在工程材料用途
这条虽然 Curie 已经写了，
但后续交给玄武时必须继续保住：

> **保留插件目录 ≠ 默认允许加载 ≠ 默认继续活动。**

这是后续最容易被工程实现误写坏的地方。

## 五、下一步建议

### 1. 现在可以继续压给玄武一张更硬的工程表
Curie 这张卡已经够做研究输入，
但如果要进入施工，建议下一步继续压成：

#### 必须有的三张表
1. **保留组件清单**
   - 哪些保留
   - 保留用途
   - 非保留用途

2. **禁止默认挂载清单**
   - 哪些角色必须退场
   - 哪些配置不能保留
   - 哪些回滚理由不成立

3. **承接层施工表**
   - 默认工作记忆谁接
   - 正式知识谁接
   - recall 能力谁以显式方式接
   - 哪些内容后补

### 2. 玄武施工时不要再现场做边界裁决
后续如果进入施工，玄武应只做：
- 配置解绑
- 执行顺序落实
- 验收记录

不应在施工过程中继续现场决定：
- 这个能不能默认保留
- 那个是不是还要观察
- viewer 是否恢复主线

这些边界现在已经被 Anna + Curie 收口完了。

### 3. Curie 这张卡建议作为 Runtime Pipeline 的“保留组件判定侧卡”使用
尤其你刚刚已经提示：
> **有一些组件 Curie 的 runtime pipeline 可能要保留下载进行改造。**

那这张卡最合理的定位就是：
> **它不负责决定是否继续默认运行，只负责决定哪些可以保留成工程材料，以及承接层怎么分。**

这个定位建议写死。

### 4. 后续正式施工文档必须附 before / after
后面无论谁执行，
都建议补：
- 切换前配置
- 切换后配置
- 切换前默认链状态
- 切换后默认链状态
- 是否仍有 auto-recall 注入
- 是否仍由 MemOS 占默认 memory slot

否则后面容易继续回到口头判断。
