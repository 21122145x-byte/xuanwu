## 一、为什么第四张选它

`cross-agent-handoff` 适合作为第四张 skill 候选，因为跨 Agent 正式交接已经有相对稳定的结构规则，但目前仍主要靠人工记忆执行：

1. **触发条件清楚**
   - 需要正式交接给另一位 Agent
   - 需要发【收到】回执
   - 需要发【任务完成】回传
   - 需要打回或重提

2. **输入输出稳定**
   - 输入：任务内容、文档路径、交接对象、当前状态
   - 输出：标准交接消息 / 标准回执 / 标准打回 / 标准重提

3. **收益明显**
   - 能减少“不算正式交接”的消息
   - 能减少缺 5 字段 / 缺路径 / 缺下一责任人 mention 的情况
   - 能让交接链显式化、可回看、可审计

4. **边界可控**
   - 它适合生成和检查正式交接格式
   - 但不适合替代谁来接、是否放行、是否熔断的最终裁决

所以它属于：
> **非常适合做模板型 / 检查型 skill，但不能做交接裁决器**

## 三、这个 skill 到底做什么

### 建议 skill 名称
- `cross-agent-handoff`

### skill 的最小职责
只做下面四件事：

1. **检查正式交接消息是否合格**
   - 是否缺 5 字段
   - 是否缺文档路径
   - 是否缺 `<@Discord数字ID>`
   - 是否缺下一步动作

2. **生成正式交接消息草稿**
   - 标准 5 字段 + 路径 + 末尾 @ 下一责任人

3. **生成标准回执消息**
   - 【收到】
   - 【任务完成】
   - 打回
   - 重提

4. **规范化已有交接消息**
   - 把散乱消息整理成正式格式

### 明确不做的事
这个 skill 不应：
- 决定任务该交给谁
- 决定是否 accepted / blocked
- 决定是否触发熔断
- 替代协调位做跨 Agent 调度判断
- 自动认定某条消息已算“被对方承接”

一句话：
> 它是正式交接消息构造器 / 检查器，不是交接裁决器。

## 四、建议的输入 / 输出

### 输入（最小版）
至少支持：
1. 当前任务摘要 + 文档路径
2. 交接对象 Discord ID
3. 当前消息草稿
4. 当前交接类型（正式交接 / 收到 / 完成 / 打回 / 重提）

### 输出（最小版）
建议支持：
1. 合格性检查结果
2. 标准正式交接消息草稿
3. 标准回执消息草稿
4. 标准打回 / 重提消息草稿

## 五、为什么适合做成 skill

因为它非常像模板层动作：
- 高频
- 重复
- 字段明确
- 错误模式稳定
- 很适合被检查和格式化

如果没有它，最容易发生：
- 只 @ 名字，不 @ 数字 ID
- 没有路径
- 没有 5 字段
- 有内容但不算正式交接
- 重提时没写修改说明

---

## 六、为什么不能全自动决定交接有效性

即使这个 skill 值得做，也必须明确：

### 不能自动做的判断
- 这个交接是否真的该发起
- 当前交给谁最合适
- 对方是否真的已经承接
- 当前内容是否真的达到可交接质量
- 这轮打回是否应触发熔断

所以：
- **格式合格** ≠ **交接合理**
- **消息完整** ≠ **任务已被正确承接**

它最合理的定位是：
> **交接结构辅助器**
而不是：
> **自动交接判官**

## 八、建议的触发语境

适合触发：
- “帮我整理成正式交接消息”
- “检查这条交接消息合不合格”
- “补一个收到回执”
- “补一个任务完成回传”
- “整理一条打回 / 重提消息”

不适合触发：
- 当前还没明确交接对象
- 当前不需要正式交接，只是在讨论
- 当前真正需要的是协调位拍板，而不是消息格式整理

---

## 九、当前建议的最小试跑方式

### 试跑样本建议
选 4 类样本：
1. 正式交接消息
2. 【收到】回执
3. 【任务完成】回传
4. 打回 / 重提消息

### 试跑要看什么
- 能否稳定识别缺项
- 生成的格式是否符合执行卡口径
- 会不会遗漏数字 ID mention
- 会不会把“消息合格”误写成“交接已被承接”

### 最小通过标准
- 至少 4 个样本无明显漏字段
- 至少 3 个样本输出明显优于手工临场补写
- 不出现“误导为已承接 / 已完成”的重大问题
