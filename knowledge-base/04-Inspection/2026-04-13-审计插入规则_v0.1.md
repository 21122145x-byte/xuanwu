## 1. 规则目标

本规则只回答四件事：
1. 审计位何时进入
2. 审计位何时默认不进入
3. 审计进入时按什么最小口径判断
4. 审计位与主入口 / runtime keeper 的边界如何区分

## 2. 审计位的默认进入时机

审计位默认只在“阶段门”进入，而不在施工过程中持续陪跑。

可进入的最小前提是：
- 当前阶段唯一主产物已经被主入口锁定
- 主入口已完成本轮结果回收与并入判断
- 当前阶段已到达预设送审门，或主入口明确发起送审

一句话口径：

> 审计位默认在阶段形成后进入，不在零散施工中途常驻插手。

## 3. 默认不进入的情形

以下情形，审计位默认不进入：
- 还处于需求对齐与计划编排阶段
- 最小单元尚在施工，结果尚未回收
- 回收已发生，但主入口尚未完成是否并入主产物的判断
- 当前只有局部草稿、补件或试探性材料，尚不足以代表当前阶段主产物
- runtime keeper 仅在做 timeout、卡断、假推进提醒

一句话口径：

> 还没到阶段门、还没形成可送审主产物时，默认不拉审计位下场。

## 4. 审计进入时的最小判断口径

审计位进入后，最小只判断以下事项：
- 当前阶段主产物是否达到送审门要求
- 是否存在越界、偷宽、证据不足、结构失真
- 当前阶段结论应为：`accepted` / `revision-needed` / `blocked`

审计位不在本规则内承担：
- 重写主产物
- 代替主入口重编排
- 对施工过程做持续细节指挥
- 代替人类拍板位做资源、边界与目标修改决策

## 5. 与主入口 Agent 的边界

主入口 Agent 负责：
- 锁定当前阶段唯一主产物
- 派发最小单元
- 回收结果并判断能否并入
- 决定何时送审
- 根据审计结果决定进入下一阶段、回本阶段重编排或暂停升级

审计位负责：
- 在送审节点判断当前阶段是否可放行
- 给出 accepted / revision-needed / blocked
- 指出越界、偷宽、证据不足、结构失真等问题

边界规则：

> 主入口管编排与收束，审计位管阶段放行，不互相吞位。

## 6. 与 runtime keeper 的边界

runtime keeper 负责：
- 卡断识别
- timeout 恢复提醒
- 假推进识别
- 阶段悬空提醒
- 停发提醒与异常分流

runtime keeper 不负责：
- 判断阶段是否 accepted / revision-needed / blocked
- 代替审计位做正式放行裁决
- 因过程异常而直接改写任务目标或送审结论

边界规则：

> runtime keeper 管运行态不断线，审计位管阶段是否过门。

## 7. 最小插入规则

在当前框架下，审计插入按以下顺序执行：
1. 主入口锁定当前阶段唯一主产物
2. 主入口完成本轮回收并确认已有可送审对象
3. 达到阶段门后，由主入口正式送审
4. 审计位进入并给出阶段结论
5. 主入口依据审计结论决定下一动作

若未达到第 2-3 步，则默认不触发审计插入。

## 8. 当前版本边界

本件只定义“审计何时进入、按什么最小口径进入”。
当前不扩展：
- 停损规则卡正文
- 完整审计 SOP
- 审计结论格式细则
- 多轮审计协作细节
