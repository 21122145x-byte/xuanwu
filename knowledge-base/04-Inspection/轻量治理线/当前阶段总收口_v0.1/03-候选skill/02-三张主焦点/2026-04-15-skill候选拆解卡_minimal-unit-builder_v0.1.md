## 一、为什么现在补它

`minimal-unit-builder` 值得现在补，是因为“最小单元机制”已经是这条主线里最稳定、最可复用的一层之一：

1. **触发条件清楚**
   - 主入口位准备派发施工单元时
   - 需要把厚问题压成可执行的最小任务时
   - 当前 prompt 过厚、任务过宽、结果不可回收时

2. **输入输出稳定**
   - 输入：当前目标、当前阶段、单元边界、期望产物
   - 输出：最小单元草稿 / 单元字段块 / 派发用最小任务结构

3. **收益明显**
   - 能减少“伪单元 / 假拆分”
   - 能降低 subagent 扩题、漂移和吞主线的概率
   - 能让派发、回收、并入更稳定

4. **边界可控**
   - 它适合生成和检查最小单元结构
   - 不适合替代主入口位决定阶段目标、主产物与最终编排策略

所以它属于：
> **非常适合 skill 化的结构生成器，但不能做自动任务总拆分器**

## 二、对应的上位机制

这个 skill 候选主要承接：

### 1. 最小单元机制
来源：
- 单一入口式 multi-agent 协作框架主文（单元 5）

当前稳定口径：
最小单元应至少满足：
- 可执行
- 可回收
- 可并入
- 可上提

并且默认不应：
- 偷宽成厚问题
- 把阶段问题伪装成单元问题
- 把 prompt 当成单元本体

### 2. 最小单元模板
来源：
- `2026-04-13-最小单元模板_v0.1.md`

当前稳定口径：
- 一次派一个最小单元
- 一个 prompt
- 一个产物
- 一次性施工完成后回收

## 三、这个 skill 到底做什么

### 建议 skill 名称
- `minimal-unit-builder`

### skill 的最小职责
只做下面三件事：

1. **检查**
   - 当前任务是否已经压到最小单元
   - 是否仍然过厚 / 过宽 / 伪单元化
   - 是否缺期望产物 / 回收方式 / 并入位置

2. **补全**
   - 基于当前上下文生成最小单元草稿
   - 给出更适合派发的单元结构

3. **规范化**
   - 将散乱的派单想法整理成统一最小单元块

### 明确不做的事
这个 skill 不应：
- 自动决定阶段目标
- 自动决定主产物
- 自动拆完整项目
- 替代主入口位决定是否并发 / 串行
- 把“能写成 prompt”误当成“已经是最小单元”

一句话：
> 它是最小单元结构生成器，不是自动项目拆分器。

---

## 四、建议的输入 / 输出

### 输入（最小版）
至少支持：
1. 当前目标 / 当前阶段
2. 当前任务草稿或派单草稿
3. 期望产物类型
4. 当前并入位置 / 回收位置（若已知）

### 输出（最小版）
建议支持：
1. 单元厚度检查结果
2. 标准最小单元草稿
3. 整理后的统一单元结构块

---

## 五、为什么适合做成 skill

因为它是典型的：
- 高频重复动作
- 结构化程度高
- 失败模式稳定
- 对协作质量影响极大
- 容易人工复核

如果没有它，最容易出现：
- 看起来拆了，其实没拆
- 一个 subagent 被塞进去太多事
- 派发前没有明确产物和回收点
- prompt 写得很长，但不是一个最小单元

---

## 六、为什么不能全自动决定“拆分正确”

即使这个 skill 值得做，也必须明确：

### 不能自动做的判断
- 当前阶段真正应该拆到哪一层
- 当前单元是否值得并发
- 某个厚问题是否该先回上层重编排
- 当前主产物是否已经被正确锁定
- 当前这轮到底该只发一个单元还是暂停推进

所以：
- **单元结构完整** ≠ **拆分策略正确**
- **字段齐全** ≠ **这就是最优最小单元**

它最合理的定位是：
> **最小单元检查器 / 生成器 / 规范器**
而不是：
> **自动任务拆解器**

## 八、建议的触发语境

适合触发：
- “帮我把这个任务压成最小单元”
- “检查这个单元是不是还太厚”
- “把这段派单想法整理成标准最小单元块”
- “这个 prompt 算不算真正的最小单元”

不适合触发：
- 当前还没锁定阶段目标
- 当前需要的是完整编排方案，而不是单元生成
- 当前问题本质上还是厚问题，需回上层重编排

---

## 九、当前建议的最小试跑方式

### 试跑样本建议
选 4 类样本：
1. 合格最小单元
2. 伪单元
3. 过厚单元
4. 已有派单但缺回收结构的单元

### 试跑要看什么
- 能否稳定识别厚度问题
- 生成的单元是否可执行、可回收、可并入
- 会不会把伪单元误判成合格单元
- 会不会把需要重编排的问题错误压成单元

### 最小通过标准
- 至少 4 个样本无明显误判为“可直接派发”
- 至少 3 个样本输出明显优于手工临场拆分
- 不出现“把厚问题伪装成最小单元”的重大问题
