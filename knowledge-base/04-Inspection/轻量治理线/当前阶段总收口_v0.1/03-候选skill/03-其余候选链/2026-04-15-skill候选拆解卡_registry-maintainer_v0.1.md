## 一、为什么现在补它

`registry-maintainer` 值得现在补，是因为注册层已经从“口头方向”推进到了：
- 有注册层整理表
- 有现役能力清单
- 有机制与 skill 映射表
- 有多张下位清单与候选卡

但当前这些维护动作仍主要靠人工回忆和手工收束，缺一个稳定的“注册层维护辅助器”。

它适合现在推进，主要因为：

1. **触发条件清楚**
   - 新机制出现后需要判断是否进入注册层
   - 新 skill 出现后需要判断是否进入注册层
   - 某机制 / skill 状态变化后需要更新清单
   - 需要把零散新增对象回写进注册层表时

2. **输入输出相对稳定**
   - 输入：对象名称、对象类型、当前状态、上位文档、适用范围、是否已验证
   - 输出：注册层回写草稿 / 状态更新建议 / 映射更新建议

3. **收益明显**
   - 能减少注册层长期靠记忆维护
   - 能减少“新对象已经存在但没挂进注册层”的漏项
   - 能降低机制 / skill / 候选卡之间不同步的问题

4. **边界必须提前卡死**
   - 它适合辅助维护注册层
   - 但不适合自动决定对象是否“正式成立”
   - 更不适合自动批量改写整套注册层而不经人工复核

所以它属于：
> **适合做注册层维护辅助 skill，但绝不能做自动注册层裁决器**

## 二、对应的上位机制

这个 skill 候选主要承接：

### 1. 注册层整理机制
来源：

当前稳定方向：

- 哪些稳定经验达到注册层候选程度

来源：

新对象出现后，至少应判断：
- 是否进入现役能力清单
- 状态是完整 / 未完成 / 慎用
- 是否已有对应 skill
- 是否只适合局部 skill 化
- 是否需要起新的候选卡 / 下位件

## 三、这个 skill 到底做什么

### 建议 skill 名称
- `registry-maintainer`

### skill 的最小职责
只做下面四件事：

1. **检查**
   - 当前对象是否已进入注册层相关文档
   - 是否缺现役能力清单回写
   - 是否缺映射表回写
   - 是否缺状态标注

2. **补全**
   - 生成一版注册层回写草稿
   - 生成状态更新建议
   - 生成映射更新建议

3. **规范化**
   - 将零散新增对象整理成统一注册层条目

4. **同步提醒**
   - 提示当前对象还缺哪些下位件或回挂动作

### 明确不做的事
这个 skill 不应：
- 自动决定某对象已正式成立
- 自动决定某机制就是完整能力
- 自动决定某 skill 一定进入现役层
- 自动批量改写所有注册层文档
- 替代 Anna / 粉雪做最终裁决与纠偏

一句话：
> 它是注册层维护辅助器，不是注册层自动裁判。

## 四、建议的输入 / 输出

### 输入（最小版）
至少支持：
1. 当前对象名称
2. 当前对象类型（机制 / skill / 候选 skill / 规则卡 / 模板件）
3. 当前已知状态
4. 当前已有挂载与上位文档
5. 当前是否已验证 / 已试跑 / 已 accepted

### 输出（最小版）
建议支持：
1. 注册层缺项检查结果
2. 现役能力清单回写草稿
3. 映射表更新草稿
4. 状态更新建议
5. 缺失下位件提醒

## 五、为什么适合做成 skill

因为它是典型的：
- 高频维护动作
- 结构化回写动作
- 错误模式重复
- 多文档同步很容易漏
- 结果容易人工复核

如果没有它，最容易出现：
- 新机制已写出，但没进现役能力清单
- 新 skill 候选已起卡，但映射表没更新
- 状态变化了，但旧表没回写
- 主文、注册表、候选卡之间不同步

---

## 六、为什么不能全自动决定注册层裁决

即使这个 skill 值得做，也必须明确：

### 不能自动做的判断
- 这个对象是否真的已经稳定
- 它是否应进入现役层而非候选层
- 它是完整、未完成还是慎用
- 它是否值得单独起下位件
- 它与哪些对象是真强关联，哪些只是弱相关

所以：
- **能回写** ≠ **应该回写成现役**
- **条目完整** ≠ **对象定位正确**

它最合理的定位是：
> **注册层维护器 / 回写辅助器 / 同步检查器**
而不是：
> **注册层自动裁决器**

## 八、建议的触发语境

适合触发：
- “把这个新机制回写进注册层”
- “检查这个对象还缺哪些注册层更新动作”
- “这个 skill 候选应该更新哪些表”
- “把这条新增对象整理成注册层条目”

不适合触发：
- 当前对象状态本身还没判断清楚
- 当前需要的是审计裁决，而不是注册层回写
- 当前要做的是大规模治理重构，而不是单对象维护

---

## 九、当前建议的最小试跑方式

### 试跑样本建议
选 4 类样本：
1. 新机制进入现役能力清单
2. 新 skill 候选进入映射表
3. 状态变化对象的回写
4. 新对象的多表同步回挂

### 试跑要看什么
- 能否稳定识别应更新哪些表
- 回写建议是否贴对象真实状态
- 会不会把候选对象过早写成现役
- 会不会遗漏关联表

### 最小通过标准
- 至少 4 个样本无明显误写现役状态
- 至少 3 个样本输出明显优于手工临场同步
- 不出现“把候选层对象错误升格”的重大问题
