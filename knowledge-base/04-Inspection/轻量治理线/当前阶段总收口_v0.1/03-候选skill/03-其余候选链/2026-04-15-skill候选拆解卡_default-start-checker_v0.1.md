## 一、为什么现在补它

`default-start-checker` 值得现在补，是因为“默认起手顺序”这一层已经在主文与规则卡里明确成立，但当前仍主要靠主入口位记忆执行：

1. **触发条件清楚**
   - 项目态任务起手前
   - 连续推进任务续接时
   - 审后回转任务准备重新接链时
   - 用户问“现在该先看什么、先做什么”时

2. **输入输出稳定**
   - 输入：任务类型、当前任务态、当前是否已有项目总表 / 注册层信息 / 相关规则卡
   - 输出：默认起手检查结果 / 默认读取顺序草稿 / 缺项提示

3. **收益明显**
   - 能减少“直接靠聊天记忆开工”
   - 能降低跳过注册层 / 项目总表的情况
   - 能让主入口位更稳定地走低返工路径

4. **边界相对可控**
   - 它适合做默认起手检查与读取顺序辅助
   - 不适合替代主入口位做真正的任务判断与结构编排

所以它属于：
> **非常适合做起手检查型 skill，但不能做自动任务编排器**

## 二、对应的上位机制

这个 skill 候选主要承接：

### 1. 系统级注册与默认起手顺序
来源：
- 系统级注册与默认起手顺序规则卡

当前稳定口径：
项目态 / 连续推进态任务默认按以下顺序起手：
1. 先确认任务目标与当前阶段
2. 再读系统级注册层
3. 再看当前项目总表
4. 再决定当前默认下一动作
5. 最后才决定是否进入派单或生产

### 2. 主文单元 3：起手顺序
来源：
- 单一入口式 multi-agent 协作框架主文

当前稳定口径：
- 不读注册层，不进入正式编排
- 项目态任务不允许跳过项目总表
- 审后回转任务不允许只看最新聊天，不回看上一阶段结论

## 三、这个 skill 到底做什么

### 建议 skill 名称
- `default-start-checker`

### skill 的最小职责
只做下面三件事：

1. **检查**
   - 当前任务是否已确认任务态
   - 是否已回看注册层
   - 是否已回到项目总表
   - 是否缺默认下一动作判断前的必要输入

2. **补全**
   - 输出一版“默认起手顺序检查结果”
   - 给出当前最小读取顺序建议

3. **规范化**
   - 将散乱的起手判断整理成统一起手检查块

### 明确不做的事
这个 skill 不应：
- 自动决定任务目标
- 自动决定任务该怎么拆阶段
- 自动决定是否一定要派单
- 替代主入口位做真正编排
- 在注册层缺失时伪装成“已经完成起手判断”

一句话：
> 它是默认起手检查器，不是自动编排器。

## 四、建议的输入 / 输出

### 输入（最小版）
至少支持：
1. 当前任务摘要
2. 当前任务态（若已知）
3. 当前已知文档链（注册层 / 总表 / 主文 / 规则卡）
4. 当前起手判断草稿

### 输出（最小版）
建议支持：
1. 起手缺项检查结果
2. 默认读取顺序建议
3. 起手检查块 / 起手结论块

## 五、为什么适合做成 skill

因为它是典型的：
- 高频判断前置动作
- 步骤顺序稳定
- 错误模式重复出现
- 能显著减少返工与误开工
- 结果容易人工复核

如果没有它，最容易出现：
- 项目态任务不回总表
- 审后回转只看最新聊天
- 注册层明明该读却被跳过
- 还没确认任务态就直接派单

---

## 六、为什么不能全自动决定起手结论

即使这个 skill 值得做，也必须明确：

### 不能自动做的判断
- 当前任务到底属于哪一类（在信息不足时）
- 现在是否应该继续推进还是先停损
- 现在是否已经足够清楚到可以派单
- 当前阶段真正的唯一主产物是什么
- 当前该走哪一条具体编排路径

所以：
- **起手检查完整** ≠ **起手判断正确**
- **默认读取顺序存在** ≠ **任务已被正确理解**

它最合理的定位是：
> **起手顺序检查器 / 读取顺序辅助器**
而不是：
> **自动任务路由器**

## 八、建议的触发语境

适合触发：
- “这个任务现在起手该先看什么”
- “帮我检查默认起手顺序有没有漏”
- “这是项目态续接，先读哪些东西”
- “这轮审后回转，现在默认应该怎么重新接链”

不适合触发：
- 当前任务目标本身还没对齐
- 当前需要的是主入口位的结构设计，而不是起手检查
- 当前用户要的是最终计划，而不是起手顺序辅助

---

## 九、当前建议的最小试跑方式

### 试跑样本建议
选 4 类样本：
1. 项目态续接任务
2. 审后回转任务
3. 新任务起手判断
4. 复盘 / 审计类任务

### 试跑要看什么
- 能否稳定识别起手缺项
- 读取顺序建议是否贴任务态
- 会不会在信息不足时假装已经判断清楚
- 会不会把“建议”写成“既定编排结论”

### 最小通过标准
- 至少 4 个样本无明显误导性起手建议
- 至少 3 个样本输出明显优于手工临场判断
- 不出现“引导跳过注册层 / 项目总表”的重大问题
