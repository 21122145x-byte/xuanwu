## 一、推进目标

本轮只推进到：
- 每张候选 skill 的试跑前置补件完成
- 每张候选 skill 至少经过一轮阶段审计
- 关键边界已做最小纠偏
- 进入真实样例试跑前形成可收口状态

本轮**不做**：
- 真实样例试跑执行
- 正式 SKILL.md 落地
- 现役 skill 升格判断

---

## 二、总体推进原则

### 1. 按最小单元逐步推进
一次只推进一个最小动作，不并行摊开三张卡。

### 2. 每张卡都走同一闭环
每张卡统一走：
1. 前置补件
2. 阶段审计
3. 纠偏
4. 阶段收口

### 3. 先低风险，再高风险
顺序固定为：
1. `minimal-unit-builder`
2. `main-artifact-judge-helper`
3. `revision-needed-router`

### 4. 到真实样例试跑前收口
本轮目标不是把它们跑起来，而是把“试跑前置条件”压清楚。

## 三、阶段划分

## Phase 1：minimal-unit-builder 试跑前收口

### Phase 1-A｜补件
目标：确认当前四件套是否已足够进入试跑前审计

最小动作：
- 回看候选拆解卡
- 回看技能说明草案
- 回看样例集
- 回看试跑骨架
- 确认是否仍缺字段级补件

通过标准：
- 不再缺新增说明件
- 当前最小单元定义、误判风险、试跑目标三者一致

### Phase 1-B｜阶段审计
目标：审计它是否已具备进入真实样例试跑前的最小清晰度

审计关注点：
- 会不会把厚问题伪装成最小单元
- 会不会把“能写成 prompt”误当成“可直接派发”
- 当前样例是否已覆盖最关键误判面

输出：
- 一条阶段审计结论
  - accepted-for-pretrial
  - revision-needed

### Phase 1-C｜纠偏
目标：若审计发现问题，只做试跑前必要纠偏，不扩题

允许纠偏范围：
- 补字段
- 补 1 处风险提醒
- 调整样例表述
- 调整试跑通过标准措辞

不允许纠偏范围：
- 重写整张候选卡
- 扩成正式 skill
- 顺手并行改后面两张卡

### Phase 1-D｜阶段收口
收口产物：
- `minimal-unit-builder` 已达到“可进入真实样例试跑前状态”
- 若未达到，则只明确仍缺什么

---

## Phase 2：main-artifact-judge-helper 试跑前收口

### Phase 2-A｜补件
目标：补上进入试跑前必须具备的“脏样本”

最小动作：
- 新增 1 类脏样本：
  - 某说明件 / 配件临时承担主线功能，但本轮不应立即升格为主产物

通过标准：
- 样例集不只会分“主产物 / 配件”
- 能覆盖“看起来像主产物，但不应直接锁定”的情况

### Phase 2-B｜阶段审计
审计关注点：
- 会不会把更长的对象直接误收成主产物
- 会不会把说明件一律压成配件
- 会不会在信息不足时伪造“已锁定主产物”

输出：
- 一条阶段审计结论
  - accepted-for-pretrial
  - revision-needed

### Phase 2-C｜纠偏
允许纠偏范围：
- 调整脏样本表述
- 补 1 条边界提醒
- 微调输出结构或风险表述

不允许纠偏范围：
- 直接引入资产上提裁决体系
- 把主产物裁决做成自动决策器

### Phase 2-D｜阶段收口
收口产物：
- `main-artifact-judge-helper` 已达到“可进入真实样例试跑前状态”
- 若未达到，则只记录仍缺的边界或样例

---

## Phase 3：revision-needed-router 试跑前收口

### Phase 3-A｜补件
目标：先补一张“最小触发边界卡”

边界卡最少包含三栏：
- 必命中
- 不命中
- 易混淆但应分流

通过标准：
- 当前不再只靠描述性触发条件理解 skill
- 能初步防止误吞 blocked / 重编排 / 普通 repair 建议

### Phase 3-B｜阶段审计
审计关注点：
- 会不会把 blocked 问题压成 revision-needed 小修补
- 会不会把建议回转层级写成既定裁决
- 会不会把需要重编排的问题误压成局部回转

输出：
- 一条阶段审计结论
  - accepted-for-pretrial
  - revision-needed

### Phase 3-C｜纠偏
允许纠偏范围：
- 调整边界卡字段
- 调整不触发条件
- 补 1 条边缘样本说明

不允许纠偏范围：
- 直接扩写为完整审计体系
- 把 router 变成裁判器

### Phase 3-D｜阶段收口
收口产物：
- `revision-needed-router` 达到“可进入真实样例试跑前状态”
- 若未达到，则只记录仍缺哪一类边界

---

## 四、阶段门口径（本轮）

每个 phase 结束时只允许两种结论：
- **accepted-for-pretrial**：已具备进入真实样例试跑前的最小条件
- **revision-needed**：仍缺试跑前必要补件，继续留在当前 phase

本轮不出现：
- accepted（正式通过）
- blocked（除非出现重大方向错误）

---

## 五、执行顺序总览

1. 先完成 `minimal-unit-builder` 的补件 → 审计 → 纠偏 → 收口
2. 再完成 `main-artifact-judge-helper` 的补件 → 审计 → 纠偏 → 收口
3. 最后完成 `revision-needed-router` 的补件 → 审计 → 纠偏 → 收口

禁止动作：
- 三张一起并行推进
- 在前一张未收口前提前跳下一张
- 把试跑前收口变成真实试跑
