## 一、机制定义

单一入口式 multi-agent 协作框架，是一种由唯一主入口 Agent 负责需求对齐、计划编排、阶段推进、结果收束与送审，其他生产位以“一次性最小单元 subagent”执行的阶段式协作机制。

这套机制的目标是：
- 先计划、后生产
- 先保主产物，再长配件
- 控制上下文厚度与返工成本
- 让任务按阶段自运行，而不是依赖人工持续值守
- 把经验、方法、模型选择与避坑判断纳入复用链，而不是每轮从零开始

---

## 二、机制要解决的问题

这套机制主要解决以下问题：

1. 聊完需求后直接起手生产，缺少整体计划与阶段结构
2. 多 agent 长上下文轮转，导致主目标失焦、上下文膨胀与空轮转增加
3. 主文长期缺位，配件先于主线膨胀
4. 生产、编排、审计混位，缺少稳定主收束位
5. 中段容易断层，阶段边界模糊，任务容易虎头蛇尾
6. 长任务无法自运行，过度依赖人工盯盘
7. 缺少成本意识、预算意识与模型选择意识
8. 沉淀的方法无法稳定复用，踩过的坑持续重复发生

---

## 三、角色结构

这套框架当前锁定 5 个角色：
- 主入口 Agent
- 一次性生产 subagent
- 审计位
- runtime keeper
- 人类拍板位

### 1. 主入口 Agent
主入口 Agent 是唯一入口与唯一主收束位，负责：
- 需求对齐
- 读取可复用的 skill、经验、注册层入口与历史方法
- 计划编排与阶段拆分
- 锁定当前阶段唯一主产物
- 单元派发、结果回收与并入
- 阶段送审
- 阶段推进、停损与复盘
- 成本与预算控制，包括模型选择

主入口 Agent 不负责：
- 长时间亲自承担大量生产单元
- 吞并审计、拍板与系统治理职责

### 2. 一次性生产 subagent
subagent 是施工单元，只负责被派发的最小单元：
- 一个最小单元
- 一个 prompt
- 一个产物
- 在当前机制下，默认完成即删除

subagent 不负责：
- 自行扩题
- 自行改阶段目标
- 自行决定进入下一阶段
- 自行承担主线收束

### 3. 审计位
审计位是阶段门正式裁决位，负责：
- 判断当前阶段是否 accepted / revision-needed / blocked
- 判断是否偷宽、越界、证据不足或结构失真

审计位不负责持续插手施工过程，也不接管主入口编排职责。

### 4. runtime keeper
runtime keeper 负责保持运行态稳定，重点关注：
- 卡断识别
- timeout 恢复提醒
- 假推进识别
- 阶段悬空提醒
- 停发提醒与异常分流

runtime keeper 不负责：
- 改任务目标
- 代替生产位完成最小单元
- 代主入口重编排
- 判断阶段是否成立
- 代审计位做放行裁决
- 代人类拍板位做资源与边界决策

### 5. 人类拍板位
人类拍板位负责：
- 边界外拍板
- 优先级取舍
- 预算与资源判断
- 任务目标修订

一句话关系：

> 主入口管结构，subagent 管施工，runtime keeper 管不断线与恢复，审计位管放行，人类拍板位管边界外决策。

## 四、起手顺序

需求对齐后，默认不直接起手生产，而是先进入计划与编排，再决定是否派发最小单元。

这一节的目的，不只是防止“聊完就开工”，还要进一步解决三件事：
- 当前任务到底该落在哪一层资产上
- 主入口位默认应先读什么、后读什么
- 本轮结果产出后默认回收到哪里

也就是说，起手顺序不只是“开始前的礼仪”，而是整个协作链避免漂移、避免乱挂、避免重复返工的第一道结构门。

统一起手顺序为：

### 1. 需求对齐
主入口位先压清：
- 任务目标是什么
- 当前不做什么
- 这一轮的阶段目标是什么
- 成功标准是什么
- 时间要求是什么
- 当前关键约束是什么

如果这些要素仍然混在聊天语气里，没有被压成清楚判断，则默认不进入后续编排。

**默认要求：**
- 目标不清，不拆阶段
- 非目标不清，不判断 scope
- 成功标准不清，不判断 done condition

**当前缺口细项：**
- 缺少一张更稳定的“需求对齐最小字段卡”，当前仍主要分散在编排卡与主入口经验中
- 对“项目态任务”和“一次性任务”的需求对齐口径虽已隐含区分，但还未形成更明确的显式判断卡

### 2. 判断任务类型与任务态
主入口位在正式读取注册层前，先判断本轮任务属于哪一类：
- 一次性任务
- 连续推进任务
- 已进入项目态任务
- 旧任务续接 / 审后回转任务
- 复盘 / 审计 / 收束类任务

这个判断的意义是：
- 一次性任务可以更轻
- 项目态任务必须回项目总表
- 审后回转任务必须优先回看阶段门与上一轮结论
- 复盘 / 审计类任务不应误走生产起手口径

**默认要求：**
- 任务态未判断清楚，不进入 spawn 判断
- 项目态任务不允许跳过项目总表
- 审后回转任务不允许只看最新聊天，不回看上一阶段结论

**当前缺口细项：**
- 任务态分类目前写在主文逻辑里，但还没有独立“任务态判断卡”
- 对“项目态续接”和“新任务起手”的差异，虽然已在规则上成立，但还未被做成统一入口模板

### 3. 读取可复用能力与系统级注册
主入口位默认先检查：
- skill
- 已有规则卡
- 已有模板件
- 历史方法
- 低成本默认路径
- 当前有哪些现役能力可直接调用

在这一层，主入口位至少要判断：
- 当前有哪些现役能力、规则卡、模板件、历史方法可以直接复用
- 默认先读谁
- 冲突时先调谁
- 哪些入口已废弃、失效或仅供参考

这一步的目的，不是把所有旧资料再读一遍，
而是避免主入口位每次都把自己当成第一次面对该问题。

**默认要求：**
- 不读注册层，不进入正式编排
- 读到可复用能力后，默认优先走低成本、低返工、低上下文厚度路径
- 已存在稳定规则件时，不应跳过后重新临场发明同类规则

**当前缺口细项：**
- 注册层目前仍是“规则已定义、清单未补齐”的状态
- 缺少现役能力清单
- 缺少默认调用顺序示例
- 缺少废弃入口标记清单
- 缺少冲突时先调谁的裁决表

### 4. 读取默认导航入口
若本轮任务涉及既有文档链、历史试跑、已有产物或跨阶段续接，主入口位还应确认默认导航入口是否可回找。

至少应检查：
- 项目总表是否能回到当前阶段
- README 或索引件是否能定位到相关文档
- 主文与规则卡之间的挂载是否足以支撑本轮读取
- 同位 / 下位文档是否存在明显断链

这一步的意义在于：
- 不让文档存在但找不到
- 不让主入口位因为回找成本过高而重新依赖聊天上下文

**默认要求：**
- 涉及历史产物链时，不确认导航入口就不开正式编排
- 若当前导航明显断链，应先补最低导航可回找性，再继续主线推进

**当前缺口细项：**
- 关键文档链路已补得较明显，但并非全量文档都完成逐份检查
- 试跑编排卡的头部统一性与全量挂载仍未完全逐份核完
- README 虽已补关键导航，但尚未成为更完整的分层导航索引

### 5. 回到项目总表确认当前阶段
如果任务已经进入项目态，主入口位不应只靠聊天记忆判断下一步，而应回到项目总表确认：
- 当前阶段
- 当前关键缺口
- 已完成项
- 默认下一动作
- 当前不做什么
- 若现在不继续推进，应回写什么状态

项目总表在项目态任务中的角色，不是辅助表，而是默认入口。
后续计划表、甘特图、聊天记录都不能替代它成为项目态任务的主入口。

**默认要求：**
- 项目态任务默认先看项目总表
- 总表未更新前，不进入新一轮生产派发
- 若本轮只是收束 / 回写，也应先确认总表状态是否与当前判断一致

**当前缺口细项：**
- 当前已把“项目总表是默认入口”立住，但还未形成更完整的项目态入口执行清单
- 总表与部分下位件之间虽已补挂，但后续仍需继续维护一致性，避免再次出现“总表已更新、正文未同步”或反向不同步

### 6. 读取主文 / 当前阶段规则件
在确定任务态并回到总表后，主入口位默认应继续读取：
- 当前主文
- 当前阶段相关规则卡
- 当前阶段相关模板件
- 如有必要，再看上一轮复盘件或审计结论

这里的核心是：
- 项目总表负责告诉你“现在在哪”
- 主文与规则卡负责告诉你“现在应怎么做”

**默认要求：**
- 总表解决阶段定位，主文 / 规则件解决动作口径
- 若当前阶段已有明确规则卡，不应跳过后直接按聊天印象执行

**当前缺口细项：**
- 现有主文、规则卡、模板件之间已开始分层，但尚未形成更完整的“按问题类型读取哪一层”的读取表
- 对不同任务（如编排、复盘、审计、停损）默认先读哪张规则件，尚未完全显式化

### 7. 判断本轮主产物应落在哪一层资产
在锁定具体主产物前，主入口位应先判断本轮产物属于哪一层：
- 主文层
- 规则卡层
- 模板件层
- 编排 / 复盘层
- 能力层 / skill 候选层
- 导航层 / 索引层

这一步的意义是：
- 避免所有产物都挤成平面文档
- 避免主产物明明是规则卡，却被误写成复盘件
- 避免模板件、复盘件、计划件抢主文位置

**默认要求：**
- 资产层级未判断清楚，不锁主产物
- 主产物层级判断先于文档命名与派发动作

**当前缺口细项：**
- 主文正文已补入“资产分层整理”方向，但尚未形成独立执行卡
- 当前团队虽已开始按主文 / 规则卡 / 模板件 / 复盘件分层，但仍缺少一张更明确的“资产层级判定表”

### 8. 锁定主入口 Agent
在当前机制下，未显式交接前，默认不更换主入口。

主入口位一旦锁定，就承担：
- 本轮编排责任
- 主产物收束责任
- 阶段送审责任
- 回写责任

这一步的意义是避免：
- 编排和收束位反复漂移
- 中途换手后责任不清
- 聊天里像是很多人都在推进，但没有唯一主收束位

**默认要求：**
- 未显式交接，不默认换主入口
- 可协助，不并位
- 可审计，不夺位

**当前缺口细项：**
- 主入口位规则已经相对清楚，但尚未形成更正式的“交接条件 / 交接格式 / 交接后责任转移规则卡”

### 9. 锁定当前阶段唯一主产物
主产物不清，不许开工。

主入口位应明确：
- 当前阶段真正要推进的唯一主产物是什么
- 哪些配件只是服务主产物，不得反客为主
- 哪些文档只是说明件，不应充当当前主推进对象

这一步的意义是把“做很多东西”收成“这一轮到底在推进什么”。

**默认要求：**
- 主产物唯一
- 配件只能服务主产物
- 若当前只有配件方向、没有主产物，不进入正式生产

**当前缺口细项：**
- 当前机制线已经明显改善“主文缺位、配件先长”的问题，但配件反客为主仍是需要长期防守的风险点
- 对某些阶段（例如复盘 / 观察 / 导航修订），主产物与配件的界线仍可继续细化

### 10. 预判本轮结果默认回收归位
在正式拆阶段前，主入口位还应提前明确：
- 本轮结果最终默认回收到哪里
- 是并入主文
- 还是形成规则卡
- 还是沉淀为模板件
- 还是记入复盘件
- 还是进入 skill / 注册层候选
- 还是只更新 README / 总表 / 索引

这一步很重要，因为它直接决定：
- 产物出来后是不是还要临时找地方挂
- 本轮是否会因为回收点不清而反复返工

**默认要求：**
- 未明确主挂点，不进入正式生产
- 回收归位口径应在派发前就先想清

**当前缺口细项：**
- 当前已有“回写总表 / 回写主文单元展开表 / README 补挂”的意识，但尚未统一收成“结果默认回收归位规则”
- 对“何时进入 skill / 注册层候选”的判断仍偏经验化

### 11. 做阶段拆分与计划编排
主入口位在进入生产前，应明确：
- 阶段目标
- 送审门
- 放行条件
- 退回条件
- 停损点
- timeout / 卡断恢复口径
- 预算策略与模型策略

如果任务已不是单次动作，而是连续项目推进，必须先落或更新项目总表，把以下内容写清：
- 当前项目处于哪一阶段
- 已完成什么
- 当前关键缺口是什么
- 默认下一步是什么
- 若当前不继续推进，应回写什么状态

**默认要求：**
- 无阶段门，不派发
- 无回收标准，不派发
- 无停损点，不进入长任务

**当前缺口细项：**
- 阶段推进机制已补强，但“更通用轮转规则”仍未沉淀为更高层件
- 某些阶段门仍然更依赖具体试跑案例，而不是更抽象可复用的统一规则表达

### 12. 判断是否需要 spawn
能直接做清楚的，不为协作而协作。

spawn 的判断标准，不是“能不能叫 subagent”，而是：
- 是否真的需要隔离上下文
- 是否真的需要并行或独立施工
- 是否真的需要更低成本 / 更合适模型去完成最小单元
- 主入口位自己做会不会让主线判断变形

**默认要求：**
- spawn 为了主产物，不是为了形式上的 multi-agent 感
- 编排不清时，先补编排，不先 spawn
- 能直接清楚完成的动作，不机械拆给 subagent

**当前缺口细项：**
- 当前机制已经证明“先编排后生产”是必要的，但对“何种场景必须 spawn / 何种场景不应 spawn”的更细分类仍可继续沉淀

### 13. 定义最小单元与模型策略
如果判断需要 spawn，主入口位应明确最小单元：
- `task goal`
- `scope`
- `deliverable`
- `done condition`
- `constraints`
- `return target`
- `suggested model / cost tier`

并同时明确：
- 这是一张施工单，不是把不确定性整体丢给 subagent
- 模型选择应服从成本、质量与阶段需要，而不是默认用最重模型

**默认要求：**
- 最小单元边界不清，不派发
- 没有 return target，不派发
- 成本与模型策略未写清，不进入长链生产

**当前缺口细项：**
- 最小单元模板已经产出，但对模型 / 成本 tier 的更稳定分层规则仍未形成更正式的通用表

### 14. 正式进入生产
到这一步，主入口位才进入正式生产或正式派发。

这里的“进入生产”并不是简单动手，而意味着：
- 任务态已清
- 注册层已读
- 导航入口可回找
- 项目总表已确认
- 主文 / 规则件已回看
- 资产层级已判断
- 主入口已锁定
- 主产物已锁定
- 默认回收归位已预判
- 阶段门 / 停损点 / spawn 判断 / 最小单元都已清楚

只有这样，这一轮生产才不是“凭聊天推进”，而是按结构推进。

### 起手顺序的一句话总规则
> 先判断任务态，再读注册层与项目总表，再锁主入口与主产物，再判断资产层级与回收归位，最后才进入阶段编排与正式生产。

### 起手顺序的默认禁忌
- 不以聊天上下文代替项目总表
- 不以临场灵感代替注册层读取
- 不在主产物未锁定时先产配件
- 不把“能 spawn”误当成“应该 spawn”
- 不把复盘件、计划件、导航件误当作当前主产物的替代物
- 不在回收归位未判断清楚时直接开工

## 五、阶段推进机制

任务进入正式生产后，默认按阶段制推进，而不是按聊天轮次推进。

这一节的目的，不只是回答“阶段怎么走”，还要回答：
- 每一阶段为什么成立
- 阶段内部到底怎样算真实推进
- 阶段何时送审、何时退回、何时停损
- 阶段结束后结果应如何回收、沉淀、上提，而不是只停在“做完了”

对照外部参考来看，阶段推进不应只是施工节奏，
还应承担一条沉淀路径：
> 当前阶段的结果，不只要被做出来，还要被回收、归位，并在必要时上提为规则、模板、能力或复盘资产。

因此，阶段推进机制既是生产机制，也是回收机制、审计机制与沉淀机制。

### 1. 阶段不是聊天轮次，而是正式工作段
一个“阶段”成立，至少意味着：
- 当前阶段目标明确
- 当前阶段唯一主产物明确
- 允许的最小单元范围明确
- 送审门明确
- 放行条件与退回条件明确
- timeout / 卡断恢复口径明确
- 停损点明确

聊天可以承载阶段信息，但聊天本身不构成阶段。
真正构成阶段的是：
- 结构边界
- 主产物边界
- 可送审边界
- 回收边界

**默认要求：**
- 无阶段目标，不成立阶段
- 无主产物，不成立阶段
- 无送审门，不进入正式生产
- 无回收边界，不进入长阶段推进

**当前缺口细项：**
- 当前“阶段”的基本定义已经成立，但对不同类型阶段（生产阶段、复盘阶段、导航修订阶段、审后回转阶段）的共性与差异仍可继续抽象
- 现有阶段门判断仍偏依赖具体试跑案例，尚未形成更高层的通用阶段定义卡

### 2. 阶段开始前，先锁定本阶段唯一主产物
每一阶段开始前，主入口位必须先锁定：
- 本阶段唯一主产物是什么
- 这一阶段做到什么程度可送审
- 哪些配件只是服务它
- 哪些内容明确不在本阶段处理

这一点和外部参考的“文档即上下文、流程要稳定入口”是一致的。
阶段不锁主产物，后续所有派发、回收、送审都会漂移。

**默认要求：**
- 不锁主产物，不开阶段
- 配件不能反向定义阶段目标
- 当前阶段若只有说明件而无主产物，不视为正式推进阶段

**当前缺口细项：**
- 对某些偏治理型阶段（例如导航修订、复盘收口），主产物和辅助件之间的边界仍有继续细化空间
- 对“阶段性说明文档”什么时候可作为主产物、什么时候只能算配件，尚未形成统一判断卡

### 3. 阶段允许范围必须先圈清
阶段开始后，并不意味着本阶段什么都能做。
主入口位应先圈清：
- 这一阶段允许哪些最小单元进入
- 哪些单元当前不许进入
- 哪些问题属于下一阶段才处理
- 哪些问题虽然相关，但暂时只记入缺口，不并行展开

这一点很关键，因为它直接决定：
- 阶段会不会失焦
- 配件会不会过早膨胀
- 审计时会不会发现阶段定义被偷宽

**默认要求：**
- 阶段范围未圈清，不派单
- 与当前主产物弱相关的内容，默认不并入阶段主链
- 新发现的重要议题，可立入口，但不自动并行拉开新主线

**当前缺口细项：**
- 当前已经在实践“立入口、暂不并行扩写”，但还没有更统一的“阶段范围管理口径卡”
- 对“发现新问题时是立缺口卡还是直接并入当前阶段”的判断，仍偏经验化

### 4. 阶段内标准推进链必须完整
阶段内的标准推进链，不应省略为“派了单 → 产物来了”。
完整口径应是：
1. 主入口 Agent 锁定本阶段唯一主产物
2. 主入口 Agent 派发最小单元
3. subagent 返回结果
4. 主入口 Agent 回收并判断是否能并入主产物
5. runtime keeper 检查是否出现 timeout、卡断、假推进或主产物停长
6. 达到阶段门后送审
7. 审计位给出 `accepted / revision-needed / blocked`
8. 主入口 Agent 决定进入下一阶段、回本阶段重编排、停损并升级、或阶段性收口后进入复盘沉淀

阶段真正的风险，从来不只在“有没有做”，而在：
- 有没有回收
- 有没有并入
- 有没有送审
- 审后能不能继续接链

**默认要求：**
- 无正式回收，不算阶段推进
- 结果未并入主产物，不算阶段增长
- 审计结论出来后无下一动作，不算阶段闭环

**当前缺口细项：**
- 审后回转链已完成最小验证，但更通用的“审后恢复推进规则”仍未沉淀为高层规则件
- 对不同阶段结束时的默认下一动作分类，还没有单独整理成统一回转表

### 5. 阶段内的“真实推进”必须可判断
阶段推进不能靠“看起来很忙”来判断。
对正式生产阶段而言，真实推进至少包括：
- 主产物有实质增长
- 新内容已并入主产物
- 主产物的关键阻塞被消除
- 主产物更接近送审状态

探针验证、排障、反例排除、导航补挂、说明校正等动作可以算辅助推进，
但不应长期替代主产物增长。

这一点与外部参考里“流程资产化”的逻辑一致：
- 过程动作有价值
- 但只有进入主承载位，才构成真正沉淀

**默认要求：**
- 忙碌不等于推进
- 辅助动作连续多轮替代主增长时，应触发断层检查或停损判断
- 若阶段内只有配件增长、没有主产物增长，不应自判阶段健康

**当前缺口细项：**
- 当前虽然已明确“主产物优先”，但对“辅助推进可持续多久后必须触发预警”还没有更细阈值
- 假推进识别仍主要依赖 runtime keeper 提醒与主入口判断，尚未沉淀为更细的预警表

### 6. 阶段送审门必须前置，而不是事后补
阶段送审不是阶段后期临时想到的动作，而应在阶段开始时就已明确：
- 到什么程度可送审
- 审计位看什么
- 什么情况判 `accepted`
- 什么情况判 `revision-needed`
- 什么情况判 `blocked`

送审门前置的意义在于：
- 阶段内的生产不会无底洞式补细节
- 主入口位知道“做到哪里该停、该送审”
- 审计位也不会被迫对模糊对象做裁决

**默认要求：**
- 无送审门，不进入正式阶段推进
- 若审计对象未成形，不应强行送审
- 阶段成熟后应送审，不应无限加细节拖延

**当前缺口细项：**
- 阶段门模板已产出，但不同类型阶段的送审标准仍可继续分层整理
- 当前 `accepted / revision-needed / blocked` 的最小判断口径已经有了，但“按阶段类型拆分的送审标准”仍未形成更细表述

### 7. revision-needed 不是失败，而是阶段内正式回转口
阶段推进机制中，`revision-needed` 不应被理解为“这一轮没用”，而应被理解为：
- 审计指出当前阶段仍需回本阶段重编排
- 主入口位应依据结论正式接链
- 当前阶段的目标、主产物、回收与送审标准需要被重新收紧

这点在我们最近的试跑里已经被验证过：
- 最小闭环试跑先经历 `revision-needed`
- 审后回转链再被最小验证为成立

**默认要求：**
- `revision-needed` 后不允许停在说明层
- 必须明确：回哪一层、补什么、谁接链、何时重送审

**当前缺口细项：**
- 当前已验证“审后回转最小成立”，但尚未形成更通用的 revision-needed 回转卡
- 不同类型 revision-needed（证据不足型、结构失真型、范围偷宽型）的回转路径尚未细分

### 8. blocked 应被理解为升级口，而不是坏消息
`blocked` 的意义不是“这事完了”，而是：
- 当前阶段继续按原方式推进的收益已经明显下降
- 需要转入升级判断、边界外拍板或更重路径

它是机制中的正式出口之一，
与“进入下一阶段”“回本阶段重编排”并列，而不是附带说明。

**默认要求：**
- `blocked` 后不继续假推进
- 必须明确：升级给谁、卡在什么边界、下一步等待什么决策

**当前缺口细项：**
- blocked 升级链仍未完成真实试跑验证
- 缺少更明确的 blocked 升级卡 / 升级后分流表

### 9. runtime keeper 在阶段里不是旁观者，而是运行态接口位
在阶段推进中， runtime keeper 的职责不是替主入口做决定，而是：
- 识别卡断
- 提醒 timeout
- 暴露假推进
- 提醒阶段悬空
- 做异常分流提醒

这意味着，阶段机制要给 keeper 一个正式位置，
否则 keeper 会长期停留在定义层，无法进入真实协作证据层。

**默认要求：**
- keeper 不代替审计、不代替编排、不代替拍板
- keeper 的作用是“把断线和假推进暴露出来”，而不是自行接管主线

**当前缺口细项：**
- keeper 介入的角色边界已较清楚，但当前缺少真实任务场景，尚未形成真实介入证据
- 缺少“keeper 触发了提醒后，主入口位默认该如何响应”的更细表述

### 10. 阶段结束时必须三选一，不允许悬空
一个阶段结束时，至少必须进入以下三种之一：
- 进入下一阶段
- 回本阶段重编排
- 阶段性收口并升级 / 复盘

阶段最危险的状态不是“出错”，而是：
- 看似结束
- 实际没有下一动作
- 没有人回写
- 也没有正式进入复盘或升级

**默认要求：**
- 阶段结束必须有结论
- 无结论的“先放这儿”不算阶段完成
- 审后无动作、回收后无回写、阶段成熟后无人送审，都应视为阶段悬空

**当前缺口细项：**
- 当前这条线已经通过试跑部分补上“审后无动作”的缺口，但对“阶段结束后的标准回写格式”仍未形成统一表
- 对收口后进入复盘 / 缺口卡 / 资产上提链的默认出口，仍可继续明确

### 11. 阶段结果必须回收，不只完成，还要归位
对照外部参考，阶段真正成熟，不只是阶段做完了，
还包括阶段结果有没有被放进正确的资产层。

一个阶段结束后，至少应判断：
- 结果是否并入主文或当前主承载位
- 是否应形成规则卡
- 是否应形成模板件
- 是否应进入复盘件
- 是否只需更新总表 / README / 索引
- 是否已达到进入 skill / 注册层候选的条件

这一步是我们和一般“任务管理线”最不同的地方。
因为我们要的不是只把任务做完，而是把经验放回可复用结构里。

**默认要求：**
- 阶段结果未归位，不算阶段真正完成
- 有效方法若已稳定，不应长期只停在单次复盘或聊天结论里

**当前缺口细项：**
- 当前已有“主文 / 规则卡 / 模板件 / 复盘件 / 注册层候选”的分层意识，但还没有统一的“阶段结果归位判定表”
- 对何种结果应进入 skill 候选、何种结果只保留为证据，仍偏经验化

### 12. 阶段推进的一句话总规则
> 阶段不是按聊天轮次往前滚，而是按“锁主产物 → 派发 → 回收 → 并入 → 送审 → 回转 / 升级 / 收口 → 归位沉淀”这条正式链路推进。

### 阶段推进的默认禁忌
- 不把聊天活跃误判为阶段推进
- 不把配件增长误判为主产物增长
- 不把送审拖延当作稳妥
- 不把 revision-needed 误当失败后搁置
- 不把 blocked 当坏消息后继续硬拖
- 不把阶段结束时的“先放着”当作正式收口
- 不让阶段结果只停在说明层，不进入回收归位与沉淀链

## 六、最小单元原则

最小单元的统一口径是：

> 在当前机制下，最小单元默认按“一个最小单元 → 一个 prompt → 一个 subagent → 一个可回收产物”组织。

但这句话本身还不够。
对照外部参考来看，最小单元不只是施工拆分问题，
它还同时关系到：
- 上下文厚度能否被控制
- 阶段回收是否稳定
- 经验是否能从单次动作上提为模板、规则或能力资产

所以“最小单元原则”要解决的，不只是“怎么拆”，还要解决：
- 什么才配叫最小单元
- 什么不该被当成最小单元派发
- 最小单元的结果如何回收、归位、复用
- 何时它会从一次性单元上提为模板件、规则件或注册层候选

### 1. 最小单元的基本定义
最小单元不是“任何一小块工作”，而是：
- 目标单一
- 产物单一
- 边界清楚
- 可独立回收
- 失败成本可控

真正的最小单元，应该允许主入口位在不重新理解整个项目的情况下：
- 派发它
- 回收它
- 判断它能否并入主产物
- 判断失败后是重做、缩小、还是回本阶段重编排

**默认要求：**
- 目标不单一，不算最小单元
- 产物不单一，不算最小单元
- 无法独立回收，不算最小单元
- 失败后牵连整阶段大面积返工的，也不应直接当作最小单元

**当前缺口细项：**
- 虽然“最小单元模板”已经产出，但当前对“什么工作只适合留在主入口位、不适合被派成单元”的反面判断还不够完整
- 对治理型任务、复盘型任务、导航修订型任务的最小单元边界仍可继续细化

### 2. 最小单元的作用，不只是拆任务，而是减厚上下文
最小单元机制的一个核心作用，是避免：
- 主入口位把一大团不确定性整体丢给 subagent
- subagent 在厚上下文里自行扩题
- 回收时主入口位需要重新吞整团内容

因此，最小单元真正应该做到的是：
- 让 subagent 只处理本轮必须处理的一小段
- 让主入口位回收时只判断这一小段能否并入主产物
- 让失败范围被限制在一个可控边界内

这与外部参考里“把经验和流程压成可控资产”的逻辑是一致的。
不是让每个执行位变强到能吞下全部复杂度，
而是让复杂度被提前切开。

**默认要求：**
- 最小单元的边界应尽量小到主入口位可直接回收判断
- 不允许把整个不确定问题打包成一个“请你自己想清楚”的伪单元
- 不允许把本应由主入口位完成的阶段判断整体下放给 subagent

**当前缺口细项：**
- 当前已在口径上明确“prompt 不是整体下放不确定性”，但对哪些场景最容易伪装成最小单元，仍可继续总结负面样例
- 对“上下文厚度过高时应如何进一步拆细”的二级拆分规则还未正式形成

### 3. 最小单元必须是可回收单元，而不是只可执行单元
一个单元能被执行，并不意味着它就是合格的最小单元。
合格的标准还包括：
- 主入口位拿到结果后，知道该怎么接
- 结果的归属层级明确
- 能判断是否并入主产物
- 能判断是否只记入复盘或转成规则件 / 模板件候选

这也是我们和很多“派单型多 agent 协作”最大的区别。
我们不要“有人去做了”，而要“做完后能稳定收回来”。

**默认要求：**
- 无 return target，不派发
- 回收口径不清，不派发
- 结果回来了却不知道挂哪，不算合格最小单元

**当前缺口细项：**
- return target 已作为模板字段存在，但“回收后默认挂哪一层”的判断仍常依赖主入口位临场经验
- 缺少更统一的“最小单元结果归位判定口径”

### 4. 每个最小单元至少应具备固定字段
每个最小单元至少应具备：
- `task goal`
- `scope`
- `deliverable`
- `done condition`
- `constraints`
- `return target`
- `suggested model / cost tier`

这些字段的意义分别是：
- `task goal`：回答这单到底干什么
- `scope`：回答边界到哪，不做什么
- `deliverable`：回答最后交回来什么东西
- `done condition`：回答做到什么程度算完成
- `constraints`：回答不能越过哪些边界
- `return target`：回答结果回收到哪里
- `suggested model / cost tier`：回答该用什么成本级别完成

**默认要求：**
- 缺字段不派发
- 关键字段由主入口位负责写清，不允许交给 subagent 自补全成阶段判断

**当前缺口细项：**
- 当前模板已能覆盖基础字段，但字段之间的优先级和冲突处理规则尚未明确
- 对不同单元类型（文档型、审计型、归档型、模板型）的字段差异，还未形成补充字段表

### 5. prompt 是施工指令，不是不确定性的垃圾桶
最小单元配套的 prompt，应被理解为：
- 施工指令
- 当前单元边界说明
- 输出要求说明

而不是：
- 把主入口位还没想清楚的问题整体扔给 subagent
- 让 subagent 替主入口位完成本应先完成的任务态判断、阶段判断、主产物判断

换句话说：
> prompt 的职责是让一个已经被定义好的单元稳定执行，而不是替代定义动作本身。

**默认要求：**
- prompt 解决施工，不解决主线定义
- 主入口位未完成的结构判断，不得整体下放给 prompt
- 一条 prompt 如果已经在事实上承担“主入口思考位”，说明单元定义失败

**当前缺口细项：**
- 当前已经有“最小单元模板”，但还缺少一张更明确的“伪单元 / 伪 prompt 识别表”
- 对“哪些问题必须留在主入口位思考，不应 prompt 化”的边界还可继续细化

### 6. subagent 是一次性施工单，不是并行常驻主脑
在当前机制下，subagent 的职责是：
- 承接一个已定义的最小单元
- 产出一个可回收结果
- 完成后退出主链

这意味着：
- subagent 不应自行扩题
- 不应自行改阶段目标
- 不应自行决定进入下一阶段
- 不应自行承担主线收束

外部参考里“可复用资产层”的启发也反向说明了这点：
长期能力应沉淀成规则、模板、skill，而不是把一次性执行位长期神化成常驻主脑。

**默认要求：**
- subagent 解决施工，不代替主入口持续思考
- 一次性单元默认完成即退出，不在主链中长期悬挂

**当前缺口细项：**
- 当前角色边界已较清楚，但对“什么情况下允许 subagent 连续承接多个紧邻单元”还没有更细口径
- 对“连续承接多个单元”何时会演变成伪主脑，也还缺负面样例总结

### 7. 模型选择跟着单元难度、风险与返工成本走
最小单元原则里，模型选择不是附属细节，而是单元边界的一部分。

同一个任务，如果最小单元划得清：
- 低成本模型可承担更多简单施工单
- 重模型只用于高风险、高歧义、高返工成本单元

这与外部参考里“流程资产化、工具层分工”的思路是一致的：
不是所有动作都用最重资源，而是按单元层级做资源匹配。

**默认要求：**
- 模型选择跟着单元走，不跟着习惯走
- 单元边界越清，越应优先用更低成本路径
- 只有在高歧义、高风险、高返工成本场景下，才优先重模型

**当前缺口细项：**
- 当前已有 `suggested model / cost tier` 字段，但尚未形成更正式的 tier 对照表
- 对“什么类型单元应该默认轻模型 / 重模型”的经验还未收成规则件

### 8. 最小单元应优先服务主产物，而不是服务配件膨胀
单元拆得再精，如果它服务的是配件膨胀，而不是当前主产物推进，那也不算好单元。

一个好单元，至少应满足：
- 它的结果能直接并入当前主产物，或
- 它在最短路径上为主产物过阶段门服务

这和外部参考强调的“结构稳定、入口稳定、资产分层”是同一个方向：
- 单元的价值，不看它是否显得忙
- 而看它是否让主承载位更接近成立

**默认要求：**
- 与主产物弱相关的单元，不应优先派发
- 配件若不能直接服务主产物，不应先于主线单元启动

**当前缺口细项：**
- 当前虽然已有“主产物优先”原则，但对“某个单元是否足够服务主产物”的前置判断，还缺少更简洁的检查口径
- 对“未来阶段配件提前生长”的识别仍较依赖主入口位经验

### 9. 最小单元不是永久定义，稳定后应考虑上提
如果某类最小单元反复出现、结构稳定、边界清楚、回收顺畅，
那它就不应长期只停在“一次性施工单”层。

后续默认应考虑它是否要上提为：
- 模板件
- 规则卡
- 提示词工作流
- skill 候选
- 注册层现役能力

这正是外部参考最值得吸收的点之一：
一次性动作不应无限重复；
稳定动作应从“单次执行”上提到“可复用资产”。

**默认要求：**
- 反复出现且稳定的单元，不应长期停留在临时派单状态
- 一旦已证明稳定，应至少进入模板件 / 规则件 / skill 候选评估

**当前缺口细项：**
- 当前主文已补入“资产分层整理 / 注册层整理”方向，但最小单元到资产上提的路径尚未形成单独规则卡
- 对“出现几次、稳定到什么程度，才值得上提”的阈值仍未明确

### 10. 最小单元的一句话总规则
> 最小单元不是把任务切小而已，而是把复杂度切到可派发、可回收、可并入、可复用的程度；切不清的复杂度，不应假装已经被定义好。

### 最小单元的默认禁忌
- 不把厚问题伪装成小单元
- 不把主入口位未完成的结构判断整体丢给 subagent
- 不把“能执行”误当成“可回收”
- 不把 prompt 当作不确定性的垃圾桶
- 不把 subagent 当作并行常驻主脑
- 不让反复稳定的单元永远停留在一次性派单状态

## 七、主产物优先原则

任何阶段都必须先保主产物，再谈配件；不能让配件增长替代主线推进。

对照外部参考来看，这一节要解决的，不只是“主文比配件重要”这种抽象判断，
而是更具体地回答：
- 当前阶段真正的主承载位是谁
- 什么增长才算主线增长
- 配件什么时候是帮主产物过关，什么时候已经开始反客为主
- 阶段结果最终应优先沉淀到哪个主承载位，而不是散成平面材料

### 1. 主产物是当前阶段唯一主承载位
主产物不是“当前最重要的一份文档”这么简单，
而是当前阶段的唯一主承载位。它至少要同时满足：
- 承载当前阶段主要目标
- 成为后续阶段继续生长的主要对象
- 成为审计位判断阶段是否成立的主要对象
- 直接代表主线有没有推进

**默认要求：**
- 每阶段只允许一个唯一主产物
- 无唯一主产物，不进入正式生产
- 若多个产物都看似重要，主入口位必须先做主次裁断

**当前缺口细项：**
- 当前虽然已经建立“唯一主产物”意识，但对治理型阶段、复盘型阶段、导航修订阶段的主承载位判定仍可继续细化
- 缺少更统一的“主产物判定表”

### 2. 主产物优先，不等于只允许主文存在
主产物优先的意思不是“除了主文都不许长”，
而是：
- 当前阶段必须有一个主承载位
- 其他配件、模板、规则卡、复盘件都只能围绕它服务
- 不能反过来把主产物挤成配角

**默认要求：**
- 可长配件，但配件必须解释自己如何直接服务主产物
- 无法解释服务关系的配件，不应优先启动

**当前缺口细项：**
- 当前已在口径上承认“配件可以长”，但对“服务关系如何判断”的检查口径仍可继续抽象
- 对“说明件 / 配件 / 主产物”之间的界线，尚未形成快速判定卡

### 3. 什么才算主线推进
对正式生产阶段而言，主线推进主要包括：
- 主产物有实质增长
- 新内容已经并入主产物
- 主产物关键阻塞被消除
- 主产物更接近送审状态

探针验证、排障、反例排除、导航补挂、说明校正等动作，可以算辅助推进，
但不能长期替代主线增长。

**默认要求：**
- 配件增长不等于主线推进
- 辅助动作连续多轮替代主产物增长时，应触发预警
- 主产物长时间不长，不应自判阶段健康

**当前缺口细项：**
- 当前仍缺“辅助推进持续多久后必须预警”的更细阈值
- 对“主产物长得慢但阶段依旧成立”的特殊例外，尚未形成细分判断

### 4. 配件只能服务主产物过关，不能替未来阶段提前铺陈
这一条是当前轻治理线里最需要长期防守的原则之一。

配件可以长，但前提是：
- 它真的在帮当前主产物过阶段门
- 它不是替未来阶段提前大面积铺陈
- 它没有开始抢夺当前阶段的注意力、预算和判断资源

**默认要求：**
- 与当前阶段送审门弱相关的配件，不应优先启动
- 未来阶段需要的材料，可立入口，但不自动并入当前主线

**当前缺口细项：**
- 当前虽然已把“先立入口、不并行扩写”实践出来，但对“未来阶段提前铺陈”的识别仍偏经验化
- 缺少一张“配件反客为主预警表”

### 5. 预算、模型与回收资源都应优先保证主产物
主产物优先原则不只作用在“写什么”，也作用在：
- 预算分配
- 模型选择
- 回收精力
- 导航维护
- 审计资源

也就是说：
- 预算优先保证主线，不为外围热闹买单
- 重模型优先给主线高风险节点，不优先给配件抛光
- 回收精力优先给主承载位，而不是把时间都耗在配件整理上

**默认要求：**
- 资源分配跟着主产物走
- 配件若开始显著挤占主线资源，应触发收缩判断

**当前缺口细项：**
- 当前已有成本意识与模型策略字段，但“主线资源优先级规则”尚未单独成卡
- 对“配件吸走预算”的识别口径还不够显式

### 6. 主产物应是阶段结果默认优先归位点
阶段结束后，最优先被更新、被并入、被回收的，应是主产物本身。
规则卡、模板件、复盘件、README、索引件可以随后更新，
但不应倒过来替代主产物成为阶段结果的主要落点。

这和外部参考里“经验要进入稳定资产层”的逻辑一致：
- 先进入正确主承载位
- 再决定是否分流到其他资产层

**默认要求：**
- 阶段结果默认先看能否并入主产物
- 若不能并入主产物，应明确说明为什么它仍然构成当前阶段主推进

**当前缺口细项：**
- 当前已有回写意识，但“主产物优先归位规则”还没有单独抽出来
- 对哪些结果应先并主文、哪些应先成规则卡，仍存在经验判断空间

### 7. 主产物稳定后，才有资格带动资产上提
当前阶段如果连主产物都没立住，就不应过早讨论：
- skill 化
- 注册层化
- 更大规模模板化

真正合理的顺序应是：
- 先让主产物成立
- 再判断哪些稳定做法值得从当前主产物中抽出
- 再把稳定方法上提为规则、模板、prompt、skill 候选或注册层能力

**默认要求：**
- 主产物未立住时，不优先讨论大规模资产上提
- 资产上提以前，应先确认主线已经稳定

**当前缺口细项：**
- 当前已在正文中补入“资产分层整理 / 注册层整理 / 复盘制度上提”，但“主产物成立到资产上提”的顺序关系仍可继续强调
- 缺少一张“资产上提前置条件表”

### 8. 主产物优先的一句话总规则
> 当前阶段先保唯一主承载位，再允许配件围绕它服务；不能让配件的热闹、说明的完整、或未来的铺陈，替代主线真实成立。

### 主产物优先的默认禁忌
- 不把配件增长当成主线推进
- 不让未来阶段材料提前反客为主
- 不让预算、模型和回收精力被配件大量吸走
- 不让 README / 说明件 / 导航件替代主产物成为阶段主承载位
- 不在主产物尚未成立时过早讨论大规模资产上提

## 八、断层点检查

断层是指协作链在关键接口上失去连续性，导致任务虽然看似还在动，但已经无法稳定地派发、回收、并入、送审、推进下一阶段或进入复盘沉淀。

对照外部参考来看，断层点检查不能只看“有没有卡住”，
还应看：
- 哪一层资产没有被正确接上
- 哪一段工作流没有进入可回找、可回收、可上提的链路
- 问题是阶段内断线，还是沉淀链断线

### 1. 当前重点检查 7 类断层
当前重点检查：
1. 需求对齐 → 计划编排
2. 计划编排 → 单元派发
3. 单元派发 → 结果回收
4. 结果回收 → 并入主产物
5. 阶段形成 → 送审进入
6. 审计结论 → 恢复推进
7. 阶段完成 → 复盘沉淀

这 7 类断层的共同点是：
- 它们都位于正式接口上
- 一旦断掉，聊天仍可能继续，但机制已经开始失真

**默认要求：**
- 断层点必须被点名，不允许模糊表达成“有点不顺”
- 一旦定位断层，应先判断断在哪个接口，而不是先怪执行位或模型

**当前缺口细项：**
- 当前这 7 类断层已较稳定，但对“导航层断层 / 资产归位断层 / 注册层调用断层”的补充分类还可继续完善
- 缺少更显式的断层分类卡

### 2. 统一断层预警信号
统一预警信号包括：
- 需求已清，但没有计划
- 计划已出，但迟迟不能派单
- 单元已派，但没有正式回收
- 回收有了，但主产物不长
- 阶段已成熟，但没人送审
- 审计已结束，但没有进入下一动作
- 任务结束了，但没有复盘沉淀
- README / 总表 / 索引已该更新，但主入口位没有回写动作
- 重复问题不断出现，但没有被对象化或上提

**默认要求：**
- 预警信号一旦连续出现，不应继续按“只是忙”理解
- 同类预警重复出现时，应从一次性问题转为治理性问题判断

**当前缺口细项：**
- 当前已能识别部分断层，但对“连续出现多少次应升级为治理性问题”尚未形成阈值
- 对导航层 / 资产层断层的预警信号仍未完全结构化

### 3. 断层后先接链，不先解释
断层出现后的默认动作，不应是：
- 先解释为什么会这样
- 先继续假装推进
- 先把锅推给执行位

而应是：
1. 先标记断层点
2. 判断断在哪一层
3. 由对应角色先接链
4. 再决定恢复原阶段、回本阶段重编排，或停损并升级

**默认要求：**
- 断层后先恢复连续性，再讨论责任与原因
- 无接链动作的解释，不算处理

**当前缺口细项：**
- 当前已在试跑里验证过部分断层补口，但“各类型断层默认由谁先接链”的更细分工尚未形成表格
- 缺少断层恢复后如何留痕的统一格式

### 4. 断层检查不只服务当前轮，也服务后续复盘制度
断层点检查的价值，不应只停留在“这轮补上了”，
还应进入：
- 复盘文档
- 问题对象化
- 高频问题统计
- 规则 / 模板 / skill 候选的后续上提判断

也就是说，断层如果反复出现，不应每次都只作为当前轮事故处理，
而应逐步成为治理资产的一部分。

**默认要求：**
- 同类断层重复出现时，应进入复盘制度的高频问题跟踪
- 断层若已有稳定补口方法，应评估是否上提为规则卡或模板件

**当前缺口细项：**
- 当前复盘制度上提方向已补入正文，但断层数据如何周期整理仍未形成正式制度件
- 缺少“断层 → 问题卡 / 规则卡 / 模板件”的上提路径卡

### 5. 断层点检查的一句话总规则
> 断层不是“看起来慢了”，而是正式接口失去连续性；断层后先接链，再恢复，再决定是否上提为治理问题。

### 断层点检查的默认禁忌
- 不把断层误说成普通延迟
- 不在未定位接口前先归因甩锅
- 不让聊天继续活跃掩盖正式链路已断
- 不把重复断层永远当作一次性偶发问题

## 九、停损点与防虎头蛇尾机制

停损点，是指当协作继续按当前方式推进，新增成本已明显高于新增收益时，系统应停止硬拖、停止假推进，转入重编排、恢复或升级判断的正式节点。

对照外部参考来看，停损并不是“放弃”，而是：
- 阻止低质量重复劳动继续堆积
- 保护主产物和主承载位
- 为后续回收、复盘、资产上提留出清晰结构

### 1. 停损判断的典型触发信号
典型信号包括：
- 主产物连续不增长
- 小单元错误开始牵连整阶段返工
- 回收成本显著高于继续生产收益
- 审计连续指出同类结构问题
- runtime keeper 连续发出卡断 / timeout / 假推进提醒
- 预算消耗开始偏离主线目标
- 任务目标已变化，但计划未重编排
- 配件开始显著反客为主
- 结果虽然越来越多，但没有稳定归位

**默认要求：**
- 触发信号连续出现时，不继续硬拖
- 停损提出应被正式记录，而不是只在聊天中口头表达

**当前缺口细项：**
- 当前已形成停损规则卡，但更细的触发阈值仍可继续明确
- 对“连续多少次 / 多长时间”进入正式停损判断，仍缺统一表述

### 2. 停损提出与裁定机制
- 主入口 Agent 可以提出停损
- runtime keeper 可以发出停损提醒
- 审计位可以提出停损建议
- 人类拍板位可以做最终停损裁定

这一层的意义是：
- 停损不是谁情绪上不想做了
- 而是正式协作角色共同识别“继续当前路径已不划算”

**默认要求：**
- 停损理由应指向结构问题、成本问题、边界问题或主线偏离问题
- 无明确停损理由，不轻易进入停损裁定

**当前缺口细项：**
- 当前角色责任已大致明确，但“停损裁定记录格式”仍未单列成卡
- 对不同停损类型（成本型、结构型、边界型、目标漂移型）的裁定差异仍可继续细分

### 3. 停损后的默认路径
停损后的默认路径包括：
- 回本阶段重编排
- 缩小粒度后恢复推进
- 暂停并升级
- 阶段性收口后转复盘

这一步很关键，因为停损不是结束，而是正式分流口。

**默认要求：**
- 停损后必须明确进入哪条路径
- “先停一下看看”不算停损收口
- 若进入暂停，应明确等待什么条件恢复

**当前缺口细项：**
- 当前停损规则已初步成立，但“停损后的恢复分流细则”仍未形成下位件
- 对暂停后恢复的前置条件还没有更显式的恢复表

### 4. 防虎头蛇尾的关键不是催快，而是强制收束
虎头蛇尾不是因为大家不努力，
而是因为：
- 阶段结束时没有正式结论
- 阶段结果没有被回收归位
- 缺口没有进入后续链路
- 说明越来越多，结构越来越少

因此，防虎头蛇尾的关键不是催快，而是要求每个阶段结束时必须正式三选一：
- 进入下一阶段
- 回本阶段重编排
- 阶段性收口并升级 / 复盘

**默认要求：**
- 阶段结束无结论，不算结束
- 无回收归位，不算收口
- 无后续路径，不算完成

**当前缺口细项：**
- 当前已通过试跑部分补强“审后无动作”问题，但对“阶段性收口的标准回写模板”尚未形成单独件
- 对收口后如何进入复盘、问题对象化与资产上提链，仍缺统一出口卡

### 5. 停损结果也应进入沉淀链
停损不应只作为“这次没做成”的说明。
它应进一步进入：
- 复盘件
- 问题卡
- 高频问题统计
- 规则 / 模板补丁判断
- 后续 skill / 注册层边界判断

换句话说，停损本身也是经验。
如果一次停损暴露了稳定结构问题，后续就不应再只作为聊天记忆存在。

**默认要求：**
- 停损后至少应进入复盘或缺口记录
- 重复停损原因应上提为治理问题，而不是反复口头说明

**当前缺口细项：**
- 当前“全局复盘整理制度”已立入口，但停损数据如何进入周期整理仍未形成制度件
- 缺少“停损原因 → 问题对象 → 规则补丁”的结构化上提路径

### 6. 停损点的一句话总规则
> 停损不是放弃，而是当继续当前路径已明显不划算时，正式停止硬拖、停止假推进，并把任务转入重编排、升级或复盘沉淀的结构口。

### 停损点的默认禁忌
- 不把停损当作情绪化中断
- 不在已明显不划算时继续硬拖
- 不把“先放着”当作停损后的正式收口
- 不让停损结果只停在聊天说明层

## 十、下位件方向

后续若需要细化，优先往下位件拆，不再默认只往主文继续堆厚。

对照外部参考来看，下位件的意义不只是“主文拆小”，
而是把已经反复出现、结构稳定、值得重复调用的内容，逐步变成：
- 规则卡
- 模板件
- 提示词工作流
- skill 候选
- 注册层现役能力

所以“下位件方向”实际上承担的是：
> 主文如何把稳定经验下沉为可复用资产。

### 1. 当前优先下沉的下位件
当前优先往下拆：
- 最小单元模板
- 阶段门模板
- 停损规则卡
- 审计插入规则

这些内容适合作为第一批下位件，是因为它们已经：
- 重复出现
- 结构相对稳定
- 明显具备复用价值

**默认要求：**
- 主文里反复被引用、反复被实践验证的结构，优先考虑下沉
- 不把还未稳定的内容过早做成重模板

**当前缺口细项：**
- 当前虽已有部分下位件，但挂载链路、资产层级与继续下沉路径仍可继续整理
- 对“什么程度算足够稳定，值得下沉”为独立件的阈值仍未完全明确

### 2. 下位件不是越多越好，而是要减少重复解释成本
下位件的目标，不是文档数量增长，
而是：
- 降低主文厚度中的重复解释
- 降低主入口位每轮重讲成本
- 提高复用与回找效率

这点和外部参考中 `documents / prompts / skills / tools / README` 的分层很一致：
真正的结构不是堆料，而是降低未来重复劳动。

**默认要求：**
- 下位件若不能显著减少重复解释成本，不应为了完整感而硬拆
- 新下位件应同时说明其服务哪一层主文逻辑

**当前缺口细项：**
- 当前已有一些规则件和模板件，但“是否真正减少了未来解释成本”的回看还未系统进行
- 缺少“下位件是否值得保留 / 合并 / 上提”的周期性回看机制

### 3. 下位件应有明确层级出口
下位件不应都停留在同一层文档里。
后续应更清楚地区分：
- 哪些只是规则卡
- 哪些应变成模板件
- 哪些适合组织为提示词工作流
- 哪些已达到 skill 候选程度
- 哪些应进入注册层清单

这一步是吸收外部参考最关键的地方之一。
它提醒我们：
- 稳定经验的终点，不该只是“又多一份文档”
- 而应进入合适的长期资产层

**默认要求：**
- 下位件形成后，应继续判断其最终长期层级
- 不让已经稳定的高频方法长期停留在一次性说明件层

**当前缺口细项：**
- 当前已在正文补入“资产分层整理”方向，但缺少“下位件层级出口表”
- 对 prompt 化 / skill 化 / 注册层化 的上提条件仍偏经验化

### 4. 下位件也应进入导航与索引链
下位件一旦形成，如果不能被稳定找到、正确调用、被主入口位优先读到，
那它的复用价值会大幅下降。

所以：
- README
- 项目总表
- 主文头部挂载
- 同位 / 下位文档关系
都应随着下位件形成而同步更新。

**默认要求：**
- 新下位件形成后，应同步补挂导航与索引
- 无法回找的下位件，不视为真正完成沉淀

**当前缺口细项：**
- 当前关键文档导航已补明显，但全量逐份检查仍未彻底做完
- README 还未完全演化为分层导航中心

### 5. 下位件方向的一句话总规则
> 下位件不是把主文拆碎，而是把已经稳定、反复出现、值得复用的结构，下沉到正确的长期资产层，减少未来重复解释与重复发明。

### 下位件方向的默认禁忌
- 不为显得完整而过早拆出重模板
- 不把不稳定内容提前固化成长期件
- 不让下位件形成后继续悬空、无导航、无层级出口


## 十一、当前结论

### 11.1 当前默认入口仍是项目总表
项目态任务的默认入口，仍应收敛到项目总表。
后续计划表、甘特图、聊天记录都不替代项目总表，只作为辅助视图或补充说明。

### 11.2 当前产物应按资产层级理解
为避免文档继续堆成平面列表，当前这条线的产物默认按以下层级理解：
- **主文层**：负责定义机制主线与判断边界
- **规则卡层**：负责承接关键规则、默认动作、边界与例外
- **模板件层**：负责支撑重复调用与最小单元复用
- **编排/复盘层**：负责记录试跑、回收、断层与验证结果
- **注册层 / skill 候选层**：负责沉淀已证明稳定、可复用、应被优先调用的能力

这不是为了增加结构负担，而是为了让已有产物更容易被回找、复用与上提。

### 11.3 注册层整理应作为后续正式方向
当前机制已经暴露出一个清晰事实：
如果没有更明确的系统级注册层，主入口位仍容易退回聊天式推进。

因此，后续应逐步补实：
- 现役能力清单
- 默认调用顺序
- 废弃入口标记
- 冲突时先调谁
- 项目态任务默认先回项目总表

当前这层方向已通过 `系统级注册与默认起手顺序规则卡_v0.1` 初步承接，但还未形成更完整注册层整理件。

### 11.4 复盘制度应从单项目上提到周期整理层
当前复盘已经不应只服务于单条机制线。
后续更合理的方向是：
- 按周期整理复盘文档
- 统计高频问题与重复断层
- 识别哪些优点或有效动作值得上提为机制
- 将经验从“记录过”推进到“可复用”

当前这一方向已立入口：`全局复盘整理制度缺口卡_v0.1`，但暂不与本主线并行扩写。

### 11.5 当前采纳边界
上述三项方向当前已补回正文，作为正式后续方向：
1. 注册层整理
2. 资产分层整理
3. 复盘制度上提

但当前不将其并行展开为新的主任务。
本阶段仍以已在项目总表中锁定的主线缺口为准。
