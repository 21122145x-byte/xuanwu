## 一、这张表回答什么

这张表只回答一件事：

> 在当前轻量治理线里，面对不同层级对象时，默认先读谁、再读谁、最后读谁。

它不负责：
- 判断某任务该不该进入这条主线
- 代替冲突裁决表做最终裁决
- 代替按任务类型读取表处理所有细分场景

一句话：
> 它是**默认调用顺序表**，不是完整裁决表。

---

## 二、默认调用总原则

默认顺序先按层级走，不先按聊天热度走。

当前固定总原则：
1. **先看项目总表 / 当前主入口**
2. **再看主文 / 正式机制承载位**
3. **再看对应规则卡 / 模板件 / 候选辅助件**
4. **最后才回看聊天、旧计划、历史参考件**

也就是说：
- 入口先于解释
- 主文先于散件
- 正式件先于聊天
- 当前件先于历史件

## 三、默认调用顺序（总表）

| 顺位 | 层级 | 默认对象 | 当前口径 |
|---|---|---|---|
| 1 | 主入口层 | 项目总表 | 项目态任务默认先回项目总表 |
| 2 | 主承载层 | 主文 | 正式机制定义、主线结构与骨架先看主文 |
| 3 | 规则件层 | 规则卡 / 模板件 | 需要具体执行口径时，再读对应规则件 |
| 4 | 候选辅助层 | 已成立候选 skill 文档链 | 当明确进入 skill 候选推进或试跑前收口时再读 |
| 5 | 辅助视图层 | README / 展开表 / 缺口表 / 计划表 | 用于导航、补看、找缺口，不反向替代前四层 |
| 6 | 历史参考层 | 聊天记录 / 旧计划 / 旧版本文档 | 仅作参考，不得反向覆盖当前正式口径 |

## 四、按对象类型的默认调用顺序

### 1. 机制主线判断
默认顺序：
1. 项目总表
2. 主文
3. 对应规则卡
4. 对应模板件
5. 聊天参考

适用：
- 现在这条机制到底怎么定义
- 当前主线骨架应该先信谁

---

### 2. 起手顺序判断
默认顺序：
1. 系统级注册与默认起手顺序规则卡
2. 项目总表
3. 主文
4. 当前阶段相关规则件

适用：
- 项目态续接
- 连续推进任务
- 审后回转重新接链

---

### 3. 具体执行判断
默认顺序：
1. 对应规则卡
2. 对应模板件
3. 主文相关单元
4. 展开表 / 缺口表

适用：
- 阶段门
- 审计插入
- 停损
- 最小单元

---

### 4. 候选 skill 推进判断
默认顺序：
1. 现役能力清单
2. 机制与skill映射表
3. 对应 skill 候选拆解卡
4. 技能说明草案
5. 样例集
6. 试跑记录 / 阶段审计记录

适用：
- 候选是否已进入链路
- 当前推进到哪一阶段
- 还缺哪类补件或边界收口

---

### 5. 注册层补件判断
默认顺序：
1. 注册层整理表
2. 现役能力清单
3. 机制与skill映射表
4. 相关候选推进文档
5. README 导航层

适用：
- 现在注册层还缺什么表
- 哪条候选线需要回写到注册层
- 哪些状态还没进入正式表件

---

## 五、明确不应采用的错误顺序

以下顺序当前默认视为错误：

### 错误 1：先看聊天，再定机制口径
错误原因：
- 聊天只能提供上下文，不应反向定义正式口径

### 错误 2：先看 README，再跳过项目总表
错误原因：
- README 是导航层，不是项目态默认入口

### 错误 3：先看候选 skill，再回头找机制本体
错误原因：
- skill 候选是辅助层，不是机制本体定义层

### 错误 4：先看缺口表，再代替主文做判断
错误原因：
- 缺口表是缺口视图，不是正式定义位

## 六、当前边界

这张表当前还没有处理的内容：
- 当项目总表与主文口径冲突时，最终以谁为准
- 当新件与旧件冲突时，先信谁
- 当不同任务类型进入不同分支时，是否要局部改写默认顺序

这些属于：
- 冲突裁决优先级表
- 按任务类型读取表

所以当前口径应理解为：
> **默认先读顺序已成立，但冲突时的最终裁决仍待补正式表。**

## 七、当前判断

这张表补上后，注册层已经不再只是在说：
- 有哪些东西
- 哪些是现役能力

而开始能回答：
- **默认先读什么**
- **默认别先读什么**

这意味着注册层开始从“静态清单”往“可操作入口”迈了一步。
