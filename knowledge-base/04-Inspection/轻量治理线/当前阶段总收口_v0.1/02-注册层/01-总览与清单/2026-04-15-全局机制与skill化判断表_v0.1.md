## 一、先纠偏：这次到底在盘什么

这次不再把以下对象直接算进“现役能力”：
- 项目总表
- README
- 主文承载位
- 缺口表
- 单元展开表

这些是：
- 入口层
- 导航层
- 承载层
- 台账层

它们很重要，但**不是能力本体**。

这次要盘的，是两类东西：
1. **机制**：团队已经形成的协作规则、流程结构、执行协议、判断闸门
2. **skill**：已经被封装成可直接调用的稳定能力单元

同时再判断：
- 哪些机制适合直接 skill 化
- 哪些机制只能把一部分 skill 化
- 哪些机制本质上不该直接 skill 化，只能作为上位约束或审计边界存在

## 二、skill 化判断总原则（纠偏版）

### 1. 适合直接 skill 化的机制
通常满足：
- 触发条件明确
- 输入输出稳定
- 边界可描述
- 执行步骤相对固定
- 成败可验证
- 不依赖高强度情境判断或跨角色裁决

### 2. 只适合局部 skill 化的机制
通常满足：
- 整体机制需要人判断 / 审计 / 拍板
- 但其中有稳定子步骤可抽成 skill
- 可被 skill 化的是“调用入口、标准模板、检查表、路由逻辑、格式化动作”
- 不可被 skill 化的是“最终裁决、越界处理、复杂升级判断”

### 3. 不适合直接 skill 化的机制
通常满足：
- 是团队宪法、上位法、边界法
- 依赖跨情境裁量
- 需要持续纠偏，不能把碰巧有效路径固化成默认正确路径
- 一旦直接 skill 化，最容易把“局部有效”误封装成“长期正确”

### 4. 当前总提醒
> 自动创建 skill 最危险的地方，不是写不出来，而是会把“碰巧有效的路径”提前固化。
>
> 所以：**机制是否能 skill 化，不等于它是否值得现在立刻 skill 化。**
>
> 默认要求应是：
> - 先机制稳定
> - 再局部 skill 化
> - skill 创建后必须经过纠偏 / 审计 / 试跑，才能进入现役层

## 三、全局机制盘点（团队级）

### A. 宪法级 / 上位治理机制

| 机制 | 来源 | 当前作用 | 是否适合直接 skill 化 | 判断 |
|---|---|---|---|---|
| 联邦式主 Agent 架构 | 团队协作宪法与治理总纲 | 定义团队基本结构与主责归属原则 | 否 | 上位法，不应直接做成执行 skill |
| P0-P3 自主 Spawn / P3+ 协同升级 | 团队协作宪法与治理总纲 | 定义何时进入更强协同结构 | 部分可 | 适合把“触发判断清单” skill 化，不适合把升级裁决整体自动化 |
| 需求理解与对齐规则 | 团队协作宪法与治理总纲 | 任务启动前压清目标、边界、交付、时间 | 部分可 | 可 skill 化“对齐字段卡 / 检查表”，不宜把对齐判断整体自动化 |
| 外部调研优先规则 | 团队协作宪法与治理总纲 | 遇未知事实先调研后结论 | 部分可 | 可做“调研触发检查 skill”，不宜自动化最终结论 |
| 遇阻默认动作规则 | 团队协作宪法与治理总纲 | 遇阻先拆问题、补证据、缩小问题面 | 部分可 | 可 skill 化“遇阻排查卡”，不宜直接替代策略判断 |
| 长文产物 5 字段 handoff 规则 | 团队协作宪法与治理总纲 | 统一正式交接格式 | 是 | 非常适合 skill 化为格式化生成 / 校验 skill |
| 文档挂载强制字段规则 | 团队协作宪法与治理总纲 | 统一文档链路位置暴露 | 是 | 适合 skill 化为挂载字段检查 / 自动补齐辅助 skill |
| 文档分层原则 | 团队协作宪法与治理总纲 | 判断内容应落在哪一层 | 部分可 | 可做“分层建议器”，但最终归位仍需人工裁决 |
| 固定主任务制度化机制 | 固定主任务制度化与主动经营SOP | 将主任务推进为可复用、可维护能力单元 | 部分可 | 很适合拆 skill，但不适合整体一键 skill 化 |

#### 宪法层结论
- **不应直接 skill 化的**：联邦结构、主权边界、团队上位法
- **可局部 skill 化的**：对齐字段卡、排查卡、分层建议器、主任务制度化模板链
- **最适合优先 skill 化的**：handoff 规则、文档挂载字段规则

### B. 跨 Agent 协作机制

| 机制 | 来源 | 当前作用 | 是否适合直接 skill 化 | 判断 |
|---|---|---|---|---|
| 跨 Agent 正式交接协议 | 跨Agent协作流转协议 / 执行卡 | 规范交接、回执、完成、打回、重提 | 部分可 | 协议本体不宜 skill 化；但交接模板、回执模板、打回检查可 skill 化 |
| 末尾 @ 下一责任人规则 | 跨Agent协作流转执行卡 | 保证正式交接显式闭环 | 是 | 适合做消息格式检查 / 模板 skill |
| 5 字段正式交接格式 | 跨Agent协作流转执行卡 | 统一交接内容结构 | 是 | 适合直接 skill 化 |
| 收到 / 完成 / 打回 / 重提规则 | 跨Agent协作流转执行卡 | 规范状态回执 | 是（模板层） | 可 skill 化为回执模板 skill |
| 协调位介入熔断规则 | 跨Agent协作流转执行卡 | 多轮无实质修改时由粉雪介入 | 部分可 | 适合做提醒与触发器，不宜自动替代介入裁决 |

#### 跨 Agent 层结论
最适合 skill 化的是：
- 交接模板
- 回执模板
- 打回校验
- 文档路径必要件检查

不适合直接 skill 化的是：
- 谁来接、是否放行、是否熔断的最终裁决

### C. Discord 半自动任务机制

| 机制 | 来源 | 当前作用 | 是否适合直接 skill 化 | 判断 |
|---|---|---|---|---|
| Discord 半自动任务协议 | 2026-04-11 半自动任务执行协议_Discord版 | 在频道前台消息持续插入时维持主链不断 | 部分可 | 协议整体不宜直接 skill 化；适合拆成角色调用 skill |
| 协调位速查卡 | Discord 版速查卡 | 协调位最短动作入口 | 是 | 很适合 skill 化为协调位调用 skill |
| 执行位速查卡 | Discord 版速查卡 | 执行位最短动作入口 | 是 | 很适合 skill 化为执行位调用 skill |
| 审计位速查卡 | Discord 版速查卡 | 审计位最短动作入口 | 是 | 很适合 skill 化为审计位调用 skill |
| 超时催办规则补丁 | Discord 版补丁 | 检查点超时后强制推进状态机 | 是（提醒 / 检查层） | 非常适合 skill 化成“超时催办器” |
| 频道回执模板 / 阶段门状态模板 | Discord 版模板 | 统一前台回执格式与状态门 | 是 | 适合直接 skill 化 |
| 一个任务只允许一条主链 | Discord 版协议 | 约束并行与乱改题 | 部分可 | 可 skill 化成检查器，不能把主链仲裁自动化 |

#### Discord 半自动层结论
这是**最适合“局部 skill 化”**的一层。
尤其适合拆成：
- 协调位调用 skill
- 执行位调用 skill
- 审计位调用 skill
- 超时催办 skill
- 回执模板 skill
- 阶段门状态校验 skill

不宜整体一次性 skill 化成“自动半自动任务总 skill”。

### D. 轻量治理 / 单一入口协作机制

| 机制 | 来源 | 当前作用 | 是否适合直接 skill 化 | 判断 |
|---|---|---|---|---|
| 单一入口式 multi-agent 协作框架 | 主文 v0.1 | 作为项目态单一入口、多生产位、阶段式送审的机制骨架 | 部分可 | 主文整体不宜 skill 化；适合拆出稳定子机制 |
| 系统级注册与默认起手顺序 | 承接规则卡 | 规定项目态默认先读注册层与总表 | 是 | 非常适合 skill 化为“默认起手检查 skill” |
| 最小单元机制 | 最小单元模板 / 主文单元 5 | 统一最小派单与回收颗粒度 | 是 | 适合 skill 化为最小单元生成器 / 校验器 |
| 阶段门机制 | 阶段门模板 / 主文单元 4 | 控制 sent-to-audit / accepted / revision-needed / blocked | 是 | 非常适合 skill 化 |
| 审计插入机制 | 审计插入规则 | 定义何时送审与如何接回 | 是（已接近 skill 化对象） | 可直接进入候选 |
| 审后回转机制 | 主文单元 4 | accepted / revision-needed 后继续接链 | 部分可 | 可 skill 化回转模板，不宜自动替代阶段判断 |
| 主产物优先机制 | 主文单元 6 | 锁定阶段唯一主产物 | 部分可 | 可做判定表 / 预警器，最终裁决仍需人工 |
| 断层补口机制 | 主文单元 7 | 处理中段断层与第二件停住 | 部分可 | 可 skill 化断层诊断卡、补口模板 |
| 停损机制 | 停损规则卡 / 主文单元 8 | 为失败或漂移建立正式分流口 | 部分可 | 可 skill 化停损记录格式 / 恢复细则，停损裁定不应自动化 |
| 下位件沉淀机制 | 主文单元 9 | 决定内容下沉为规则卡 / 模板 / skill 候选 | 部分可 | 可做沉淀建议器，不宜自动代替上提裁决 |
| 注册层整理机制 | 主文单元 10 / 注册层整理表 | 收现役机制与 skill 的清单、顺序、裁决 | 部分可 | 可 skill 化维护动作，不宜一键自动改写注册层 |

#### 轻治理层结论
最适合优先 skill 化的是：
- 默认起手检查
- 最小单元模板
- 阶段门模板
- 审计插入模板
- 断层诊断卡
- 停损记录格式

最不适合直接 skill 化的是：
- 主文整体
- 最终阶段裁决
- 停损裁定
- 下位件上提裁决

## 五、全局 skill 化判断（收口版）

### 1. 已经存在 skill，说明这类东西天然适合 skill 化
- 搜索类
- 外部平台接入类
- 媒体处理类
- 明确输入输出的编程 / 执行桥接类
- 长任务编排器 / 调度器
- 结构化知识管理器

### 2. 当前最值得继续 skill 化的机制对象
按优先级，我建议是：

#### 第一优先级：模板化、检查化、格式化机制
- 5 字段 handoff 生成 / 校验
- 文档挂载字段检查
- 跨 Agent 正式交接模板
- Discord 协调位 / 执行位 / 审计位调用 skill
- 超时催办器
- 默认起手检查器
- 最小单元生成器
- 阶段门状态校验器

#### 第二优先级：局部路由与建议机制
- 分层归位建议器
- 主产物判定辅助器
- 断层诊断卡生成器
- 停损记录格式器
- 注册层维护助手

#### 第三优先级：需要慎重推进的机制
- revision-needed 回转器
- 下位件沉淀建议器
- 固定主任务制度化助手

这些可以做，但必须先经过更强纠偏，不要把“碰巧好用”直接固化。

### 3. 当前不建议直接 skill 化的机制对象
- 团队宪法本体
- 联邦主权与角色边界本体
- P0-P3 / P3+ 升级裁决本体
- 审计最终放行裁决
- 停损最终裁定
- 人类拍板边界

这些适合：
- 写规则
- 写检查表
- 写提醒器
- 写辅助判定

但不适合：
- 直接封成“自动正确”的 skill
