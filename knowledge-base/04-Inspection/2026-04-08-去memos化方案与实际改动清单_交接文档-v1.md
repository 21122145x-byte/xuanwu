## 一、这份交接文档解决什么问题

当前已经明确两件事：

1. **MemOS 不能继续作为默认 memory slot 与默认 auto-recall 注入链保留。**
2. **Curie 的 OpenClaw Runtime Pipeline 某些组件后续可能仍需保留下载、局部改造、作为 Phase 1 工程材料继续使用。**

因此当前不能把“去 memos 化”写成：
- 一刀切全删
- 把所有相关组件一并清空
- 先做物理清理，再考虑哪些能力还要保留

当前更准确的收口应是：

> **把 MemOS 从默认运行链中退场，但保留可用于研究、改造、验证、迁移的组件和材料；执行上先切默认绑定，再冻结，再迁挂，再决定物理归档。**

---

## 二、当前正式判断

### 1. 必须退场的，不是“所有 MemOS 资产”，而是“默认运行角色”
必须退场的是：
- 默认 memory slot 绑定
- 默认 auto-recall 注入
- 默认上下文信任地位

不要求当前立刻退场的是：
- 历史数据库
- 历史插件目录
- viewer 相关残留
- 可用于改造/验证的组件与文档

### 2. 方案目标是“双收口”
#### 收口 A：运行链退场
让 MemOS 退出默认运行链。

#### 收口 B：资产保留与改造边界明确
对仍可能被 Curie 的 Runtime Pipeline 保留、下载、改造的组件，明确：
- 可以保留什么
- 不可以继续默认挂什么
- 哪些只能作为研究/改造材料存在

---

## 三、方案总览（供交接时快速理解）

### 主线目标
1. 让 MemOS 退出默认 memory slot
2. 让 auto-recall / 默认注入退场
3. 用本地替代层接住默认工作闭环
4. 保留可用于 Runtime Pipeline 改造的组件与文档
5. 不在本轮做破坏性物理删除

### 本地替代层承接
默认承接改为：
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `knowledge_base/`
- `memory_search / memory_get / memory_timeline / memory_write_public`
- 本地 skill / rules / SOP 文件化承载

### 原则
> **退默认绑定，不退历史资产；退自动注入，不退可改造材料。**

## 四、实际改动清单（执行顺序版）

以下清单按执行顺序排列，供后续真正进入切换窗口时使用。

### Phase 0：执行前冻结（不改配置）
#### 0.1 治理口径冻结
统一口径：
- MemOS = 待迁出旧依赖 + 当前高风险默认记忆位
- 不再视为默认可信记忆基础设施
- 版本升级不再作为主解法

#### 0.2 主线投入冻结
- 停止继续围绕 viewer / auto-recall 做主线修理
- 保留证据，但不继续追加主投入

#### 0.3 物理删除冻结
- 不删库
- 不删插件目录
- 不清 backup / hold / debug logs

---

### Phase 1：替代层接管（先接工作闭环）
#### 1.1 默认工作记忆切到文件层
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`

#### 1.2 正式知识切到显式 knowledge_base
- 审计意见
- RCA
- 执行卡
- 规则 / SOP
- design / topic / report

#### 1.3 检索改为显式调用
- `memory_search`
- `memory_get`
- `memory_timeline`
- `memory_write_public`

#### 1.4 skill / 规则去插件依赖
- 经验进 skill / references / scripts / KB 文件
- 不再经 MemOS 插件返注上下文

---

### Phase 2：切默认绑定（核心改动）
#### 2.1 解绑 `plugins.slots.memory`
- 目标：默认 memory slot 不再指向 `memos-local-openclaw-plugin`

#### 2.2 禁用 `plugins.entries["memos-local-openclaw-plugin"]`
- 目标：不再处于活动 entry 态

#### 2.3 从 `plugins.allow` 中移除 `memos-local-openclaw-plugin`
- 目标：不再保留默认允许参与加载的地位

> 顺序必须固定：**slot unbind → entry disable → allow remove**

---

### Phase 3：台账修正（v2 新增强制项）
#### 3.1 修正插件安装记录漂移
当前已知问题：
- `plugins.installs_version_drift`
- recorded `1.0.7`
- installed `1.0.8`

目标：
- 修正版本记录
- 避免后续继续污染审计与升级判断

#### 3.2 记录切换前后状态
至少记录：
- 切换前是否默认挂 MemOS
- 切换后是否已解绑
- 切换前是否 auto-recall 注入
- 切换后是否已停止默认注入

---

### Phase 4：后处理（保留改造材料，不做破坏性清除）
#### 4.1 backend 冻结
保留：
- `~/.openclaw/memos-local/memos.db`
- 旧数据与历史样本

#### 4.2 插件目录冻结
保留：
- `~/.openclaw/extensions/memos-local-openclaw-plugin`

用途限定为：
- 研究
- 对照
- 改造参考
- Runtime Pipeline 组件保留下载与后续工程比对

#### 4.3 文档与 skill 资产迁挂
保留并迁挂：
- MemOS 治理文档
- 污染样本
- 审计链
- skill 经验

改挂到：
- knowledge_base
- 本地 skill
- rules / SOP / archive

## 五、哪些组件可以保留下载 / 保留改造

结合 Curie 的 Runtime Pipeline 线，当前建议保留的，不是“默认运行态”，而是“工程材料态”。

### 可保留项
1. **插件源码/目录**
   - 用于后续结构研究、接口比对、局部改造参考
2. **历史 DB / recall 样本 / 调试日志**
   - 用于污染样本回看、行为复验、before/after 对照
3. **MemOS 相关治理文档与 RCA**
   - 用于总结问题边界、设计新路径时避坑
4. **相关 skill / SOP / 操作卡**
   - 用于迁移成新规则层 / 新技能层

### 保留前提
> **它们只能作为“研究/改造材料”保留，不能继续以“默认活动运行位”存在。**

---

## 六、明确不保留的东西

以下角色当前不应继续保留：

1. **MemOS 作为默认 memory slot**
2. **MemOS 作为默认 auto-recall 注入层**
3. **MemOS 作为默认正式知识入口**
4. **MemOS 作为默认可信记忆基础设施**

一句话：
> **可以保留组件，不可以保留默认统治地位。**

---

## 七、验收标准

这份方案+改动清单成立，只看以下 6 条：

1. 默认启动与工作闭环不再依赖 MemOS
2. 默认主会话不再接收 MemOS auto-recall 注入
3. 正式知识已切到显式 knowledge_base
4. MemOS 已不在默认 memory slot
5. 版本台账漂移已被记录并进入修正项
6. 保留组件已明确降格为研究/改造材料，而非默认运行组件

---

## 八、当前不允许的错误推进方式

### 不允许 1
- 先删库 / 删目录 / 清空文档，再看系统还缺什么

### 不允许 2
- 因为 Curie 可能还要研究组件，就继续默认挂 MemOS

### 不允许 3
- 因为 1.0.8 已升级，就再给 MemOS 一段默认观察期

### 不允许 4
- 把“组件保留下载”写成“默认运行继续保留”

这些都会把边界写坏。

---

### 1. 先按本交接文档继续补“实际改动清单 + 验收记录模板”
让后续执行时不需要再临场拼动作。

### 3. 切换窗口执行时，必须先退默认绑定，再谈后续改造
不能反过来。
