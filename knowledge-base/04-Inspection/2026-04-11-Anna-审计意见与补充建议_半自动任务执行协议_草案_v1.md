- **审计日期**：2026-04-11
- **审计角色**：Anna
- **审计口径**：结构漏洞 / 角色边界漏洞 / 执行可落地性 / 协作防卡断机制
- **审计目的**：判断该草案是否足以防止“后台执行已推进，但阶段流转、审计承接、对外回传再次卡断”的机制性问题。

---

## 一、已确认情况

当前草案已经有 4 个成立点：

### 1. 主问题抓得对
它没有把问题误写成“worker 不够勤快”或“谁忘了跟进”，而是明确识别为：
- 产物完成 ≠ 已被接住
- heartbeat 保活 ≠ 保闭环
- 前台消息 ≠ 后台状态
- tool 边界 ≠ 角色边界

这是对的。

### 2. 角色边界主方向对
尤其这两条是对的：
- worker 只负责产出，不负责跨环交接
- 主 Agent 对阶段流转负责

这比把“谁能做”与“谁该负责”混在一起强很多。

### 3. 阶段门思路是必要的
把半自动任务压成显式状态，而不是靠聊天上下文猜当前进度，这一条是必须的。

### 4. heartbeat 被降回“发现停滞 / 升级提醒”而不是业务兜底
这一点也是对的。heartbeat 不能替代 accept / revise / send-to-audit 判断。

---

## 二、主要缺口

### 缺口 1：当前协议没有“审计承接规则”，导致 sent-to-audit 之后仍可能悬空
现在草案写到：
- 主 Agent 可以把状态推到 `sent-to-audit`
- 审计位接到任务后进入 `audit-in-progress` 或 `returned`

问题是，中间缺了一层非常关键的机制：

> **什么叫“已送审”成立？什么叫“审计位已承接”成立？**

如果这层不补，后续仍会出现老问题：
- 主 Agent 以为自己送出去了
- 审计位没看到 / 没接住 / 没承认承接
- heartbeat 只能看到停滞，不能区分是“没送到”还是“送到了但没开审”

这会继续卡断。

---

### 缺口 2：当前协议没有“打回规则”，returned 会重新退化成口头状态
现在草案有：
- `returned`
- `revision-needed`

但没写清：
- 谁有权打回
- 打回时必须携带什么字段
- 什么情况下是“退回修改”，什么情况下是“拒绝承接”
- 打回后状态如何重新进入下一轮

如果这层不补，`returned` 最后还是会退化成：
- 聊天里一句“你再改改”
- 没有最小修复动作
- 没有回收 owner / next_owner / next_action

然后主链再次断在返修环。

---

### 缺口 3：当前协议没有“放行判断规则”，accepted 容易被写成礼貌性通过
现在草案有：
- `accepted`
- `announced`
- `closed`

但没写：
- 什么情况下允许 accepted
- accepted 是否等于质量通过 / 仅等于可进入下一环
- 谁有最终放行权
- accepted 与 announced / closed 的关系是否必须顺序成立

如果这层不补，accepted 会变成最危险的伪稳定词：
- 看起来结论已定
- 实际只是“差不多先过”
- 然后直接进入 announce / close

这是最容易让后台协作再次卡断且不可追责的点。

---

### 缺口 4：状态门缺少“承接失败 / 等待确认 / 阻塞”类异常状态，导致异常只能伪装成停滞
当前状态门是：
- drafting
- revision-needed
- review-ready
- sent-to-audit
- audit-in-progress
- accepted
- returned
- announced
- closed

问题是，这套状态只覆盖了理想正向链，没有覆盖几类真实常见异常：
- 已送出但未被承接
- 等待人工确认
- 受外部依赖阻塞
- 审计位拒绝承接但未进入正式返修

结果就是：
> 很多异常不会被记录成状态，只会伪装成“停滞太久”。

这会降低 heartbeat 和主 Agent 判断的分辨率。

---

### 缺口 5：当前强制字段还不够，缺“依据 / 判定原因 / 承接方式”三个关键信息
现在强制字段只有：
- `status`
- `owner`
- `next_owner`
- `next_action`
- `updated_at`
- `artifact_where`
- `known_issues`

这不够。至少还缺：
- `decision_basis`：这次 accept / revise / return / announce 依据什么做的
- `handoff_mode`：是已投递、已引用文档、已消息通知，还是仅内部标记
- `blocking_reason`：若非正常推进，为何卡住

没有这三类字段，后续复盘时仍然会回到：
- 状态知道了
- 但不知道为什么这么判
- 也不知道下一环到底有没有真接住

---

### 缺口 6：前后台解耦规则仍缺“消息优先级裁定”，容易再次在新消息插入时失真
你现在写了：
- 后台主链以阶段状态为准
- 前台旧问题不应拉回早期阶段

但没写：
> **当用户新消息确实改变任务范围时，谁来裁定“继续当前链”还是“起新链 / 分支链”？**

如果这条没有，后续很容易出现两种失真：
1. 明明该切分支，却硬说“后台继续按原状态跑”
2. 明明不该打断，却被一句新消息拉回重开

这不是态度问题，是缺仲裁规则。

---

## 三、必补条款

以下 3 组条款，我认为必须直接补进草案，否则这份协议仍不足以进入试跑。

---

### A. 审计承接规则（必须新增）

#### 规则 A1：`sent-to-audit` 不等于审计已承接
只有当审计位明确写出以下任一状态，才算送审成功进入下一环：
- `audit-in-progress`
- `returned`
- `audit-blocked`

否则：
> **`sent-to-audit` 只代表主 Agent 已发起送审，不代表审计链已承接。**

#### 规则 A2：送审必须携带最小承接包
主 Agent 送审时，必须至少提供：
- 主文路径
- supporting artifacts 路径
- 当前状态
- 本轮已知问题
- 希望审计位重点核什么

缺少上述任一项，审计位可拒绝承接并打回为：
- `returned`
- 或 `audit-blocked`

#### 规则 A3：审计位承接后必须改写状态
审计位一旦开始处理，不允许继续让主链停留在 `sent-to-audit`。必须显式改成：
- `audit-in-progress`
- 或 `audit-blocked`

否则默认视为：
> **未完成承接。**

---

### B. 打回规则（必须新增）

#### 规则 B1：`returned` 必须携带最小修复信息
任何打回必须至少补：
- `return_reason`
- `minimum_fix_path`
- `who_should_rework`
- `reentry_condition`

没有这 4 项的 returned，不算有效打回，只算口头反馈。

#### 规则 B2：打回要区分两类
- **审计退回**：内容质量 / 证据链 / 规则边界不达标
- **承接退回**：材料不齐 / handoff 不合格 / 无法正式进入审计

这两类不能混写。否则后续会分不清：
- 是内容要改
- 还是交接方式本身就不合格

#### 规则 B3：打回后必须重新落到明确状态
`returned` 不是终点，打回后必须在短时间内重新落到：
- `revision-needed`
- 或 `review-ready`
- 或 `closed`（明确放弃）

不允许长期挂在 `returned` 上。

---

### C. 放行判断规则（必须新增）

#### 规则 C1：`accepted` 只代表当前阶段放行，不代表全链最终关闭
必须明确：
> **accepted = 当前环节通过，可以进入下一环；不等于整条链已 closed。**

#### 规则 C2：放行判断必须携带依据
任何 `accepted` 至少补：
- `accept_scope`：放行的是哪一层
- `accept_basis`：依据什么放行
- `known_limits`：放行但仍保留哪些边界

否则 accepted 容易退化成礼貌性通过。

#### 规则 C3：`accepted → announced → closed` 不能自动连跳
只有当：
- 已达到对外回传条件
- 或已达到正式关闭条件

才允许从 `accepted` 进入下一步。

不能因为 accepted 成立，就自动视为可以 announced / closed。

## 四、建议修订动作

### 修订动作 1：扩状态门，补异常状态
建议至少补 3 个状态：
- `audit-blocked`
- `awaiting-human-confirmation`
- `blocked`

这样异常不再只能伪装成“超时停滞”。

### 修订动作 2：扩强制字段
建议把最小强制字段扩成：
- `status`
- `owner`
- `next_owner`
- `next_action`
- `updated_at`
- `artifact_where`
- `known_issues`
- `decision_basis`
- `handoff_mode`
- `blocking_reason`（如适用）

### 修订动作 3：在“标准流转”里显式插入审计承接与打回分叉
现在流转仍偏主干，建议补成：
- review-ready
- 主 Agent accept / revise
- sent-to-audit
- audit-in-progress / audit-blocked / returned
- accepted / revision-needed / announced / closed

让异常分支不是附注，而是主协议的一部分。

### 修订动作 4：补“新消息改范围”的仲裁条款
建议增加一条：
> 当用户新消息实质改变任务范围时，主 Agent 必须显式判断：
> - 继续当前链
> - 新开分支链
> - 中止当前链
>
> 不允许既口头承认变更，又维持原状态链不重写。

### 修订动作 5：补模板件
当前下位文档里写了“待补”，这不是小问题。至少应补 3 张模板：
- 阶段门状态模板
- 后台任务看板模板
- Anna 审计承接模板

否则协议很容易停留在文档层，落不到执行层。

## 五、是否可进入试跑

**当前判断：可在补完关键条款后进入小范围试跑；不建议原样直接试跑。**

### 原因
当前草案已经有主框架，但仍缺：
- 审计承接规则
- 打回规则
- 放行判断规则
- 异常状态补口

如果不先补这几条，试跑时最容易再次卡在：
- sent-to-audit 看似已送，其实无人承接
- returned 看似已打回，其实没有最小修复路径
- accepted 看似已放行，其实没有放行边界
- 前台新消息进来后，主链状态与实际任务范围再次错位

### 我的结论
- **原样：不建议直接试跑**
- **补完上述必补条款后：可以做一条真实任务的小范围试跑**
