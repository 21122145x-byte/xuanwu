## 一、这次为什么必须重写方案

v1 方案成立，但现在已经不够。

原因不是“方案方向错了”，而是：

> **MemOS 从 `1.0.7` 升到 `1.0.8` 后，去 memos 化这件事已经不只是“上下文漂移治理”问题，而是同时变成了“默认记忆位暴露面治理”问题。**

升级后新增的确定性输入包括：
- `plugins.slots.memory = memos-local-openclaw-plugin` 仍在
- `allowPromptInjection=true, auto-recall enabled`
- `Memory: unavailable`
- viewer 自启动失败与端口冲突仍在
- `plugins.code_safety` 继续标红
- `plugins.installs_version_drift`（记录 1.0.7，实际 1.0.8）
- 更大的系统组合风险仍然存在：
  - `groupPolicy = open`
  - `exec.security = full`
  - `sandbox = off`

所以当前主判断必须升级为：

> **去 memos 化不再只是为了“修 recall 漂移”，而是为了把“默认 memory slot + 自动注入 + 开放暴露面”这组高风险组合拆开。**

---

## 二、当前正式判断

### 1. MemOS 的治理定位继续下调
当前统一口径应改为：

> **MemOS = 待迁出旧依赖 + 当前高风险默认记忆位。**

不再允许继续写成：
- 默认可信记忆基础设施
- 当前可继续主挂的默认 memory provider
- 只需 patch 升级即可恢复信任的组件

### 2. 当前不再建议继续围绕 MemOS 修补主线
以下线继续投入，只保留最小证据，不再作为主线：
- viewer 修补
- auto-recall 修补
- “看看 1.0.8 还能不能再救一下”

### 3. 当前方案目标改成“双目标”
本轮去 memos 化执行方案不再只追一个目标，而是两个：

#### 目标 A：解除默认记忆绑定
- 让 MemOS 退出默认 memory slot
- 让 auto-recall 不再承担默认上下文注入责任

#### 目标 B：降低暴露面联动风险
- 避免开放群策略 + 高权限执行 + 无沙箱 + 默认自动记忆注入继续叠加

---

## 三、v2 方案的四层拆法（保留，但调整优先级）

四层拆法仍然成立：
1. viewer / UI 层
2. auto-recall / 注入层
3. memory backend / 历史数据层
4. 技能 / 知识资产层

但优先级必须调整。

### 新优先级
#### P0：auto-recall / 注入层
这是当前最先退场的一层。

原因：
- 它直接承担默认上下文注入
- 它和 `allowPromptInjection=true` 同域
- 它是漂移、串带、错误召回的主要入口
- 在当前开放暴露面下，它不只是“效果不好”，而是“风险放大器”

#### P1：默认 memory slot 绑定层
这层必须尽快解除。

原因：
- 只要还挂在默认 memory slot 上，系统就还在默认信任 MemOS
- 这会继续让配置、审计、运行口径混乱

#### P2：viewer / UI 层
这层不再是主目标，只做证据保留。

原因：
- viewer 起不来不是最核心问题
- 当前最危险的是默认挂载和自动注入，不是看板好不好用

#### P3：backend / 历史数据层与技能资产层
这两层先冻结、再迁出、最后归档。

原因：
- 有历史价值
- 有治理经验价值
- 但不该继续留在默认运行链里

---

## 四、重新定义执行顺序

### Phase 0：治理冻结（立即生效）
先做口径冻结，不等待配置改动。

#### 立即冻结 1
统一口径：
> **MemOS 不再是默认可信记忆基础设施。**

#### 立即冻结 2
统一口径：
> **版本升级不是当前 recall 漂移 / viewer / 暴露面问题的主解法。**

#### 立即冻结 3
统一口径：
> **后续主投入改为“替代层接管 + 默认绑定退场”，而不是继续修 MemOS。**

---

### Phase 1：先立替代层（不动数据，先接闭环）
这一阶段延续 v1，但要求更明确。

#### 1. 文件记忆层接管默认工作记忆
默认主层固定为：
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`

#### 2. 显式知识层接管正式结论
正式承载固定为：
- `knowledge_base/`
- 审计意见 / RCA / SOP / 执行卡 / topic / design

#### 3. 可控检索层只显式触发
固定使用：
- `memory_search`
- `memory_get`
- `memory_timeline`
- `memory_write_public`

原则：
> **检索变显式，不再默认注入。**

#### 4. 技能与规则层去插件依赖
原则：
- 经验进文件、规则、skill
- 不再通过 MemOS 插件“偷偷返注”进上下文

---

### Phase 2：切默认绑定（这是 v2 的真正核心）
这一步必须明确比 v1 更硬。

#### 动作顺序必须固定
1. **先解绑 `plugins.slots.memory`**
2. **再禁用 `plugins.entries["memos-local-openclaw-plugin"].enabled`**
3. **最后从 `plugins.allow` 中移除 `memos-local-openclaw-plugin`**

这条顺序不能乱。

#### 为什么现在必须强调这一步
因为当前已知：
- MemOS 仍然是默认 memory slot
- plugin 已初始化
- auto-recall 仍开着
- `allowPromptInjection=true`

所以如果只做“治理口径降级”而不切默认绑定，
那只是文档收口，不是实际退场。

---

### Phase 3：修正台账漂移（v2 新增强制项）
这是 v2 相比 v1 新增的必须项。

#### 当前必须补的台账项
- `plugins.installs_version_drift`
- recorded `1.0.7`
- installed `1.0.8`

#### 当前判断
这不是“顺手修一下”的小事，
而是：
> **必须补的事实台账修正项。**

原因：
- 它已经造成过版本判断错误
- 会继续污染后续升级决策
- 会继续污染去 memos 化执行边界

---

### Phase 4：冻结 backend 与历史资产，不做物理删除
这一阶段基本延续 v1。

#### 允许
- 保留 `~/.openclaw/memos-local/memos.db`
- 保留旧插件目录
- 保留日志与 backup / hold
- 保留历史 MemOS 治理文档与技能材料

#### 不允许
- 直接删库
- 直接删插件目录
- 直接清空所有 MemOS 文档
- 把治理经验当垃圾一起扔掉

原因：
> **我们现在要移除的是默认运行依赖，不是销毁历史证据。**

---

## 五、v2 里新增的风险边界

### 1. 去 memos 化必须与暴露面收紧联动
现在不能再把这两条线拆开写。

原因：
- `groupPolicy=open`
- `exec.security=full`
- `sandbox=off`
- `memos` 默认挂载 + auto-recall

四件事叠在一起时，MemOS 已经不只是“体验差”，而是风险组合中的一环。

所以建议收口为：
> **去 memos 化 = 运行治理动作 + 暴露面减压动作。**

### 2. viewer 问题不再主导执行优先级
viewer 启不起来、端口冲突、self-starting，这些都记录，但不再主导方案。

### 3. 版本讨论退到次级位
1.0.7、1.0.8，乃至更高 patch，后续都不再作为主线争论点。

主线只看：
- 是否默认挂载
- 是否自动注入
- 是否污染上下文
- 是否扩大暴露面

---

## 六、当前建议的执行节奏

### 第一批：马上推进
1. 统一治理口径：MemOS = 待迁出旧依赖 + 当前高风险默认记忆位
2. 明确本地替代层为正式主线
3. 准备切换窗口，先做 slot unbind / entry disable / allow remove 的执行清单
4. 将版本台账漂移列入必须修正项

### 第二批：切换窗口执行
1. 解除默认 memory slot 绑定
2. 禁用 MemOS plugin entry
3. 从 allow 中移除 MemOS
4. 验证默认工作闭环已由本地替代层接住

### 第三批：后处理
1. backend 冻结
2. 历史技能/文档资产迁挂
3. viewer / gateway / memory unavailable 并行排查，仅作为善后，不再反向拉回主线

---

## 七、当前验收标准（v2 版）

v2 方案是否成立，只看以下 6 条：

1. **默认启动与工作不再依赖 MemOS**
2. **默认主会话不再接收 MemOS auto-recall 注入**
3. **正式知识已完全切到显式 knowledge_base 承载**
4. **检索改为手动 / 场景触发，不再默认背景注入**
5. **MemOS 不再处于默认 memory slot**
6. **高风险组合被拆开至少一环**
   - 最优先是拆掉“默认挂载 + 自动注入”这一环

---

## 八、不放行的写法

当前明确不允许继续写成：
- “再升一个 patch 看看 MemOS 能不能恢复”
- “viewer 修好后再决定是否迁出”
- “先保留默认 memory slot，再慢慢观察”
- “等所有历史资产整理完再退场”

这些都会导致：
> **默认高风险链继续运行。**

---

## 九、结论

> **基于 MemOS `1.0.7 → 1.0.8` 升级后安全扫描，当前“移除 memos”的执行方案必须升级为 v2：核心不再只是修 recall 漂移，而是尽快让 MemOS 退出默认 memory slot 与默认自动注入链，同时用本地文件记忆、显式知识库、可控检索工具与技能规则层接管工作闭环。**
