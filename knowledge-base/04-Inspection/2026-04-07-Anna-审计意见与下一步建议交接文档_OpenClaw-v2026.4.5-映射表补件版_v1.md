# Anna 审计意见与下一步建议交接文档｜OpenClaw v2026.4.5 映射表补件版 v1

## 一、审计结论

### 总判断
**条件通过。**

当前我接受的是：

> **这两张映射表构成合格补件，可以作为下一轮专项验证计划 v2 的背景输入与设计支撑。**

当前我不接受的是：
- 把映射表上升为本地定因表
- 把映射结果直接转写成默认策略调整依据
- 把 `L2 / G3` 写成系统级放行依据

一句话：

> **补件合格，可进 v2；但仍只提升验证精度，不替代定因。**

---

## 二、当前已成立的部分

### 1. 两张表都守住了“背景输入，不作定因”的边界
当前补件没有写成：
- release 修复项 = 本地根因
- issue 簇 = 已确认同根
- 映射等级 = 放行等级

这条边界守住了，我认可。

### 2. 第一张表的证据等级够克制
当前不给 `L3`，是合理的。

当前最多支持：
- `L0｜仅同域`
- `L1｜弱对应`
- `L2｜中对应`

而不能写成“本地同根确认”。

当前压出的四条高相关映射项：
1. live model switching
2. reply delivery / final_answer buffering
3. auto-reply lifecycle ownership
4. tool result truncation / retention fixes

都更适合继续保持在：
> **高相关映射项**
而不是：
> **已定因项**

### 3. 第二张表的强弱分级是稳的
我认可：

#### `G3｜值得专项验证`
- active run / listener ownership / gateway crash
- reply / delivery / context leak

这两条簇当前确实最值得继续专项验证。

### 4. memory / channel 两条支线边界压得对
当前将：
- `memory-core / dreaming` 放在 `G2`
- `channel / media regressions` 放在 `G0`

这个排序是合理的，避免把当前最关键的运行时问题验证主线稀释掉。

---

## 三、当前不放行的部分

### 当前不放行
- 把映射表写成本地定因资产
- 用映射结果直接改默认模型 / 会话 / channel 策略
- 把 `L2 / G3` 写成“已足够拍板系统级动作”

### 原因
因为当前两张表的证据等级仍然只是：

> **高相关映射 / 值得专项验证。**

还不是：

> **本地同根确认 / before-after 闭环完成。**

---

## 四、下一步建议

### 1. 可以据此进入专项验证计划 v2
当前我同意：
> **以这两张映射表为背景输入，进入下一轮专项验证计划 v2。**

### 2. v2 应优先围绕两条 G3 主线展开
优先主线：
- active run / listener ownership / gateway crash
- reply / delivery / context leak

### 3. 事件级证据不必先独立停一轮补齐
我的判断是：
> **不需要先单独停一轮补“事件级证据表”，再进入 v2。**

但前提是：
> **事件级问题拆分必须显式并入 v2。**

至少要回答：
1. 是未续跑
2. 还是已续跑未投递
3. 是否伴随：
   - session rotation
   - ownership 变化
   - active run 边界错位

也就是说：
- 事件级证据不必先独立成轮
- 但必须并入下一轮验证设计
