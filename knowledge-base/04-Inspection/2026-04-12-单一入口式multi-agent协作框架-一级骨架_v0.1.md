## 一、机制目标

这套框架要解决的，不只是“多 agent 怎么协作”这个表面问题，而是要系统性纠偏当前多 agent 协作里反复出现的 8 类结构性问题。

### 1. 主目标失焦
在连续轮转中，任务经常从“主文 / 主骨架 / 主方案”偏移到零散补件、解释件、过渡件，导致：
- 主产物长期缺位
- 配件增长快于主线推进
- 看起来很忙，实际上主任务没有被真正推进

### 2. 先生产、后编排，缺少整体计划
当前常见问题是：
- 需求刚聊完，就直接起手生产
- 没有先锁阶段目标、主产物、阶段门、停损点
- 整个任务或整个项目缺少整体规划，只能“做一步看一步”

这会直接导致：
- 中途频繁改方向
- 配件与主文顺序错位
- 任务推进越来越依赖临场判断
- 后续越做越散，难以收束

### 3. 上下文持续膨胀
传统多 agent 长链轮转，会让：
- 每个 agent 都背越来越厚的历史上下文
- 后续理解成本越来越高
- 轮转成本高于推进收益
- 最终出现空轮转、假推进、无效交接

### 4. 生产、编排、审计混位
如果没有单一入口与清晰分工，就容易出现：
- 没有稳定主位盯主线
- 编排位被生产细节拖走
- 审计位过早插入或持续打断
- 没人真正负责阶段推进与总收束

### 5. 中段断层与虎头蛇尾
很多任务开头有热情、有讨论，到了中段就开始：
- 派发和回收脱节
- 产物不能并入主文
- 审计没有明确阶段门
- 后续推进越来越虚

最后表现为：
- 开头像在做项目
- 中段像在补聊天
- 结尾没有真正交付

### 6. 无法自运行，过度依赖人工值守
当前一旦任务变长、变复杂，经常出现：
- agent 不会自己按计划继续跑
- 到检查点不会主动回执
- 没有阶段门时不会自行切换状态
- 只要人不盯着，任务就容易悬空或失焦

这说明当前缺的不是“更努力”，而是：
- 可持续的自编排结构
- 默认的阶段推进规则
- 无需人工持续盯盘的回收与送审机制

### 7. 没有成本意识、预算意识与模型选择意识
当前协作里还有一个被低估的问题：
- 没有任务预算概念
- 不区分高成本任务与低成本任务
- 不会根据任务粒度与难度选择模型
- 低成本任务也可能默认用昂贵模型执行

直接后果是：
- 资源浪费
- 复杂任务真正需要预算时反而没有余量
- 团队对协作成本没有真实感知

### 8. 复用不足，坑反复踩
之前已经沉淀过的方法、模板、规则、失败教训，常常没有被真正复用：
- 做过的编排方法不复用
- 踩过的坑没有变成默认避坑规则
- spawn 时不参考历史经验
- 每轮都像从头再来

这意味着团队缺少的，不只是文档，而是：
- 能被调用的历史方法
- 能进入编排的成本与经验约束
- 能避免重复踩坑的默认规则

---

### 这套机制要达成的目标

基于以上问题，这套“单一入口式 multi-agent 协作框架”要达成的目标，不是追求形式上的多 agent 热闹，而是实现以下 6 个结果：

#### 目标 1：先计划、后生产
需求对齐后，先进入计划与编排，不允许直接跳入生产。

#### 目标 2：主产物优先
每阶段都先锁唯一主产物，禁止配件膨胀替代主线推进。

#### 目标 3：上下文受控
通过一次性最小单元 subagent，把单元上下文控制在可管理范围内，避免持续滚厚。

#### 目标 4：阶段式自运行
让主入口 Agent 能按阶段自行推进、回收、送审、继续，而不是必须依赖人工全程值守。

#### 目标 5：成本与预算可控
任务在编排时就要考虑：
- 是否值得 spawn
- 用什么粒度 spawn
- 用什么模型执行
- 哪些任务应走低成本模型
- 哪些任务才值得用高成本模型

#### 目标 6：方法沉淀可复用
让成功方法、失败经验、阶段规则、模型选择与避坑判断进入默认编排，而不是每次重来。

---

一句话定义：

> **单一入口式 multi-agent 协作框架，是一种由唯一主入口 Agent 负责需求对齐、计划编排、阶段推进、结果收束与送审，其他生产位以“一次性最小单元 subagent”执行的阶段式协作机制；它的目标是先计划、控主线、控上下文、可自运行、控成本、可复用。**

## 二、角色结构

这套框架的核心，不是“多几个 agent 一起干活”，而是把不同角色的职责拆开，避免编排、生产、审计、拍板重新混成一团。

当前建议至少锁定 5 个角色：
- 主入口 Agent
- 一次性生产 subagent
- 审计位
- runtime keeper
- 人类拍板位

---

### 1. 主入口 Agent

主入口 Agent 是整条协作链的唯一入口与唯一主收束位。

它不是“参与者之一”，而是：
> **计划发起者、任务编排者、阶段推进者、结果回收者、送审发起者、总收束者。**

#### 主入口 Agent 的职责
主入口 Agent 必须负责：

1. **需求对齐**
   - 把任务目标、不做什么、当前阶段目标、成功标准、时间要求、关键约束压清

2. **计划编排**
   - 在生产前先出任务结构
   - 先读取可复用的 skill、经验、注册层入口与历史方法，而不是默认从零发明
   - 锁定当前阶段唯一主产物
   - 判断是否需要多 agent
   - 判断哪些工作自己做，哪些工作下发为最小单元

3. **阶段拆分**
   - 定义每一阶段的目标、送审门、放行条件、退回条件
   - 防止整个任务“做一步看一步”

4. **单元派发**
   - 把任务压成最小单元
   - 明确每个单元的目标、范围、交付、约束、回收点
   - 遵循：一个最小单元 → 一个 prompt → 一个 subagent，完成即删除，不保留为长链上下文节点

5. **结果回收与并入**
   - 回收 subagent 结果
   - 判断是否合格
   - 判断是否能并入当前阶段主产物
   - 不合格时重发、重切或退回

6. **阶段送审**
   - 到阶段门后统一送审
   - 不把审计位拉进生产过程持续打断

7. **阶段推进或停损**
   - 审计通过后推进下一阶段
   - 发现空转、偏移、回收变重时暂停继续派发，先重编排

8. **成本与预算控制**
   - 判断任务是否值得 spawn
   - 判断单元应使用什么模型
   - 控制高成本模型只用于高价值任务
   - 避免低成本任务使用高成本配置
   - 避免因为单个最小单元失误，导致整个阶段任务被迫重跑

9. **经验复用与最终复盘**
   - 复用已有方法、模板、规则与避坑经验
   - 对可重复使用的方法，主动推动沉淀为 skill，而不是停留在聊天或一次性文档里
   - 任务结束后完成复盘，不让经验再次散失

#### 主入口 Agent 不负责什么
主入口 Agent 不应负责：
- 长时间亲自下场做大量生产单元
- 把自己的上下文拖进所有施工细节
- 用持续聊天代替正式计划与编排
- 把审计职责、拍板职责、系统治理职责一并吞掉

#### 主入口 Agent 的核心边界
主入口 Agent 的核心边界是：

> **它负责“让任务有结构地往前走”，但不负责替所有生产位亲自施工，也不负责替审计位和人类拍板位做越权决策。**

---

### 2. 一次性生产 subagent

一次性生产 subagent 是施工单元，不是小型主 Agent。

它的存在目的，是把复杂任务拆成低上下文、低耦合、可回收的最小生产动作。

#### 一次性生产 subagent 的职责
一次性生产 subagent 负责：

1. 承接一个最小单元
2. 在清晰边界内完成一个 prompt、一个产物
3. 按要求返回可验证结果
4. 完成后退出，不保留跨阶段长链上下文

#### 一次性生产 subagent 不负责什么
它不负责：
- 自行扩题
- 自行改变任务目标
- 自行补做额外配件
- 自行判断进入下一阶段
- 自行承担主线收束
- 自行替主入口位做多轮编排

#### 一次性生产 subagent 的边界原则
它的边界原则是：

> **只解决被派发的那个最小单元，不把自己变成新的主线。**

如果一个 subagent 回来时带着：
- 多个核心产物
- 多个方向的扩展建议
- 超出单元范围的大量新需求

那通常说明：
- 单元切太大了
- 或边界没有写清
- 或 subagent 越界了

---

### 3. 审计位

审计位不是生产链里的持续陪跑者，而是阶段门的正式裁决位。

#### 审计位的职责
审计位负责：

1. 在阶段门判断是否放行
2. 判断是否偷宽、越界、证据不足、结构失真
3. 对阶段产物给出正式结论：
   - accepted
   - revision-needed
   - blocked
4. 必要时指出：
   - 是产物问题
   - 是结构问题
   - 是边界问题
   - 还是应升级拍板的问题

#### 审计位不负责什么
审计位不应负责：
- 接管主入口 Agent 的任务编排职责
- 持续插手每个最小单元施工
- 在没有到阶段门前频繁打断生产链
- 代替人类做优先级与资源取舍

#### 审计位的核心边界
审计位的核心边界是：

> **它负责“判断这一阶段能不能过”，而不是替团队继续把后面的路全部写完。**

---

### 4. runtime keeper

runtime keeper，负责的不是生产内容本身，而是协作链在运行中的“不断线、不悬空、不假推进”。

这个角色补的是：
- 卡断识别
- timeout 后恢复
- 状态维持
- 异常分流
- 停发提醒

#### runtime keeper的职责
runtime keeper负责：

1. **状态检查**
   - 检查最小单元是否按计划返回
   - 检查当前阶段是否仍在正常推进
   - 检查是否出现“有动作但无正式回执”的状态

2. **卡断识别**
   - 判断当前卡点属于：
     - 单元超时未回
     - 回收失败
     - 阶段门未送审
     - 审计后未恢复推进
     - 主产物停长、配件乱长

3. **timeout 恢复触发**
   - 到超时点后，推动恢复动作
   - 包括：拉状态、提醒回执、建议重发、建议重切单元、建议回主入口重编排

4. **异常分流**
   - 判断这是：
     - 执行卡断
     - 编排问题
     - 审计卡断
     - 系统异常
     - 还是应升级给人类拍板的问题

5. **停发与防假推进提醒**
   - 当发现：
     - 主产物不再增长
     - 配件代替主线推进
     - 连续回收无效结果
     - timeout 连续发生
   - 应提醒暂停继续派发，先重编排

#### runtime keeper不负责什么
runtime keeper不负责：
- 代替主入口 Agent 重新定义任务目标
- 代替生产位亲自完成最小单元
- 代替审计位做 accepted / blocked 裁决
- 代替人类拍板位做优先级与资源决策

#### runtime keeper的定位
它的定位不是“再多一个生产角色”，而是：

> **专门保证协作链不断线、可恢复、不过度依赖人工盯盘的运行职责位。**

#### 默认归属与升级条件
- 在轻量任务中，runtime keeper 职责可默认挂在主入口 Agent 下
- 在长任务、多阶段任务、容易 timeout 的任务里，应允许显式设为独立角色

一句话口径：

> **主入口管计划推进，runtime keeper管不断线与可恢复。**

---

### 5. 人类拍板位

人类拍板位不是日常微操位，而是边界外判断与资源裁决位。

#### 人类拍板位的职责
人类拍板位负责：

1. 边界外拍板
2. 优先级取舍
3. 资源分配
4. 任务目标修订
5. 当现有规则无法覆盖时给出最终方向判断

#### 应升级到人类拍板位的典型场景
包括但不限于：
- 当前任务目标发生实质变化
- 需要调整阶段优先级
- 需要决定是否继续投入预算
- 需要决定是否接受当前风险
- 现有规则之间发生冲突且无法自行裁决

#### 人类拍板位不负责什么
人类拍板位不应负责：
- 替主入口 Agent 做日常派发
- 替 subagent 写每个最小单元 prompt
- 替审计位做逐阶段事实判断

---

### 6. 五个角色之间的流转关系

这套框架的基本流转关系应是：

1. **主入口 Agent** 负责对齐需求与编排计划
2. **一次性生产 subagent** 按最小单元完成施工
3. **主入口 Agent** 回收结果并并入当前阶段主产物
4. **runtime keeper** 持续检查是否出现卡断、timeout、无效推进与恢复缺口
5. 到阶段门后，由 **审计位** 做正式放行判断
6. 遇到超边界、优先级、预算、目标变更问题时，升级 **人类拍板位**
7. 再由 **主入口 Agent** 继续推进下一阶段或调整计划

一句话压缩：

> **主入口管结构，subagent 管施工，runtime keeper管不断线与恢复，审计位管放行，人类拍板位管边界外决策。**

---

### 7. 当前角色结构的关键约束

为了避免角色重新混位，建议先锁死以下约束：

#### 约束 1：主入口唯一
未显式交接前，不更换主入口位。

#### 约束 2：生产位一次性
生产位默认按一次性最小单元运行，不保留长链。

#### 约束 3：审计按阶段门进入
审计位不持续混在施工过程里。

#### 约束 4：runtime keeper 职责不能缺位
即使不单独设岗，也必须明确由谁承担卡断识别、timeout 恢复与停发提醒。

#### 约束 5：拍板问题不上移为日常流程
不是所有问题都要交人类，只有超出既定边界与规则覆盖范围时才升级。

#### 约束 6：成本控制默认归主入口
spawn 与模型选择，不应成为无人负责的灰区，默认由主入口 Agent 承担判断责任。

## 三、起手顺序

这套框架的第一道硬约束是：

> **需求对齐后，不允许直接起手生产。必须先进入计划与编排，再决定是否派发最小单元。**

也就是说，起手顺序不是：
- 聊完需求 → 觉得差不多 → 直接开做

而必须是：
- 需求对齐 → 读取可复用能力与经验 → 明确主目标与阶段 → 做计划编排 → 判断是否需要 spawn → 再进入生产

这一步的意义，是避免任务从第一分钟开始就陷入：
- 没有整体规划
- 没有阶段感
- 没有主产物
- 没有停损点
- 没有成本判断
- 没有复用意识

---

### 步骤 1：需求对齐

需求对齐阶段至少要压清 6 件事：

1. **任务目标是什么**
2. **这轮不做什么**
3. **当前阶段要交什么**
4. **什么算成功 / 什么算未完成**
5. **时间要求是什么**
6. **关键约束是什么**

如果这 6 件事没有压清，就不应进入后续编排。

#### 这一步的核心作用
- 防止边做边改题
- 防止不同 agent 各自理解不同任务
- 防止主文与配件顺序倒挂

---

### 步骤 2：读取可复用的 skill、经验与注册层入口

在正式编排前，主入口 Agent 必须先判断：

- 这类任务有没有现成 skill
- 有没有已沉淀的方法、模板、规则卡、阶段卡
- 注册层有没有推荐入口
- 过去是否踩过类似坑
- 是否已有更低成本、更高确定性的默认做法

#### 这一阶段的原则
不是“先做，做完再想起原来有方法”，而是：

> **先查可复用能力，再决定怎么编排。**

#### 这一步要解决的问题
- 避免从零发明流程
- 避免重复踩坑
- 避免忽视历史经验
- 避免把可以 skill 化的工作再次临场手搓

如果这一阶段跳过，后面很容易出现：
- 明明有 skill 却不用
- 明明有阶段卡却重新临场编排
- 明明已有低成本路径却走了高成本路径

---

### 步骤 3：锁定唯一主入口 Agent

在正式编排前，必须明确：
- 谁是这轮任务的唯一主入口 Agent
- 谁负责编排、派发、回收、送审、收口
- 未显式交接前，不允许主入口漂移

#### 这一步的核心作用
- 防止多入口并行导致任务发散
- 防止“大家都在推进，但没人负责总收束”
- 防止生产位反过来接管编排

如果主入口不锁定，后续所有阶段都容易虚化。

---

### 步骤 4：锁定当前阶段唯一主产物

这是起手顺序里的关键动作。

主入口 Agent 必须先明确：
- 当前阶段唯一主产物是什么
- 当前阶段禁止先长什么
- 哪些内容即使有价值，也不是这一阶段优先项

#### 典型主产物例子
- 主文轮廓
- 主文骨架
- 可做压力测试的方案骨架
- 阶段性可审版正文

#### 这一阶段必须避免
- 主产物还没立住，就先长一堆配件
- 一开始就同时追多个核心产物
- 边做边决定这阶段到底要什么

一句话规则：

> **先锁主产物，再谈生产；主产物不清，不许开工。**

---

### 步骤 5：做阶段拆分与计划编排

只有在主目标、复用能力、主入口、当前主产物都明确后，才进入正式编排。

编排至少要包含：
- 当前任务分几阶段
- 当前先推进哪一阶段
- 每阶段的目标是什么
- 每阶段的送审门在哪里
- 每阶段的放行条件与退回条件是什么
- 哪些工作必须自己做
- 哪些工作可以派发为最小单元
- 当前阶段的停损点是什么
- 当前阶段的预算与模型策略是什么

#### 这一步的核心作用
把“任务理解”变成“任务结构”，避免整个项目只靠临场判断往前推。

如果没有这一步，任务就会退化成：
- 做一步看一步
- 今天补这个，明天补那个
- 配件先于主文膨胀
- 中段开始失速

---

### 步骤 6：判断是否需要 spawn

不是所有任务都值得进入 multi-agent 派发。

主入口 Agent 需要先判断：
- 这件事是否真的需要 subagent
- 还是自己直接做更快、更稳、更省上下文
- 是否值得为这个任务付出额外协作成本
- 是否已经有 skill / 模板可以直接完成

#### 应谨慎 spawn 的情况
- 任务本身很小，主入口自己做更快
- 任务边界还不清
- 主产物还没锁定
- 阶段计划还没成形
- 单元拆分还很模糊

#### 更适合 spawn 的情况
- 任务已能压成清晰最小单元
- 单元结果可以明确回收
- 主入口继续亲做会拖慢主线推进
- 该单元适合低上下文一次性施工

一句话规则：

> **能直接做清楚的，不为协作而协作；只有值得派发的，才派发。**

---

### 步骤 7：定义最小单元并确定模型策略

决定 spawn 后，不是立刻派发，而是先定义：
- 每个最小单元的 task goal
- scope
- deliverable
- done condition
- constraints
- return target
- 建议模型或成本等级

#### 这一阶段要特别解决两个问题

##### 问题 1：单元切得假小真大
表面一个单元，实际塞了多个目标、多种产物、多阶段任务。

##### 问题 2：模型选择失控
低成本任务也默认跑昂贵模型，高价值任务反而没有预算空间。

#### 这里的硬口径
- 一个最小单元 → 一个 prompt → 一个 subagent
- 单元完成即删除，不保留成长链节点
- 模型选择必须服务任务难度与成本控制，而不是默认用最贵的

---

### 步骤 8：正式进入生产

只有前 7 步完成后，才允许进入生产阶段。

这意味着，正式生产的前提至少包括：
- 需求已对齐
- 可复用能力已检查
- 主入口已锁定
- 当前阶段唯一主产物已锁定
- 阶段计划已存在
- 是否 spawn 已判断
- 最小单元与模型策略已明确

如果这些前提不成立，就不应以“先做起来再说”作为借口直接开工。

---

### 当前起手顺序的核心结论

这套框架的起手顺序，本质上就是把“生产冲动”往后压，把“计划、复用、结构、预算”往前提。

一句话总结：

> **先对齐需求，先查可复用能力，先锁主入口与主产物，先做阶段计划，再判断是否 spawn；只有这些成立后，才进入生产。**

## 四、阶段推进机制

起手顺序解决的是“怎么开始”，阶段推进机制解决的是“开始之后怎么不散、不空转、不虎头蛇尾”。

这套框架的核心要求是：

> **任务一旦进入正式生产，就必须按阶段推进，而不是靠聊天热度、临场感觉或人肉盯盘往前拖。**

也就是说，任务推进不能是：
- 做一点看一点
- 想到什么补什么
- 做完一个单元再临场决定下一步
- 出现卡点后大家等着看谁先说话

而必须是：
- 每阶段有明确目标
- 每阶段有唯一主产物
- 每阶段有明确回收口
- 每阶段有送审门
- 每阶段有放行、退回、卡断恢复与停损机制

---

### 1. 阶段制是默认推进方式

进入正式生产后，任务默认按阶段制推进，而不是按“消息轮次”推进。

#### 阶段制的意义
阶段制的作用是：
- 让任务推进以产物和状态为核心，而不是以聊天轮次为核心
- 让主入口 Agent 能稳定管理主线
- 让审计位只在该进场的时候进场
- 让 runtime keeper 有可观察的状态点
- 让中途卡断时有明确恢复位置

#### 每阶段至少要有的要素
每个阶段至少要有：
- 阶段名称
- 阶段目标
- 当前阶段唯一主产物
- 允许的最小单元范围
- 当前阶段允许生长的配件范围
- 送审门
- 放行条件
- 退回条件
- timeout / 卡断恢复口径
- 停损点

如果一个阶段没有这些最小要素，它就更像“模糊工作段”，而不是可治理阶段。

---

### 2. 阶段内的标准推进链路

每一阶段，默认都按同一条推进链路执行：

1. **主入口 Agent** 锁定本阶段唯一主产物
2. 主入口 Agent 确认当前阶段允许做哪些最小单元、禁止先长哪些配件
3. 主入口 Agent 派发一个或多个最小单元给一次性 subagent
4. subagent 完成后回收结果
5. 主入口 Agent 判断：
   - 结果是否合格
   - 是否能并入当前阶段主产物
   - 是否需要重发 / 重切 / 退回
6. **runtime keeper** 持续检查是否出现：
   - timeout
   - 卡断
   - 假推进
   - 主产物停长
7. 当阶段达到送审条件时，进入审计位
8. 审计位给出 accepted / revision-needed / blocked
9. 主入口 Agent 根据审计结论：
   - 进入下一阶段
   - 重编排本阶段
   - 或暂停并升级

一句话压缩：

> **派发—回收—并入—监测—送审—放行/退回/阻塞，是阶段内的标准推进链。**

---

### 3. 阶段内回收原则

阶段推进最容易空转的地方，不在派发，而在回收。

所以这套机制要求：

#### 回收对象优先是“产物”，不是“过程聊天”
主入口 Agent 回收的，应优先是：
- 一个可验证的结果对象
- 一个可以并入主产物的内容块
- 一个明确的阻塞说明

而不是：
- 大段过程解释
- 长串未收束讨论
- 含糊的“我在做 / 快好了 / 先给你看一半”

#### 回收后必须做三步判断
每次回收后，主入口 Agent 至少要判断：
1. 这是不是当前单元要求的产物
2. 这能不能并入当前阶段唯一主产物
3. 这次回收是推进了主线，还是只是制造了更多边角材料

#### 回收不合格的处理方式
回收不合格时，不应默认继续堆下一单元，而应先判断：
- 是单元切太大
- 是 prompt 边界不清
- 是 subagent 越界
- 是阶段目标本身不清

一句话规则：

> **不能并入主产物的回收，不算有效推进。**

---

### 4. 阶段门与送审机制

审计位不应随时插在生产链里，而应在阶段门进入。

#### 阶段门的作用
阶段门负责判断：
- 当前阶段是否达到最小可审状态
- 主产物是否已经长到足以判断放行与否
- 是不是该继续推进下一阶段，而不是继续无边界补细节

#### 什么情况下进入送审
通常至少满足以下条件时，才应送审：
- 当前阶段唯一主产物已形成可审对象
- 本阶段最小单元已基本回收完毕
- 当前缺口已经暴露，而不是仍在混乱生成中
- 再继续补细节的收益已低于进入审计判断的收益

#### 审计的三个典型结论
- **accepted**：当前阶段成立，可以进入下一阶段
- **revision-needed**：当前阶段方向可保留，但需要回本阶段补正
- **blocked**：当前阶段暂不能继续，需要补证据、补边界、补拍板或补系统支持

#### 送审后的原则
送审不是任务结束，而是阶段切换点。

也就是说：
- accepted 后继续推进
- revision-needed 后回本阶段重编排
- blocked 后暂停，并进入恢复或升级路径

---

### 5. runtime keeper 的介入点

runtime keeper 不参与所有内容判断，但它必须对阶段运行状态持续敏感。

#### runtime keeper 应关注的信号
包括但不限于：
- 单元超时未回
- 有动作但无正式回执
- 回收持续不合格
- 主产物长时间无增长
- 配件增长快于主产物
- 已到阶段门却未送审
- 审计退回后迟迟未恢复推进

#### runtime keeper 的典型动作
包括：
- 发出状态提醒
- 提醒主入口 Agent 检查当前阶段是否已卡断
- 建议重发单元 / 重切单元 / 重编排
- 建议停发而不是继续堆单元
- 在必要时建议升级系统问题或人类拍板

一句话口径：

> **runtime keeper 不决定做什么内容，但负责提醒这条链是否还在正常跑。**

---

### 6. 阶段推进中的防空转机制

多 agent 协作最容易出现的假象，是“大家都在动，但主线没有前进”。

这套机制要求，至少要防以下 4 种空转：

#### 空转 1：单元在跑，但主产物不长
表现：不断有产出，但都并不进主产物。

#### 空转 2：不断补配件，主文始终缺位
表现：配件越来越多，阶段主目标一直没真正交付。

#### 空转 3：不断有状态更新，但没有正式结果
表现：大家一直在说“在推进”，但没有可审对象。

#### 空转 4：审计退回后重复犯同类问题
表现：阶段看似在继续，实际上只是重复失败。

#### 防空转的判断标准
如果出现以下任一情况，应视为进入空转风险：
- 连续回收内容无法并入主产物
- 两轮以上没有推进阶段唯一主产物
- 新增长内容以配件为主
- runtime keeper 连续发出卡断 / timeout 提醒
- 审计反复指出相同结构问题

---

### 7. 阶段推进中的恢复与停损

阶段推进机制不能只写“怎么跑顺”，还要写“跑不顺时怎么办”。

#### 出现卡断时，先做恢复判断
优先判断：
- 是单元问题
- 是编排问题
- 是审计阻塞
- 是系统异常
- 是目标变化

#### 恢复动作的典型顺序
可按以下顺序判断：
1. 提醒正式回执
2. 检查当前单元是否需要缩小或重切
3. 判断是否需要重发该最小单元
4. 判断是否应暂停新增派发，先回主入口重编排
5. 若已超出当前阶段边界，则升级审计位 / 系统位 / 人类拍板位

#### 什么时候该停损
出现以下情况时，应停止继续按原计划派发：
- 当前阶段主产物连续不增长
- 小单元错误开始牵连整阶段返工
- 回收成本已明显高于继续生产收益
- 当前阻塞已经不是施工问题，而是边界/资源/系统问题

一句话规则：

> **恢复优先于硬拖，停损优先于假推进。**

---

### 8. 阶段完成的判断

一个阶段完成，不是因为“聊得差不多了”，而是因为它满足了明确状态条件。

当前建议，阶段完成至少要满足：
- 阶段唯一主产物已经形成
- 阶段内需要的关键单元已回收
- 当前主要缺口已暴露
- 审计已给出 accepted 或可进入下一阶段的明确放行口径
- 主入口 Agent 已明确下一阶段入口

如果没有这些条件，就不应把阶段口头宣布为“差不多完成”。

---

### 当前阶段推进机制的核心结论

这套机制想解决的，不只是“让大家一起协作”，而是让每一阶段都具备：
- 可推进
- 可观察
- 可回收
- 可审计
- 可恢复
- 可停损

一句话总结：

> **阶段推进不是边做边聊，而是围绕主产物，按派发、回收、并入、监测、送审、恢复、停损这一整条链来运行。**

## 五、最小单元原则

“最小单元”是这套框架里最容易说对、最容易做错的地方。

很多时候，表面上看是在拆小，实际上只是把一个大任务换了个更碎的名字，结果：
- prompt 还是很长
- 目标还是很多
- 回收还是很模糊
- subagent 还是在带长上下文干活

所以这里必须先把口径锁死：

> **最小单元，不是“看起来小”，而是“目标单一、产物单一、边界单一、可独立回收、不会把上下文重新拖厚”的施工单元。**

---

### 1. 最小单元的核心定义

当前统一定义为：

> **一个最小单元 → 一个 prompt → 一个 subagent → 一个可回收产物。**

并且：
- subagent 完成后即删除
- 不保留为长链上下文节点
- 不负责跨阶段延续
- 不承担额外配件生产任务

这条定义的核心，不是形式整齐，而是为了硬控：
- 上下文厚度
- 单元边界
- 回收清晰度
- 返工成本

---

### 2. 什么样的单元才算“真小”

一个最小单元至少应满足以下 5 个条件：

#### 条件 1：只服务一个明确目标
它必须只回答一个问题、完成一个动作，或补一个清晰缺口。

#### 条件 2：只交一个核心产物
不能一个单元同时交：
- 正文片段
- 配件说明
- 审计建议
- 额外延展方案

如果一个单元天然会交多个核心产物，通常说明单元切大了。

#### 条件 3：边界清楚
必须明确：
- 做什么
- 不做什么
- 产物交到哪里
- 什么算完成

#### 条件 4：可独立回收
主入口 Agent 在不重新读整条历史聊天的前提下，也能判断：
- 这个产物是不是合格
- 能不能并入当前主产物
- 要不要重发或重切

#### 条件 5：失败成本可控
一个最小单元即使失败，也不应轻易拖垮整个阶段，更不应导致整阶段从头重跑。

---

### 3. 假小真大的常见表现

当前最需要警惕的，不是“没拆”，而是“假装拆了”。

以下情况通常都属于假小真大：

#### 情况 1：一个单元塞多个目标
例如：
- 既补主文，又顺手补配件
- 既收证，又做判断，还给建议
- 既写框架，又带着补实施细则

#### 情况 2：一个单元塞多个产物
例如：
- 同时交正文、表格、补充说明、下一步方案
- 一次返回多个方向的半成品

#### 情况 3：任务描述仍然依赖大段历史上下文
如果一个单元必须靠：
- 大量前情摘要
- 多轮聊天背景
- 多份未收束材料
才能理解，那它通常就不是真正意义上的最小单元。

#### 情况 4：单元做完后仍需要长解释才能回收
如果一个 subagent 回来后，主入口 Agent 还得花大量成本去听它解释“我到底做了什么”，那说明这个单元并没有被切到可回收粒度。

一句话判断：

> **真正的小单元，应该让回收比理解更轻，而不是理解比回收更重。**

---

### 4. 最小单元的固定字段

为了避免“口头觉得差不多”，每个最小单元至少应具备固定字段。

当前建议最少包括：
- `task goal`
- `scope`
- `deliverable`
- `done condition`
- `constraints`
- `return target`
- `suggested model / cost tier`

#### 各字段的作用
- `task goal`：只说这一个单元要解决什么
- `scope`：锁定范围，避免扩题
- `deliverable`：明确产物形态
- `done condition`：告诉 subagent 什么才算完成
- `constraints`：防止越界与误解
- `return target`：告诉它交回给谁、回哪一阶段
- `suggested model / cost tier`：把模型选择与成本控制前置，而不是默认最贵

---

### 5. prompt 的控制原则

一个最小单元虽然是“一个 prompt”，但不代表 prompt 可以无限膨胀。

#### prompt 应该承担什么
它应该承担：
- 说明目标
- 给出必要边界
- 提供最少必要输入
- 明确交付格式

#### prompt 不应承担什么
它不应承担：
- 把整条项目历史重新灌给 subagent
- 同时交代多个阶段目标
- 把“你自己理解着办”写成默认策略
- 把主入口 Agent 没想清楚的事转嫁给 subagent 自行判断

一句话口径：

> **prompt 是施工指令，不是把不确定性整体下放。**

---

### 6. subagent 的生命周期原则

这套框架里的 subagent，默认必须是一次性消耗品。

#### 生命周期规则
- 领一个最小单元
- 完成一个产物
- 交回主入口 Agent
- 完成后删除

#### 为什么必须完成即删
因为一旦保留：
- 它会开始积累自己的历史上下文
- 它会逐渐从施工单元变成新的长链节点
- 协作成本会重新上升
- 主入口 Agent 的回收与管理也会变重

所以这里的口径不能含糊：

> **subagent 是一次性工作单，不是并行常驻主脑。**

---

### 7. 最小单元与模型选择

最小单元原则不仅是任务拆分原则，也是成本控制原则。

#### 为什么模型选择必须绑在单元上
因为不绑在单元上，就容易出现：
- 小任务也默认跑高成本模型
- 低风险任务也消耗高预算
- 真正需要高质量推理的阶段反而预算被提前吃掉

#### 当前建议的基本原则
- 低复杂度、格式化、收集型任务 → 优先低成本模型
- 高歧义、高推理、高压缩成本任务 → 才考虑更高成本模型
- 单元失败可能导致整阶段返工时，应优先选择更稳妥方案，而不是只看单次便宜

一句话口径：

> **模型选择跟着单元难度、风险与返工成本走，不跟着习惯走。**

---

### 8. 最小单元的复用与 skill 化关系

最小单元不只是施工颗粒，也是 skill 沉淀的来源。

如果一类最小单元反复出现，并且：
- 目标稳定
- 输入稳定
- 输出稳定
- 约束稳定
- 回收标准稳定

那它就不应该一直停留在“临时派发单元”，而应主动考虑：
- 是否能沉淀成 skill
- 是否能成为注册层现役入口
- 是否能让后续编排不再每次重新描述

一句话规则：

> **反复出现的稳定最小单元，应尽快脱离人工临场派发，进入 skill 化与注册层复用。**

---

### 当前最小单元原则的核心结论

最小单元的目的，不是把任务切碎得好看，而是为了真正做到：
- 易派发
- 易执行
- 易回收
- 易删除
- 易控成本
- 易沉淀复用

一句话总结：

> **真正的最小单元，是一个目标、一个 prompt、一个 subagent、一个产物，完成即删；它既要小到可控，也要稳到可回收。**

## 六、主产物优先原则

这套框架里最需要被硬性守住的一条，就是：

> **任何阶段都必须先保主产物，再谈配件；不能让配件增长替代主线推进。**

因为多 agent 协作最常见、也最隐蔽的跑偏方式，不是彻底不做，而是：
- 大家都在产出
- 文件越来越多
- 配件越来越细
- 说明越来越充分
- 但主产物始终没有真正站起来

也就是说，真正需要被防的，不只是“没做”，而是：

> **看起来一直在做，但主线没长出来。**

---

### 1. 什么是主产物

主产物，不是“今天最顺手写的那个文件”，而是当前阶段真正承载主任务推进的核心对象。

它通常具有 4 个特征：

#### 特征 1：它承载当前阶段的主要目标
例如：
- 主文轮廓
- 主文骨架
- 可做压力测试的方案骨架
- 阶段性可审版正文

#### 特征 2：它是后续阶段继续生长的主承载位
也就是说，后续阶段不是绕开它另起炉灶，而是继续在它之上推进。

#### 特征 3：它是审计位判断的主要对象
审计位在阶段门做判断时，主要看的应是主产物是否成立，而不是配件是否写得多。

#### 特征 4：它最直接代表“主线有没有推进”
如果一个阶段结束时，主产物没有明显长出来，那通常不能说主线已经有效推进。

一句话口径：

> **主产物是阶段主目标的主承载位，不是材料堆里随手最先长出来的那份文件。**

---

### 2. 为什么必须主产物优先

主产物优先，不是写作偏好，而是为了防止协作系统自动滑向“配件膨胀、主线失踪”。

如果没有这条原则，协作往往会滑向以下路径：
- 主文还没立住
- 但大家已经开始补说明、补卡片、补边界、补附表
- 配件数量迅速增长
- 主入口 Agent 回收了很多内容，却很难并入主线
- 最终看起来很忙，实际上阶段主目标并没完成

#### 这条原则要解决的核心问题
- 防止主文长期缺位
- 防止配件先于主线膨胀
- 防止“材料增长”冒充“主线推进”
- 防止回收成本被边角件拖高
- 防止审计面对的是一堆附件，而不是可判断的主对象

---

### 3. 每阶段必须先锁唯一主产物

主产物优先原则，首先要求：

> **每个阶段开始前，主入口 Agent 必须先明确“这一阶段唯一主产物是什么”。**

不能出现以下情况：
- 一阶段同时追两个主产物
- 做着做着才临时决定主产物是谁
- 主产物与配件没有边界
- 配件反过来占据阶段中心

#### 锁主产物时至少要说清
- 这一阶段的唯一主产物名称
- 它要长到什么程度才算达到送审条件
- 哪些补件可以服务它
- 哪些内容即使重要，也不在这一阶段先做

一句话规则：

> **主产物不锁，阶段就没有中心；阶段没有中心，协作就会自动发散。**

---

### 4. 配件的合法性来自“服务主产物”

配件不是天然不该做，而是必须服从主产物优先原则。

一个配件只有在以下条件满足时，才算“当前阶段合法生长”：
- 它明确服务当前阶段主产物
- 它解决的是主产物推进中的真实阻塞
- 它不会反客为主，拖走阶段重点
- 它的回收成本低于它对主产物推进的帮助

#### 什么样的配件属于“当前阶段可接受”
例如：
- 为主文骨架补一张必要结构表
- 为阶段送审补一张必须的边界卡
- 为主方案补一个阻塞判断件

#### 什么样的配件属于“当前阶段不应优先”
例如：
- 主文还没立住，就先补一整套附录
- 主方案还没可审，就先补大量实施细则
- 当前阶段只需要一个判断件，却顺手长出多个外围说明件

一句话口径：

> **配件只有在服务主产物时才有优先权；不能服务主产物的配件，即使有价值，也应延后。**

---

### 5. 什么才算“有效推进主线”

在这套框架里，不是所有新增内容都算推进。

当前建议，只有满足以下任一条件，才算有效推进主线：
- 当前阶段唯一主产物有实质增长
- 某个新增内容已被并入主产物
- 某个阻塞主产物推进的关键问题被真正消除
- 主产物已达到更接近送审的状态

反过来，以下情况通常不应算主线推进：
- 只新增了大量解释文字
- 只新增了外围说明件
- 只补了未来阶段才需要的配件
- 只是产生了更多“可以再讨论”的中间材料

一句话规则：

> **能并入主产物、能推动主产物、能解除主产物阻塞的，才算主线推进。**

---

### 6. 主产物停长时，默认视为风险信号

主产物优先原则要求，不只要看“有没有东西在增长”，而要看“主产物有没有在增长”。

如果出现以下情况，应视为风险信号：
- 配件数量在涨，主产物不涨
- 讨论很多，主产物不涨
- 单元一直在派发，主产物不涨
- 审计前材料越来越多，但主对象仍不可审

#### 这类风险信号说明什么
通常说明：
- 阶段重心已经偏了
- 配件开始替代主线
- 回收没有真正并入主产物
- 主入口 Agent 需要停下来重看编排

runtime keeper 在这里应特别敏感，因为：

> **主产物停长，往往比单个单元 timeout 更能说明系统已经开始失焦。**

---

### 7. 主产物优先与审计关系

主产物优先原则，也决定了审计位应该看什么。

审计位在阶段门判断时，优先检查的应是：
- 当前阶段唯一主产物是否成立
- 主产物是否已达到最小可审状态
- 当前配件是否真的服务了主产物
- 有没有出现“配件写得很多，但主文仍未站住”的情况

也就是说：
- 配件不能代替主产物接受审计
- 说明件不能代替主方案被放行
- 附表不能代替正文承担阶段完成判断

一句话口径：

> **审计看的是主产物是否成立，而不是附件是否热闹。**

---

### 8. 主产物优先与成本控制关系

主产物优先原则，也是一条成本控制原则。

因为如果不先保主产物，就很容易出现：
- 预算被外围配件消耗掉
- 高成本模型先被拿去跑边角任务
- 真正主线推进时，预算反而不足
- 单元失误不只是浪费一次调用，而是浪费整个阶段的编排时间

#### 这条原则带来的成本约束
- 模型预算应优先保证主产物推进
- 派发预算应优先给主产物相关单元
- 配件类工作只有在服务主产物时才值得占用预算

一句话规则：

> **预算优先保证主线，不为外围热闹买单。**

---

### 当前主产物优先原则的核心结论

主产物优先原则，本质上是给多 agent 协作加一个“主线收束器”。

它要解决的不是“配件能不能做”，而是：
- 什么时候必须先保主文
- 什么内容算真的推进
- 什么增长只是看起来忙
- 什么时候应该停下来重编排

一句话总结：

> **主产物优先，不等于不要配件；而是所有配件、预算、派发与审计，都必须围绕当前阶段唯一主产物服务。**

## 七、断层点检查

这套框架里，“断层”不是指任务慢，而是指：

> **协作链在某个关键接口上失去连续性，导致任务虽然看似还在动，但已经无法稳定地派发、回收、并入、送审或推进下一阶段。**

断层的危险在于，它往往不是立刻表现为“彻底停机”，而是表现为：
- 有人在说话
- 有文件在增长
- 有单元在派发
- 但真正的主线流转已经断了

所以，断层点检查的目的，不是事后复盘“哪里出问题了”，而是：

> **尽可能在主线真正失控前，看见断层信号，并把它接回来。**

---

### 1. 断层不是单点故障，而是流转失真

在这套框架里，断层不只是“某个 subagent 超时没回”这么简单。

更常见的是以下几类流转失真：
- 任务理解有了，但没进入计划编排
- 计划写了，但没变成可执行单元
- 单元做了，但没被有效回收
- 回收有了，但没并入主产物
- 阶段到了，但没人送审
- 审计退回了，但没人恢复推进
- 任务做完了，但没有复盘沉淀

也就是说，断层不是“有没有动作”，而是：

> **动作还能不能继续被主线吸收。**

---

### 2. 第一类断层：需求对齐 → 计划编排

#### 典型表现
- 需求聊清楚了，但没有形成计划
- 大家开始直接起手生产
- 任务看起来在推进，实际上没有阶段结构
- 主入口 Agent 没有把任务压成任务树或阶段链

#### 风险
一旦这里断层，后面通常会直接滑向：
- 做一步看一步
- 主产物不清
- 配件先长
- 后续返工成本高

#### 默认接法
- 主入口 Agent 应暂停生产冲动
- 先补计划编排
- 明确阶段目标、主产物、送审门、停损点

#### 谁应敏感
- 主入口 Agent
- runtime keeper（如果已发现直接起手生产趋势）

---

### 3. 第二类断层：计划编排 → 单元派发

#### 典型表现
- 计划有了，但没有被压成最小单元
- 任务还是太大、太糊、太依赖上下文
- 大家知道“要做什么”，却不知道“怎么发第一单”

#### 风险
这里一断，常见后果是：
- 计划停留在纸面
- 生产位拿不到可执行单元
- 主入口 Agent 被迫亲自下场做所有事情
- multi-agent 框架退化成单人硬扛

#### 默认接法
- 检查当前阶段是否已锁唯一主产物
- 重切任务颗粒度
- 给出正式最小单元字段，而不是口头交代

#### 谁应敏感
- 主入口 Agent
- runtime keeper（发现阶段计划迟迟无法转派发时）

---

### 4. 第三类断层：单元派发 → 结果回收

#### 典型表现
- subagent 已启动，但长时间无正式回执
- 有动作，但没有可回收产物
- 只回了一堆过程解释
- 主入口 Agent 不知道这单算完成还是没完成

#### 风险
这是最常见的运行断层之一，因为：
- 派发看起来已经发生
- 但回收没有成立
- 后面所有并入、送审都无从谈起

#### 默认接法
- runtime keeper 先识别是 timeout、假推进还是回收格式不成立
- 主入口 Agent 判断是重发、重切还是正式阻塞说明
- 禁止在回收不成立时继续盲目堆下一单元

#### 谁应敏感
- runtime keeper（首责）
- 主入口 Agent

---

### 5. 第四类断层：结果回收 → 并入主产物

#### 典型表现
- 回收到了内容
- 但主入口 Agent 发现无法并入当前阶段主产物
- 结果越来越多，主产物却不长
- 大量外围说明件开始堆积

#### 风险
这是最隐蔽的一类断层，因为表面看起来“产出了很多”，但实际上：
- 主线没有被推进
- 配件正在替代主产物
- 回收成本会越来越高

#### 默认接法
- 主入口 Agent 先判断：
  - 是单元切太大
  - 是 prompt 边界不清
  - 是配件越位
  - 还是主产物定义不清
- 若不能并入，应尽快停下来，而不是继续追加更多边角内容

#### 谁应敏感
- 主入口 Agent（首责）
- runtime keeper（监测主产物停长）

---

### 6. 第五类断层：阶段形成 → 送审进入

#### 典型表现
- 当前阶段主产物其实已经形成
- 但没人明确触发送审
- 团队继续补细节、补配件、补解释
- 阶段迟迟不进入裁决点

#### 风险
这里不断，会导致：
- 阶段边界越来越模糊
- 审计位迟迟无法进入
- 当前阶段本该结束却被无限拉长
- 下阶段被整体拖延

#### 默认接法
- 主入口 Agent 主动判断是否已达最小可审状态
- runtime keeper 关注“阶段已成熟却未送审”的悬空状态
- 一旦满足送审条件，应优先送审，而不是继续无限补细节

#### 谁应敏感
- 主入口 Agent（首责）
- runtime keeper

---

### 7. 第六类断层：审计结论 → 恢复推进

#### 典型表现
- 审计已经给出 accepted / revision-needed / blocked
- 但结论没有真正转成后续动作
- revision-needed 后没人重编排
- blocked 后没人分流或升级

#### 风险
这里断层后，会出现一种很典型的悬空：
- 形式上已经审过了
- 实际上链路没有继续跑
- 团队误以为“审过就算处理了”

#### 默认接法
- 主入口 Agent 必须把审计结论翻译成下一动作
- runtime keeper 应检查审后状态是否被真正接走
- blocked 情况下应尽快判断：补证据、补系统、补拍板，还是暂停

#### 谁应敏感
- 主入口 Agent（首责）
- runtime keeper
- 审计位（可提醒结论未被接续）

---

### 8. 第七类断层：阶段完成 → 复盘沉淀

#### 典型表现
- 任务做完了
- 但没有复盘
- 没有把方法、坑点、成本判断、模型经验沉淀下来
- 下一轮任务又从头再来

#### 风险
这类断层不会立刻拖垮当前任务，但会持续拖垮未来任务。

因为它意味着：
- 坑还会重复踩
- skill 不会长出来
- 注册层不会变强
- 成本意识不会累积

#### 默认接法
- 主入口 Agent 在阶段或任务结束后补复盘
- 对稳定方法推动 skill 化
- 把应进入注册层的入口显式补上

#### 谁应敏感
- 主入口 Agent（首责）
- 人类拍板位（若长期未沉淀、反复踩坑）

---

### 9. 断层信号的统一判断

为了避免“感觉不对但说不清”，当前建议把以下信号统一视为断层预警：

- 需求已清，但没有计划
- 计划已出，但迟迟不能派单
- 单元已派，但没有正式回收
- 回收有了，但主产物不长
- 阶段已成熟，但没人送审
- 审计已结束，但没有进入下一动作
- 任务结束了，但没有复盘沉淀

如果其中任一信号持续存在，就不应继续假装主线还在正常运行。

---

### 10. 断层后的默认处理原则

出现断层后，默认不是“继续往前顶”，而是先判断断在什么层：
- 断在编排层
- 断在生产层
- 断在回收层
- 断在审计层
- 断在恢复层
- 断在沉淀层

#### 默认处理顺序
1. 先标记断层点
2. 判断当前断层归属哪一层
3. 由对应角色先接：
   - 编排问题 → 主入口 Agent
   - timeout / 卡断 / 悬空 → runtime keeper
   - 放行判断 → 审计位
   - 资源 / 边界 / 优先级 → 人类拍板位
4. 接回后再决定：
   - 恢复原阶段
   - 回本阶段重编排
   - 停损并升级

一句话规则：

> **断层后先接链，不先假装继续推进。**

---

### 当前断层点检查的核心结论

断层点检查，真正要防的不是“完全停摆”，而是：
- 看起来还在动
- 实际已经失去连续流转
- 最后把项目拖进高成本空转

一句话总结：

> **断层点检查，就是持续检查这条链有没有从需求、编排、派发、回收、并入、送审、恢复到复盘一路接住；任何一环接不住，都不该再被包装成正常推进。**

## 八、停损点与防虎头蛇尾机制

多 agent 协作最危险的，不只是“做不出来”，而是：
- 开头很快
- 中段很忙
- 后段很虚
- 产物很多
- 主线很弱
- 最后没有人明确说该停、该收、该重编排

所以，这套框架必须把“停损”写成正式机制，而不是临场感觉。

一句话定义：

> **停损点，是指当协作继续按当前方式推进，新增成本已明显高于新增收益时，系统应停止硬拖、停止假推进，转入重编排、恢复或升级判断的正式节点。**

而“防虎头蛇尾机制”，要防的就是：
- 起手很漂亮
- 中间不断补细节
- 后面阶段迟迟不收
- 最终既没主交付，也没正式止损

---

### 1. 为什么必须显式设置停损点

如果没有停损点，协作会天然滑向两种坏结果：

#### 坏结果 1：硬拖
已经明显失焦、返工、卡断，仍然继续派发、继续补件、继续堆上下文。

#### 坏结果 2：软烂尾
没有人明确宣布失败或暂停，但任务已经失去主线，只剩零散解释、补件和悬空状态。

所以停损点的意义不是“保守”，而是：
- 阻止低质量惯性继续烧预算
- 阻止阶段错误放大为整任务错误
- 阻止大家把“还在动”误当成“还在推进”

---

### 2. 停损点不是失败，而是成本控制节点

在这套机制里，停损点不应被理解成“任务失败”，而应被理解成：

> **当前推进路径已不再经济、不再稳定、不再值得继续按原方式跑。**

也就是说，到了停损点，不是直接放弃任务，而是必须改问：
- 该不该继续按原计划跑
- 是重切单元，还是重做阶段编排
- 是恢复后继续，还是暂停升级
- 是不是已经触碰预算边界、结构边界或系统边界

---

### 3. 哪些信号应触发停损判断

当前建议，出现以下任一类信号，就应进入停损判断，而不是继续惯性推进：

#### 信号 1：主产物连续不增长
表现：
- 单元持续在派发
- 配件持续在增长
- 讨论持续在推进
- 但当前阶段唯一主产物连续两轮以上没有实质增长

#### 信号 2：小单元错误开始牵连整阶段返工
表现：
- 单个单元失误，已经不是局部返工
- 而是影响主产物结构、阶段判断或整体送审条件

#### 信号 3：回收成本显著高于继续生产收益
表现：
- 主入口 Agent 花越来越多时间在理解、解释、清理回收物
- 新增单元带来的有效推进越来越少

#### 信号 4：审计连续指出同类结构问题
表现：
- revision-needed 连续发生
- 且退回原因高度相似
- 说明当前不是施工偶发问题，而是编排路径有误

#### 信号 5：runtime keeper 连续发出卡断 / timeout / 假推进提醒
表现：
- 超时不是偶发，而是连续出现
- 阶段成熟却迟迟不送审
- 回收长期不成立
- 主线长期悬空

#### 信号 6：预算消耗开始偏离主线目标
表现：
- 大量预算被外围配件、解释件、重复返工吃掉
- 真正主产物推进反而预算不足

#### 信号 7：任务目标已变化，但计划未重编排
表现：
- 人类拍板位或现实约束已改变目标
- 但执行链还在按旧目标惯性运行

---

### 4. 停损判断应由谁提出

停损不能变成“大家都觉得不对，但没人先说”。

当前建议：

#### 主入口 Agent 可以提出停损
当发现：
- 主产物停长
- 回收无法并入
- 编排路径明显偏离
- 审计退回已不是局部补正可解

#### runtime keeper 可以提出停损提醒
当发现：
- timeout 连续发生
- 卡断持续存在
- 假推进明显
- 阶段悬空迟迟未收

#### 审计位可以提出停损建议
当发现：
- 当前问题已不再是“再补一点就能过”
- 而是结构层、边界层、阶段层错误

#### 人类拍板位可以做最终停损裁定
当发现：
- 当前继续推进不值得
- 或需要改优先级、改预算、改目标

一句话口径：

> **主入口能提停，runtime keeper 能报警，审计位能建议，人类拍板位能裁定。**

---

### 5. 停损后的默认动作

触发停损判断后，默认不能继续“边跑边看”，而应进入明确分流。

当前建议的默认分流是：

#### 路径 A：回本阶段重编排
适用于：
- 阶段目标还成立
- 但当前单元设计、回收方式或节奏判断有误

#### 路径 B：缩小粒度后恢复推进
适用于：
- 主要问题在于单元切太大、prompt 太糊、回收不清
- 核心主线仍然成立

#### 路径 C：暂停并升级
适用于：
- 已超出编排问题，进入资源 / 系统 / 边界 / 目标变更问题

#### 路径 D：阶段性收口后转复盘
适用于：
- 当前阶段继续推进已不划算
- 但已有结果足以形成阶段结论与复盘输入

一句话规则：

> **停损后必须转入明确路径，不能停在“感觉不对但先放着”。**

---

### 6. 防虎头蛇尾的核心，不是催快，而是阶段收束

“虎头蛇尾”在这里，通常不是因为不努力，而是因为：
- 开头结构感很强
- 中段不断加料
- 后段没人判断“这一阶段到底该不该收”
- 结果阶段越拖越虚，任务越来越像悬空项目

所以防虎头蛇尾的关键，不是催更快，而是：

> **每个阶段都必须能被明确收束，而不是无限续命。**

---

### 7. 防虎头蛇尾的 5 条机制约束

#### 约束 1：每阶段必须有明确收口条件
不能只写阶段目标，不写“达到什么状态就该收”。

#### 约束 2：阶段一旦满足送审条件，不应无限补细节
否则很容易从“快可审”拖成“永远差一点”。

#### 约束 3：连续两轮未推进主产物，必须复查是否已进入虎头蛇尾区
这条不是绝对停止线，但必须触发检查。

#### 约束 4：外围配件增长快于主产物时，应视为阶段失焦信号
说明协作重心已经开始漂移。

#### 约束 5：阶段结束时必须明确三选一
- 进入下一阶段
- 回本阶段重编排
- 阶段性收口并升级 / 复盘

不能停在“先这样吧，后面再说”。

---

### 8. 停损与预算控制的关系

停损机制同时也是预算保护机制。

因为如果没有停损：
- 小错误会滚成大返工
- 配件会挤占主线预算
- 昂贵模型会被低价值任务消耗掉
- 最后不是没花钱，而是花了钱却没推进主产物

所以停损判断里，预算问题要被显式纳入：
- 当前投入是否还在服务主产物
- 当前返工是否值得继续承受
- 是否应该切到更低成本路径
- 是否需要保留预算给后续高价值阶段

一句话口径：

> **停损不是为了少做事，而是为了不把预算烧在错误路径上。**

---

### 9. 停损与复盘沉淀的关系

如果一个阶段触发了停损，但最后没有复盘，那这次停损就只救了当前，没有救未来。

所以一旦发生停损，后续至少应回答：
- 是什么信号触发了停损
- 断在了哪一层
- 是编排问题、单元问题、审计问题还是系统问题
- 下次应在更前面加什么规则、模板或 skill

一句话规则：

> **停损如果不沉淀，下次还会在同一个地方再停一次。**

---

### 当前停损点与防虎头蛇尾机制的核心结论

停损点与防虎头蛇尾机制，本质上是给协作链加一套“防惯性失控装置”。

它要解决的不是“如何永远不停”，而是：
- 什么时候必须承认当前路径不对
- 谁来发出停损信号
- 停下来以后走哪条路
- 如何避免阶段无限续命

一句话总结：

> **停损不是中断推进，而是阻止错误路径继续冒充推进；防虎头蛇尾，不是逼任务一直跑，而是逼每个阶段都能被明确收住。**

## 九、配件生长原则

这一板不展开成重规则层，只保留一条轻原则，避免它反过来限制正常生产。

一句话规则：

> **配件可以长，但前提是它真的在帮当前主产物过关，而不是在替未来阶段提前铺陈，反客为主。**

这条原则的作用只有两个：
- 提醒主入口 Agent 先看主产物，再判断配件是否值得当前就长
- 防止配件膨胀重新吞掉主线

当前口径先到这里，不再往下压成更重的配件治理细则。

## 十、当前需要继续对齐的关键问题

这份一级骨架后续还需要继续压清至少 5 个问题：

5. 复盘输出是独立阶段，还是并入最终收口
