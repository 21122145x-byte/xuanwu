# 玄武｜升级异常正式收证｜OpenClaw v2026.4.5｜网关拉起失败、weixin 兼容断裂与残留清理 v1

- 升级对象：`OpenClaw v2026.4.5`
- 当前运行版本（升级后）：`2026.4.5 (3e72c03)`

## 一、现象

本次升级到 `2026.4.5` 后，**更新完成后的第一时间 gateway 无法正常拉起**。
随后，外部工具（antigravity / Claude Code）介入，对配置与插件做了一轮修补；但从正式收证角度看，**真正与启动失败最贴近的硬故障样本，集中在 `openclaw-weixin` 插件兼容断裂**。

最终在将 `openclaw-weixin` 扩展残留目录彻底清理后，当前状态恢复为：
- gateway 正常在线
- `openclaw doctor` 插件错误数为 `0`
- `openclaw-weixin` 扩展目录已不存在

---

## 二、四个显式问题与修补动作

## 问题 1｜配置校验失败：Discord channels 仍使用 legacy `allow`
### 现象
control-ui 报错：
> `channels.discord.accounts.<id>.guilds.<id>.channels.<id>.allow is legacy; use ... enabled instead`

### 外部工具修补
Claude Code 将 `openclaw.json` 中 6 个 Discord 账号的 channel 配置从：
- `"allow": true`
替换为：
- `"enabled": true`

### 当前判断
这是一条**升级后配置口径兼容问题**，需要修，但它更像是**配置校验层阻塞项**，不是后续 `weixin` 启动崩溃的唯一主因。

---

## 问题 2｜配置 invalid：`talk.apiKey` 属于 legacy 键
### 现象
control-ui 报错：
> `talk.voiceId/talk.voiceAliases/talk.modelId/talk.outputFormat/talk.apiKey are legacy; use talk.providers.<provider>`

### 外部工具修补
Claude Code 移除了顶层：
- `talk.apiKey`
保留：
- `talk.providers.elevenlabs.apiKey`

### 当前判断
这也是一条**升级后配置 schema 兼容问题**。它属于必须清理的 legacy 项，但仍更偏**配置口径修复**，不是后续 gateway 长时间拉不起的最硬主故障源。

---

## 问题 3｜Discord `groupPolicy` 被改写
### 现象
外部工具将 Discord 顶层及多账号：
- `groupPolicy: "allowlist"`
改成：
- `groupPolicy: "open"`

### 当前判断
这不是启动失败的直接日志根因，反而是**升级期间被动动过的配置面**。

从系统治理角度，这条需要单独复核，因为它牵涉：
- 群触达边界
- 触发面扩大
- 安全审计口径变化

因此：
> **这条不能混入“gateway 拉不起”的主故障结论，必须作为并行配置改动单独复核。**

---

## 问题 4｜gateway 仍拉不起：`openclaw-weixin` 插件兼容断裂
### 现象
在 `gateway.err.log` 中，升级后启动阶段反复出现：

```text
[plugins] openclaw-weixin failed to load from /Users/xiaojingbo/.openclaw/extensions/openclaw-weixin/index.ts:
TypeError: (0 , _pluginSdk.resolvePreferredOpenClawTmpDir) is not a function
```

时间集中在：
- `2026-04-07T07:49:49+08:00` 起
- `07:59` 至 `08:01` 仍持续出现

### 邻接证据
`node.err.log` 同时出现大量：

```text
node host gateway connect failed: connect ECONNREFUSED 127.0.0.1:18789
node host gateway closed (1006)
node host gateway closed (1012): service restart
```

这说明在该时间段内：
- gateway 端口确实未正常拉起
- node 侧持续重试连接
- 启动链伴随 service restart 循环

### 根因判断
当前最硬判断是：
> **升级到 `2026.4.5` 后，外部扩展 `openclaw-weixin` 仍依赖旧 plugin-sdk 兼容层中的函数 `resolvePreferredOpenClawTmpDir`，该函数在当前版本口径下已不可用，导致插件在启动期同步抛出 TypeError，从而拖垮 gateway 启动链。**

这条是本次“gateway 拉不起”的**主故障源候选**，且证据强于前 3 条配置兼容问题。

## 五、并行异常与次级问题

除主故障外，本轮还看到三类并行问题：

### 1. 旧更新检查脚本路径失效
`update-check.log` 中反复出现：
```text
/bin/sh: /Users/xiaojingbo/.openclaw/cron/check-update.sh: No such file or directory
```
这说明升级后仍有**旧脚本路径残留**。

### 2. topic judge 404
`gateway.err.log` 中出现：
```text
judgeNewTopic failed ... Anthropic topic-judge failed (404)
```
这说明升级后还有一条**topic judge provider / endpoint** 异常支线。

### 3. cron list 表格渲染 `padEnd` 异常
#### 现象
在 `09:00` 后的新样本里，`openclaw cron list` 持续出现：
```text
TypeError: Cannot read properties of undefined (reading 'padEnd')
```

#### 已锁到的具体文件
- `/opt/homebrew/lib/node_modules/openclaw/dist/cron-cli-Dtvnx5y1.js`

#### 已锁到的关键实现
- `149`：`const pad = (value, width) => value.padEnd(width);`
- `194-202`：`cron list` 表头列
- `206-236`：逐 job 渲染表格行

#### 已锁到的具体脏记录
通过 `openclaw cron list --json` 与底层存储文件交叉核对，确认：
- 旧 `oa-collect` job 为**无 id 脏记录**
- 新增规范 `oa-collect` 后，旧脏记录仍未被 `cron rm oa-collect` 正常移除
- `cron rm oa-collect` 返回：`{ ok: true, removed: false }`

#### 真实落点
- 持久化存储：`/Users/xiaojingbo/.openclaw/cron/jobs.json`
- 在该文件中，旧脏记录 `oa-collect` 位于旧索引位置，且**缺少 `id` 字段**
- 规范重建后，新记录获得：`id = 17181a1b-d760-4d62-be62-e9bbd89c1a1b`

#### 已执行修复
1. 备份旧记录：
   - `/Users/xiaojingbo/.openclaw/backup/oa-collect-before-readd-2026-04-07_1010.json`
   - `/Users/xiaojingbo/.openclaw/backup/jobs.json.before-oa-collect-clean-2026-04-07_1015`
2. 手动从 `~/.openclaw/cron/jobs.json` 中删除**无 id 的旧 `oa-collect` 脏记录**
3. 保留新规范记录
4. 重启 gateway
5. 复测 `openclaw cron list`

#### 修复后结果
- `openclaw cron list` 表格恢复正常
- `openclaw cron list --json` 中 `oa-collect` 只剩一条规范记录
- cron store jobs 数从 `52` 回到 `51`

#### 当前判断
> **这条 `padEnd` 已可压实为：`openclaw cron list` 的 CLI 表格渲染层，在处理 cron 存储里的无 id 脏 job 记录时崩溃；手动清除旧 `oa-collect` 脏记录并重启 gateway 后，问题消失。**

### 当前判断
上述三条都应记账，但其中：
- `openclaw-weixin` 启动断裂 = 本轮主故障链
- `padEnd` = 已形成“脏数据 + CLI 渲染未兜底”的独立闭环样本
- `check-update.sh` 路径残留、topic judge 404 = 后续支线问题

## 六、正式判断

### 已成立
1. 升级到 `2026.4.5` 后，gateway 第一时间确实无法正常拉起；

5. `openclaw cron list` 的 `padEnd` 异常已经形成独立闭环样本：旧 `oa-collect` 无 id 脏记录是直接触发条件，手工清理并重启 gateway 后恢复。

## 九、一句话收口
