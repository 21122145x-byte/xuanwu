## 一、总原则

这张表回答的是：
- MemOS 退出默认位后，谁来接住工作闭环
- 哪些能力必须马上接
- 哪些能力可以显式保留但不自动接
- 哪些内容不进第一版监控层

**第一版监控层目标：看得见、判得出、接得住。**
不要顺手把 retrieval policy / durable memory 治理一起做厚。

## 二、承接层施工总表

| 能力 / 角色 | 是否必须承接 | 优先级 | 承接层 | 当前承接口径 |
|---|---|---:|---|---|
| 默认工作记忆 | 是 | P0 | 文件层 | `MEMORY.md` + `memory/YYYY-MM-DD.md` |
| 正式知识入口 | 是 | P0 | 显式 KB | `knowledge_base/` |
| 默认 recall 注入 | 否（默认链必须拆） | P0 | 不自动承接 | 先停止默认链 |
| recall 能力本身 | 是（能力保留） | P1 | 显式 memory tools | `memory_search / memory_get / memory_timeline` |
| 经验 / skill 返注 | 是 | P1 | 本地 skill / KB / rules | 不走插件默认返注 |
| progress monitoring | 是 | P0 | 第一版监控层 | progress-aware monitoring |
| no-progress / stuck signal | 是 | P0 | 第一版监控层 | idle_warning / stuck_candidate / stuck |
| recovery entry | 是 | P0 | 第一版监控层 + artifact 层 | 最小 RecoveryArtifact |
| runtime judge | 否（方向保留） | P2 | 后续 governance 层 | 当前只写承接方向 |
| durable memory 治理 | 否（后补） | P2 | 后续 memory / governance 专题 | 不进首版监控层 |
| retrieval policy 自动化 | 否（后补） | P2 | 后续 governance / memory 专题 | 先保持显式调用 |
| viewer / UI 善后 | 否 | Drop | 非主线 | 冻结 |

## 三、P0：玄武进入施工前必须先接住的部分

### 1. 文件层工作闭环
必须可用：
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `knowledge_base/`

目的：
- 拆掉默认 slot 后，不让日常工作闭环悬空

### 2. 监控层最小职责
第一版监控层只先接：
- progress-aware monitoring
- no-progress 检测
- stuck signal 检测
- 最小状态切换
- 最小 recovery entry

目的：
- 先让系统看得见、判得出、接得住
- 不把监控层提前做成 memory policy 中枢

### 3. RecoveryArtifact 最小入口
要求：
- 能在 stuck / timeout / interrupted 时生成最小恢复入口
- 不要求一开始就做完整恢复系统

---

## 四、P1：保留能力，但改成显式方式接

### recall 能力
当前口径：
- 保留能力
- 拆默认链
- 改成显式调用

允许：
- `memory_search`
- `memory_get`
- `memory_timeline`

不允许：
- 默认 auto-recall 注入
- 默认可信 recall 地位

### skill / 经验返注
当前口径：
- 进入本地 skill / KB / rules / SOP
- 不再经旧插件自动返注上下文

---

## 五、P2：后补项，不进明天第一版监控层

### 1. runtime judge / governance 自动判定
- 当前只能写成承接方向
- 不能写成已落地能力

### 2. durable memory 治理
- 不进第一版监控层
- 后续单独接入 memory / governance 专题

### 3. retrieval policy 自动化
- 不进第一版监控层
- 后续单独收口

### 4. 更复杂的 lineage / policy-aware resume
- 当前不作为去默认化窗口的先决条件
- 可在监控层跑通后再继续扩

---

## 六、施工时禁止的混接

### 禁止 1
把“监控层”做成“新的默认 recall 自动注入层”

### 禁止 2
把“显式 recall 能力保留”误写成“默认 recall 继续存在”

### 禁止 3
因为后面要做 governance，就提前把 durable memory / retrieval policy 全塞进第一版

### 禁止 4
把 viewer / UI 善后重新带回主线，干扰去默认化窗口

---

## 七、执行顺序建议

### Step 1
先按禁止默认挂载清单，完成 slot / entry / allow / auto-recall 退场

### Step 2
确认文件层工作闭环已接住：
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `knowledge_base/`

### Step 3
让第一版监控层只接：
- progress
- no-progress
- stuck signal
- recovery entry

### Step 4
把 recall / memory policy / durable memory 留在显式调用或后补层，不抢进第一版

---

## 八、一句话收口

> 去 memos 化后的第一版承接层，不是“再造一个新 memos”，而是先用文件层接工作闭环，再让监控层接住 progress / stuck / recovery 这条最小治理链，其余 retrieval / durable memory 职责后补。
