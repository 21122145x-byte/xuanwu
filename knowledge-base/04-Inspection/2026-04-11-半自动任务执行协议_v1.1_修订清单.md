## 一、修订目标

本次 v1.1 不做全面重写，只修 4 个最高优先级问题：

1. **上下位协议冲突**
2. **SSOT（唯一真实状态源）缺失**
3. **`blocked` 分类不足**
4. **状态转移矩阵缺“生效条件列”**

本轮原则：
> **只修会直接导致试跑再断链、或让执行者不知道该听哪份规则的问题。**

## 二、修订项 1：上下位协议冲突

### 当前问题
v1 第七节流转 4 默认使用 `session_send` 做交付后提醒回执；
但上位《跨Agent协作流转执行卡》明确要求：
- Discord 协作面的正式定向触达，应优先使用 `<@Discord数字ID>`
- 不应把 `sessions_send` 当作默认正式唤醒手段

这导致下位协议与上位执行卡出现直接冲突。

### 风险
- 执行者不知道该优先遵循哪条规则
- 交付动作可能在 Discord 面与 session 面之间来回摇摆
- 一旦断链，无法判断是“执行不当”还是“制度冲突”

### v1.1 修订动作
#### 规则改写
将“交付后提醒回执”的默认优先级改为：
1. **交接频道 `<@Discord数字ID>` 正式提醒**
2. **`session_send` 作为补充提醒 / 兜底唤醒**
3. **超时后升级粉雪**

#### 新口径
- `session_send` 不再是 Discord 场景的默认正式唤醒手段
- 它只作为：
  - 已完成正式交接后的补充提醒
  - 或对方 session 明显 idle / announce 链不稳定时的兜底提醒

### 建议写入位置
- 协议第七节 流转 4
- 协议第三节 原则 0（上位规则叠加适用）
- `handoff_mode` / `ack_mode` 字段说明处

## 三、修订项 2：SSOT 缺失

### 当前问题
v1 定义了大量字段与模板，但没有明确：
- 每条半自动任务到底写在哪一份状态记录里
- 哪份记录是唯一真实来源（single source of truth）
- 多角色同时改状态时以谁为准

### 风险
可能同时出现：
- 主 Agent 状态卡一版
- 审计模板一版
- 看板一版
- heartbeat 记录一版
- 聊天口头状态一版

最终出现多套状态并行，无从裁定。

### v1.1 修订动作
#### 规则 1：每条半自动任务必须有且仅有一张状态卡
建议固定为：
- **状态卡文件 = 该任务唯一真实状态源（SSOT）**

#### 规则 2：其他载体均为派生视图，不是主状态
- 看板：派生视图
- 审计模板：审计动作模板 / 补充结构
- heartbeat 记录：观察记录
- Discord / session 消息：通知与回执，不是 SSOT

#### 规则 3：状态冲突时的裁定原则
若同一任务出现多处状态不一致：
1. 先以状态卡为准
2. 若状态卡被并发修改，以 `updated_at` 较晚者为准
3. 后写入者必须在 `decision_basis` 中显式说明：
   - 是否覆盖前一状态
   - 为什么覆盖

#### 规则 4：每次状态更新必须回写状态卡
无论是：
- 主 Agent
- 审计位
- 粉雪
都不得只改聊天或只改看板而不改状态卡

### 建议写入位置
- 协议新增独立章节：**状态记录与 SSOT 规则**
- 模板件说明部分同步写明：本模板是否为 SSOT / 派生件

## 四、修订项 3：`blocked` 分类不足

### 当前问题
v1 中 `blocked` 可以从多个状态进入，也能回到多个状态；
但没有定义：
- 哪类阻塞才能进入 `blocked`
- 阻塞类型如何区分
- 不同阻塞的解除路径与升级路径是什么

### 风险
`blocked` 会变成万能垃圾桶：
- 遇到任何困难都可以合法停在这里
- heartbeat 也不知道该怎么升级
- 长期会演变成制度化拖延位

### v1.1 修订动作
#### 新增字段
- `blocking_type`

#### 初始枚举
- `human-dependency`：等待特定人类 / Agent 回应
- `tool-limitation`：工具权限、运行时能力不足
- `external-dependency`：外部 API / 平台 / 网络依赖异常
- `scope-dispute`：范围争议、任务边界未定

#### 新规则
进入 `blocked` 时，必须同时写出：
- `blocking_type`
- `blocking_reason`
- `unblock_condition`
- `max_wait`
- `upgrade_target`

#### 解除规则
解除 `blocked` 时，必须明确：
- 是谁判定已解除
- 依据是什么
- 回到哪个合法状态

#### heartbeat 补丁
heartbeat 对 `blocked` 不再只看是否还活着，而要检查：
- 是否超过 `max_wait`
- 是否已满足 `unblock_condition` 但未推进

### 建议写入位置
- 协议第五节阶段门说明
- 协议第六节强制字段
- 协议第十一节 heartbeat 升级规则
- 状态模板 v1.1

## 五、修订项 4：状态转移矩阵缺“生效条件列”

### 当前问题
v1 的状态转移矩阵只有：
- 当前状态
- 允许转向
- 触发条件
- 责任位
- 禁止转向示例

但没有区分：
- 谁执行这次转移
- 谁批准这次转移真正生效

### 风险
在这些节点最容易出现争议：
- `sent-to-audit → returned`
- `sent-to-audit → audit-blocked`
- `audit-in-progress → accepted`

例如：
- Anna 写了 `returned`
- Curie 说“我还没正式接收这个打回”
- 那这个状态到底算不算成立？

### v1.1 修订动作
#### 矩阵新增一列
- **生效条件**

#### 初始判定口径
- `review-ready → sent-to-audit`
  - 生效条件：主 Agent 已正式交付，且状态卡已更新
- `sent-to-audit → audit-in-progress`
  - 生效条件：审计位已显式承接并改写状态卡
- `sent-to-audit → returned`
  - 生效条件：审计位已显式给出打回，状态即刻成立；主 Agent 后续必须接续推进，但不构成 returned 的生效前提
- `sent-to-audit → audit-blocked`
  - 生效条件：审计位已显式写出阻塞原因与补齐要求
- `audit-in-progress → accepted`
  - 生效条件：审计位明确放行并补全放行依据字段
- `returned → revision-needed`
  - 生效条件：主 Agent 或指定返修责任位已正式接单并回写状态卡

#### 目标
把“状态已经成立”与“后续还需要谁接动作”明确拆开，避免争议。

### 建议写入位置
- 协议第五节状态转移矩阵
- 第八 / 九 / 十节对应规则里补“单方生效 / 需对方动作接续”的边界说明
