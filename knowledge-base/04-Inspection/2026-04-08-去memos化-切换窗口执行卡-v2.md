## 一、这次切换窗口的目标

v2 版切换窗口只做一件事，但这件事必须写清：

> **让 MemOS 退出默认 memory slot 与默认自动注入链，不再继续处于系统默认运行位。**

这次切换不是为了：
- 修好 viewer
- 继续观察 1.0.8 是否还能救
- 给 MemOS 再争取一个 patch 观察期

这次切换是为了：
- 解除默认记忆绑定
- 停止默认 auto-recall 注入
- 把默认工作闭环切回本地替代层
- 顺手拆掉“默认挂载 + 自动注入”这条高风险链路

---

## 二、切换前统一冻结项

### 冻结 1：治理定位冻结
统一口径：
> **MemOS = 待迁出旧依赖 + 当前高风险默认记忆位。**

### 冻结 2：主线投入冻结
统一口径：
> **viewer / auto-recall 修补不再是主线投入。**

### 冻结 3：默认信任冻结
统一口径：
> **在切换完成前，不再把 MemOS 视为默认可信记忆基础设施。**

### 冻结 4：物理删除冻结
统一口径：
> **本窗口不做删库、删目录、清日志。**

原因：
- 当前要退的是默认运行角色
- 不是先销毁证据

---

## 三、切换前先决条件

以下条件未满足前，不进入配置切换：

### 条件 1：本地替代层已立住
至少确认：
- `MEMORY.md` 可作为长期记忆主层
- `memory/YYYY-MM-DD.md` 可作为当日记忆主层
- `knowledge_base/` 可作为正式知识承载主层
- `memory_search` 等工具仅作显式检索补位

### 条件 2：默认工作闭环可在无 MemOS 情况下运行
至少确认：
- 启动读 `SOUL.md / USER.md / MEMORY.md / memory/YYYY-MM-DD.md`
- 需要知识时读 `knowledge_base`
- 需要回查时手动检索
- 有结论时显式落文

### 条件 3：接受 viewer / memory unavailable 不作为回滚理由
切换前必须先定死：
- viewer 是否可用，不影响退场决定
- `Memory: unavailable` 不是保留默认挂载的理由

## 四、v2 切换动作顺序（必须按序）

### Step 1：解除默认 memory slot 绑定
目标：
- 将 `plugins.slots.memory` 从 `memos-local-openclaw-plugin` 解绑

这是 v2 的首动作，不能后置。

原因：
> **只要 memory slot 还指着 MemOS，系统就仍在默认信任它。**

---

### Step 2：禁用 MemOS plugin entry
目标：
- `plugins.entries["memos-local-openclaw-plugin"].enabled = false`

原因：
> **slot 解绑后，必须紧接着让 entry 退出活动态，避免继续被加载。**

---

### Step 3：从 `plugins.allow` 移除 MemOS
目标：
- 从 `plugins.allow` 中移除 `memos-local-openclaw-plugin`

原因：
> **allow 层不移除，就仍会给后续状态判断留下“理论可加载 / 理论可参与”的混乱空间。**

---

### Step 4：校正安装记录漂移
目标：
- 把 `plugins.installs_version_drift` 单独记录并修正
- 避免 recorded `1.0.7` / installed `1.0.8` 继续污染口径

原因：
> **版本台账漂移已经造成过事实判断错误，v2 里必须纳入执行卡，不再放到“以后顺手修”。**

---

## 五、切换时明确不做的动作

### 当前不做
- 不删除 `~/.openclaw/memos-local/memos.db`
- 不删除 `~/.openclaw/extensions/memos-local-openclaw-plugin`
- 不删除 backup / hold / debug log
- 不清空历史 MemOS 治理文档
- 不立刻迁掉所有 memos 相关 skill 资产

### 原因
因为当前切换目标是：
> **先让 MemOS 退出默认运行链，再处理历史资产。**

不是：
> **先做大扫除，再看系统会不会倒。**

---

## 六、切换后验收项（v2 版）

切换后至少要验这 6 条：

### 验收 1：默认启动不再依赖 MemOS
- 启动读取仍能走：
  - `SOUL.md`
  - `USER.md`
  - `MEMORY.md`
  - `memory/YYYY-MM-DD.md`

### 验收 2：默认主会话不再出现 MemOS auto-recall 注入
- 主会话默认不再出现 prependContext 式的 MemOS 注入

### 验收 3：正式知识完全由显式 knowledge_base 承接
- 不再把 MemOS 当默认正式知识入口

### 验收 4：检索改为显式触发
- 需要时手动用 `memory_search` / `memory_get`
- 不再依赖默认 recall

### 验收 5：MemOS 已不在默认 memory slot
- 配置态明确不再指向 `memos-local-openclaw-plugin`

### 验收 6：高风险组合至少被拆掉一环
最低要求：
> **“默认挂载 + 自动注入” 这一环必须被拆掉。**

---

## 七、回滚边界（v2 版）

### 允许回滚的条件
只有以下情况成立时，才考虑临时回滚：
1. 本地替代层未接住默认工作闭环
2. 无 MemOS 时默认工作无法启动 / 无法形成最小工作链
3. 且短期没有其他显式替代路径

### 当前不允许作为回滚理由的事项
以下都不构成回滚理由：
- “viewer 也许还能修”
- “Memory unavailable 可能只是暂时的”
- “1.0.8 再观察几天也许会更稳”
- “自动注入省事，先留着”
- “历史技能还没迁完，所以先不退” 

这些都只会导致：
> **默认高风险链继续挂着。**

---

## 八、后处理顺序

切换完成后，后处理按以下顺序走：

### 后处理 1：backend 冻结
- 数据保留
- 不再作为默认运行依赖

### 后处理 2：技能与知识资产迁挂
- 保留经验
- 去插件依赖
- 把治理经验迁到 knowledge_base / rules / skills

### 后处理 3：viewer / gateway / memory unavailable 并行收证
- 作为善后问题排查
- 不得反向决定默认挂载恢复

---

## 九、当前一句话执行原则

> **v2 切换窗口的核心不是“看 MemOS 还能不能修”，而是“先让它退出默认 memory slot 与默认自动注入链”；先退场，后冻结，再迁挂，最后才考虑物理归档。**
