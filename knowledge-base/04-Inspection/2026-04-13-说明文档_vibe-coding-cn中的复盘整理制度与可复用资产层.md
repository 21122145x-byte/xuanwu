## 一、先说结论

`vibe-coding-cn` 里并没有一份直接叫“复盘整理制度”的正式制度文档。

但它实际上已经通过以下组合，形成了一种**弱显式但很真实的整理制度**：
- 方法论文档层（documents）
- 可调用提示词层（prompts）
- 长期能力封装层（skills）
- 批量维护工具层（prompts-library）
- 结构/目录/README 索引层

换句话说，这个仓库不是把“复盘制度”写成一张规则卡，
而是把：
> **经验 → 方法 → 模板 → 提示词 → 技能 → 工具链**
这条沉淀路径先做出来了。

这正是它对我们最有价值的地方。

---

## 二、它的“复盘整理制度”不在一张文档里，而在沉淀路径里

### 1. documents 层：经验先被写成可阅读的方法论
从仓库结构看，`i18n/zh/documents/` 被明确定位为：
- 流程文档
- 架构说明
- 开发经验
- 最佳实践

这里的关键不是“有很多文档”，而是：
- `vibe-coding-经验收集.md`
- `开发经验.md`
- `系统提示词构建原则.md`
- `胶水编程.md`
- `通用项目架构模板.md`

这些名字本身就说明，它会把经验和原则**先落成文档化知识**。

也就是说，它的第一步不是“问题库对象化”，而是：
> 先让经验离开聊天，进入文档层。

### 2. prompts 层：文档知识再被压成可调用工作流
`i18n/zh/prompts/README.md` 很关键，它明确写了：
- 系统提示词用于约束 AI 边界
- coding_prompts 用于跑「需求澄清 → 计划 → 执行 → 复盘」流程
- 当某领域反复做同类工作，就把“提示词 + 资料”升级为 skill

这说明它的第二步不是停在“经验文档”，而是：
> 把经验压成能重复调用的提示词工作流。

这就是一种非常实在的“复盘整理制度”——
不是只回顾发生了什么，而是把复盘里形成的方法转成可反复调用的流程脚本。

### 3. skills 层：高频经验再上提为长期能力资产
`i18n/zh/skills/README.md` 明确区分了：
- Prompts = 单次任务指令
- Skills = 完整能力封装

并且明确说：
- 高频领域能力要沉淀为 Skills
- 每个 Skill 是目录，不只是单文件
- 包含主文件、assets、scripts、references

这说明它的第三步是：
> 当一个方法反复有效，就不再只把它当提示词，而是上提为长期能力资产。

这对我们特别关键，因为这正对应你说的：
- 什么优点可以变成机制
- 去复用

在 `vibe-coding-cn` 里，这件事已经不是口头想法，而是体现在结构上的：
- 临时提示词 ≠ 长期能力
- 长期能力应被单独编目、封装、引用

## 三、它的“可复用资产层”其实是分层设计，不是单一仓库堆料

### 1. 第一层：documents = 方法与经验资产
用途：
- 承接经验
- 记录原则
- 固化结构判断

它解决的是：
- 这个经验值不值得留下
- 这个原则是什么
- 为什么要这么做

### 2. 第二层：prompts = 可执行工作流资产
用途：
- 把方法变成可调用流程
- 让 AI 不是靠猜，而是按脚本跑

它解决的是：
- 下一次遇到类似任务怎么执行
- 哪些动作可以直接复用

### 3. 第三层：skills = 长期能力资产
用途：
- 把高频稳定方法封装成领域能力
- 让复用不再靠复制粘贴，而靠能力调用

它解决的是：
- 哪些做法已经稳定到值得长期保留
- 哪些不该每次重讲一次

### 4. 第四层：tools / prompts-library = 资产维护工具层
用途：
- 支撑批量维护
- 维护索引
- 提供格式转换

它解决的是：
- 资产多起来后如何维护
- 如何避免资产管理本身崩掉

### 5. 第五层：README / 目录结构 = 可导航层
用途：
- 让人和 AI 知道从哪开始读
- 让资产不变成仓库里的暗知识

它解决的是：
- 可回找
- 可定位
- 可协作

所以这个仓库真正强的不是“资料多”，而是：
> 它已经把资产分成“经验层 / 工作流层 / 能力层 / 工具层 / 导航层”。

## 四、这对“复盘整理制度”的真正启发是什么

### 启发 1：复盘的终点不是归档，而是上提
`vibe-coding-cn` 的做法说明：
- 经验如果只停在文档里，价值有限
- 真正的整理制度应该能把经验继续上提

也就是说，复盘整理至少应有一条默认上提路径：
- 复盘原文
- 问题 / 优点抽取
- 方法总结
- 提示词 / 模板化
- skill / 规则化

这正是我们当前还没完全补实的地方。

### 启发 2：问题对象化之外，还要有“可复用资产化”
Curie 的设计卡已经抓住了“问题对象化”。
但 `vibe-coding-cn` 进一步提醒我们：
如果只做问题库，而不做可复用资产层，最终会出现：
- 问题越来越会统计
- 但有效做法仍然不会自动复用

所以我们后续制度不应只问：
- 这是哪个问题？
- 升级去哪？

还应问：
- 这里有没有可抽象成模板 / prompt / skill 的稳定做法？

### 启发 3：复盘制度本质上需要“层级出口”
`vibe-coding-cn` 实际上给了不同出口：
- documents
- prompts
- skills
- templates/resources

这对我们很重要。
因为我们现在经常只有两种处理：
- 写进复盘
- 写进主文

但真正成熟的制度，出口应该更细：
- 保留为证据文档
- 提炼为规则卡
- 下沉为模板件
- 组织为提示词
- 上提为 skill
- 收入注册层

### 启发 4：整理制度要兼顾“索引能力”
这个仓库在 README、子目录 README、目录结构说明上花了很多力气。
这说明它不是只关心“有没有资产”，而是关心：
- 能不能找到
- 能不能被人或 AI 正确调用

所以我们的复盘制度后续也不能只有：
- 卡片
- 周升级

还必须考虑：
- 索引怎么挂
- 哪类资产挂哪层
- 默认从哪进入

## 五、如果把它翻译成我们团队语境

### 1. 对“全局复盘整理制度”的启发
后续制度不应只做：
- 问题卡
- 周升级会

还应内含一条明确原则：
> 复盘整理的目标，不只是识别问题，还要把稳定有效做法上提为可复用资产。

### 2. 对“可复用资产层”的启发
我们后续至少可以按这几个层级来理解现有产物：
- **证据层**：复盘原文、审计记录、观察记录
- **判断层**：问题卡、优点卡、缺口卡
- **规则层**：规则卡、边界卡、默认动作卡
- **模板层**：模板件、编排卡、周升级模板
- **能力层**：skill 候选、注册层能力、长期可调用能力
- **导航层**：README、总表、索引件

### 3. 对当前轻治理线的直接影响
当前主文里已经补回：
- 注册层整理
- 资产分层整理
- 复盘制度上提

而 `vibe-coding-cn` 的进一步深读，等于把这三条变得更扎实了：
- 注册层整理不是抽象概念，而是能力目录化、入口稳定化
- 资产分层整理不是分类游戏，而是沉淀路径的必要结构
- 复盘制度上提不是周会而已，而是“问题对象化 + 资产上提路径”两条一起做

## 六、当前我认为最值得直接吸收的两条口径

### 口径 1
> 复盘整理制度的目标，不只是让问题更容易回看，而是让经验更容易上提为可复用资产。

### 口径 2
> 可复用资产层不应只理解为“资料堆放区”，而应理解为经验经过筛选、模板化、能力化之后的长期调用层。

## 七、当前边界

- 本说明文档只解释 `vibe-coding-cn` 的真实结构启发，不代表其内部已经有我们想要的一整套“问题库制度”成文规则。
- 当前结论是“从结构中读出制度路径”，不是声称其仓库内已经存在一份完整周升级规范。
- 对我们来说，重点不是照搬其目录，而是吸收其“经验 → prompts → skills → 可维护资产”的沉淀逻辑。
