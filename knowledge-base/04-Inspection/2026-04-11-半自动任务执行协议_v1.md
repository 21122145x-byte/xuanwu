## 一、协议目标

本协议适用于以下类型任务：
- 执行时间较长
- 先由 worker / subagent 产生产物
- 由主 Agent 验收与收口
- 产物需要进入下一环（审计、回传、归档）
- 执行期间用户可能继续在前台发消息

本协议的目标不是“自动跑起来就算完”，而是：

> **半自动任务即使遇到前台消息干扰、announce 被跳过、worker 已完成但无人继续接手、审计未稳定承接等情况，也能稳定完成阶段流转与闭环。**

---

## 二、当前适用问题模型

本协议主要解决以下典型断链：

1. worker 已完成，但结果没有进入下一环
2. 主 Agent 已 review-ready，但外部体感像“停住了”
3. heartbeat 证明链路还活着，但不能推动交接
4. 前台 Discord 新消息抢占后台 announce / 回传链
5. `sent-to-audit`、`returned`、`accepted` 三节点定义不硬，导致链路再次断裂

---

## 三、核心原则

### 原则 0：上位跨 Agent 协作规则同时生效
凡涉及送审、打回、回执、任务完成回传的跨 Agent 交接，除遵循本协议外，还必须同时遵循上位《跨Agent协作流转执行卡》。

本协议与上位执行卡的关系是：
- 上位执行卡规定通用交接格式、回执与熔断规则
- 本协议补充半自动任务专属的状态门、阶段流转、异常状态与 heartbeat 规则
- 两者为叠加关系，不是二选一


### 原则 1：后台执行不以“有人记得接”为前提
每一环必须显式落状态，而不是靠聊天上下文或个人记忆维持流转。

### 原则 2：worker 只负责产出，不负责跨环 handoff
worker 默认职责仅限：
- 写稿
- 修订
- 打包结果
- 写明当前状态

worker 默认不负责：
- 送审
- 对外 announce
- 回答前台确认消息

### 原则 3：主 Agent 对阶段流转负责
主 Agent 必须承担：
- 接住 worker 产物
- 做 accept / revise 判断
- 推下一环
- 决定是否对外回传

### 原则 4：审计位负责裁决，不负责猜进度
Anna 等审计位负责：
- 审计承接
- 放行 / 打回
- 输出审计意见

审计位不应承担“自己去猜当前主链进度”的成本。

### 原则 5：heartbeat 负责发现停滞，不负责代替业务判断
heartbeat 只负责：
- 提醒
- 记录
- 升级

不替代主链做 accept / revise / send-to-audit 判断。

## 四、角色边界

### 1. Worker / Subagent
负责：
- 产出主文与 supporting artifacts
- 写明结果包位置
- 把状态推进到 `review-ready` 或 `revision-needed`

不负责：
- 送 Anna 审计
- 对外 announce
- 处理前台消息仲裁

### 2. 主 Agent
负责：
- 接住 worker 结果
- 验收与修订决策
- 送审
- 主链状态维护
- 对外回传判断

### 3. 审计 Agent（如 Anna）
负责：
- 审计承接
- 放行 / 打回 / 阻塞判断
- 审计意见归档

### 4. Heartbeat 子链
负责：
- 盯阶段停滞
- 标记“活着但没流转”
- 提醒与升级

## 五、标准阶段门（含异常状态）

所有半自动任务必须显式处于以下状态之一：

- `drafting`
- `revision-needed`
- `review-ready`
- `sent-to-audit`
- `audit-in-progress`
- `audit-blocked`
- `awaiting-human-confirmation`
- `blocked`
- `accepted`
- `returned`
- `announced`
- `closed`

禁止使用“差不多好了”“已经推进了”这类模糊表达代替阶段门。

### 状态转移矩阵（试跑初版）

| 当前状态 | 允许转向 | 触发条件 | 责任位 | 禁止转向示例 |
|---|---|---|---|---|
| `drafting` | `review-ready`, `revision-needed`, `blocked` | 产物完成 / 发现需返修 / 执行受阻 | worker | 直接到 `sent-to-audit`、`accepted` |
| `revision-needed` | `drafting`, `review-ready`, `blocked`, `closed` | 进入修订 / 修订完成 / 阻塞 / 明确终止 | worker / 主 Agent | 直接到 `accepted` |
| `review-ready` | `sent-to-audit`, `revision-needed`, `blocked`, `closed` | 主 Agent 验收通过并送审 / 决定返修 / 阻塞 / 终止 | 主 Agent | 长期悬停不判断；直接到 `accepted` |
| `sent-to-audit` | `audit-in-progress`, `audit-blocked`, `returned` | 审计位接住 / 无法承接 / 直接打回 | 审计位 | 主 Agent 自行改成 `accepted` |
| `audit-in-progress` | `accepted`, `returned`, `audit-blocked` | 审计完成放行 / 打回 / 审计受阻 | 审计位 | 跳过判断直接 `closed` |
| `audit-blocked` | `sent-to-audit`, `revision-needed`, `closed` | 补齐后重送 / 改回修订 / 明确终止 | 主 Agent / 审计位 | 无说明解除阻塞 |
| `awaiting-human-confirmation` | `review-ready`, `accepted`, `closed`, `blocked` | 收到人类确认 / 人类确认后放行 / 人类明确终止 / 超时阻塞 | 主 Agent | 长期无升级地挂起 |
| `blocked` | `drafting`, `revision-needed`, `review-ready`, `sent-to-audit`, `closed` | 阻塞解除后回到最近合法阶段 / 明确终止 | 当前 owner / 主 Agent | 未写解除依据直接跳到更后阶段 |
| `accepted` | `announced`, `awaiting-human-confirmation`, `closed` | 进入回传 / 需人类确认 / 该阶段无需回传且正式关闭 | 主 Agent | 自动连跳到 `closed` |
| `returned` | `revision-needed`, `review-ready`, `closed` | 进入修订 / 补齐后重审 / 终止 | 主 Agent | 长期悬停 |
| `announced` | `closed`, `awaiting-human-confirmation` | 回传完成 / 仍待确认 | 主 Agent | 回到更早审计态 |
| `closed` | 无 | 任务正式结束 | 主 Agent / 粉雪 | 重新开启但不建新状态记录 |

如需偏离本矩阵，必须由粉雪显式裁定，并在 `decision_basis` 中写明理由。

## 六、每次状态更新的强制字段

每次状态流转至少要记录：

- `status`
- `owner`
- `next_owner`
- `next_action`
- `updated_at`
- `artifact_where`
- `known_issues`
- `decision_basis`
- `handoff_mode`
- `blocking_reason`（如适用）
- `stalled_at_gate`（如适用）
- `flow_state`（如适用，`flowing` / `not-flowing`）
- `alive_signal`（heartbeat 观察时必填）
- `flow_signal`（heartbeat 观察时必填）

### `handoff_mode` 初始枚举
- `main-agent-send`：主 Agent 主动投递给下一责任位
- `session-send-reminder`：完成交接后，通过 `session_send` 向下一责任位发送提醒，要求其以 `session_send` 形式回复确认
- `document-reference`：仅提供文档路径 / 文件路径，不主动投递
- `message-notify`：通过外部消息渠道提醒
- `internal-mark`：仅内部留痕，不构成正式交接

推荐格式：

```md
status: review-ready
owner: curie
next_owner: anna
next_action: audit
updated_at: 2026-04-11 01:30 Asia/Shanghai
artifact_where:
- /path/mainline-v0.2.md
- /path/supporting-card-1.md
known_issues:
- section 2 still needs audit on scope boundary
decision_basis:
- worker package complete
- key evidence chain present
handoff_mode: main-agent-send
blocking_reason:
```

## 七、标准流转

### 流转 1：worker 完成
worker 完成后必须：
1. 产出主文与 supporting artifacts
2. 写 `status=review-ready`
3. 写 `next_owner=主 Agent`
4. 停止继续外放

### 流转 2：主 Agent 验收
主 Agent 在 `review-ready` 状态只允许两种正式动作：
- `accept` → 进入 `sent-to-audit`
- `revise` → 进入 `revision-needed`

不得长期停留在 `review-ready` 而不作判断。

### 流转 3：送审
送审必须携带最小承接包：
- 主文路径
- supporting artifacts
- 当前状态
- 已知问题
- 希望重点核查项

送审同时适用上位《跨Agent协作流转执行卡》的正式交接格式：
- Where
- What
- How to verify
- Known Issues
- Next Steps

默认由主 Agent 承担送审动作，不由 worker 承担。

### 流转 4：跨 Agent 交付后的提醒回执
凡主 Agent 已完成跨 Agent 交付，必须追加一次提醒动作，防止交接停留在“已发出但未被接住”。

最小要求：
1. 交付完成后，向下一责任位发送提醒
2. 提醒默认使用 `session_send`
3. 提醒内容必须要求对方以 `session_send` 形式回复【收到】
4. 未收到回执前，不得把 `sent-to-audit` 误判为已承接

这条规则的目标不是重复交接，而是显式补齐“交付后提醒 + 回执确认”这一段，防止断链。

## 八、审计承接规则（必补）

### 规则 1：`sent-to-audit` 不等于已承接
只有当审计位显式将状态改为以下之一，才算审计成功接住：
- `audit-in-progress`
- `returned`
- `audit-blocked`

### 规则 2：审计位承接后必须改写状态
不允许主链长期停留在 `sent-to-audit` 而没有承接反馈。

### 规则 3：审计位拒绝承接也必须显式落状态
若审计位因信息不足、输入不完整或范围不清无法承接，必须进入：
- `audit-blocked`

并写明：
- `blocking_reason`
- 需要补什么
- 由谁补

---

## 九、打回规则（必补）

### 规则 1：`returned` 必须携带最小修复信息
每次打回至少写出：
- `return_reason`
- `minimum_fix_path`
- `who_should_rework`
- `reentry_condition`

### 规则 2：区分两类打回
- **审计退回**：审计认为内容不满足放行条件
- **承接退回**：审计位无法承接，要求主 Agent 补齐输入后再送审

### 规则 3：打回后必须进入明确状态
`returned` 不是结束态。打回后必须继续进入以下之一：
- `revision-needed`
- `review-ready`
- `closed`

不得让主链长期悬停在 `returned`。

---

## 十、放行判断规则（必补）

### 规则 1：`accepted` 只代表当前阶段放行
`accepted` 只表示：
- 当前审计阶段通过

不等于：
- 已 announce
- 已 closed
- 整条链自然完成

### 规则 2：放行必须带依据
每次 `accepted` 至少要补：
- `accept_scope`
- `accept_basis`
- `known_limits`

### 规则 3：`accepted → announced → closed` 不能自动连跳
三者必须单独满足条件，不能默认 accepted 就自动完成全部后续动作。

## 十一、Heartbeat 升级规则

heartbeat 不只检查“还活着没”，还检查“流转了没”。

### 规则 0：以下时间阈值均为试跑初始值
当前 20 分钟 / 30 分钟 / 15 分钟阈值，均视为试跑初始值，不视为最终固定制度参数。

- 试跑结束后，应根据实际停滞数据回顾并调整
- 调整权归粉雪
- 长周期状态（尤其 `awaiting-human-confirmation`）可使用更长阈值，但必须显式写明

### 规则 1：review-ready 停滞
- `review-ready > 20 分钟` 未变化 → 提醒主 Agent

### 规则 2：sent-to-audit 停滞
- `sent-to-audit > 30 分钟` 未承接 → 提醒审计位
- 再超 30 分钟无变化 → 升级粉雪

### 规则 3：accepted 未回传
- `accepted > 15 分钟` 未进入 `announced` → 标记为回传链异常

### 规则 4：returned 未重分配
- `returned > 20 分钟` 未进入 `revision-needed` / `review-ready` / `closed` → 提醒主 Agent

### 规则 5：awaiting-human-confirmation 超时
- 默认不适用短阈值
- 必须在进入该状态时显式写明超时阈值与升级对象
- 超时后至少进入 `blocked` 或提交粉雪裁定

## 十二、前后台解耦规则

### 原则
后台主链以阶段门为准，不以前台 Discord 会话节奏为准。

### 规则 1：前台旧问题不得反向打断更后阶段
若后台已推进到更后阶段，前台旧问题不得把主链拉回更早状态。

### 规则 2：前台确认只允许出现在显式节点
只有进入 `awaiting-human-confirmation` 时，才允许前台向用户请求确认。

### 规则 3：announce 被跳过不等于主链未执行
若 announce 失败、被打断或 `ANNOUNCE_SKIP`，必须以阶段门判断真实进度，不得直接判断为任务未推进。

## 十三、新消息改范围的仲裁条款（必补）

当前台用户新消息真的改变任务范围时，必须由主 Agent 做三选一仲裁，并显式落状态：

### 方案 A：继续当前链
适用于：
- 新消息不改变主任务边界
- 只是补充说明

### 方案 B：新开分支链
适用于：
- 新消息引入新的子任务
- 原链仍应继续保留

### 方案 C：中止当前链
适用于：
- 用户明确改变方向
- 原链已不再需要继续

禁止出现：
- 口头上承认任务已变化
- 但状态链不更新

## 十五、异常熔断规则

### 规则 1：连续打回熔断
同一任务若在同一环节连续出现 `returned → revision-needed / review-ready → returned` 3 次，触发熔断。

### 规则 2：粉雪介入后仍无法推进
若粉雪已介入，但一轮方向性判断后仍未恢复合法流转，必须由粉雪明确裁定以下之一：
- 继续
- 重新定义任务
- 中止

不得继续无限循环。

### 规则 3：熔断必须留痕
触发熔断时，至少补：
- `blocking_reason`
- `decision_basis`
- 下一责任位
- 是否继续 / 重定义 / 中止
