## 一、为什么要出这份修订版

前序版本审计链中，已经形成了：
- 版本风险扫描
- 映射表补件
- 映射表补件版审计意见

这些文档本身方向没有错，
但当前已确认一个关键口径必须钉死：

> **当前运行版本** 和 **目标研究版本** 不是同一个对象。 

如果这条不单独收口，后续容易持续混淆：
- 本机现在到底跑什么版本
- 玄武/Anna 正在审哪个版本
- 上游 release / issue 风险到底是背景输入，还是本机事实

因此需要一份最小修订版，把对象重新钉清。

### 2. 目标研究版本

这条用于：

- 是否值得继续做专项验证设计

## 三、必须守住的边界

### 边界 1：不得混案
以下两件事必须分开：
- `2026.4.2` 的本地运行事实
- `2026.4.5` 的上游风险扫描与研究判断

不得把：
> `2026.4.5` 的 release / issue 风险
直接写成：
> `2026.4.2` 本机已发生事实

---

### 边界 2：上游背景输入不等于本地定因
当前对 `2026.4.5` 的风险扫描、映射表、补件审计，
都只能成立为：

> **背景输入 / 高相关参考 / 值得专项验证**

不能直接成立为：
- 本地根因
- 本地修复依据
- 升级放行依据
- 回滚放行依据

---

### 边界 3：release 修复项不等于本机已自动解决
即使 `2026.4.5` release note 修过某类问题，
当前也不能直接写成：
> 本机相应问题已经自然解决。

因为：
- 本机还在 `2026.4.2`
- 且本地 before / after 验证未完成
