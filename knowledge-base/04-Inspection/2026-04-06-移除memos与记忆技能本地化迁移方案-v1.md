# 移除 memos 与记忆/技能本地化迁移方案 v1

- 日期：2026-04-06
- 主责：Anna
- 状态：已启动 / 第一轮盘点完成

---

## 一、当前判断

当前问题已经不适合再按“继续修 memos 单点”来处理。

更准确的判断是：

> **memos 当前同时绑定了 memory backend、auto-recall 注入、viewer/UI、历史插件修补、知识库文档体系与若干技能资产。**
> **如果要移除，必须按层拆，不应一刀切盲拔。**

因此当前主线不应是：
- 继续围着 memos viewer 修
- 继续把漂移当局部 bug

而应改成：

> **把 memos 从“默认记忆基础设施”降级为“待迁出旧依赖”，同时建设本地记忆/技能/知识承载替代层。**

---

## 二、第一轮盘点结论

## 1. 运行配置层绑定仍在
当前 `~/.openclaw/openclaw.json` 中仍明确存在：
- `plugins.allow` 包含 `memos-local-openclaw-plugin`
- `plugins.slots.memory = memos-local-openclaw-plugin`
- `plugins.entries["memos-local-openclaw-plugin"].enabled = true`
- memos 插件安装记录仍在 `plugins.installs`

这说明：
> **memos 仍是当前默认 memory slot。**

---

## 2. 本地数据与插件残留仍在
当前本地仍存在：
- `~/.openclaw/extensions/memos-local-openclaw-plugin`
- `~/.openclaw/memos-local/memos.db`
- `~/.openclaw/memos.db`
- `~/.openclaw/logs/memos-auto-recall-debug.log`
- 多份历史 backup / hold 目录

这说明：
> **物理层并未迁出。**

---

## 3. 知识库与治理文档耦合很深
当前 workspace-anna / team KB 中已有大量与 memos 绑定的：
- 当前状态文档
- RCA / SOP
- 幽灵上下文修复文档
- autorecall 轻审计包
- MemOS 索引页
- Spawn 试跑样本与治理文档
- 系统治理与 Topic Judge 设计稿

这说明：
> **知识层不能简单删除，只能做“转义、归档、改挂载关系”。**

---

## 4. 技能与操作知识也有 memos 依赖残留
当前可见：
- `~/.openclaw/skills/memos-memory-guide`
- `workspace/skills/memos-memory-guide`
- `skills-store/` 下多份 memos 相关 skill / SOP / 升级验证 / 校准技能

这说明：
> **技能层也需要去 memos 化，不只是配置层。**

---

## 三、迁移时必须拆成 4 层

## 层 A：viewer / UI 层
这是最先可切断的一层。

因为当前最不稳、最影响体验的是：
- viewer 起不来
- UI 不可达
- service startup 时序异常

### 当前建议
- 不再把 viewer 当默认可依赖入口
- 即使 backend 暂未移除，也先废除对 memos viewer 的使用预期
- 后续改为知识库文件 + 本地索引 + memory_search / memory_write_public 的组合承载

---

## 层 B：auto-recall / 注入层
这是当前最该尽快切断的一层。

因为当前最严重的问题不是“看不见记忆”，而是：
- 低充分度注入
- 上下文漂移
- cross-agent 串带
- runtime/system block 污染

### 当前建议
- 将 memos 的 auto-recall / prependContext 视为首要退场对象
- 默认不再允许它继续承担主会话上下文注入责任
- 后续改为：
  - 手动 memory_search
  - 本地 memory 文件
  - 规则化 knowledge retrieval

---

## 层 C：memory backend / 历史数据层
这层不能盲删。

因为它仍可能承载：
- 历史 chunks
- 历史 recall 样本
- 调试留样
- 需要回收的记忆资产

### 当前建议
- 先冻结，不先删库
- 先盘点哪些数据值得导出
- 优先迁移“可解释、可验证、可文本化”的部分到本地知识体系
- 只有完成导出/替代后，才考虑彻底停用 backend

---

## 层 D：技能 / 知识资产层
这层不应跟着插件一起废弃。

因为很多资产虽然以 memos 为主题，但其真实价值是：
- 治理经验
- 污染样本
- 边界规则
- 修复路径
- 架构认知

### 当前建议
- “去插件依赖”，不是“删经验”
- 将有长期价值的内容改挂到：
  - local memory architecture
  - consistency governance
  - runtime pipeline
  - KB maintenance
- 将纯历史维修链归档到集中 archive 区

---

## 四、第一阶段：现在就能做的动作（非破坏性）

## 动作 1：正式降级 memos 的治理定位
统一口径改为：

> **memos 不再视为默认可信记忆基础设施，而是待迁出旧依赖。**

这是立刻可生效的治理动作。

---

## 动作 2：停止继续围绕 viewer 修理做主线投入
viewer / service startup 线不再作为主投入方向，
只保留必要证据，不再继续加大修补投入。

---

## 动作 3：建立“本地承载替代层”
替代层建议由三部分组成：
1. **workspace memory 文件层**
   - `MEMORY.md`
   - `memory/YYYY-MM-DD.md`
2. **knowledge_base 显式知识层**
   - 索引、topic、note、report
3. **memory_search / memory_write_public / public memory**
   - 用作本地可控的共享记忆工具层

也就是说：
> **把“默认记忆”从 memos 插件，迁回文件化 + 本地工具化的组合。**

---

## 五、第二阶段：需要切换窗口的动作（高风险）

以下动作属于真正切换，不建议无验收直接做：

### 1. 修改 `openclaw.json`
目标包括：
- 从 `plugins.allow` 移除 `memos-local-openclaw-plugin`
- 解除 `plugins.slots.memory` 对 memos 的绑定
- 禁用对应 `plugins.entries`

### 2. 停止使用 memos 数据路径作为活动路径
包括：
- `~/.openclaw/memos-local/`
- `~/.openclaw/memos.db`

### 3. 处理历史 plugin 目录与 backup
包括：
- 扩展目录
- backup/hold 目录
- 调试日志目录

### 4. 迁出/冻结 memos 相关技能资产
包括：
- `memos-memory-guide`
- skills-store 中的 memos 系技能

这些都需要：
> **先盘点 → 再冻结 → 再迁移 / 归档 → 最后才禁用。**

## 七、当前最合理的推进顺序

### Step 1
先冻结 memos 的“默认可信地位”与“主线修理地位”。

### Step 2
先把本地替代承载层立稳：
- 文件记忆
- 显式知识库
- 本地 memory 工具

### Step 3
盘点哪些历史数据 / skill / 经验要迁出保留。

### Step 4
进入切换窗口，修改配置，解除 memory slot 绑定。

### Step 5
最后才决定是否物理停用或归档插件与数据库。

## 八、一句话收口

> **移除 memos 现在不能按“拔插件”处理，而应按“降级旧依赖 → 建本地替代层 → 迁移知识与技能 → 再切换配置”处理。当前我已经启动第一轮盘点，下一步应先推进非破坏性的本地承载替代方案，再决定禁用与迁出窗口。**
