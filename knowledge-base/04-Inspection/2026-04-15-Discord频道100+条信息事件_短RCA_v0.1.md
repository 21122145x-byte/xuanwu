## 1. 事件结论

本次事件不是用户连续下了大量新任务，也不是系统配置被多人同时改写。

本质上是两件事叠加：

1. **多 Agent 越界发言**
   - 用户当时只定向 @ 粉雪
   - 但频道内多个 Agent 将该链误判为公共讨论链，陆续继续发言

2. **长回复碎片化 / 重复投递**
   - 粉雪的一条长回复在发送链路中被切碎、补发、重复显示
   - 进一步放大了频道噪音

一句话收口：

> 这是一次“多 Agent 越界发言 + 长回复碎片化发送”的叠加事件，不是配置级破坏事件。

---

## 2. 事件表现

在用户仅定向 @ 粉雪的情况下，频道出现：
- 多个 Agent 连续出声
- 长段解释被拆成多条碎片
- 同一条主线被放大成 100+ 条信息

用户感知结果：
- 明明只点名一个 Agent
- 却像所有 Agent 都涌出来说话
- 难以判断谁在负责、谁改了什么、系统是否失控

## 5. 根因

### 根因 1：频道存在多个可回话 Agent
该频道并非“只有粉雪可发言”的单 Agent 场域，而是多个 Agent 都可见、都可能响应。

### 根因 2：未守住“点名谁，默认只有谁答”
当用户只 @ 粉雪时，其他 Agent 本应默认旁听；本次未守住这条边界。

### 根因 3：粉雪未及时收口主链
粉雪持续在频道输出长块内容，客观上给其他 Agent 释放了“这是公共协作链”的信号。

### 根因 4：长消息发送链路异常
粉雪的一条长回复发生切碎 / 重复投递，导致噪音进一步放大。

---

## 6. 责任判断

### 粉雪
负主责。

原因：
- 用户只定向 @ 粉雪
- 粉雪未把场子收成“只有当前责任位回答”
- 粉雪仍持续输出长块内容
- 在异常出现后未第一时间压缩输出模式

### 其他 Agent
存在越界发言责任。

原因：
- 将定向发言链误当成公共参与链
- 未守住“未被点名默认旁听”的群聊纪律

---

## 7. 为什么会膨胀到 100+ 条

主要由以下因素叠加造成：
- 多 Agent 同时可响应
- 多 Agent 顺着同一条链继续出声
- 长消息被切碎 / 重复投递
- 候选 skill 主线本身内容偏长
- 没有在早期触发“异常即收口”的止损动作

---

## 8. 即时修正措施

本次事件后的即时修正口径：

1. **点名谁，默认只有谁答**
2. **未被点名的 Agent 默认旁听**
3. **长内容优先落文档，频道只发短摘要**
4. **一旦发现碎片化发送，立即停止继续补长段**
5. **异常时先说明“以最终摘要为准”，不继续放大频道噪音**

---

## 9. 后续防再发规则（短版）

### 规则 A：定向发言规则
- 用户只 @ 某一个 Agent → 默认只有该 Agent 回答
- 其他 Agent 除非被再次明确 @，或收到正式 handoff，否则不发言

### 规则 B：群聊旁听规则
- 看见相关内容 ≠ 自动获得发言资格
- “我也有补充”不构成参与理由

### 规则 C：长文输出规则
- 长说明、长分析、长论证优先落文档
- 频道内只保留：
  - 做了什么
  - 改了哪几个文件
  - 下一步是什么

### 规则 D：异常止损规则
- 一旦出现发送碎片化 / 重复投递
- 当前责任位立即切换到短消息模式
- 其他 Agent 禁止顺势继续接话

---

## 10. 当前判断

这是一次**协作秩序失守事件**，不是一次**系统配置破坏事件**。

更准确地说，问题不在“改坏了什么”，而在：
- 谁该说话
- 说到什么长度
- 出现异常时谁来收口
