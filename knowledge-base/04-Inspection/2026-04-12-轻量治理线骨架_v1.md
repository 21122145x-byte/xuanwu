## 一、这条线到底要解决什么

轻量治理线要解决的，不是“再写一篇 SOP”，而是解决以下 4 类现场问题：

1. **已有原则、已有能力、已有材料，但现场不会接，不会挂，不会升级**
2. **材料产出速度快于主线承接速度，导致有件无纲**
3. **现场任务需要最小治理闭环，但团队宪法太上位、系统治理线太重、runtime pipeline 太技术化**
4. **出现遗漏、误判、卡点后，没有统一的轻量回写与升级口径**

一句话定义：

> **轻量治理线，是团队在正式重治理与完整 runtime pipeline 之间，用来承接现场任务、压清入口、约束流转、暴露卡点、完成最小升级闭环的主线。**

---

## 二、这条线不解决什么

轻量治理线当前**不负责**：

- 替代团队宪法
- 替代系统治理线
- 替代 runtime pipeline 技术实现与技术验证本体
- 替代本地 skill / 模板 / 协议本体
- 宣称“长期稳定运行已验证”
- 覆盖所有 agent 的全量生命周期治理

这条线只做：

> **把现场治理层立起来，并把已有主文、配件、审计、接口层收成一条可运行主线。**

## 三、边界划分

### 1. 与团队宪法的边界
团队宪法负责：
- 上位原则
- 红线
- 角色分工
- 冲突裁决依据

轻量治理线负责：
- 现场默认入口
- 主文与配件的挂位
- 轻量流转与纠偏
- 最小升级判断

**边界结论：**
轻量治理线承接宪法，但不替代宪法。

### 2. 与系统治理线的边界
系统治理线负责：
- runtime / config / routing / memory consistency / service health
- 运行异常归因
- 高风险修复与系统机制归属判断

轻量治理线负责：
- 现场任务是否已形成最小治理闭环
- 哪类问题应继续留在轻量治理层
- 哪类问题必须升级到系统治理位

**边界结论：**
轻量治理线暴露系统问题，但不吞系统问题。

### 3. 与 runtime pipeline 的边界
runtime pipeline 负责：
- 技术实现链
- 自动化/半自动化运行机制
- 真实执行链路的稳定性与可重复性

轻量治理线负责：
- runtime 相关任务在前台的接线层
- 现场任务的默认入口、流转与纠偏
- 在 pipeline 未完全稳定前的过渡治理

**边界结论：**
轻量治理线是现场治理层，不是 pipeline 本体。

### 4. 与本地 skill / 模板 / 协议层的边界
skill / 模板 / 协议负责：
- 具体能力怎么做
- 触发条件
- 操作模板
- 输出规范

轻量治理线负责：
- 什么时候该想到已有 skill / 模板 / 协议
- 没想到时如何纠偏
- 没能力时如何回退到最小治理流程

**边界结论：**
轻量治理线负责“找对入口与收口”，不负责替代能力本体。

## 六、最小运行链路

轻量治理线当前最小运行链路建议统一为：

1. **任务进入**
   - 先判断：这是不是轻量治理线应承接的问题
2. **主文定位**
   - 查主文与默认入口，不从聊天临场发明流程
3. **配件调用**
   - 根据问题类型调用对应配件：注册层 / 字段块 / 运行补口 / 回写分类 / 实施说明
4. **审计判断**
   - 遇到放行、偷宽、边界不清问题时，进入 Anna 审计位
5. **系统边界判断**
   - 遇到 runtime / config / 运行异常 / 机制归属问题时，升级玄武
6. **收口与回写**
   - 结果回主线，明确是否继续留在轻量治理层，还是上提更重层

一句话压缩：

> **任务进入 → 主文定位 → 配件调用 → 必要时审计 / 系统升级 → 回主线收口。**

## 七、最小升级边界

### 留在轻量治理线的问题
- 默认入口不清
- 主文与配件挂位不清
- 轻量流转、补口、责任字段、回写分类不清
- 现场执行中出现的轻量纠偏与最小闭环问题

- 是否 accepted / 是否可放行
- 是否把阶段性成立偷宽成长期成立
- 是否越过已锁死边界
- 是否需要给出正式审计口径

### 升级玄武的问题
- runtime / gateway / routing / config / memory consistency
- 服务异常、执行异常、机制归属不清
- 已超出现场治理层、进入系统治理层的问题
