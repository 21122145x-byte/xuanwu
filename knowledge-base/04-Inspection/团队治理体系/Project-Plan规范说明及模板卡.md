## 一、定位

Project Plan 不是宪章，不是 Program，也不是 SOP。

它只承担一个职责：

> **把一个具体任务压成当前周期内可进入、可推进、可分段、可交付、可裁决的项目级推进结构。**

在五层结构中：

```text
Charter
  ↓ 定义原则与边界
Program
  ↓ 定义团队默认闭环结构、运行机制与承载口径
Project Plan
  ↓ 定义本轮推进目标、阶段、责任、交付与验收
SOP / Runbook / Checklist
  ↓ 定义具体动作标准做法、场景处置与检查门槛
Prompt
  ↓ 定义单轮执行指令
```

一句话区分：

- **Charter**：什么不能错位  
- **Program**：系统怎么跑  
- **Project Plan**：这一轮具体打什么、怎么分段打、谁来打、打到什么算过  
- **SOP / Runbook / Checklist**：某一步怎么稳地做、怎么处置、怎么检查  
- **Prompt**：这一手具体怎么发出去

---

## 二、Project Plan 解决什么问题

当一个正式任务已经具备基本进入条件后，Project Plan 负责把以下内容压清：

1. **本轮到底做什么，不做什么**  
2. **本轮分几段推进，先后顺序是什么**  
3. **每一阶段的主责、支持、复核是谁**  
4. **每一阶段交付什么、放在哪里、如何验证**  
5. **阶段出口是什么，何时交接，交给谁**  
6. **什么情况放行、打回、暂停、阻塞或失败**  
7. **本轮发生范围变化时，如何回写与裁决**

因此，Project Plan 的本质不是“项目介绍”，而是：

> **当前周期的项目级推进与裁决框架。**

---

## 三、Project Plan 不解决什么问题

### 1. 不重写团队上位原则
那是 Charter 的职责。

### 2. 不定义团队级闭环机制
那是 Program 主协议的职责。

### 3. 不替代任务级 `program.md`
如任务需要独立运行闭环、正式状态流转、特化交接结构或 runtime / harness 接入，应由任务级 `program.md` 承担该层职责。

### 4. 不直接下沉为 SOP / Runbook / Checklist
Project Plan 只定义“做什么、谁做、做到什么算过”，不写动作级步骤细节、场景处置路径与检查项制度。

### 5. 不承担单轮 Prompt 功能
Prompt 只服务单次执行单元调用，不承担项目推进编排。

### 6. 不吞并施工日志 / 归档 / 复盘
过程记录、失败复盘、历史归档应下沉到相应文档，不应把 Plan 写成流水账。

### 7. 不定义运行结构实现
Plan 可以声明本轮是否依赖 Skill、MCP、Harness、Runtime Pipeline，但不负责定义这些运行结构本身。

---

## 四、何时必须建立 Project Plan

出现以下任一情况时，原则上应建立 Project Plan：

1. 任务不是单步完成，而是需要跨多轮推进  
2. 任务至少包含两个及以上阶段性交付  
3. 任务涉及多人或多 Agent 协作  
4. 任务存在依赖链、阻塞链或阶段性验收点  
5. 任务周期会跨越单次对话窗口，存在上下文漂移风险  
6. 任务需要明确“本轮目标 / 非目标 / 阶段出口 / 放行标准”  
7. 任务虽不一定需要独立 `program.md`，但已经不能只靠 SOP / Prompt 或即时执行承载

如果只是单轮、单步、单执行者的轻量动作，可不立 Project Plan，直接由 SOP / Prompt 或即时执行承担。

---

## 五、Project Plan 的进入条件

Project Plan 不是想到就写的说明文；要进入正式项目态，至少应满足以下条件：

1. **任务对象可指认**：当前对象是什么，不能模糊  
2. **本轮目标可表述**：这一轮要解决什么问题  
3. **本轮非目标可区分**：明确哪些内容不并入本轮  
4. **主责任主体可识别**：至少能明确项目主责  
5. **最小交付物可指认**：本轮至少会产出什么  
6. **存在基本裁决口径**：完成 / 待核 / 打回 / 暂停 / 阻塞 等判断方式可说明

若上述条件缺失到足以影响推进，Project Plan 可以停留在 `Draft / Scoped`，但不得直接宣称进入稳定执行态。

## 六、Project Plan 与 Program 的关系

### 6.1 默认关系
Project Plan 默认受团队级 Program 主协议约束。

如果任务**不单独建立任务级 `program.md`**，则该 Plan 默认继承：
- 团队级默认状态骨架  
- 默认交接底线与验证底线  
- 默认放行 / 打回 / 暂停 / 阻塞的基本裁决口径  
- 默认最小执行单元原则

### 6.2 何时仅有 Plan 即可
以下任务通常可仅有 Project Plan，而不另建任务级 `program.md`：

1. 本轮需要阶段化推进，但不需要独立运行闭环  
2. 协作关系有限，不需要自定义状态机  
3. 无需 runtime pipeline 深度接入  
4. 团队级 Program 已足够覆盖其运行机制

### 6.3 何时必须同时具备 Plan + 任务级 `program.md`
当任务满足以下任一条件时，原则上应同时具备：
- 任务需独立、持续、可恢复地运行  
- 任务有正式状态机、review / rework / pause 机制  
- 任务需 runtime pipeline 或 harness 承载  
- 任务失败成本高，不能只靠项目计划维持秩序  
- 多角色交接结构较复杂，需为该任务单独定义闭环规则或下位字段结构

一句话说：

> **Program 解决“系统怎么跑”，Plan 解决“这一轮怎么推”。**

## 七、Project Plan 生命周期与状态映射

Project Plan 不是静态文档，而是项目推进状态的承载件之一。

推荐采用以下状态：

- **Draft**：已有任务意向，但进入条件尚未压清  
- **Scoped**：目标、范围、非目标、主责、最小交付物已基本压清  
- **Active**：阶段已排布，责任已分配，当前可正式推进  
- **Review**：当前轮产物进入复核  
- **Rework**：当前轮被打回，进入最小修复  
- **Blocked**：因依赖、权限、输入、接口等问题卡住  
- **Pause**：当前不宜继续推进，但不宣告失败  
- **Failed**：本轮推进在当前周期内不可继续，或已失去推进价值  
- **Done**：本轮交付已通过验证，并形成可读取结果  
- **Archived**：已退出主链，仅保留历史事实

### 7.1 与 Program 状态的对应原则
- Project Plan 应尽量与 Program 主协议状态兼容，而不是自造另一套状态体系  
- 如 Plan 使用更简化状态，必须能映射回 Program 状态  
- 如任务级 `program.md` 自定义了更细状态，Plan 应至少反映阶段性主状态  
- 更细状态解释、回写字段与运行实现，不由 Plan 本身承担

---

## 八、Project Plan 的核心编写原则

### 1. 面向“本轮推进”，不是面向永久制度

### 2. 面向可继续执行，不面向展示
后续执行者应能据此接住，而不是只看懂背景。

### 3. 目标、非目标、范围、阶段、责任、验收必须分开写
不得混成一段叙述。

### 4. 阶段必须绑定交付物、位置与验证方式
没有这三项，阶段容易失真。

### 5. 阶段出口必须明确
阶段结束时，要么进入下一阶段，要么进入 Review / Blocked / Pause / Rework，不得悬空。

### 6. 责任必须可定位
一个阶段可以多人参与，但主责、复核、裁决必须能明确指认。

### 7. 关键风险必须前置暴露
尤其是会影响方向、范围、验收、节奏与外部依赖的风险。

### 8. Plan 不替下位件写动作细节
Plan 写到“做什么、谁做、做到什么算过”为止。

### 9. 变化不能只留在聊天里
凡影响目标、范围、阶段、交付、验收或责任的变化，必须回写 Plan。

### 10. 与上位件冲突时，以上位件为准
Plan 不能反向覆盖 Charter 或 Program。

### 11. Plan 默认管理阶段，不默认管理动作级实现
动作标准做法、场景处置路径、接口级运行实现与单轮调用，不由 Plan 主体承担。

## 九、Phase 与 Work Unit 的关系

Project Plan 默认管理的是**阶段（Phase）层**，而非每一步动作层。

但当任务复杂到单个阶段已无法稳定承载时，应进一步拆成 **Work Unit（最小执行单元）**。

### 9.1 何时只写到 Phase 即可
- 阶段内动作较少  
- 主责关系清晰  
- 不需要多人并行接力  
- 阶段内无需正式单元级交接

### 9.2 何时必须下钻到 Work Unit
- 单一阶段内部已存在多个正式交接点  
- 单一阶段涉及多个责任人或多 Agent 串联  
- 需要更细的阻塞追踪、验证或重做控制  
- 阶段内容已经复杂到“只有阶段标题不足以承载执行”

### 9.3 Work Unit 最低挂接原则
当 Plan 下钻到 Work Unit 时，应至少说明：
- 当前阶段是否已拆单元  
- 单元记录挂在哪里  
- 单元的目标、主责、交付、验证、依赖、去向与状态如何被承接  
- 当前轮最关键单元有哪些

字段命名、schema 结构、运行字段与任务特化模板，下沉至任务级 `program.md`、Work Unit 清单或接口件承担。

---

## 十、阶段性交接与交付要求

凡阶段出口涉及“把当前产物转为下一责任方输入”，都应视为正式交接。

### 10.1 Plan 中必须写清的交接信息
Plan 至少应写清：
- 哪些阶段出口必须触发正式交接  
- 交接给谁  
- 交接后预期进入什么状态  
- 若交接失败，如何回写为 Rework / Blocked / Pause

### 10.2 正式交接底线
阶段性交接必须包含足以支持下一责任方继续推进的最小必要信息，例如：
- 当前产物位置或对象  
- 已完成内容  
- 验证方式或验证依据  
- 当前未解项 / 已知问题  
- 下一责任方可接续的动作

固定字段模板、频道格式与任务特化交接结构，下沉至任务级 `program.md`、交接单或交接规范卡承担。

### 10.3 阶段完成 ≠ 交接完成
某阶段产物已生成，不等于该阶段已经完成；  
**只有产物可读取、验证方式明确、下一接收方可接住时，阶段出口才算成立。**

---

## 十一、交付、证据与位置原则

Project Plan 不只写“有什么交付物”，还必须尽量写清以下内容：
- 交付物名称  
- 交付物位置  
- 验证证据或验证入口  
- 下一接收方  
- 必要时的归档去向

最小原则：

> **一个不可指认位置的交付物，不足以支持稳定交接。**

字段表、交付清单模板与审计记录格式，下沉至交付单、验收单、审计单或任务级执行件承担。

---

## 十二、阻塞、暂停、恢复与失败原则

当项目进入 Blocked / Pause / Failed 时，Plan 至少应说明：
- 原因是什么  
- 影响范围是什么  
- 恢复或重开条件是什么  
- 当前停留在哪一阶段

更细的状态记录项、恢复路径与任务专属回写模板，下沉至任务级 `program.md`、运行记录件或执行卡承担。

---

## 十三、裁决与权限原则

Project Plan 至少应明确：
- 项目主责是谁  
- 阶段主责是谁  
- 阶段复核由谁承担  
- 整体放行由谁裁决  
- 打回由谁发起  
- 暂停由谁有权触发

更细的权限结构、回执制度与任务专属裁决机制，下沉至任务级 `program.md`、验收卡或治理执行卡承担。

---

## 十四、下位件清单

Project Plan 负责定义当前周期的项目级推进结构，不直接替代下位执行件；当任务需要进一步落地时，可派生、约束或调用以下下位件：

### 14.1 Program 层特化件
- **任务级 `program.md`**：用于定义某一具体任务的特化闭环、状态展开、运行节奏、承载方式与交接结构

### 14.2 Project 层下的补充件
- **Phase 卡**：用于展开单个阶段的目标、责任、交付与出口  
- **Work Unit 清单**：用于承接阶段下钻后的单元拆分  
- **交付清单 / 依赖清单 / 风险清单**：用于承接项目级辅助结构

### 14.3 执行层下位件
- **SOP / Runbook / Checklist**：用于定义动作标准做法、场景处置与检查门槛  
- **Prompt**：用于定义单轮执行指令  
- **交接单 / 验收单 / 审计单 / 施工日志**：用于承接某次具体执行、交付、复核与留痕

### 14.4 使用原则
- Project Plan 只定义这些下位件为什么被调用，不替它们把实现细节一次写完  
- 下位件可以特化，但不得覆盖 Project Plan 的本轮推进结构与上位件约束  
- 当同一内容已有明确下位件承担时，不应继续堆积在 Plan 主体中

---

## 十五、总括

Project Plan 不是为了让项目多一张说明文，而是为了让当前周期的任务推进：
- 目标清楚  
- 范围可控  
- 阶段明确  
- 责任可定位  
- 交付可指认  
- 验收可判断  
- 风险可暴露  
- 变化可回写

一句话说：

> **Program 负责系统怎么跑，Project Plan 负责这一轮怎么推；Plan 只管阶段推进结构，不替任务级运行协议和下位执行件写实现细节。**
