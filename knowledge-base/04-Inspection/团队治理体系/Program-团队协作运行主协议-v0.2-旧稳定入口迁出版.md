# Program：团队协作运行主协议
## v0.2（Multi-Agent 场景适配版）

- **类型**：Program / 团队协作运行主协议
- **适用对象**：需要拆分、交接、复核、持续推进的团队任务
- **重点场景**：Multi-Agent 协作任务
- **上位依据**：`团队协作宪法与治理总纲 v2.1（减法收口版）`
- **状态**：draft
- **定位说明**：本文件定义团队任务如何进入闭环、如何持续运行、如何交接、如何验证、如何放行 / 打回 / 暂停。`Multi-Agent` 是本 Program 的重点适配场景之一，而非本 Program 的上位概念。本文件不替代项目计划、SOP、Prompt 与项目文档。

---

## 一、Program 定位

`Program` 不是 prompt，不是 SOP，也不是项目计划书。

它的职责是：

> **把团队任务从一次性指令，组织成可持续推进、可验证、可回收的运行闭环。**

对团队而言，Program 主要回答以下问题：

1. 任务如何进入正式闭环  
2. 闭环如何持续运行而不失真  
3. 任务如何被拆成可执行单元  
4. 单元之间如何交接与复核  
5. 什么时候放行、打回、暂停或收口  
6. 哪些结果应沉淀，哪些只属于当前项目

因此，Program 的本质不是“某类任务说明书”，而是：

> **团队协作的运行主协议。**

---

## 二、与其他层级的关系

在团队治理五层中，Program 位于 Charter 之下、Project Plan 之上：

```text
Constitution / Charter
        ↓
Program
        ↓
Project Plan
        ↓
SOP
        ↓
Prompt
```

各层分工如下：

- **Charter**：定义上位原则与边界  
- **Program**：定义任务闭环与运行机制  
- **Project Plan**：定义本次任务的阶段排布  
- **SOP**：定义高频动作的标准做法  
- **Prompt**：定义单轮执行指令

Program 不重写上位原则，也不下沉到逐步操作细则。

## 三、适用范围

### 3.1 默认适用任务

本 Program 适用于以下任务：

- 任务需要跨多轮推进
- 任务可以拆成多个最小执行单元
- 单元之间存在正式输入 / 输出关系
- 产物需要被他人接续、复核或审计
- 任务存在质量门槛、证据门槛或放行门槛

### 3.2 重点适配场景：Multi-Agent 协作

当任务涉及多个 Agent 之间的拆分、指派、交接、复核与持续推进时，本 Program 进入其最典型使用场景。

在该场景下，Program 重点解决：
- 一单元一主责
- 多角色分工不失真
- 正式交接可继续接住
- Review / Rework / Pause / Done 可裁决
- 运行中不断点、不失忆、不靠口头续命

### 3.3 不适用范围

以下场景不建议直接套用本 Program：
- 单 Agent 单步即可完成的简单任务
- 目标尚未形成、边界极度模糊的纯讨论任务
- 无法留痕、无法验证、无法回退的高风险操作
- 一次性问答、无需交接的轻量动作

对不适用场景，默认采用更轻执行方式，而非强行套 Program。

## 四、Program 核心目标

本 Program 的目标，不是让更多人同时开工，而是让团队任务具备以下能力：

1. **可进入**：任务进入前有最小边界，不靠模糊启动。  
2. **可拆分**：复杂任务能被压成明确的最小执行单元。  
3. **可指派**：每个单元都有唯一主责。  
4. **可交接**：一个执行者的产物能稳定成为下一个执行者的输入。  
5. **可验证**：完成、待核、打回、暂停有明确判定口径。  
6. **可持续**：协作能持续运行，而不依赖单点人物反复续命。  
7. **可回收**：阶段产物、关键判断与失败原因不会无痕蒸发。

Program 服务于一句话：

> **进入 → 拆分 → 指派 → 产出 → 交接 → 复核 → 放行/打回/暂停 → 继续推进或收口**

## 五、核心治理原则

### 原则 1：先压边界，再进闭环
任务未形成最小进入条件前，不进入正式运行闭环。

最小进入条件至少包括：
- 目标能被描述
- 主责任主体能被识别
- 本轮交付物能被指认
- 范围与非范围能被区分
- 存在基本裁决口径

### 原则 2：先拆最小单元，再分配责任
不把“大任务”整体抛给某个执行者，必须先压成最小执行单元，再决定由谁处理。

每个最小单元至少应回答：
- 做什么
- 交付什么
- 放在哪里
- 如何验证
- 完成后交给谁

### 原则 3：一个单元只允许一个主责
每个最小执行单元只能有一个主责角色。

可以存在支持位、复核位、审计位，但不能出现“共同负责”而无人负责到底。

### 原则 4：正式交接必须结构化
任何进入正式链路的交接，都不得使用“你看一下”“差不多好了”之类模糊表达。

正式交接必须至少包含：
- **Where**：产物位置
- **What**：已完成内容
- **How to verify**：验证方式
- **Known Issues**：已知问题
- **Next Steps**：接收方下一步动作

### 原则 5：无证据，不放行
任何“完成”“可用”“通过”“可进入下游”的判断，都必须附着于可读取的产物、证据或判定标准。

### 原则 6：打回是制度动作，不是情绪动作
复核发现问题时，应给出明确打回原因与最小修复路径，而不是笼统否定。

### 原则 7：暂停优于带病推进
当前提缺失、路径冲突、权限不足或继续推进会显著扩大错误成本时，暂停优先于强推。

### 原则 8：可沉淀事项不得长期停留在聊天里
重复性结论、稳定边界、反复出现的交接格式与验证口径，应沉淀为下位制度或知识条目，而不是每轮重新发明。

## 六、角色模型

本 Program 默认区分以下角色。一个 Agent 可以兼任多个角色，但**单次动作必须存在唯一主责角色**。

### 6.1 主任务位（Task Owner）
**职责**：
- 对当前任务结果负责
- 决定任务是否进入正式闭环
- 确认关键协作关系
- 对最终放行 / 暂停 / 收口承担主判断责任

### 6.2 编排位（Orchestrator）
**职责**：
- 拆分任务单元
- 建立状态流转
- 指派责任角色
- 跟踪阻塞与交接
- 维护协作秩序

### 6.3 执行位（Builder / Operator）
**职责**：
- 完成被分配的最小执行单元
- 产出明确交付物
- 按协议发起正式交接

### 6.4 复核位（Reviewer / Auditor）
**职责**：
- 验证产物是否达到本轮目标
- 检查证据、边界、逻辑与风险
- 给出放行 / 打回 / 暂停判断

### 6.5 收口位（Keeper / Curator）
**职责**：
- 对值得保留的规则、结论、模板与知识进行沉淀
- 完成必要归档、回挂与版本留痕

## 七、任务生命周期

默认状态流转如下：

```text
Inbox → Scoped → Assigned → In Progress → Review → Done
                    ↑            ↓            ↘ Rework ↗
                    └──── Blocked ────────→ Pause / Failed
```

### 7.1 Inbox
任务已进入系统，但尚未形成最小边界。

### 7.2 Scoped
任务边界已被压清，至少明确：要做什么、本轮不做什么、最小交付物、初步风险点、是否可拆分。

### 7.3 Assigned
任务已被拆成最小执行单元，并明确单元主责、产物位置、验证方式与交接要求。

### 7.4 In Progress
执行者正在产出。开工即进入该状态；遇到关键阻塞必须报码点；完成后必须发起正式 handoff。

### 7.5 Review
产物进入复核。输出结论仅允许：放行、打回、暂停。

### 7.6 Rework
被打回后，执行者按最小修复路径进行修复，不得把明确问题重新模糊化。

### 7.7 Blocked
任务因依赖缺失、权限不足、输入不明、接口未就绪等原因卡住，必须写清阻塞点、缺失项与解除条件。

### 7.8 Pause
任务当前不宜继续推进，但不宣告失败。

### 7.9 Failed
任务在当前周期内不可继续推进，或继续推进已不具价值，必须记录失败原因。

### 7.10 Done
任务已通过本轮验证，且必要交付已完成。Done 的最低条件是：产物存在、结论存在、验证结论存在、下一层输入可读取。

---

## 八、最小执行单元规范

一个最小执行单元（Work Unit）至少应包含以下字段：

- **Unit ID**：单元标识
- **Owner**：主责角色
- **Goal**：本单元目标
- **Deliverable**：最小交付物
- **Location**：产物位置
- **Verify**：验证方法
- **Depends On**：依赖项
- **Next Receiver**：下一接收方
- **Status**：当前状态

若以上字段无法说明，则该单元仍不够清楚，不应直接派发。

---

## 九、正式交接协议（Handoff Card）

所有正式 handoff 统一使用以下最小结构：

```text
[HANDOFF]

Where:
产物路径 / 文档链接 / 文件位置

What:
本轮已完成内容

How to verify:
如何验证 / 看什么 / 用什么标准判断

Known Issues:
当前已知问题 / 限制 / 未覆盖部分

Next Steps:
明确希望接收方下一步做什么
```

满足以下条件，才可视为有效正式交接：
- 接收方知道去哪里看
- 接收方知道这轮做完了什么
- 接收方知道如何判断其是否可用
- 接收方知道当前还有什么问题
- 接收方知道自己接下来该做什么

## 十、复核与裁决协议

### 10.1 复核检查项
复核至少检查以下五类问题：
- **目标达成**：是否满足本轮目标
- **证据充分**：是否存在足够证据支撑结论
- **边界正确**：是否越界或漏掉关键范围
- **交付可用**：下游是否能继续接住
- **风险已说清**：关键限制是否被显式暴露

### 10.2 裁决输出
复核结论只允许以下三类：

#### 放行（Pass）
达到本轮目标，证据充分，不存在足以推翻完成判断的关键未解项。

#### 打回（Rework）
必须说明：打回原因、未达标点、最小修复路径、回来时应补什么证据。

#### 暂停（Pause）
必须说明：暂停原因、当前不能继续推进的关键前提、解除暂停需要什么条件。

## 十一、任务级 Program.md 的适用条件与模板边界

团队级 `Program-团队协作运行主协议.md` 用于定义 Program 的通用原则与运行机制；
当某一任务需要被组织成独立、持续、可裁决的运行闭环时，应为该任务建立任务级 `program.md`。

### 11.1 哪些任务必须具备独立 `program.md`
当任务满足以下任一条件时，原则上应建立任务级 `program.md`：
- 任务跨越多轮，而非单轮即可完成
- 任务涉及多个 Agent、多个角色或明确交接界面
- 任务存在正式 review / rework / pause / done 状态
- 任务需要持续运行、断点恢复或 runtime pipeline 接入
- 任务失败成本较高，不能依赖临场口头协作推进
- 任务需要被审计、追踪或阶段性回看

### 11.2 哪些任务可以不单独建立 `program.md`
以下任务通常可不单独建立任务级 `program.md``：
- 单轮即可完成的轻量任务
- 单执行者直接处理、无需正式交接的任务
- 已可由 Project Plan、SOP 或 Prompt 直接承载的低复杂度动作
- 不需要持续运行、状态流转与复核机制的简单执行

不单独建立任务级 `program.md`，不等于不受 Program 约束；而是表示该任务可直接服从团队级主协议，以轻模式运行。

### 11.3 任务级 `program.md` 的最小结构
任务级 `program.md` 至少应包含以下部分：
1. **任务目标**：本任务要解决什么问题  
2. **进入条件**：任务在什么前提下进入正式闭环  
3. **角色结构**：主任务位、执行位、复核位等如何挂接  
4. **状态流转**：任务采用何种状态机推进  
5. **最小执行单元**：如何拆分、如何交接  
6. **交付与验证**：最小交付物、验证方法与裁决口径  
7. **暂停与恢复**：何时暂停、如何恢复  
8. **与 runtime 的关系**：哪些运行状态由 runtime pipeline 承载

### 11.4 边界说明
任务级 `program.md` 的边界如下：
- **不重写 Charter**：不得重新定义团队上位原则  
- **不替代 Project Plan**：不得承担本次任务的阶段排布与里程碑设计  
- **不下沉为 SOP**：不得写成逐步操作手册  
- **不压缩成 Prompt**：不得直接承担单轮执行指令功能

一句话说，任务级 `program.md` 负责定义：

> **这一个任务如何被组织成可持续运行的闭环。**

## 十二、Program 与 runtime pipeline 的接口关系

Program 定义任务应如何运行；runtime pipeline 负责把这种运行真实承载起来。

两者关系不是替代关系，而是：

> **Program 是运行主协议，runtime pipeline 是运行承载层与反向验真层。**

### 12.1 Program 向 runtime pipeline 提供什么
当任务接入 runtime pipeline 时，Program 至少应向运行层提供以下内容：
- **任务标识**：任务 / Program 的唯一标识
- **状态模型**：该任务采用何种状态流转
- **角色结构**：谁是主任务位、执行位、复核位等
- **执行单元定义**：最小执行单元如何拆分与派发
- **交接规则**：handoff 的最小要求是什么
- **裁决规则**：何时 pass / rework / pause / done
- **阻塞口径**：什么情况记为 blocked，如何解除

### 12.2 runtime pipeline 向 Program 回传什么
runtime pipeline 至少应向 Program 回传以下信息：
- **当前状态**：任务当前推进到哪里
- **执行结果**：某个执行单元是否已完成
- **日志与轨迹**：运行中发生了什么
- **阻塞原因**：卡点在哪里、缺什么、谁能解除
- **恢复点**：从哪里可以继续接续任务
- **摩擦信号**：哪些地方反复断点、失真、卡死或依赖单点人物

### 12.3 runtime pipeline 如何服务任务
runtime pipeline 对任务的服务，至少体现在以下五类能力上：

#### 1. 承载状态流转
把 Program 中定义的状态真实记录下来，而不是只停留在文档里。

#### 2. 承载持续推进
支持任务跨轮继续运行，避免任务每次都靠人工重新说明上下文。

#### 3. 承载执行单元调度
支持最小执行单元的派发、回收、交接与再进入 review。

#### 4. 承载日志与追踪
为任务提供最小可追溯信息，支撑验证、审计与断点恢复。

#### 5. 承载制度验真
暴露 Program 在真实运行中的断点、漂移与摩擦，作为反向修订 Program 与 Charter 的依据。

### 12.4 边界说明
- Program 不负责实现 runtime pipeline 的技术细节  
- runtime pipeline 不负责替 Program 发明运行规则  
- 若 runtime 的现实约束与 Program 设计冲突，应显式回传并触发 Program 修订，而不是由实现层静默改写主协议

一句话说：

> **Program 负责定义怎么跑，runtime pipeline 负责让它真正跑起来，并告诉我们这套跑法哪里还不成立。**

## 十三、升级与降级规则

### 13.1 何时升级为正式协作模式
当出现以下任一情况时，任务可升级为正式团队协作模式：
- 单执行者无法稳定完成全部链路
- 任务需要独立复核或审计
- 任务存在明确分工界面
- 任务跨多轮且需要状态管理
- 任务失败成本较高，需要更强边界控制

### 13.2 何时降级为轻模式
当出现以下情况时，可退出 Program 重模式：
- 任务已压缩为单执行者可直接完成的小任务
- 当前轮不需要正式交接与复核
- 结构化管理成本高于任务本身成本

降级不是放弃治理，而是避免治理开销压过任务价值。

## 十四、沉淀与回挂规则

以下内容不应长期只停留在聊天里：
- 重复出现的交接格式
- 多次复用的验证标准
- 已稳定的角色边界
- 高频风险点与打回口径
- 可复用的执行模板

满足以下条件之一者，应进入沉淀：
- 已重复使用两次以上
- 已成为多人默认依赖
- 若不沉淀将持续造成失真或返工

沉淀位置应服从文档层级：
- 团队级原则 → Charter
- 闭环结构与角色协议 → Program / 执行卡
- 具体操作步骤 → SOP / runbook / 模板
- 当前项目事实 → 项目文档

## 十五、与 Multi-Agent 场景的关系

Multi-Agent 协作不是 Program 的上位概念，而是 Program 的重点适配场景之一。

也就是说：
- **不是**“因为有多个 Agent，所以临时拼一套 Program”
- **而是**“团队先有运行主协议，多 Agent 协作只是该主协议最典型、最复杂、最需要被验证的使用场景之一”

因此，本 Program 可以用于：
- 多 Agent 协作任务
- 多角色串联的人机协作任务
- 单 Agent 但需跨多轮、需正式交接与复核的复杂任务

## 十七、总括

本 Program 的最低要求，不是“把任务分给几个人”，而是：

> **任务先压边界，再拆最小单元；单元一事一主责；交接必须结构化；复核必须能裁决；问题能打回，风险可暂停，结果可继续接住。**

若一项团队协作同时满足以下条件：
- 进入前边界清楚
- 拆分后责任明确
- 交接时字段完整
- 复核时证据充分
- 打回时修复路径清楚
- 运行中不中断于单点续命
- 收口时产物可回收

即可视为符合本 Program。
