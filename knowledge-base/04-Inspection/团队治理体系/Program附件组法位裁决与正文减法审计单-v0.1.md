## 二、Program 正文必须承担什么

以下内容建议明确保留在 `Program-团队协作运行主协议.md` 正文中，不再外移：

### 2.1 团队级 Program 的定位
必须保留：
- Program 是团队默认运行主协议  
- Program 不等于 Project Plan / SOP / Prompt  
- Program 负责默认闭环、默认角色机制、默认裁决口径与运行结构法位

### 2.2 团队级 Program 与任务级 `program.md` 的边界
必须保留：
- 团队级 Program 管“默认怎么跑”  
- 任务级 `program.md` 管“具体任务怎么落地跑”  
- 任务级可细化但不覆盖上位原则

### 2.3 默认状态骨架与默认交接 / 验证底线
必须保留：
- 默认状态骨架  
- 默认正式交接底线  
- 默认验证与裁决底线

说明：
- 这些是 Program 作为主协议的核心，不可完全外包给附件

### 2.4 能力层 / 接口层 / 承载框架的最小法位
必须保留：
- Skill 的法位  
- MCP 的法位  
- Harness 的法位  
- Runtime Pipeline 的法位  

说明：
- Program 正文不需要展开技术实现  
- 但必须在正文里保留最小法位确认，否则下位件无法统一引用

### 2.5 下位件清单
必须保留：
- Program 层会派生哪些下位件  
- 哪些属于任务级 `program.md`、接口卡、交接卡、裁决卡等

说明：
- 这是 Program 保持减法边界的关键结构，不建议删掉

## 三、Program 正文建议继续做的减法

以下内容不建议继续在 Program 正文中膨胀，应控制在“最小默认协议”范围内：

### 3.1 不继续展开固定字段模板
Program 不应再新增：
- 固定 handoff 字段表  
- 固定 Work Unit schema  
- 固定 runtime 字段结构

### 3.2 不继续细化任务级状态机
Program 保留默认状态骨架即可，不应继续长成任务级状态说明书。

### 3.3 不继续补运行接线细节
Program 不应直接吸收：
- runtime pipeline 的具体接线方式  
- MCP 接入的具体配置方法  
- Harness 的实现细节

### 3.4 不继续吸收术语澄清件全文
Program 可以保留术语最小法位，但不应把关系卡全文并回正文。

## 四、附件组逐件裁决

### 4.1 `任务级Program文件规范卡-v0.1.md`
#### 当前角色判断
Program 层的**正式下位规则卡候选**。

#### 裁决
**建议保留为正式附件。**

#### 原因
- 它承担的是“任务级 `program.md` 何时需要建立、如何承担对象”的规则  
- 这类内容若全部并入 Program 正文，会使主文继续增厚  
- 但它又不是纯说明件，而是直接约束 Program 下位特化的规范卡

#### 后续建议
- 可在 Program 正文中保留简述 + 链接  
- 本卡保留为 Program 正式附件之一

---

### 4.2 `Program与runtime-pipeline接口卡-v0.1.md`
#### 当前角色判断
Program 层的**接口说明附件候选**。

#### 裁决
**建议暂保留为说明附件，不并入正文。**

#### 原因
- 它更偏 Program 与 runtime 接入之间的接口说明  
- 过早并入正文，会把 Program 拉回到接线级文档  
- 但在当前体系尚未完全稳定前，仍有保留价值

#### 后续建议
- 二轮再判断是否：
  1. 长期作为说明附件保留  
  2. 并入 runtime 相关接口总卡  
  3. 或并入任务级 Program 规范附件组
