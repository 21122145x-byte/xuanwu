# Project Plan：团队治理体系建设
## v0.1

- **类型**：Project Plan / 项目计划
- **状态**：draft
- **用途**：用于定义“团队治理体系建设”这一当前主任务的阶段目标、推进顺序、责任分配、里程碑与阶段出口。
- **适用范围**：当前团队治理体系建设任务
- **上位依据**：
  - `团队协作宪法与治理总纲-v2.1-减法收口版.md`
  - `Program-团队协作运行主协议.md`

---

## 一、项目定位

本 Project Plan 服务的不是某一条单独文档，而是当前正在进行的**团队治理体系建设任务**。

该任务的目标，不是继续堆叠概念，而是把团队治理从历史混合线路中拆层、收口，并建立一条新的正式治理链路：

```text
Constitution / Charter
→ Program
→ Project Plan
→ SOP
→ Prompt
→ runtime pipeline
```

本 Plan 负责回答：

1. 当前治理建设任务分几阶段推进  
2. 哪些文档优先落地  
3. 哪些能力属于当前范围，哪些暂不进入本轮  
4. 每个阶段完成的判定标准是什么  
5. 哪些旧体系内容应迁移，哪些应归档退役

---

## 二、项目目标

### 2.1 总目标
建立一套层级清楚、入口单一、可持续运行、可被 runtime pipeline 反向验真的团队治理体系。

### 2.2 当前阶段目标
本轮优先完成以下事项：

1. **上位法收口**  
   完成 Charter 主文收口，并明确其为团队上位法。

2. **Program 主协议成型**  
   完成 Program 主协议，并补齐任务级 `program.md` 规范与 runtime 接口关系。

3. **轻治理线退役分流**  
   将历史“轻治理线”内容拆分迁入 Program / Project Plan / SOP / runtime / 归档。

4. **Project Plan 层建立**  
   明确当前治理建设任务本身的阶段排布与里程碑。

5. **SOP 层开始建骨架**  
   至少明确首批应建设的 SOP 清单与优先顺序。

6. **runtime pipeline 接口对齐**  
   明确 Program 与 runtime pipeline 的关系，准备后续运行验真接入。

---

## 三、本轮范围与非范围

### 3.1 本轮范围
本轮范围包括：

- Charter 主文收口与定位压清
- Program 主协议收口与唯一入口确定
- Project Plan 层建立
- 轻治理线内容迁移分流
- 首批 SOP 的选型与排序
- Program 与 runtime pipeline 的接口口径建立
- 团队治理相关图卡与配套卡建设

### 3.2 非本轮范围
以下事项暂不作为本轮必须完成项：

- 一次性补全全部 SOP 正文
- 全量 Prompt 体系一次性建完
- runtime pipeline 的全部技术实现细节
- 所有历史治理文档的立刻归档整理
- 所有团队任务同时切换到新治理链路

### 3.3 范围控制原则
若新增事项不能直接服务于“团队治理体系主链建立”，原则上不并入本轮。

---

## 四、阶段拆分

本项目当前建议拆为五个阶段。

---

## 阶段 A：上位法与主协议定型

### 目标
建立新的正式治理主链的前两层：
- Charter
- Program

### 当前应交付
- `团队协作宪法与治理总纲-v2.1-减法收口版.md`
- `Program-团队协作运行主协议.md`
- Program 配套卡：
  - `任务级Program文件规范卡-v0.1.md`
  - `Program与runtime-pipeline接口卡-v0.1.md`

### 完成判定
- Charter 作为上位法定位明确
- Program 作为运行主协议定位明确
- Program 具备任务级 Program 规范与 runtime 接口说明
- 主入口不再并行分叉

---

## 阶段 B：旧治理线路拆解与迁移

### 目标
停止“轻治理线”作为独立主线继续存在，并将其内容拆层迁入新治理链路。

### 当前应交付
- `轻治理线内容迁移表-v0.1.md`
- 对旧治理线内容的去向判断

### 完成判定
- 轻治理线不再作为默认主入口
- Program / Plan / SOP / runtime / 归档 的分流口径明确
- 旧线路从“主线”降为“待吸收 / 待归档对象”

---

## 阶段 C：Project Plan 层建立

### 目标
为当前治理建设任务本身建立任务级推进计划。

### 当前应交付
- `Project-Plan-团队治理体系建设-v0.1.md`

### 完成判定
- 当前任务的阶段、顺序、责任、里程碑被明确定义
- Program 与 Plan 的层级边界成立
- 当前推进不再依赖临场口头排序

---

## 阶段 D：SOP 层首批建骨架

### 目标
确定首批最必要的 SOP，并开始从动作层补齐治理体系。

### 首批建议 SOP
1. 正式交接执行卡  
2. 放行 / 打回 / 暂停裁决卡  
3. 需求对齐执行卡  
4. prompt 转写执行卡  
5. 审计与回写执行卡

### 完成判定
- 首批 SOP 清单明确
- SOP 的优先顺序明确
- 哪些动作为 SOP 层、哪些仍属于 Program / Plan 已压清

---

## 阶段 E：runtime pipeline 对接与验真准备

### 目标
将治理体系从文档链推向真实运行承载链。

### 当前应交付
- Program 与 runtime pipeline 的接口口径明确
- 后续 runtime 接入的最小验证面清单

### 完成判定
- 明确 runtime pipeline 如何承载状态、调度、日志、恢复点与制度验真
- 明确下一步要如何拿真实任务去反向校验 Program 与 Charter

---

## 五、当前里程碑

### M1：治理主链成型
判定条件：
- Charter 已建立
- Program 已建立
- Project Plan 已建立
- 五层关系已压清

### M2：旧线退役完成
判定条件：
- 轻治理线已完成分流判断
- 不再作为正式主线继续使用

### M3：动作层开始落地
判定条件：
- 首批 SOP 已明确并开始建设

### M4：运行层准备接入
判定条件：
- Program 与 runtime pipeline 接口已可被引用
- 已形成 runtime 验真准备条件

## 六、责任分配

### 主任务位
当前主任务位：**团队治理体系建设主线**  
本 Plan 默认服务于该主线推进。

### 当前主要工作责任
- **Charter / Program / Plan 收口**：当前主线推进责任

## 八、当前风险点

### 风险 1：旧线路回流
若不及时压住，轻治理线可能继续以“熟路径”身份回流为并行主线。

### 风险 2：Program 再次空心化
若 Project Plan、SOP、runtime 不继续补齐，Program 可能再次停留在概念层。

### 风险 3：层级重新混线
若任务级内容继续写进 Program，或动作模板继续堆进 Plan，治理体系会重新长胖。

### 风险 4：runtime 接口悬空
若不尽快把运行承载层接上，治理体系会停留在文档链，无法完成反向验真。

---

## 九、下一步动作

当前建议的下一步顺序如下：

1. 确认本 Project Plan 作为当前任务级计划入口  
2. 按优先级起草首批 SOP 之一  
3. 明确 runtime pipeline 的最小接入验证面  
4. 逐步将旧轻治理线内容迁出并降级归档

---

## 十、阶段出口标准

本 Project Plan v0.1 的出口，不是“体系全部完成”，而是满足以下条件：

1. 当前治理建设任务已有明确计划层承载  
2. Program 不再承担 Project Plan 职责  
3. 旧轻治理线不再承担主线职责  
4. SOP 层已明确首批建设顺序  
5. runtime pipeline 已进入正式对接准备

满足以上条件后，可进入 Project Plan v0.2 或转入首批 SOP 建设阶段。

---
