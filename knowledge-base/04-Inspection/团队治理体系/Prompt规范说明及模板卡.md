## 一、定位

在团队治理五层中，Prompt 位于最下层。

```text
Charter
  ↓ 原则与边界
Program
  ↓ 团队默认闭环结构、运行机制与承载口径
Project Plan
  ↓ 本轮推进目标、阶段、责任、交付与验收
SOP / Runbook / Checklist
  ↓ 动作标准做法、场景处置与检查门槛
Prompt
  ↓ 单轮执行指令
```

Prompt 的唯一职责是：

> **把当前这一次调用、这一步动作、这个执行单元，压成一个可直接发起的调用壳。**

Prompt 以下不再继续分治理层，只进入具体执行与输出。

---

## 二、功能

Prompt 只解决以下问题：

1. **当前这次调用具体做什么**  
2. **当前输入材料是什么**  
3. **应按什么边界处理**  
4. **要输出什么结果**  
5. **什么样的结果才算合格**

一句话说：

> **Prompt 负责把一次执行发准。**

---

## 三、边界

### 1. Prompt 不是 Program
Prompt 不定义角色关系、状态机、交接制度、放行 / 打回 / 暂停机制。

### 2. Prompt 不是 Project Plan
Prompt 不定义本轮项目目标、阶段排布、责任分工、交付链路与整体验收。

### 3. Prompt 不是 SOP
Prompt 不承担长期稳定动作规范，不定义默认标准做法。

### 4. Prompt 不是 Runbook
Prompt 不定义完整场景处置路径与分支结构。

### 5. Prompt 不是 Checklist
Prompt 不承担正式检查制度，只能引用已有检查项完成当前一次检查动作。

### 6. Prompt 可引用上位件，但不承接其职责
Prompt 可以引用 Project Plan、SOP、Runbook、Checklist 的结果、约束或局部要求，但不替这些文档承担其制度职责。

一句话压缩：

> **Prompt 只负责这一次怎么发，不负责整个系统怎么跑、这轮怎么推进、这步平时怎么做。**

---

## 四、何时使用 Prompt

出现以下任一情况时，应使用 Prompt：

1. 当前需要发起一次明确执行  
2. 当前动作已有清晰输入与预期输出  
3. 当前动作需要明确边界，避免自由发挥过度  
4. 当前结果需要有可判断的输出格式或完成标准

---

## 五、何时不能只靠 Prompt

出现以下任一情况时，不应只靠 Prompt：

1. 任务需要多轮持续推进  
2. 任务需要定义阶段、责任、交付与验收  
3. 动作会高频重复，应沉淀为 SOP  
4. 场景复杂，需写 Runbook  
5. 需要门槛检查，需补 Checklist

---

## 六、编写原则

1. **只面向当前执行单元，不面向整个项目**  
2. **输入、动作、输出分开写**  
3. **边界与限制必须具体**  
4. **输出要求必须可判断**  
5. **信息不足时应允许显式说明卡点，不得硬答**  
6. **能短则短，能直则直，不把上位文档整本塞进 Prompt**

---

## 七、最小组成

一个最小可用 Prompt，至少应包含：

1. **任务目标**  
2. **输入材料**  
3. **执行动作**  
4. **输出要求**  
5. **边界 / 限制**  
6. **完成判断或失败处理**

少于这六块，Prompt 很容易退化成空泛指令。

## 九、一句话结论

Prompt 不是制度件，不是项目件，也不是动作规范件。

它只是：

> **把当前这一次调用，压成一个输入明确、边界清楚、输出可判定的调用壳。**
