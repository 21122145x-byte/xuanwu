# Project Plan 规范说明及模板卡
## v0.1

- **状态**：draft  
- **性质**：项目层规范 + 模板卡  
- **适用范围**：需要跨步骤推进、跨角色协作或需阶段性交付的正式任务  
- **上位关系**：受《团队协作宪法与治理总纲 v2.1》约束，运行机制服从 Program 层，不能与上位原则冲突  
- **下位关系**：可派生 SOP、执行单、检查单、Prompt、交接单、验收单等具体执行件

---

## 一、定位

Project Plan 不是宪章，不是 Program，也不是 SOP。

它的职责只有一个：

> **把某一个具体任务，压成一份可推进、可分阶段、可分责任、可验收的项目级作战图。**

如果再压一句：

> **宪章解决“按什么原则做”，Program 解决“系统怎么跑”，Project Plan 解决“这一轮具体打什么、怎么分段打、谁来打、打到什么算过”。**

---

## 二、Project Plan 解决什么问题

一个正式主任务进入执行时，通常会同时出现四类问题：

1. **这次到底做哪一件事，不做哪一件事？**  
2. **这次分几段推进，先后顺序是什么？**  
3. **每一段谁主责，谁支持，交付物是什么？**  
4. **做到什么程度算本轮完成，什么情况算待核 / 打回 / 暂停？**

这些问题，不应由宪章承担，也不应全部压给 SOP。

因此需要 Project Plan 这一层，专门承担：

- 本轮目标定义
- 本轮范围界定
- 本轮阶段拆解
- 责任分配
- 交付物设计
- 依赖与风险识别
- 阶段验收与整体验收口径

---

## 三、Project Plan 不解决什么问题

### 1. 不定义团队上位原则
那是宪章的职责。

### 2. 不定义协作闭环总机制
那是 Program 的职责。

### 3. 不承担逐步操作细节
那是 SOP / runbook / checklist 的职责。

### 4. 不直接替代 prompt
Prompt 只服务某一步的执行与调用，不负责项目级编排。

### 5. 不吞并实施记录与历史归档
施工日志、复盘、归档文档应下沉到对应对象中，不应混入 Plan 正文。

---

## 四、在 5 层结构中的位置

推荐层级如下：

1. **Constitution / Charter**：定义上位原则与边界  
2. **Program.md**：定义闭环机制、角色关系、流转与裁决结构  
3. **Project Plan**：定义本轮项目目标、阶段、责任、交付与验收  
4. **SOP / Runbook / Checklist**：定义某一环节怎么具体做  
5. **Prompt**：定义某一个执行单元如何向 Agent 发起调用

对应关系可以压成一句：

- **宪章**回答：什么不能错位  
- **Program**回答：系统怎么跑  
- **Plan**回答：这轮打什么  
- **SOP**回答：这步怎么做  
- **Prompt**回答：这一手怎么发出去

---

## 五、Project Plan 的最低成立条件

一份 Project Plan 至少应回答清楚以下问题：

1. **任务对象是什么**  
2. **本轮目标是什么**  
3. **本轮明确不做什么**  
4. **至少分为哪几个阶段**  
5. **每个阶段的交付物是什么**  
6. **每个阶段谁主责、谁支持**  
7. **阶段完成如何判断**  
8. **整体验收如何判断**  
9. **当前已知依赖和风险是什么**  
10. **如果中途阻塞，往哪里打回、升级或暂停**

如果这些问题大面积缺失，则该任务虽然可以有想法，但还不能算进入稳定项目态。

---

## 六、Project Plan 的编写原则

### 1. 面向“本轮任务”，不是面向永久制度
Project Plan 只服务当前对象、当前阶段、当前目标。

### 2. 面向推进，不面向展示
不是写得漂亮，而是让后续执行者能据此继续推进。

### 3. 目标、范围、阶段、责任、验收必须分开写
不要混成一坨叙述。

### 4. 能阶段化就不要一把梭
长链任务必须拆阶段，不允许所有内容都堆在一个“总目标”里。

### 5. 阶段必须绑定交付物
没有交付物的阶段，不算有效阶段。

### 6. 阶段必须绑定验收口径
没有验收口径的阶段，容易长期漂移。

### 7. 责任必须可定位
一个阶段可以多人参与，但主责必须唯一或至少清晰。

### 8. 风险不写满，但关键风险必须前置暴露
尤其是会影响方向、范围、验收与节奏的风险。

### 9. 不替下位件写细节
Plan 写到“做什么、谁做、做成什么样”，不要直接膨胀成 SOP。

### 10. 与上位件冲突时，以上位件为准
如 Plan 与宪章 / Program 冲突，应先修正 Plan，不得反向覆盖上位规则。

---

## 七、Project Plan 的推荐结构

一份标准 Project Plan，建议由以下九段组成：

### 1. 项目卡头
用于标识对象、状态、版本、负责人、关联上位件。

### 2. 任务定义
说明这次任务到底要解决什么问题。

### 3. 本轮目标与非目标
把“做什么 / 不做什么”同时写清。

### 4. 阶段拆解
把任务压成若干阶段，每一阶段有清晰产物。

### 5. 责任分配
标明主责、支持、审计 / 验收角色。

### 6. 交付物清单
列出本轮应产出的正式对象。

### 7. 验收标准
分别写阶段验收与整体放行标准。

### 8. 依赖与风险
列出关键依赖、关键阻塞点与已知风险。

### 9. 状态流转说明
说明当前处于什么状态，下一步进入什么状态，什么情况下打回 / 暂停 / 升级。

---

## 八、与其他文档的边界表

| 文档层 | 核心问题 | 不该承担什么 |
|---|---|---|
| Constitution / Charter | 原则、边界、裁决基准 | 不写项目细节 |
| Program.md | 闭环结构、协作机制、角色流转 | 不写某个项目的阶段计划 |
| Project Plan | 当前项目的目标、阶段、责任、交付、验收 | 不写逐步操作细节 |
| SOP / Runbook | 某一步具体怎么执行 | 不定义项目目标与整体分工 |
| Prompt | 某一执行单元如何发起 | 不承担项目编排与制度解释 |

---

## 九、适用触发条件

出现以下任一情况时，建议立 Project Plan：

1. 任务预计不是一次性完成，而是需要多步推进  
2. 任务涉及两个及以上阶段性交付  
3. 任务涉及多人或多 Agent 协作  
4. 任务存在明确依赖链或阻塞链  
5. 任务需要阶段性验收，而非一次性交付  
6. 任务周期会跨越单次对话窗口，需防止上下文漂移

如果只是一次性小任务、单步执行、无需阶段管理，可不立 Plan，直接由 SOP / Prompt 或即时执行承担。

## 十一、最小使用口径

如果要极简落地，Project Plan 至少保留以下六块：

1. 本轮目标  
2. 非目标  
3. 阶段拆解  
4. 责任分配  
5. 交付物  
6. 验收口径

少于这六块，Plan 很容易退化成普通说明文。

---

## 十二、常见失真方式

### 1. 把 Plan 写成愿景文案
只有方向，没有阶段和交付。

### 2. 把 Plan 写成 SOP
步骤很细，但缺目标、边界、责任和验收。

### 3. 把 Plan 写成施工日志
全是发生了什么，没有“接下来按什么结构推进”。

### 4. 把 Plan 写成任务堆积表
罗列事项很多，但没有阶段关系与优先级。

### 5. 把 Plan 写成制度件
把本轮特定问题泛化成永久规则，导致文档层级混乱。

### 6. 只有总目标，没有非目标
最后最容易范围漂移。

### 7. 只有执行人，没有验收人或验收口径
最后难以放行。

---

## 十三、结语

Project Plan 的价值，不在于把事情说得更完整，
而在于把一项长链任务压成：

- **知道现在在做什么**  
- **知道下一段要做什么**  
- **知道谁主责**  
- **知道做到什么算过**  
- **知道哪里可能卡住**

如果 Program 决定了“系统怎么跑”，
那么 Project Plan 决定的就是：

> **这一轮，系统到底在跑哪一条具体战线。**
