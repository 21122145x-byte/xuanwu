# 任务级 Program 文件规范卡
## 当前稳定入口（现由 v0.1 承接）

- **类型**：规范卡 / 配套卡
- **状态**：draft
- **稳定入口说明**：本文件为当前稳定名入口，现阶段由对应 v0.1 版本实体承接内容；后续升级优先更新本入口。
- **用途**：用于说明何时必须为任务建立独立 `program.md`，以及任务级 `program.md` 的最小结构与边界。
- **适用范围**：Project Plan 编写、复杂任务启动、SOP 分流、runtime pipeline 接入前置判断
- **上位依据**：`Program-团队协作运行主协议.md`

---

## 一、核心结论

团队级 `Program-团队协作运行主协议.md` 定义的是通用运行规则；
当某一任务需要被组织成**独立、持续、可裁决的运行闭环**时，应为该任务建立任务级 `program.md`。

换句话说：

> **主协议解决“Program 是什么”，任务级 `program.md` 解决“这个任务如何按 Program 跑起来”。**

---

## 二、何时必须具备独立 `program.md`

当任务满足以下任一条件时，原则上应建立独立 `program.md`：

1. **任务跨多轮推进**  
   不是单轮即可完成，而是需要持续运行。

2. **任务涉及多个 Agent / 多个角色 / 明确交接界面**  
   需要主任务位、执行位、复核位、审计位等明确协作关系。

3. **任务存在正式状态流转**  
   包含 review / rework / pause / done / blocked 等状态，而非一次性交付。

4. **任务需要 runtime pipeline 接入**  
   需要真实运行承载、日志追踪、状态恢复与持续推进能力。

5. **任务失败成本较高**  
   不能依赖临场口头协作，必须有稳定运行结构。

6. **任务需要审计、复盘或阶段性回看**  
   需要可追溯、可解释、可回收的运行记录。

---

## 三、何时可以不单独建立 `program.md`

以下任务通常可不单独建立任务级 `program.md`：

1. **单轮即可完成的轻量任务**  
2. **单执行者直接处理、无需正式交接的任务**  
3. **已可由 Project Plan、SOP 或 Prompt 直接承载的低复杂度动作**  
4. **不需要持续运行、状态流转与复核机制的简单执行**

注意：

> **不单独建立任务级 `program.md`，不等于不受 Program 约束。**

它只是表示：当前任务可直接服从团队级主协议，以轻模式运行。

---

## 四、任务级 `program.md` 的最小结构

一个合格的任务级 `program.md`，至少应包含以下八部分：

1. **任务目标**  
   - 本任务要解决什么问题

2. **进入条件**  
   - 任务在什么前提下进入正式闭环

3. **角色结构**  
   - 主任务位、执行位、复核位等如何挂接

4. **状态流转**  
   - 本任务采用何种状态机推进

5. **最小执行单元**  
   - 如何拆分、如何交接、如何串联

6. **交付与验证**  
   - 最小交付物是什么，如何验证，如何裁决

7. **暂停与恢复**  
   - 何时暂停，如何恢复，阻塞如何解除

8. **与 runtime 的关系**  
   - 哪些状态、日志、调度、恢复动作由 runtime pipeline 承载

---

## 五、边界说明

任务级 `program.md`` 的边界如下：

### 1. 不重写 Charter
不得重新定义团队上位原则、裁决口径与文档层级规则。

### 2. 不替代 Project Plan
不得承担本次任务的阶段排布、里程碑、节奏与阶段出口设计。

### 3. 不下沉为 SOP
不得写成逐步操作手册，不负责“动作具体怎么做”。

### 4. 不压缩成 Prompt
不得直接承担单轮执行指令功能，不负责当前回合输入输出的最小执行壳。

一句话说：

> **任务级 `program.md` 负责定义：这一个任务如何被组织成可持续运行的闭环。**

---

## 六、与其他层的关系

```text
Charter
  ↓ 定义原则与边界
Task-level program.md
  ↓ 定义本任务闭环
Project Plan
  ↓ 定义本次推进阶段与里程碑
SOP
  ↓ 定义动作标准做法
Prompt
  ↓ 定义单轮执行指令
runtime pipeline
  ↓ 承载真实运行
```

其中：
- `program.md` 解决“本任务怎么跑”
- `Project Plan` 解决“本任务这次怎么推进”
- `SOP` 解决“高频动作标准怎么做”
- `Prompt` 解决“这一轮具体怎么执行”

---

## 七、最短判断法

如果当前任务满足以下三个条件中的两个以上，就应优先考虑建立独立 `program.md`：

- **跨多轮**
- **多角色 / 多 Agent**
- **需要 review / pause / runtime 接入**

---

## 八、最短总结

> **独立 `program.md` 不是每个任务都要有，但凡任务需要独立运行闭环、正式状态流转、交接复核与持续推进，就应建立。**
