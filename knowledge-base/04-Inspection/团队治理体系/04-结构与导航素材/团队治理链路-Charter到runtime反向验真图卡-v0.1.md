# 团队治理链路图卡：Charter → Program → Plan → SOP → Prompt → runtime → 反向验真

- **类型**：结构图卡 / 治理链路说明卡
- **状态**：draft
- **用途**：用于说明团队治理体系不是单向写文档，而是“上位定义、下位派生、运行验真、问题反溯”的闭环系统。
- **适用范围**：团队治理设计、下位件规划、runtime pipeline 接入、宪章修订讨论

---

## 一、核心结论

团队治理体系不是一条单向文档链，而是一个双向闭环：

- **正向派生**：从上位原则一路下沉到真实运行
- **反向验真**：从真实运行暴露的问题一路回溯到上位原则

因此，`Constitution / Charter` 不是“写完即正确”，而是需要经过 `Program → Project Plan → SOP → Prompt → runtime pipeline` 的连续派生与反向校验，才能逐步被证真。

## 三、反向验真图

```text
runtime pipeline 出现断点 / 漂移 / 摩擦
                  │
                  ▼
         1. Prompt 是否压错？
            - 角色不清
            - 输出不清
            - 边界缺失
            - 回传方式不明

                  │ 若不是
                  ▼
         2. SOP 是否缺失或不够稳定？
            - 没有正式交接法
            - 没有裁决卡
            - 高频动作仍靠临场发挥

                  │ 若不是
                  ▼
         3. Project Plan 是否排错？
            - 阶段顺序不对
            - 分工挂法不清
            - 里程碑不成立
            - 本轮范围过大或过乱

                  │ 若不是
                  ▼
         4. Program 是否设计不足？
            - 状态流转不完整
            - 主责机制不稳
            - Review / Rework / Pause 口径不足
            - 交接协议不足

                  │ 若不是
                  ▼
         5. Charter 是否需要修订？
            - 原则过虚
            - 原则互相打架
            - 上位边界不清
            - 不停机原则与暂停原则冲突
            - 某些条文应上提 / 下沉 / 合并
```

## 四、每一层解决什么问题

### 1. Constitution / Charter
**角色**：团队上位法  
**解决的问题**：什么是合法协作、什么不能越界、什么叫完成、什么叫真实、什么叫可验证。

### 2. Program.md
**角色**：协作运行机制  
**解决的问题**：任务如何进入闭环、如何拆分、如何交接、如何复核、何时打回 / 暂停 / 放行。

### 3. Project Plan
**角色**：本次任务的阶段排布  
**解决的问题**：这次任务分几阶段、谁做什么、先后顺序是什么、里程碑是什么。

### 4. SOP
**角色**：标准动作手册  
**解决的问题**：高频动作的固定做法是什么，交接怎么写，裁决怎么做，归档怎么落。

### 5. Prompt
**角色**：单轮执行壳  
**解决的问题**：这一轮具体让哪个 Agent 做什么、输入是什么、输出是什么、如何交回。

### 6. runtime pipeline
**角色**：真实压力测试场  
**解决的问题**：整套治理结构在真实执行中是否稳定，是否存在断点、单点、漂移与制度摩擦。

---

## 五、这条链路真正表达的关系

这条链路不是简单的“文档清单”，而是两种方向同时成立：

### 正向派生
```text
Charter → Program → Plan → SOP → Prompt → runtime
```
含义：上位原则不断下沉，直到进入真实运行。

### 反向验真
```text
runtime → Prompt → SOP → Plan → Program → Charter
```
含义：真实运行暴露的问题不断回溯，最终校验上位制度是否真的成立。

---

## 六、对宪章正确性的含义

因此，一个 `Constitution / Charter` 是否真正正确，不能只看文本是否通顺、结构是否完整，还要看：

1. 它能否稳定派生出 `Program.md`  
2. 它能否被 `Project Plan` 实例化而不失真  
3. 它能否沉淀为可执行的 `SOP`  
4. 它能否被压缩为清晰的 `Prompt`  
5. 它能否在 `runtime pipeline` 中经受真实运行考验

若不能，则说明 Charter 仍处于“文本成型、系统待验”的阶段，而非最终正确版。

---

## 七、最短总结

> **Charter 负责定义原则，Program 负责定义闭环，Plan 负责定义本次战役，SOP 负责定义标准动作，Prompt 负责定义单轮执行，runtime 负责暴露真实问题；而真实问题最终会反向校验 Charter 是否真的成立。**
