# Program 目录最终结构建议
## v0.1

- **状态**：draft  
- **性质**：结构建议文档 / Program 目录收口建议  
- **适用范围**：`Program-团队协作运行主协议.md` 及其相关附件、历史件、归档件  
- **目标**：明确 Program 线的主文、附件、历史件与归档件最终应如何收口，避免 Program 周边继续长成副主链。  
- **形成依据**：基于 `Program-团队协作运行主协议.md`、Program 附件组终局裁决与当前团队治理体系目录结构给出的收口建议。

---

## 一、设计目标

Program 相关目录最终应满足以下目标：

1. **唯一主入口清楚**  
2. **正式附件数量极少**  
3. **说明附件不与主文争入口**  
4. **历史版本件可回看但不污染当前主链**  
5. **过渡澄清件有退场路径，不长期挂在主链周边**

---

## 二、最终法位结构建议

建议 Program 线最终收成以下 5 层对象：

```text
1. 主文（唯一正式入口）
2. 正式附件
3. 说明附件
4. 历史版本件
5. 归档 / 过渡澄清件
```

---

## 三、最终建议清单

### A. 主文（唯一正式入口）

#### 建议保留
- `Program-团队协作运行主协议.md`

#### 法位
- **唯一正式入口**  
- **当前生效 Program 主文**  
- 团队默认引用对象

#### 使用规则
- 后续所有下位件默认引用它  
- 任何版本升级，优先更新此文件内容  
- 不允许再让带版本号文件长期承担入口位

---

### B. 正式附件

#### 建议保留
- `任务级Program文件规范卡.md`  
  - 现阶段可先由 `任务级Program文件规范卡-v0.1.md` 过渡承载  
  - 后续建议稳定名化

#### 法位
- **Program 正式附件**  
- 负责回答：
  - 何时必须建立任务级 `program.md`  
  - 任务级 `program.md` 的最小结构与边界

#### 保留理由
这张卡承担的是 Program 层向任务级 Program 的正式下位规范，不是解释性材料。

#### 使用规则
- 正式附件数量尽量控制在 **1 张**  
- 没有特别充分理由，不再新增第二张正式附件

---

### C. 说明附件

#### 建议保留
- `Program与runtime-pipeline接口卡.md`  
  - 现阶段可先由 `Program与runtime-pipeline接口卡-v0.1.md` 过渡承载  
  - 后续建议稳定名化

#### 法位
- **Program 说明附件**  
- 负责回答：
  - Program 与 runtime pipeline 如何分工  
  - 两者之间交换什么  
  - runtime 如何承载 Program 的运行闭环

#### 保留理由
这类内容对 Program 有支撑价值，但不应并入主文，否则主文会再次厚化。

#### 使用规则
- 说明附件只保留与 Program 高耦合、且短期内确有使用价值的内容  
- 原则上控制在 **1 张**  
- 再新增前，必须先判断：
  - 能不能并回 runtime 体系文档  
  - 能不能直接删除

---

### D. 历史版本件

#### 建议保留
- `Program-团队协作运行主协议-v0.3-轻纠偏版.md`  
- `Program-团队协作运行主协议-v0.2-旧稳定入口迁出版.md`

#### 法位
- **历史版本件 / 版本实体**  
- 只承担：
  - 对照  
  - diff  
  - 复盘  
  - 回滚参考

#### 使用规则
- 历史版本件不得继续承担默认入口位  
- 历史版本件不得被下位件默认引用  
- 历史版本件应显式标明：
  - 版本号  
  - 状态（historical / migrated / archived）  
  - 与当前稳定入口的关系

---

### E. 归档 / 过渡澄清件

#### 当前建议放入这一层
- `Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1.md`

#### 法位
- **过渡术语澄清件**  
- 不再算 Program 正式附件  
- 不再算 Program 说明附件

#### 去向建议
后续二选一：

##### 方案 1：并入统一术语体系
若团队后续建设统一术语卡 / 术语手册：
- 把它迁过去  
- 退出 Program 目录核心区

##### 方案 2：直接归档
若后续 Program 主文和接口卡已经足够稳定：
- 把这张卡标记为 archived  
- 保留历史参考价值  
- 不再参与默认并读

#### 使用规则
- 这类文件不能长期卡在“也重要、也不是主链”的中间地带  
- 过渡澄清件必须有退场机制

## 四、推荐的最终目录心智模型

如果按“看一眼就知道怎么用”的方式，建议将 Program 相关对象理解为：

```text
Program/
├── Program-团队协作运行主协议.md              # 唯一正式入口
├── 任务级Program文件规范卡.md                 # 正式附件
├── Program与runtime-pipeline接口卡.md         # 说明附件
├── history/
│   ├── Program-团队协作运行主协议-v0.3-轻纠偏版.md
│   └── Program-团队协作运行主协议-v0.2-旧稳定入口迁出版.md
└── archive-or-terms/
    └── Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1.md
```

若当前暂不调整子目录结构，至少也应在文件法位说明中达到同样效果。

---

## 五、当前不搬目录时的最低要求

即使当前不新建 `history/` / `archive/` 子目录，最低也应满足以下 4 条：

1. 主文文件名稳定：  
   - `Program-团队协作运行主协议.md`

2. 附件文件名和法位明确：  
   - `任务级Program文件规范卡-v0.1.md` → 正式附件  
   - `Program与runtime-pipeline接口卡-v0.1.md` → 说明附件

3. 历史件明确标注历史角色：  
   - `Program-团队协作运行主协议-v0.3-轻纠偏版.md` → 历史版本件  
   - `Program-团队协作运行主协议-v0.2-旧稳定入口迁出版.md` → 迁出版 / 历史件

4. 术语澄清件退出核心附件组：  
   - `Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1.md` 不再挂在 Program 核心附件口径里

---

## 六、推荐的最终命名口径

为降低后续阅读负担，建议命名最终统一成两类：

### 1. 当前生效件：稳定名
- `Program-团队协作运行主协议.md`  
- `任务级Program文件规范卡.md`  
- `Program与runtime-pipeline接口卡.md`

### 2. 非当前生效件：带角色标识
例如：
- `Program-团队协作运行主协议-v0.3-轻纠偏版.md`  
- `Program-团队协作运行主协议-v0.2-旧稳定入口迁出版.md`  
- `Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1-术语澄清归档候选.md`

若暂时不改文件名，至少应在文内头部明确角色。

---

## 七、最终使用规则

为防止 Program 线再次回胖，建议固定以下使用规则：

### 规则 1：默认只读 1+1
正常使用时，默认只读：
- 主文  
- 必要时加 1 张附件

不要默认主文 + 全附件组并读。

### 规则 2：附件不能反向拖厚主文
如果附件内容开始频繁进入主文，应先判断：
- 是否主文过度吸收  
- 是否附件本身该退出

### 规则 3：术语件不是永久豁免对象
术语澄清件只在体系不稳定时有高价值；术语一旦稳定，就应并入统一术语体系或归档。

### 规则 4：历史件只服务回看，不服务当前默认执行
任何默认执行、引用、派生，都不应再指向历史版本件。

## 九、一句话结论

> **Program 最终应收成：一个主文、一个正式附件、一个说明附件；历史件回到历史位，术语件退出核心附件组。**
