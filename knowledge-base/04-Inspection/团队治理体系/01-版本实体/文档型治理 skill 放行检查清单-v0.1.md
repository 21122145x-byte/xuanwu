## 一、用途

本清单用于判断：

> 一个对象若被命名为 `skill`，但本质属于**文档型治理 skill**，是否已经具备进入团队试用阶段的最小放行条件。

本清单不用于：
- 判断程序型 skill 的代码质量
- 替代单次业务审计
- 替代正式收编裁决

---

## 二、放行检查清单

### A. 身份声明
- [ ] 已明确声明该对象是 **文档型治理 skill**，而非程序型 skill
- [ ] 未用模糊措辞暗示其天然应带 `run.py` / CLI / 独立执行器
- [ ] 身份声明可在 `SKILL.md`、`README.md` 或等效入口说明中直接找到

### B. 入口说明
- [ ] 已明确说明调用入口是什么
- [ ] 已明确说明由谁调用、按什么文件调用
- [ ] 若当前不提供 `run.py`，已明确写死“当前不提供程序执行入口”
- [ ] 至少存在一份稳定入口说明件（如 `README.md` / `references/call-pattern.md`）

### C. 有效运行产物
- [ ] 已明确什么算一次有效运行
- [ ] 已明确运行后至少沉淀哪些可复核产物
- [ ] 产物定义不以聊天结论替代正式回写
- [ ] 至少列出一类以上有效产物（如 scan record / execution record / governance judgment / cleanup writeback）

### D. 试用边界
- [ ] 已明确当前是“可试用”还是“已正式收编 / 已模板化”
- [ ] 已明确允许什么动作
- [ ] 已明确暂不放行什么动作
- [ ] 未将“可试用”偷换为“已稳定通用”

### E. 真实样本
- [ ] 已有至少一轮真实样本可复核
- [ ] 若曾被打回，已能对应到整改后复审结果
- [ ] 样本结果能支持“有条件通过”或“不通过”的明确裁决

---

## 三、放行判定规则

### 1. 不通过
出现以下任一情况，直接不通过：
- A / B / C / D 任一组存在关键缺项
- 无法判断什么算有效运行
- 无法判断当前边界
- 只有聊天说明，没有正式可复核产物

### 2. 有条件通过
满足以下条件时，可判有条件通过：
- A / B / C / D 四组均成立
- 已有最小真实样本
- 但样本量仍不足以支持直接模板化 / 正式收编

### 3. 可进入正式收编评估
满足以下条件时，可进入正式收编评估：
- 已经过多轮真实样本验证
- 输出质量稳定
- 边界未再反复漂移
- 已证明具有跨对象复用价值

---

## 四、默认审查顺序

默认按以下顺序检查：

1. 先审身份声明
2. 再审入口说明
3. 再审有效运行产物
4. 再审试用边界
5. 最后审真实样本

若前四步任一步不成立：
- 不进入试用放行
- 优先整改后复审

---

## 五、一句话结论

> 文档型治理 skill 的放行，不看它像不像程序包，而看它是否把“身份、入口、有效运行产物、试用边界、真实样本”五件事压清并留下可复核产物。
