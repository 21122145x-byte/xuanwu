# 轻治理线内容迁移表
## v0.1

- **类型**：迁移表 / 治理重构分流表
- **状态**：draft
- **用途**：用于将“轻治理线”中的内容按新治理体系重新分流，避免其继续作为独立主线存在。
- **适用范围**：团队治理体系重构、Program / Project Plan / SOP / Prompt / runtime pipeline 建设阶段

---

## 一、迁移目标

“轻治理线”不再作为独立治理主线继续存在。

其内容应按照新的团队治理链路重新拆分，并分别迁入以下层级：

- **Program**：承接跨任务通用的运行规则
- **Project Plan**：承接当前治理建设任务的阶段安排
- **SOP**：承接高频、可复用的标准动作
- **runtime pipeline**：承接真实运行中的状态推进与系统支撑
- **归档层**：保留历史过渡价值，不再进入现行主链

本迁移表的目的，不是删除旧内容，而是消除“轻治理线”作为并行入口与混合主线的地位。

---

## 二、迁移原则

### 原则 1：跨任务通用规则进 Program
凡属于任务进入、最小单元、主责机制、交接、复核、放行 / 打回 / 暂停等通用运行规则者，应迁入 Program。

### 原则 2：当前任务排布进 Project Plan
凡只属于本轮治理建设任务的阶段目标、里程碑、顺序与责任安排者，应迁入当前 Project Plan。

### 原则 3：高频动作模板进 SOP
凡可重复复用、具有标准动作形态的内容，应下沉为 SOP，而不继续滞留在主协议或项目计划中。

### 原则 4：真实运行支撑进 runtime pipeline
凡需要系统承载、状态推进、执行调度、断点恢复与不停机能力支撑的内容，不只写文档，还应进入 runtime pipeline。

### 原则 5：历史阶段说明留作归档
凡仅具过渡背景、历史解释、阶段性补丁与旧线路说明价值的内容，应归档保留，不再参与当前主链路裁决。

## 五、建议迁移顺序

### 第一步：先迁 Program
先把轻治理线中最稳定、最通用的闭环规则吸入 Program，避免主协议长期空心化。

### 第二步：再补 Project Plan
将“当前治理建设任务”的阶段目标与顺序安排放入 Project Plan，避免任务级内容污染 Program。

### 第三步：拆 SOP
把高频动作独立成 SOP / 执行卡，避免动作模板继续堆积在主协议中。

### 第四步：对接 runtime pipeline
将需要真实系统承载的状态推进与断点恢复能力迁入 runtime pipeline，避免“不停机”“可持续推进”停留在纸面。

### 第五步：最后归档轻治理线
待前四步完成后，再将旧轻治理线降级为历史归档，而不是继续作为当前入口与主线使用。

## 六、结论

“轻治理线”不应继续作为独立治理主线存在。

其内容应按以下方式拆散并重组：

- **通用闭环规则** → Program  
- **当前治理建设任务安排** → Project Plan  
- **高频标准动作** → SOP  
- **真实运行支撑** → runtime pipeline  
- **历史过渡内容** → 归档

换言之：

> **轻治理线应被拆散，而不是被保留。**
