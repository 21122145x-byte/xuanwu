## 一、当前问题

当前体系已经完成主链收口，但仍存在一类文件：

> **它们解决的是术语澄清与结构关系理解问题，而不是主链默认执行问题。**

这类文件若长期挂在主链或 Program 附件组周边，容易造成：
- 主链周边继续增厚  
- 附件组职责再次模糊  
- 阅读者把“术语解释”误当“默认执行入口”

因此，本轮整理的目标不是补更多术语，而是先回答：

> **术语件应该放哪里、什么时候退出、谁来承载。**

### 当前最明确的术语澄清件
- `Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1.md`

### 当前相关但不再按纯术语件理解的文件
- `Program与runtime-pipeline接口卡.md`

  - 侧重点是 Program 与 runtime pipeline 的接口分工，不等同于统一术语件

---

## 三、统一术语体系的最低承载原则

若后续建设统一术语卡 / 术语手册，建议至少遵循以下原则：

### 原则 1：术语体系不承担默认执行入口位
术语件回答“概念如何分工”，不回答“当前任务默认怎么执行”。

### 原则 2：术语体系不反向拖厚主链正文
主链正文只保留最小法位确认；更细术语解释放入统一术语体系，不反向灌回主文。

### 原则 3：术语体系优先集中承载跨层通用概念
例如：
- Skill  
- MCP  
- Harness  
- Runtime Pipeline  
- Program / Project Plan / SOP / Prompt 的最小边界关系（若后续仍需集中澄清）

### 原则 4：阶段性术语件必须有退场机制
术语澄清件不能长期以“暂时但一直存在”的状态挂在主目录核心区。

## 四、当前术语件去向判断

### 对 `Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1.md` 的判断

#### 当前法位
- **过渡术语澄清件**

#### 当前承载价值
- 仍有价值  
- 但价值主要在于“帮助系统语义收口”，而不是承担长期 Program 附件职责

#### 去向建议
当前建议采用：

### 路径 A（优先）
**后续并入统一术语卡 / 术语手册**

适用条件：
- 后续确实需要建设统一术语体系  
- 团队希望把跨层通用概念集中收纳

#### 路径 B（备选）
**若统一术语体系迟迟不建，则标记 archived，转入归档候选件**

适用条件：
- Program 主文和 Program 说明附件已足够稳定  
- 当前术语卡不再承担新决策输入职责

### 当前不建议路径
- 不建议回到 Program 核心附件组  
- 不建议并回 Program 主文  
- 不建议继续以“长期悬挂说明件”存在而无下一步安排

---

## 五、当前推荐的最小实施方案

在不新增大型术语手册的前提下，当前最稳的方案是：

### 方案 1（当前执行）
- 保留 `Skill-MCP-Runtime-Pipeline-Harness关系卡-v0.1.md`  
- 明确其法位为：过渡术语澄清件  
- 在目录法位表、总说明与文内头部都标清其角色  
- 暂不让它参与默认并读

### 方案 2（后续触发）
当满足以下任一条件时，再正式升级动作：
1. 团队决定建设统一术语卡 / 术语手册  
2. 五层主链进一步封板，需要清理主目录中所有过渡件

---

## 六、后续触发条件

统一术语体系的正式建设，建议以以下条件触发，而不是立即强行发起：

1. 跨层术语持续被重复解释  
2. 新成员或新 Agent 频繁在同一概念上混淆  
3. Program / runtime / capability 相关概念再次发生边界漂移  
4. 主目录即将封板，需要系统清退过渡件
