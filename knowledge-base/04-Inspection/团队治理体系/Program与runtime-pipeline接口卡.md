# Program 与 runtime pipeline 接口卡
## 当前稳定入口（现由 v0.1 承接）

- **类型**：接口卡 / 配套卡
- **状态**：draft
- **稳定入口说明**：本文件为当前稳定名入口，现阶段由对应 v0.1 版本实体承接内容；后续升级优先更新本入口。
- **用途**：用于说明 Program 与 runtime pipeline 之间的分工、接口、回传关系，以及 runtime 如何服务任务运行。
- **适用范围**：runtime pipeline 设计、任务接入 runtime、Project Plan 对接、SOP 与执行机制分流
- **上位依据**：`Program-团队协作运行主协议.md`

---

## 一、核心结论

Program 与 runtime pipeline 不是替代关系，而是：

> **Program 定义任务应如何运行；runtime pipeline 负责把这种运行真实承载起来。**

更准确地说：

- **Program** 是运行主协议
- **runtime pipeline** 是运行承载层与反向验真层

Program 负责定义“怎么跑”，runtime pipeline 负责让它真正跑起来，并暴露这套跑法在真实环境中的断点与摩擦。

---

## 二、两者分工

### 1. Program 负责什么
Program 负责定义：
- 任务进入条件
- 状态流转模型
- 角色结构
- 最小执行单元
- 正式交接规则
- 复核与裁决口径
- 暂停 / 恢复 / 阻塞的制度条件

### 2. runtime pipeline 负责什么
runtime pipeline 负责承载：
- 真实状态推进
- 执行单元派发与回收
- 运行日志与轨迹
- 阻塞记录与恢复点
- 持续推进与不停机支撑
- 对 Program 的反向压力测试

---

## 三、Program 向 runtime pipeline 提供什么

当任务接入 runtime pipeline 时，Program 至少应向运行层提供以下信息：

1. **任务标识**  
   - 任务 / Program 的唯一标识

2. **状态模型**  
   - 任务采用何种状态流转

3. **角色结构**  
   - 主任务位、执行位、复核位等如何挂接

4. **执行单元定义**  
   - 最小执行单元如何拆分、如何派发

5. **交接规则**  
   - handoff 的最小要求是什么

6. **裁决规则**  
   - 何时 pass / rework / pause / done

7. **阻塞口径**  
   - 什么情况记为 blocked，如何解除

---

## 四、runtime pipeline 向 Program 回传什么

runtime pipeline 至少应向 Program 回传以下信息：

1. **当前状态**  
   - 任务当前推进到哪里

2. **执行结果**  
   - 某个执行单元是否已完成、是否需返工

3. **日志与轨迹**  
   - 运行中发生了什么，是否形成最小可追溯记录

4. **阻塞原因**  
   - 卡点在哪里、缺什么、谁能解除

5. **恢复点**  
   - 从哪里可以继续接续任务

6. **摩擦信号**  
   - 哪些地方反复断点、失真、卡死或依赖单点人物

---

## 五、runtime pipeline 如何服务任务

runtime pipeline 对任务的服务，至少体现在以下五类能力上：

### 1. 承载状态流转
把 Program 中定义的状态真实记录下来，而不是只停留在文档里。

### 2. 承载持续推进
支持任务跨轮继续运行，避免任务每次都靠人工重新说明上下文。

### 3. 承载执行单元调度
支持最小执行单元的派发、回收、交接与再次进入 review。

### 4. 承载日志与追踪
为任务提供最小可追溯信息，支撑验证、审计与断点恢复。

### 5. 承载制度验真
暴露 Program 在真实运行中的断点、漂移与摩擦，作为反向修订 Program 与 Charter 的依据。

---

## 六、边界说明

### 1. Program 不负责实现 runtime 的技术细节
Program 只定义制度与接口，不承担系统实现本身。

### 2. runtime pipeline 不负责替 Program 发明规则
实现层不得静默改写状态机、交接口径与裁决标准。

### 3. 冲突必须显式回传
若 runtime 的现实约束与 Program 设计冲突，应显式回传并触发 Program 修订，而不是由实现层私自吞掉。

## 七、最小接口图

```text
Program
  ├─ 提供：状态模型
  ├─ 提供：角色结构
  ├─ 提供：执行单元定义
  ├─ 提供：交接与裁决规则
  ▼
runtime pipeline
  ├─ 承载：状态推进
  ├─ 承载：调度与回收
  ├─ 承载：日志与恢复点
  ├─ 回传：阻塞、结果、摩擦信号
  ▼
Program / Charter 修订
```

## 八、最短总结

> **Program 定义怎么跑，runtime pipeline 负责让它真正跑起来，并把真实运行中的问题反向回传给 Program。**
