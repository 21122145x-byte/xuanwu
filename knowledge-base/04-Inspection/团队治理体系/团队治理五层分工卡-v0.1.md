# 团队治理五层分工卡

- **类型**：结构图卡 / 分工说明卡
- **状态**：draft
- **用途**：用于说明团队治理体系中 `Constitution / Charter`、`Program.md`、`Project Plan`、`SOP`、`Prompt` 五层各自承担什么问题、处于什么层级、彼此如何衔接。
- **适用范围**：团队治理设计、文档分层、下位件建设、任务启动前对齐

---

## 一、核心结论

团队治理不是把所有规则都塞进一份文档，而是分成五层，各自承担不同问题：

1. **Constitution / Charter**：定义上位原则与边界  
2. **Program.md**：定义协作闭环与运行机制  
3. **Project Plan**：定义本次任务的阶段排布与推进顺序  
4. **SOP**：定义高频动作的标准做法  
5. **Prompt**：定义单轮执行的当前指令

五层关系不是并列，而是：

```text
Charter → Program → Project Plan → SOP → Prompt
```

层级越高，越稳定、越抽象、越少变化；  
层级越低，越具体、越临时、越贴近执行。

## 三、逐层定义

## 1. Constitution / Charter

### 角色
团队上位法。

### 负责回答
- 什么是合法协作
- 什么不能越界
- 什么叫真实
- 什么叫可验证
- 什么情况下放行、打回、暂停
- 文档层级怎么分

### 典型内容
- 协作模型
- 任务进入原则
- 真实性原则
- 最小必要与执行边界原则
- 交付与交接原则
- 验证与裁决原则
- 反膨胀原则
- 文档层级原则

### 不负责
- 不负责具体项目排期
- 不负责高频动作步骤
- 不负责某一轮的执行指令

### 一句话定位
**定义团队为什么这样协作。**

---

## 2. Program.md

### 角色
协作运行主协议。

### 负责回答
- 任务如何进入闭环
- 如何拆成最小执行单元
- 如何分配主责
- 如何交接
- 如何 review / rework / pause / done
- 如何持续推进并收口

### 典型内容
- 进入条件
- 状态流转
- 角色模型
- 最小执行单元规范
- 正式交接协议
- 复核与裁决机制
- 沉淀与回挂规则

### 不负责
- 不负责某个项目的具体阶段排期
- 不负责每个动作的逐步操作说明
- 不负责某一轮 prompt 内容

### 一句话定位
**定义团队怎么把任务跑成闭环。**

---

## 3. Project Plan

### 角色
当前任务 / 当前项目的战役计划。

### 负责回答
- 这次任务目标是什么
- 分几阶段推进
- 每阶段的里程碑是什么
- 谁主责、谁支持
- 哪些先做，哪些后做
- 本轮范围与非范围是什么

### 典型内容
- 任务背景
- 本轮目标
- 阶段拆分
- 里程碑
- 责任分配
- 节奏安排
- 风险点
- 阶段出口条件

### 不负责
- 不重写团队上位原则
- 不替代 Program 的状态机
- 不承担具体动作模板

### 一句话定位
**定义这一次具体怎么打。**

---

## 4. SOP

### 角色
标准动作手册。

### 负责回答
- 某类动作固定怎么做
- 正式交接怎么写
- 放行 / 打回 / 暂停怎么执行
- 归档 / 回挂怎么落
- 哪些检查项是固定要过的

### 典型内容
- 交接执行卡
- 裁决执行卡
- 角色边界执行卡
- 归档 / 回挂 SOP
- 审计检查表

### 不负责
- 不定义上位原则
- 不定义整个任务状态机
- 不定义某次项目的阶段计划

### 一句话定位
**定义高频动作标准怎么做。**

---

## 5. Prompt

### 角色
单轮执行指令。

### 负责回答
- 这一轮让哪个 Agent 做什么
- 当前输入是什么
- 输出格式是什么
- 本轮边界是什么
- 做完如何回传

### 典型内容
- 角色指定
- 当前目标
- 输入材料
- 输出要求
- 禁止事项
- 完成后的交回方式

### 不负责
- 不定义制度
- 不定义协作总闭环
- 不长期承担规则沉淀

### 一句话定位
**定义这一轮具体怎么执行。**

---

## 四、五层之间最容易混淆的边界

### 1. Charter 和 Program 的边界
- **Charter** 管原则、边界、裁决口径
- **Program** 管闭环机制、状态流转、角色协作

判断法：
如果一句话在回答“什么是合法协作”，更像 Charter；  
如果一句话在回答“任务怎么跑起来”，更像 Program。

### 2. Program 和 Project Plan 的边界
- **Program** 管通用运行机制
- **Project Plan** 管这次任务具体怎么排

判断法：
如果这条内容可以跨项目复用，更像 Program；  
如果它只属于当前任务，更像 Plan。

### 3. Project Plan 和 SOP 的边界
- **Plan** 管阶段、顺序、里程碑
- **SOP** 管动作做法

判断法：
如果是在回答“先做哪一步”，更像 Plan；  
如果是在回答“这一步标准怎么做”，更像 SOP。

### 4. SOP 和 Prompt 的边界
- **SOP** 管标准动作模板
- **Prompt** 管当前这一轮具体执行

判断法：
如果内容能反复复用，更像 SOP；  
如果内容只服务这一轮，更像 Prompt。

---

## 五、五层的稳定度与变动频率

### 最高稳定层
**Charter**  
- 最少改
- 改一次影响最大
- 必须慎重

### 中稳定层
**Program / SOP**  
- 会随着运行经验调整
- 但不应每轮都变

### 高变动层
**Project Plan / Prompt**  
- 随任务变化而变化
- 生命周期更短
- 允许快速迭代

---

## 六、最短判断法

如果当前写一份文档时，不知道该放哪层，可以用下面这组问题判断：

1. **这是在定义原则，还是定义动作？**  
   - 原则 → Charter
   - 动作 → 往下看

2. **这是在定义通用闭环，还是当前项目？**  
   - 通用闭环 → Program
   - 当前项目 → 往下看

3. **这是在定义阶段排布，还是定义标准做法？**  
   - 阶段排布 → Project Plan
   - 标准做法 → 往下看

4. **这是可复用动作模板，还是当前这一轮指令？**  
   - 可复用 → SOP
   - 当前这一轮 → Prompt

---

## 七、最短总结

> **Charter 定义原则，Program 定义闭环，Plan 定义战役，SOP 定义动作，Prompt 定义当前回合。**

如果五层不分，团队会出现三种典型问题：
- 上位法过胖
- 下位件互相抢位
- 运行中不断靠临场补口

五层分清，治理体系才可能既稳定又能跑。
