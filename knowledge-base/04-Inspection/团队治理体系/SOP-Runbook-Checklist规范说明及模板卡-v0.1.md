# SOP / Runbook / Checklist 规范说明及模板卡
## v0.1

- **状态**：draft  
- **性质**：执行层规范 + 模板卡  
- **适用范围**：需要把某类重复动作、标准做法、处置路径、执行检查点压成可重复执行件的任务场景  
- **上位依据**：
  - `团队协作宪法与治理总纲-v2.1-减法收口版.md`
  - `Program-团队协作运行主协议.md`
  - `Project-Plan规范说明及模板卡-v0.2.md`
- **下位关系**：可派生具体 SOP、具体 runbook、具体 checklist、对应 prompt、执行记录、验收记录等  
- **定位说明**：本文件用于明确 SOP / Runbook / Checklist 在五层结构中的职责边界、使用时机、彼此区别与模板结构，防止这一层内部混写。

---

## 一、定位

在五层结构中：

```text
Charter
  ↓ 定义原则与边界
Program
  ↓ 定义闭环结构与运行机制
Project Plan
  ↓ 定义本轮推进目标、阶段、责任、交付与验收
SOP / Runbook / Checklist
  ↓ 定义具体动作标准做法、处置路径与检查点
Prompt
  ↓ 定义单轮执行指令
```

这一层的职责，不是回答“这一轮做什么”，而是回答：

> **当某一步或某类动作需要被稳定、重复、低失真地执行时，具体应该怎么做、怎么检查、怎么在特定场景里处置。**

一句话区分：

- **Project Plan** 解决：这轮打什么  
- **SOP / Runbook / Checklist** 解决：这一步怎么稳地做  
- **Prompt** 解决：这一手怎么发出去

---

## 二、这一层解决什么问题

当团队已经知道：
- 本轮目标是什么
- 阶段怎么分
- 谁主责
- 交付物是什么

但仍然容易在执行中出现以下问题时，就需要这一层：

1. **同类动作每次做法都不一样**  
2. **步骤顺序不稳定，容易漏项**  
3. **同一动作在特定场景下有固定处置路径**  
4. **执行者知道目标，但不知道标准做法**  
5. **交付前需要明确检查项，否则质量容易漂**

因此，这一层本质上解决的是：

> **把动作级经验，压成可重复、可检查、可交接、可训练的执行规范。**

## 三、三者分别是什么

### 3.1 SOP 是什么
SOP（Standard Operating Procedure）用于定义：

> **某类高频、相对稳定、应尽量标准化的动作，默认按什么步骤执行。**

SOP 重点回答：
- 这类动作通常怎么做
- 顺序是什么
- 每一步产出什么
- 哪些边界不能越
- 什么情况下不适用

SOP 适合：
- 高频重复动作
- 稳定流程
- 默认标准做法
- 新人 / 新 Agent 也要能照着执行的动作

---

### 3.2 Runbook 是什么
Runbook 用于定义：

> **当处在某个具体运行场景、异常场景或操作场景时，应该按什么路径处置。**

Runbook 重点回答：
- 当前场景是什么
- 遇到这种情况先看什么
- 先做哪一步，再做哪一步
- 分支判断是什么
- 什么情况下升级、暂停、回退或恢复

Runbook 适合：
- 故障处理
- 应急响应
- 值班处置
- 场景化操作
- 恢复 / 回滚 / 排查 / 切换

---

### 3.3 Checklist 是什么
Checklist 用于定义：

> **在执行前、执行中或交付前，必须逐项检查什么。**

Checklist 重点回答：
- 哪些项必须检查
- 每项通过标准是什么
- 不通过时不能进入下一步还是必须打回

Checklist 适合：
- 交付前检查
- 发布前检查
- 正式交接前检查
- 安全检查
- 审核 / 验收 / 收口检查

---

## 四、三者的区别

| 类型 | 核心问题 | 最适合的场景 | 典型结构 |
|---|---|---|---|
| SOP | 默认应该怎么做 | 高频、稳定、重复性动作 | 步骤式标准流程 |
| Runbook | 遇到这个场景怎么处置 | 故障、异常、特定运行情境 | 场景 + 判断分支 + 处置路径 |
| Checklist | 哪些项必须检查 | 交付、切换、发布、验收、交接 | 勾选项 + 通过标准 |

一句话压缩：

- **SOP**：标准做法  
- **Runbook**：场景处置  
- **Checklist**：检查门槛

## 五、与其他层的边界

### 5.1 与 Program 的边界
Program 负责闭环结构、角色关系、状态流转、交接与裁决机制。  
SOP / Runbook / Checklist 不重写这些上位规则。

换句话说：
- Program 说“什么时候进入 Review / Rework / Pause”
- 这一层说“进入某一步后具体怎么做、怎么查、怎么处置”

### 5.2 与 Project Plan 的边界
Project Plan 负责本轮目标、阶段、责任、交付、验收。  
SOP / Runbook / Checklist 不负责定义“这一轮到底做什么”。

换句话说：
- Plan 说“Phase 2 要完成验证”
- SOP / Runbook / Checklist 说“验证这件事具体怎么做”

### 5.3 与 Prompt 的边界
Prompt 负责单轮调用与当前回合输入输出。  
SOP / Runbook / Checklist 不应直接压缩成一句 prompt，也不应把 prompt 写成完整 SOP。

换句话说：
- SOP / Runbook / Checklist 是**动作说明件**
- Prompt 是**调用壳**

## 六、何时应该写 SOP / Runbook / Checklist

### 6.1 应写 SOP 的情况
满足以下任一情况时，建议写 SOP：

1. 同类动作会反复出现  
2. 存在较稳定的默认顺序  
3. 不按标准做法容易反复返工  
4. 需要降低执行者差异带来的失真  
5. 希望把经验沉淀成可复用动作规范

### 6.2 应写 Runbook 的情况
满足以下任一情况时，建议写 Runbook：

1. 动作强依赖具体场景  
2. 存在故障、异常、告警、阻塞、恢复等处置路径  
3. 同一目标下会因为不同症状走不同分支  
4. 需要明确升级、回退、暂停、恢复的路径

### 6.3 应写 Checklist 的情况
满足以下任一情况时，建议写 Checklist：

1. 执行前有必须确认的门槛  
2. 交付前有必须逐项验收的项  
3. 漏掉任一项都会显著增加错误成本  
4. 需要用最短结构做质量门控

---

## 七、何时不必单独写这一层文档

以下情况通常不必单独立 SOP / Runbook / Checklist：

1. 动作只发生一次，且无需复用  
2. 动作极简单，Prompt 已足够承载  
3. 步骤高度依赖即时判断，暂时还未形成稳定做法  
4. 任务还停留在目标未清、边界未清阶段

注意：

> **不单独写这一层文档，不等于动作没有规范；只是说明当前还没到值得制度化沉淀的程度。**

---

## 八、编写原则

### 1. 面向动作，不面向项目总目标

### 2. 面向稳定复用，不面向一次性表演

### 3. 只写最小必要步骤
避免把简单动作写成膨胀文件。

### 4. 必须写清适用范围与不适用范围
否则下游容易误用。

### 5. 步骤必须能执行，不写空话
避免“注意质量”“保持谨慎”这类不可执行表述单独充当步骤。

### 6. 检查项必须可判断
Checklist 中每一项都应能判断通过 / 不通过。

### 7. 场景处置必须给分支
Runbook 不能只写“出了问题及时处理”。

### 8. 与上位件冲突时，以上位件为准

## 十二、如何选用

### 12.1 只需要一种时
- 有稳定步骤 → 优先用 **SOP**
- 有场景分支 → 优先用 **Runbook**
- 只需要门槛检查 → 优先用 **Checklist**

### 12.2 需要组合使用时
常见组合如下：

1. **SOP + Checklist**
   - 先按标准步骤执行，再按检查表验收

2. **Runbook + Checklist**
   - 先按场景处置，再用检查表确认是否恢复 / 是否可放行

3. **Project Plan + SOP + Prompt**
   - Plan 定义阶段目标
   - SOP 定义动作标准做法
   - Prompt 承担单轮执行调用

4. **Project Plan + Runbook**
   - Plan 定义推进结构
   - Runbook 处理过程中出现的特定运行场景或故障场景

## 十三、常见失真方式

### 1. 把 SOP 写成项目计划
开始讲背景、阶段、里程碑，却不讲动作步骤。

### 2. 把 Runbook 写成情绪口号
只有“保持冷静”“尽快排查”，没有分支和处置路径。

### 3. 把 Checklist 写成愿望清单
每项都很正确，但没有通过标准和不通过处理。

### 4. 把 Prompt 直接当 SOP
导致 prompt 过长、过重、难复用，也无法稳定训练执行习惯。

### 5. 把 SOP 写成制度件
把动作规范上浮成团队原则，造成层级污染。

### 6. 三者混在一份文档里不分层
最后既没有清楚步骤，也没有清楚场景，也没有清楚检查门槛。

---

## 十四、一句话结论

这一层的职责不是定义“项目怎么跑”，而是定义：

> **当项目已经知道要做什么时，某一步具体如何稳定执行、如何在特定场景处置、如何在关键节点做质量门控。**

因此：

- **SOP** = 标准做法  
- **Runbook** = 场景处置  
- **Checklist** = 检查门槛
