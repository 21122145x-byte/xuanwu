## 一、定位

在五层结构中：

```text
Charter
  ↓ 定义原则与边界
Program
  ↓ 定义闭环结构与运行机制
Project Plan
  ↓ 定义本轮推进目标、阶段、责任、交付与验收
SOP / Runbook / Checklist
  ↓ 定义动作标准做法、处置路径与检查门槛
Prompt
  ↓ 定义单轮执行指令
```

Prompt 的职责不是定义系统，不是定义项目，也不是定义动作规范。

Prompt 只承担一个职责：

> **把某一轮、某一步、某个最小执行单元，压成一个可立即调用、可直接执行、可判断输出是否合格的执行壳。**

一句话区分：

- **Program**：系统怎么跑  
- **Project Plan**：这轮打什么  
- **SOP / Runbook / Checklist**：这一步平时怎么做 / 遇事怎么处置 / 到点检查什么  
- **Prompt**：这一轮具体怎么发出去

---

## 二、Prompt 解决什么问题

当团队已经知道：
- 当前目标是什么
- 当前动作属于哪个阶段
- 当前动作应按什么标准做

但还需要把一次具体执行发出去时，Prompt 负责解决：

1. **这一轮到底要执行什么**  
2. **输入材料有哪些**  
3. **输出物应该长什么样**  
4. **本轮边界和限制是什么**  
5. **本轮成功 / 失败如何判断**

因此，Prompt 的本质不是“对模型说点什么”，而是：

> **当前执行单元的最小调用协议。**

---

## 三、Prompt 不解决什么问题

### 1. 不定义上位原则
那是 Charter 的职责。

### 2. 不定义闭环机制与状态流转
那是 Program 的职责。

### 3. 不定义本轮项目目标、阶段与责任
那是 Project Plan 的职责。

### 4. 不承担稳定动作规范沉淀
那是 SOP / Runbook / Checklist 的职责。

### 5. 不替代施工日志、验收记录与归档
Prompt 用于发起执行，不承担历史事实沉淀。

一句话说：

> **Prompt 只负责把一次执行发准，不负责把整个系统讲完。**

---

## 四、何时应该写 Prompt

满足以下任一情况时，应使用 Prompt：

1. 当前需要发起一次明确执行  
2. 当前动作已有清晰输入与预期输出  
3. 当前执行需要明确约束，避免自由发挥过度  
4. 当前动作需要在多次调用中保持相对稳定的输出结构  
5. 当前动作已属于 Project Plan / SOP / Runbook / Checklist 下的具体执行一步

### 4.1 何时只靠 Prompt 即可
以下情况通常可以只用 Prompt：
- 单轮即可完成  
- 动作简单  
- 无需单独写 SOP 也能稳定复现  
- 不涉及复杂协作、正式交接与阶段推进

### 4.2 何时不能只靠 Prompt
以下情况不应只靠 Prompt：
- 任务需要多轮持续推进  
- 任务需要明确阶段、责任、交付与验收  
- 动作会高频重复，值得沉淀为 SOP  
- 场景复杂，需写 Runbook 分支处置  
- 需要门槛检查，需补 Checklist

## 五、与其他层的边界

### 5.1 与 Program 的边界
Program 负责运行闭环。Prompt 不定义状态机、交接制度、角色关系与裁决结构。

换句话说：
- Program 说“任务怎么跑”
- Prompt 说“这一轮怎么发”

### 5.2 与 Project Plan 的边界
Project Plan 负责本轮目标、阶段、责任、交付、验收。Prompt 不定义本轮阶段策略。

换句话说：
- Plan 说“Phase 2 要完成验证”
- Prompt 说“现在这一轮请按这些输入，完成这次验证动作”

### 5.3 与 SOP 的边界
SOP 负责标准做法；Prompt 负责一次调用。

换句话说：
- SOP 是**动作说明件**
- Prompt 是**动作发起件**

### 5.4 与 Runbook 的边界
Runbook 负责场景化处置路径；Prompt 只负责当前分支下的一次执行。

### 5.5 与 Checklist 的边界
Checklist 负责检查门槛；Prompt 可以要求“按某个 Checklist 检查”，但不应把完整检查制度直接混成一段随手 prompt。

## 六、Prompt 的最小组成

一个合格 Prompt，至少应包含以下六块：

1. **任务目标**：这一轮要完成什么  
2. **输入材料**：当前可用材料是什么  
3. **执行要求**：应按什么边界处理  
4. **输出要求**：要交付什么格式 / 结构  
5. **限制条件**：不能做什么  
6. **完成判断**：什么样的输出才算合格

如果这六块大面积缺失，Prompt 很容易退化为：
- 空泛指令
- 情绪性命令
- 过长制度堆砌
- 高自由度漂移输入

---

## 七、Prompt 的编写原则

### 1. 面向当前执行单元，不面向整个项目

### 2. 只给当前轮真正需要的信息
不要把上位文档整本塞进 Prompt。

### 3. 输入、动作、输出必须分开写
避免混成一段大白话。

### 4. 约束必须具体
少写“认真一点”“尽量更好”，多写可执行边界。

### 5. 输出要求必须可判断
最好明确格式、结构、长度、字段或完成标准。

### 6. 不把 Prompt 写成 SOP
步骤很多、适用范围很广、讲长期默认做法时，应上升为 SOP。

### 7. 不把 Prompt 写成 Project Plan
涉及阶段、责任、交付链与验收结构时，应交给 Plan。

### 8. 不把 Prompt 写成制度文本
原则、治理口径、团队规则应留在上位件。

### 9. 能短则短，能直则直
Prompt 首先是调用壳，不是文章。

### 10. 失败条件最好显式化
尤其在易漂移动作里，明确“不满足什么就不要宣称完成”。

---

## 八、Prompt 的推荐结构

推荐采用以下八段：

1. **任务说明**：当前轮做什么  
2. **上下文输入**：可用材料与背景  
3. **执行动作**：要执行的具体动作  
4. **输出要求**：最终输出格式  
5. **边界与限制**：不允许越界的点  
6. **质量标准**：怎样算合格  
7. **失败 / 不确定处理**：何时应说明卡点、停止硬答  
8. **交付方式**：结果如何返回 / 放到哪里

## 十、几种常见 Prompt 类型

### 10.1 执行型 Prompt
用于直接完成某个动作。

适合：
- 写一个摘要
- 清洗一段文本
- 产出一个结构化表格
- 根据现有规范完成一步动作

### 10.2 提取型 Prompt
用于从输入材料中提取关键信息。

适合：
- 提取结论
- 提取字段
- 提取问题清单
- 提取结构化事实

### 10.3 转写型 Prompt
用于把一种结构转成另一种结构。

适合：
- 会议记录转任务清单
- 自由文本转模板字段
- 文档转表格

### 10.4 审核型 Prompt
用于按既有标准检查输出。

适合：
- 按 Checklist 审核
- 对照 SOP 查漏项
- 对照 Plan 查是否越界

### 10.5 交接型 Prompt
用于把已有结果整理成可交给下一责任方的结构。

适合：
- 整理 Where / What / How to verify / Known Issues / Next Steps
- 生成正式交接摘要

## 十一、Prompt 与 SOP / Plan 的挂接方式

### 11.1 从 SOP 派生 Prompt
当某一 SOP 中的某一步需要实际调用执行时，可从该步骤派生 Prompt。

关系应是：
- SOP 定义默认做法
- Prompt 取其中当前所需的那一步，压成一次调用

### 11.2 从 Project Plan 派生 Prompt
当 Project Plan 中某一阶段需要实际执行时，可从当前 Phase 派生 Prompt。

关系应是：
- Plan 定义当前阶段目标、责任、交付
- Prompt 承担该阶段中的一个执行单元

### 11.3 从 Runbook 派生 Prompt
当 Runbook 已确定当前走哪条分支后，可为该分支中的动作生成 Prompt。

### 11.4 从 Checklist 派生 Prompt
当需要把检查动作本身交给执行者完成时，可把 Checklist 中的检查要求压成审核型 Prompt。

---

## 十二、交付与可验证性要求

Prompt 本身不是交付物，但 Prompt 产出的结果必须可验证。

至少应尽量明确：
- 输出放在哪里  
- 由谁接收  
- 如何判断是否合格  
- 若不合格进入什么状态（如 Rework / Blocked / Pause）

最小原则：

> **没有输出标准的 Prompt，通常不是好 Prompt；没有失败处理的 Prompt，通常不是稳 Prompt。**

---

## 十三、常见失真方式

### 1. 把 Prompt 写成一段口号
只有“请认真完成以下任务”，没有输入、输出、限制和标准。

### 2. 把 Prompt 写成 Program
开始讲角色模型、状态机、长期运行机制，导致过重。

### 3. 把 Prompt 写成 Project Plan
开始讲阶段、分工、里程碑、验收链路，导致混层。

### 4. 把 Prompt 写成 SOP
把长期默认步骤、适用范围、不适用范围全塞进一次调用里。

### 5. 输入材料不清
执行者不知道该依据什么做判断。

### 6. 输出标准不清
最后只能“看感觉”。

### 7. 限制条件不清
导致执行者自由发挥过度，越界输出。

### 8. 没有失败处理
信息不足时仍硬答，最容易失真。

---

## 十四、最短判断法

如果当前你要解决的问题是：

- “系统怎么跑” → 去找 **Program**  
- “这轮打什么” → 去找 **Project Plan**  
- “这步平时怎么做” → 去找 **SOP**  
- “遇到这个场景怎么处置” → 去找 **Runbook**  
- “到点查什么” → 去找 **Checklist**  
- “这一轮具体怎么发出去” → 写 **Prompt**

---

## 十五、一句话结论

Prompt 不是制度，不是项目，不是流程，不是经验库。

它只是：

> **把当前这一轮执行，压成一个边界清楚、输入明确、输出可判定的调用壳。**
