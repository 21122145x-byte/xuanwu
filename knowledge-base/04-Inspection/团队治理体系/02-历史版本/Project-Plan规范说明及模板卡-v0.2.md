# Project Plan 规范说明及模板卡
## v0.2

- **当前文件角色**：历史版本件
- **使用限制**：本文件仅用于审计、diff、回滚与复盘，不再作为默认引用入口。

- **状态**：draft  
- **性质**：项目层规范 + 模板卡  
- **适用范围**：需要跨步骤推进、跨角色协作、阶段性交付、正式状态流转或需防止长链漂移的任务  
- **上位依据**：
  - `团队协作宪法与治理总纲-v2.1-减法收口版.md`
  - `Program-团队协作运行主协议.md`
  - `任务级Program文件规范卡-v0.1.md`
- **下位关系**：可派生 SOP、runbook、checklist、prompt、交接单、验收单、施工日志等执行件  
- **定位说明**：本文件定义“某一具体任务在当前周期内如何推进”，不定义团队上位原则，不替代 Program 主协议，也不下沉为逐步操作说明。

---

## 一、定位

Project Plan 不是宪章，不是 Program，也不是 SOP。

它只承担一个职责：

> **把一个具体任务压成当前周期内可进入、可推进、可分段、可交付、可裁决的项目级推进结构。**

在五层结构中：

```text
Charter
  ↓ 定义原则与边界
Program
  ↓ 定义闭环结构与运行机制
Project Plan
  ↓ 定义本轮推进目标、阶段、责任、交付与验收
SOP / Runbook / Checklist
  ↓ 定义具体动作标准做法
Prompt
  ↓ 定义单轮执行指令
```

一句话区分：

- **Charter**：什么不能错位  
- **Program**：系统怎么跑  
- **Project Plan**：这一轮具体打什么、怎么分段打、谁来打、打到什么算过  
- **SOP**：某一步具体怎么做  
- **Prompt**：这一手具体怎么发出去

---

## 二、Project Plan 解决什么问题

当一个正式任务已经具备基本进入条件后，Project Plan 负责把以下内容压清：

1. **本轮到底做什么，不做什么**  
2. **本轮分几段推进，先后顺序是什么**  
3. **每一阶段的主责、支持、复核是谁**  
4. **每一阶段交付什么、放在哪里、如何验证**  
5. **阶段出口是什么，何时交接，交给谁**  
6. **什么情况放行、打回、暂停、阻塞或失败**  
7. **本轮发生范围变化时，如何回写与裁决**

因此，Project Plan 的本质不是“项目介绍”，而是：

> **当前周期的项目级推进与裁决框架。**

---

## 三、Project Plan 不解决什么问题

### 1. 不重写团队上位原则
那是 Charter 的职责。

### 2. 不定义团队级闭环机制
那是 Program 主协议的职责。

### 3. 不替代任务级 `program.md`
如任务需要独立运行闭环、正式状态流转与 runtime 接入，应由任务级 `program.md` 承担该层职责。

### 4. 不直接下沉为 SOP
Project Plan 只定义“做什么、谁做、做到什么算过”，不写动作级步骤细节。

### 5. 不承担单轮 prompt 功能
Prompt 只服务单次执行单元调用，不承担项目推进编排。

### 6. 不吞并施工日志 / 归档 / 复盘
过程记录、失败复盘、历史归档应下沉到相应文档，不应把 Plan 写成流水账。

---

## 四、何时必须建立 Project Plan

出现以下任一情况时，原则上应建立 Project Plan：

1. 任务不是单步完成，而是需要跨多轮推进  
2. 任务至少包含两个及以上阶段性交付  
3. 任务涉及多人或多 Agent 协作  
4. 任务存在依赖链、阻塞链或阶段性验收点  
5. 任务周期会跨越单次对话窗口，存在上下文漂移风险  
6. 任务需要明确“本轮目标 / 非目标 / 阶段出口 / 放行标准”  
7. 任务虽不一定需要独立 `program.md`，但已经不能只靠 SOP 或 Prompt 承载

如果只是单轮、单步、单执行者的轻量动作，可不立 Project Plan，直接由 SOP / Prompt 或即时执行承担。

---

## 五、Project Plan 的进入条件

Project Plan 不是想到就写的说明文；要进入正式项目态，至少应满足以下条件：

1. **任务对象可指认**：当前对象是什么，不能模糊  
2. **本轮目标可表述**：这一轮要解决什么问题  
3. **本轮非目标可区分**：明确哪些内容不并入本轮  
4. **主责任主体可识别**：至少能明确项目主责  
5. **最小交付物可指认**：本轮至少会产出什么  
6. **存在基本裁决口径**：完成 / 待核 / 打回 / 暂停 / 阻塞 等判断方式可说明

若上述条件缺失到足以影响推进，Project Plan 可以停留在 `Draft / Scoped`，但不得直接宣称进入稳定执行态。

## 六、Project Plan 与 Program 的关系

### 6.1 默认关系
Project Plan 默认受团队级 Program 主协议约束。

如果任务**不单独建立任务级 `program.md`**，则该 Plan 默认继承：
- 团队级状态流转逻辑
- 正式交接字段
- 放行 / 打回 / 暂停 / 阻塞的基本裁决口径
- 最小执行单元的基本规范

### 6.2 何时仅有 Plan 即可
以下任务通常可仅有 Project Plan，而不另建任务级 `program.md`：

1. 本轮需要阶段化推进，但不需要独立运行闭环  
2. 协作关系有限，不需要自定义状态机  
3. 无需 runtime pipeline 深度接入  
4. 团队级 Program 已足够覆盖其运行机制

### 6.3 何时必须同时具备 Plan + 任务级 `program.md`
当任务满足以下任一条件时，原则上应同时具备：
- 任务需独立、持续、可恢复地运行  
- 任务有正式状态机、review / rework / pause 机制  
- 任务需 runtime pipeline 承载  
- 任务失败成本高，不能只靠项目计划维持秩序  
- 多角色交接结构较复杂，需为该任务单独定义闭环规则

一句话说：

> **Program 解决“这个任务如何被组织成运行闭环”，Plan 解决“这个任务这次怎么推进”。**

## 七、Project Plan 生命周期与状态映射

Project Plan 不是静态文档，而是项目推进状态的承载件之一。

推荐采用以下状态：

- **Draft**：已有任务意向，但进入条件尚未压清  
- **Scoped**：目标、范围、非目标、主责、最小交付物已基本压清  
- **Active**：阶段已排布，责任已分配，当前可正式推进  
- **Review**：当前轮产物进入复核  
- **Rework**：当前轮被打回，进入最小修复  
- **Blocked**：因依赖、权限、输入、接口等问题卡住  
- **Pause**：当前不宜继续推进，但不宣告失败  
- **Failed**：本轮推进在当前周期内不可继续，或已失去推进价值  
- **Done**：本轮交付已通过验证，并形成可读取结果  
- **Archived**：已退出主链，仅保留历史事实

### 7.1 与 Program 状态的对应原则
- Project Plan 应尽量与 Program 主协议状态兼容，而不是自造另一套状态体系  
- 如 Plan 使用更简化状态，必须能映射回 Program 状态  
- 如任务级 `program.md` 自定义了更细状态，Plan 应至少反映阶段性主状态

## 八、Project Plan 的核心编写原则

### 1. 面向“本轮推进”，不是面向永久制度

### 2. 面向可继续执行，不面向展示
后续执行者应能据此接住，而不是只看懂背景。

### 3. 目标、非目标、范围、阶段、责任、验收必须分开写
不得混成一段叙述。

### 4. 阶段必须绑定交付物、位置与验证方式
没有这三项，阶段容易失真。

### 5. 阶段出口必须明确
阶段结束时，要么进入下一阶段，要么进入 Review / Blocked / Pause / Rework，不得悬空。

### 6. 责任必须可定位
一个阶段可以多人参与，但主责、复核、裁决必须能明确指认。

### 7. 关键风险必须前置暴露
尤其是会影响方向、范围、验收、节奏与外部依赖的风险。

### 8. Plan 不替下位件写动作细节
Plan 写到“做什么、谁做、做到什么算过”为止。

### 9. 变化不能只留在聊天里
凡影响目标、范围、阶段、交付、验收或责任的变化，必须回写 Plan。

### 10. 与上位件冲突时，以上位件为准
Plan 不能反向覆盖 Charter 或 Program。

## 九、Phase 与 Work Unit 的关系

Project Plan 默认管理的是**阶段（Phase）层**，而非每一步动作层。

但当任务复杂到单个阶段已无法稳定承载时，应进一步拆成 **Work Unit（最小执行单元）**。

### 9.1 何时只写到 Phase 即可
- 阶段内动作较少  
- 主责关系清晰  
- 不需要多人并行接力  
- 阶段内无需正式单元级交接

### 9.2 何时必须下钻到 Work Unit
- 单一阶段内部已存在多个正式交接点  
- 单一阶段涉及多个责任人或多 Agent 串联  
- 需要更细的阻塞追踪、验证或重做控制  
- 阶段内容已经复杂到“只有阶段标题不足以承载执行”

### 9.3 Work Unit 最低挂接要求
当 Plan 下钻到 Work Unit 时，每个单元至少应补充：
- Unit ID  
- Owner  
- Goal  
- Deliverable  
- Location  
- Verify  
- Depends On  
- Next Receiver  
- Status

Plan 不一定要把全部 Work Unit 正文都塞进主体，但必须说明：
- 当前阶段是否已拆单元  
- 单元记录挂在哪里  
- 当前轮最关键单元有哪些

---

## 十、正式交接要求

凡阶段出口涉及“把当前产物转为下一责任方输入”，都应视为正式交接。

正式交接至少应包含以下五字段：

1. **Where**：产物位置  
2. **What**：已完成内容  
3. **How to verify**：验证方式  
4. **Known Issues**：已知问题  
5. **Next Steps**：接收方下一步动作

### 10.1 Plan 中必须写清的交接信息
- 哪些阶段出口必须触发正式交接  
- 交接给谁  
- 交接后预期进入什么状态  
- 若交接失败，如何回写为 Rework / Blocked / Pause

### 10.2 阶段完成 ≠ 交接完成
某阶段产物已生成，不等于该阶段已经完成；
**只有产物可读取、验证方式明确、下一接收方可接住时，阶段出口才算成立。**

---

## 十一、交付、证据与位置要求

Project Plan 不只写“有什么交付物”，还必须尽量写清以下信息：

1. **Deliverable**：交付物名称  
2. **Location**：交付物位置  
3. **Evidence**：验证证据或验证入口  
4. **Receiver**：下一接收方  
5. **Archive**：必要时的归档位置

最小原则：

> **一个不可指认位置的交付物，不足以支持稳定交接。**

---

## 十二、阻塞、暂停、恢复与失败

### 12.1 Blocked
当项目因依赖缺失、权限不足、输入不明、接口未就绪等原因无法继续推进时，必须至少写清：
- 阻塞点是什么  
- 缺失项是什么  
- 谁能解除  
- 解除条件是什么

### 12.2 Pause
当当前不宜继续推进，但尚不宣告失败时，应写清：
- 暂停原因  
- 暂停范围  
- 恢复前提  
- 恢复后回到哪个状态

### 12.3 Failed
当本轮推进在当前周期内不可继续、或继续推进已不具价值时，可进入 Failed，但必须记录：
- 失败原因  
- 已停留到哪一阶段  
- 是否保留后续重开条件

---

## 十三、裁决与权限

Project Plan 至少应明确以下裁决关系：

1. **项目主责是谁**  
2. **阶段主责是谁**  
3. **阶段复核由谁承担**  
4. **整体放行由谁裁决**  
5. **打回由谁发起**  
6. **暂停由谁有权触发**

最小原则：

- 支持位不接管主任务裁决  
- 复核位可以发起打回 / 暂停判断，但不自动改写项目主责  
- 若出现冲突，应按上位 Program / Charter 的责任原则处理

---

## 十四、变更控制

凡涉及以下内容的变化，原则上都属于正式变更，应回写 Plan：

- 本轮目标变化  
- 非目标变化  
- 范围扩大或缩减  
- 阶段调整  
- 关键交付物变化  
- 验收口径变化  
- 主责 / 复核 / 接收关系变化

### 14.1 最低变更记录
每次正式变更至少应记录：
- 变更项  
- 变更原因  
- 谁确认  
- 影响到哪些阶段 / 交付 / 验收

### 14.2 不允许只口头漂移
如果变更已经影响执行与验收，而 Plan 没有回写，则该 Plan 已经失真。

---

## 十五、与 runtime / 执行链路的接口关系

Project Plan 不是 runtime 配置文件，但不能与真实执行链路脱节。

应至少区分：

### 15.1 由 Plan 承担
- 本轮目标与非目标  
- 阶段拆解  
- 责任分配  
- 阶段出口  
- 验收口径  
- 正式变更记录

### 15.2 由 runtime / 执行链路承载
- 真实调度  
- 执行日志  
- 状态回写  
- 恢复机制  
- 调用记录与运行证据

### 15.3 接口原则
- 若任务已进入 runtime pipeline，Plan 应至少指向对应运行入口或状态来源  
- 若 runtime 状态已改变到影响项目判断，Plan 应回写关键状态变化  
- Plan 不替 runtime 记流水，但必须能说明“去哪里看真实运行状态”

---

## 十六、Project Plan 的推荐结构

一份标准 Project Plan，建议至少包含以下十二段：

1. 项目卡头  
2. 任务定义  
3. 本轮目标 / 非目标  
4. 范围说明  
5. 进入条件与当前状态  
6. 阶段拆解  
7. Work Unit 挂接说明（如需要）  
8. 交付物 / 位置 / 证据 / 接收方  
9. 依赖 / 风险 / 阻塞  
10. 验收 / 打回 / 暂停口径  
11. 裁决与权限  
12. 变更记录 / 运行入口 / 当前状态摘要

## 十八、最小使用口径

如果要极简落地，Project Plan 至少不得缺以下八块：

1. 本轮目标 / 非目标  
2. 当前状态  
3. 阶段拆解  
4. 阶段主责  
5. 交付物与位置  
6. 验证方式  
7. 裁决口径  
8. 当前阻塞 / 风险

少于这八块，Project Plan 很容易退化成普通说明文。

## 十九、常见失真方式

### 1. 把 Plan 写成愿景文案
只有方向，没有阶段、责任、交付与验收。

### 2. 把 Plan 写成 SOP
步骤很细，但缺项目级边界和裁决结构。

### 3. 把 Plan 写成施工日志
只有“发生了什么”，没有“接下来按什么结构推进”。

### 4. 只有阶段标题，没有阶段出口
阶段会长期悬空，无法判断是否真正完成。

### 5. 只有交付物名称，没有位置与证据
下游接不住，复核也无法落地。

### 6. 状态只写 active / done
中间的 review / rework / blocked / pause 被吞掉，项目真实状态失真。

### 7. 只在聊天里变更，不回写 Plan
文档与真实推进脱节。

### 8. 把 Program、Plan、SOP 混写成一个东西
导致层级混乱、文档失控。

## 二十、结语

Project Plan 的价值，不在于把事情写得更完整，
而在于把一项具体任务压成：

- **知道本轮做什么 / 不做什么**  
- **知道现在处于什么状态**  
- **知道下一阶段由谁主责**  
- **知道交付物放在哪里**  
- **知道做到什么算过**  
- **知道卡住后怎么处理**

如果 Program 决定了“任务如何跑起来”，
那么 Project Plan 决定的就是：

> **这一轮，任务具体按什么推进结构往前走。**
