# Harness v0.2 小收束说明

- **状态**：收束说明 / 过渡件
- **目的**：在不继续扩搜旧版本运行时细节的前提下，对 `Harness法位与治理-运行解耦说明-v0.2-修订建议稿.md` 做一次最小收束，明确当前可保留判断、暂缓验证项与下一步转入 runtime 文档整合的边界。
- **对应主文**：`Harness法位与治理-运行解耦说明-v0.2-修订建议稿.md`

---

## 一、为什么先做小收束，不直接扩 runtime 验证

当前本机运行版本为 `v2026.4.15`，但外部已存在更高版本（用户当前关注 `v2026.4.22`）。

这意味着：
- 若现在继续围绕 `v2026.4.15` 做大量“当前运行时可验证行为”扩搜，容易把版本差异误当结构结论
- ACP 相关 bug（尤其 thread-bound persistent / 外部承载链路）是否已在新版本修复，当前尚未重新核实
- 因此本轮更合适的动作，不是继续放大旧版本验证，而是先把当前 Harness 思路压稳，再把 runtime pipeline 文档与施工记录整合成“可更新、可复核”的检查入口

一句话：

> **当前优先级从“继续扩大旧版本验证”切换为“先收束结构判断，再准备版本更新后的复核入口”。**

---

## 二、v0.2 当前可保留的结构判断

以下判断当前仍可保留，不因运行版本升级而失效：

### 1. Harness 更适合作为治理视角下的外部运行承载框架抽象
可保留。

原因：
- 这是分层判断，不依赖某个单版本实现细节
- 即使 OpenClaw 新版本内建了更多能力，也不等于治理抽象层自动消失

### 2. workflow ≠ 长期生命运行体的宿主承载层
可保留。

原因：
- 这是结构边界判断
- 新版本能力增强不自动推翻该边界

### 3. gateway 是控制中枢与主要接入点之一，但不应默认等于唯一运行宿主
可保留。

原因：
- 本地 docs 与现有 runtime 文档都支持其 control-plane 核心法位
- 新版本若扩了 ACP / autorecall / memory，也只会强化“分层更细”，不会反向证明 gateway = 唯一宿主

### 4. runtime pipeline 更像 Harness 内部推进 / 编排机制
可保留。

原因：
- 现有 runtime/system 主案中，pipeline / executor / queue / writeback 已明显分层
- 该判断当前仍成立

---

## 三、v0.2 中暂不继续放大的判断

以下判断目前不删除，但暂不继续扩大为“强结论”：

### 1. ACP bug 的现状
当前只能写成：
- 在 `v2026.4.15` 时点，相关链路曾构成系统级阻塞或不稳定项
- 是否在 `v2026.4.22` 已修复，待版本更新后复核

不宜继续写成：
- ACP 现在仍不可用
- 外部独立承载当前仍然不成立

### 2. OpenClaw 哪些能力仍必须外置
当前只能写成待核问题：
- 新版本已内建到什么程度
- 哪些仍是插件位 / 框架留位 / 外部运行位

不宜基于旧文章或旧运行版本直接下全局结论。

### 3. resident executor 与 Harness 的映射强度
当前仍以较稳口径保留：

> resident executor 更像 Harness 执行承载层的早期形态或组成部分。

但不再继续上推为：
- resident executor = Harness
- 当前 runtime 实现已等价完整 Harness

---

## 四、当前主线切换

从这一刻起，当前主线不再是：
- 继续围绕旧版本 runtime 行为做放大式调研
- 继续围绕外部文章做解释性延展

而切换为：

### 新主线 A：整合 runtime pipeline 方案与施工记录
目标：
- 把 `runtime/system/` 中的主案、差距、实施、施工记录压成稳定入口
- 为版本更新后的复核保留统一挂载点

### 新主线 B：形成版本更新后的检查单
目标：
- 等版本切到新版本后，逐项核对 ACP / sessions / autorecall / memory / external runner 等行为
- 防止“旧版本结论污染新版本判断”

### 新主线 C：形成重新接线图
目标：
- 把治理层、pipeline、gateway、ACP、executor、host 的连接关系重新画清
- 让版本变动只影响“某条线可不可用”，不再冲击整体结构理解

---

## 五、对 Harness v0.2 的当前一句准话

> **Harness v0.2 当前先保留为治理视角下的外部运行承载框架抽象说明；本轮不继续放大旧版本 runtime 细节验证，而转入 runtime pipeline 文档整合、版本更新后检查单与重新接线图三项收束动作。**
