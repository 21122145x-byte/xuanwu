# Program：团队协作运行主协议
## v0.3（轻纠偏版）

- **类型**：Program / 团队协作运行主协议
- **适用对象**：需要拆分、交接、复核、持续推进的团队任务
- **重点场景**：Multi-Agent 协作任务
- **上位依据**：`团队协作宪法与治理总纲-v2.3-能力前置补丁版.md`
- **状态**：draft
- **定位说明**：本文件定义团队任务如何进入闭环、如何持续运行、如何交接、如何验证、如何放行 / 打回 / 暂停，以及团队默认的运行结构与承载口径。`Multi-Agent` 是本 Program 的重点适配场景之一，而非本 Program 的上位概念。本文件不替代项目计划、任务级 `program.md`、SOP、Prompt 与项目文档。

---

## 一、Program 定位

`Program` 不是 prompt，不是 SOP，也不是项目计划书。

它的职责是：

> **把团队任务从一次性指令，组织成可持续推进、可验证、可回收的运行闭环。**

对团队而言，Program 主要回答以下问题：

1. 任务如何进入正式闭环  
2. 闭环如何持续运行而不失真  
3. 任务如何被拆成可执行单元  
4. 单元之间如何交接与复核  
5. 什么时候放行、打回、暂停或收口  
6. 哪些结果应沉淀，哪些只属于当前项目  
7. 正式闭环由什么承载，能力通过什么方式进入运行体系

因此，Program 的本质不是“某类任务说明书”，而是：

> **团队协作的运行主协议。**

---

## 二、与其他层级的关系

在团队治理五层中，Program 位于 Charter 之下、Project Plan 之上：

```text
Charter
  ↓ 定义上位原则与边界
Program
  ↓ 定义团队默认闭环结构、运行机制与承载口径
Project Plan
  ↓ 定义本轮目标、阶段、责任、交付与验收
SOP / Runbook / Checklist
  ↓ 定义动作标准做法、场景处置与检查门槛
Prompt
  ↓ 定义单轮执行指令
```

各层分工如下：

- **Charter**：定义上位原则与边界  
- **Program**：定义团队默认闭环、角色机制、裁决口径与运行结构  
- **Project Plan**：定义本次任务的阶段排布与本轮推进安排  
- **SOP / Runbook / Checklist**：定义高频动作的标准做法、场景处置与检查门槛  
- **Prompt**：定义单轮执行指令

Program 不重写上位原则，也不下沉到逐步操作细则。

## 三、团队级 Program 与任务级 `program.md` 的关系

团队级 Program 与任务级 `program.md` 同属 Program 层，但承担对象不同。

### 3.1 团队级 Program
团队级 Program 负责定义：
- 团队默认运行主协议
- 默认闭环结构与默认状态骨架
- 默认角色机制与默认裁决口径
- 默认交接底线与验证底线
- 团队能力沉淀、接口接入与运行承载的基本法位

### 3.2 任务级 `program.md`
任务级 `program.md` 负责定义：
- 某一具体任务的运行闭环如何落地
- 该任务是否采用更细状态、字段、接口与节奏
- 该任务的 runtime 接入、承载方式与特殊约束
- 该任务的正式交接格式、单元结构或特化规则

### 3.3 边界规则
- 任务级 `program.md` 默认继承团队级 Program  
- 任务级 `program.md` 可以细化，但不得覆盖上位原则与核心裁决口径  
- 团队级 Program 不替任务级 `program.md` 把所有字段、模板、状态细节一次写死  
- 当任务复杂到需要独立持续运行、恢复、回收与特化状态机时，应建立任务级 `program.md`

一句话：

> **团队级 Program 定义“团队默认怎么跑”，任务级 `program.md` 定义“这个任务具体怎么落地跑”。**

## 四、适用范围

### 4.1 默认适用任务

本 Program 适用于以下任务：

- 任务需要跨多轮推进
- 任务可以拆成多个最小执行单元
- 单元之间存在正式输入 / 输出关系
- 产物需要被他人接续、复核或审计
- 任务存在质量门槛、证据门槛或放行门槛

### 4.2 重点适配场景：Multi-Agent 协作

当任务涉及多个 Agent 之间的拆分、指派、交接、复核与持续推进时，本 Program 进入其最典型使用场景。

在该场景下，Program 重点解决：
- 一单元一主责
- 多角色分工不失真
- 正式交接可继续接住
- Review / Rework / Pause / Done 可裁决
- 运行中不断点、不失忆、不靠口头续命

### 4.3 不适用范围

以下场景不建议直接套用本 Program：
- 单 Agent 单步即可完成的简单任务
- 目标尚未形成、边界极度模糊的纯讨论任务
- 无法留痕、无法验证、无法回退的高风险操作
- 一次性问答、无需交接的轻量动作

对不适用场景，默认采用更轻执行方式，而非强行套 Program。

## 五、Program 核心目标

本 Program 的目标，不是让更多人同时开工，而是让团队任务具备以下能力：

1. **可进入**：任务进入前有最小边界，不靠模糊启动。  
2. **可拆分**：复杂任务能被压成明确的最小执行单元。  
3. **可指派**：每个单元都有唯一主责。  
4. **可交接**：一个执行者的产物能稳定成为下一个执行者的输入。  
5. **可验证**：完成、待核、打回、暂停有明确判定口径。  
6. **可持续**：协作能持续运行，而不依赖单点人物反复续命。  
7. **可回收**：阶段产物、关键判断与失败原因不会无痕蒸发。  
8. **可承载**：正式闭环能够进入明确的运行承载结构，而不是长期悬空在聊天中。  
9. **可沉淀**：适合长期复用的稳定能力，不长期停留在一次性 prompt、聊天或个人手法中。

Program 服务于一句话：

> **进入 → 拆分 → 指派 → 产出 → 交接 → 复核 → 放行/打回/暂停 → 继续推进或收口**

## 六、核心治理原则

### 原则 1：先压边界，再进闭环
任务未形成最小进入条件前，不进入正式运行闭环。

最小进入条件至少包括：
- 目标能被描述
- 主责任主体能被识别
- 本轮交付物能被指认
- 范围与非范围能被区分
- 存在基本裁决口径

### 原则 2：先拆最小单元，再分配责任
不把“大任务”整体抛给某个执行者，必须先压成最小执行单元，再决定由谁处理。

每个最小单元至少应回答：
- 做什么
- 交付什么
- 放在哪里
- 如何验证
- 完成后交给谁

### 原则 3：一个单元只允许一个主责
每个最小执行单元只能有一个主责角色。

可以存在支持位、复核位、审计位，但不能出现“共同负责”而无人负责到底。

### 原则 4：正式交接必须可继续接住
任何进入正式链路的交接，都不得停留在模糊表达。

正式交接必须包含足以支持下一责任方继续推进的最小必要信息，包括但不限于：
- 产物位置或对象
- 已完成内容
- 验证方式或验证依据
- 当前未解项 / 已知问题
- 下一责任方可接续的动作

固定字段模板、频道格式与任务特化交接结构，下沉至任务级 `program.md`、交接规范卡或执行卡承担。

### 原则 5：无证据，不放行
任何“完成”“可用”“通过”“可进入下游”的判断，都必须附着于可读取的产物、证据或判定标准。

### 原则 6：打回是制度动作，不是情绪动作
复核发现问题时，应给出明确打回原因与最小修复路径，而不是笼统否定。

### 原则 7：暂停优于带病推进
当前提缺失、路径冲突、权限不足或继续推进会显著扩大错误成本时，暂停优先于强推。

### 原则 8：稳定能力优先进入能力层
对适合长期复用的稳定能力，团队默认优先沉淀为 Skill 或其他可调用能力单元，而不是长期停留在聊天、一次性 prompt 或个人手法中。

### 原则 9：正式闭环应有承载结构
正式闭环不应长期悬空在口头协作中；凡需要持续推进、交接、恢复、验证与回收的任务，原则上应进入明确的承载框架。

## 七、运行结构口径

为避免把团队运行协议写成纯抽象流程，本 Program 对当前系统中的几类运行对象作最小法位确认。

### 7.1 Skill
Skill 是团队内部沉淀的可调用能力单元，属于能力层。

它回答的是：
> 哪些稳定能力应被前置沉淀，而不是每轮重新临场发明。

### 7.2 MCP
MCP 是能力接入协议 / 标准接口层。

它回答的是：
> 某类外部能力或服务通过什么标准方式接入运行体系。

### 7.3 Harness
Harness 是正式闭环的承载框架。

它回答的是：
> 某类任务最终由什么执行壳 / 容器 / 承载框架组织起来。

### 7.4 Runtime Pipeline
Runtime Pipeline 是 Harness 内部的推进链路。

它回答的是：
> 任务在承载框架内部如何完成状态流转、执行推进、交接、恢复、验证与回收。

### 7.5 最小关系
在当前系统里，最稳的关系口径是：

> **Skill 是能力包，MCP 是接口层，Harness 是承载框架，Runtime Pipeline 是 Harness 内部的推进链。**

本 Program 只确认它们的法位与分工，不在本文件中展开技术实现、字段 schema 或具体接线方式。

## 八、角色模型

本 Program 默认区分以下角色。一个 Agent 可以兼任多个角色，但**单次动作必须存在唯一主责角色**。

### 8.1 主任务位（Task Owner）
对当前任务结果负责，并对是否进入正式闭环、是否放行 / 暂停 / 收口承担主判断责任。

### 8.2 编排位（Orchestrator）
负责拆分任务单元、建立协作秩序、跟踪阻塞与维护交接连续性。

### 8.3 执行位（Builder / Operator）
负责完成被分配的最小执行单元，产出明确交付物，并按协议发起正式交接。

### 8.4 复核位（Reviewer / Auditor）
负责验证产物是否达到本轮目标，并给出放行 / 打回 / 暂停判断。

### 8.5 收口位（Keeper / Curator）
负责对值得保留的规则、结论、模板、能力或知识进行沉淀、回挂与留痕。

本节只定义角色最小边界，不展开岗位说明书级别职责细目。

## 九、任务生命周期

团队级 Program 只定义默认状态骨架与基本裁决方向，不替任务级 `program.md` 展开全部状态实现细节。

默认状态流转如下：

```text
Inbox → Scoped → Assigned → In Progress → Review → Done
                    ↑            ↓            ↘ Rework ↗
                    └──── Blocked ────────→ Pause / Failed
```

### 9.1 默认主状态
- **Inbox**：任务已进入系统，但尚未形成最小边界  
- **Scoped**：目标、范围、最小交付物与基本裁决口径已被压清  
- **Assigned**：任务已拆分并明确主责与交接关系  
- **In Progress**：执行中  
- **Review**：进入复核  
- **Rework**：被打回后进入修复  
- **Blocked**：因依赖、权限、输入或接口问题卡住  
- **Pause**：当前不宜继续推进，但不宣告失败  
- **Failed**：当前周期内不可继续推进，或继续推进已不具价值  
- **Done**：通过本轮验证，形成可读取结果

### 9.2 状态使用原则
- 团队级 Program 只定义默认状态骨架  
- 任务级 `program.md` 可以在不违背上位裁决口径前提下做更细状态展开  
- runtime pipeline 的具体状态实现，不由本文件展开承担

## 十、最小执行单元原则

团队级 Program 只定义最小执行单元应具备的信息要求，不在本文件中固定字段 schema。

一个最小执行单元至少应具备：
- 明确目标
- 唯一主责
- 可指认交付物
- 可读取产物位置或对象
- 可说明验证方式
- 必要依赖关系
- 明确下一接收方或下一去向
- 当前状态可判断

字段命名、落盘结构、runtime schema 与任务特化模板，下沉至任务级 `program.md`、接口卡或执行卡承担。

## 十一、交付、验证与裁决

### 11.1 交付底线
交付不是“说过了”，而是形成可被下一责任方读取、接续、验证的对象。

### 11.2 验证底线
凡声称已完成的事项，原则上应有明确验证方式、证据载体或判定标准。

### 11.3 放行
放行适用于：
- 当前产物达到本轮目标
- 关键验证已完成
- 不存在足以推翻完成判断的关键未解项

### 11.4 打回
打回适用于：
- 目标未达成
- 证据不足
- 交付物不可用
- 关键风险未处理
- 声称完成但无法验证

### 11.5 暂停
暂停适用于：
- 前提缺失
- 路径冲突
- 权限不足
- 风险未澄清
- 继续推进将显著扩大错误成本

### 11.6 任务级特化
更细的验收口径、审计方式、交接字段与任务专属判定标准，下沉至任务级 `program.md`、Project Plan、SOP / Runbook / Checklist 或验收卡承担。

## 十二、与 Project Plan 的关系

Program 解决的是：

> **系统怎么跑。**

Project Plan 解决的是：

> **这一轮怎么推。**

因此：
- Program 不写本轮具体目标、阶段排布与项目时序  
- Project Plan 不自定义团队级运行机制  
- 当任务需要独立运行闭环时，Plan 应与任务级 `program.md` 配套，而不是单独替代 Program 层职责

---

## 十三、下位件清单

Program 负责定义团队默认运行主协议，不直接替代下位执行件；当任务需要进一步落地时，可派生、约束或调用以下下位件：

### 13.1 Program 层下的特化件
- **任务级 `program.md`**：用于定义某一具体任务的特化闭环、状态展开、运行节奏、交接结构与承载方式  
- **Program 接口卡 / runtime 接线卡**：用于说明 runtime pipeline、harness、外部接口或系统接线关系  
- **交接规范卡**：用于定义正式 handoff 的固定字段、频道格式、回执要求或任务专属交接结构  
- **裁决 / 验收卡**：用于定义某类任务更细的放行、打回、暂停、待核标准

### 13.2 Project 层与执行层下位件
- **Project Plan**：用于定义某一轮具体任务的目标、阶段、责任、交付与验收  
- **SOP / Runbook / Checklist**：用于定义动作标准做法、场景处置与检查门槛  
- **Prompt**：用于定义单轮执行指令  
- **验收单 / 交接单 / 审计单 / 施工日志**：用于承接某次具体执行、交付、复核与留痕

### 13.3 能力层与接口层下位件
- **Skill**：用于承接可长期复用的稳定能力  
- **MCP / 接口配置件**：用于承接标准接口接入、服务暴露与能力接线方式  
- **Harness / runtime 说明件**：用于说明承载框架、内部 pipeline 与运行环境的具体实现

### 13.4 使用原则
- 团队级 Program 只定义这些下位件各自承担什么，不替它们把实现细节一次写完  
- 下位件可以特化，但不得覆盖团队级 Program 的默认运行协议  
- 当同一内容已经有明确下位件承担时，不应继续堆积在团队级 Program 正文中

## 十四、变化、回写与沉淀

### 14.1 变化不能只留在聊天里
凡影响目标、范围、责任、交接、裁决、承载方式或运行结构的变化，均应回写正式文档，而不应只停留在临时对话中。

### 14.2 应沉淀什么
以下内容应优先考虑沉淀：
- 稳定能力  
- 稳定接口  
- 稳定交接底线  
- 稳定验证口径  
- 稳定承载方式  
- 反复出现且已被证明有效的运行结构

### 14.3 沉淀去向
- 团队级默认规则 → 团队级 Program 或上位件  
- 任务专属规则 → 任务级 `program.md` 或 Project Plan  
- 动作标准做法 → SOP / Runbook / Checklist  
- 单轮调用指令 → Prompt  
- 可长期复用的稳定能力 → Skill 或其他能力层单元  
- 接口接入方式 → MCP / 接口卡 / 运行接线文档

---

## 十五、总括

本文件不是为了让团队多一层文档，而是为了让正式任务进入一种可持续、可验证、可承载、可回收的运行状态。

团队级 Program 的最低要求，不是把所有任务细节一次写满，而是保证：

- 任务能进入正式闭环  
- 角色与主责不失真  
- 交接能继续接住  
- 裁决有一致口径  
- 正式闭环有承载结构  
- 稳定能力能进入能力层  
- 下位特化不覆盖上位协议

一句话说：

> **Program 负责让团队默认会跑；任务级 `program.md` 负责让具体任务真正跑起来。**
