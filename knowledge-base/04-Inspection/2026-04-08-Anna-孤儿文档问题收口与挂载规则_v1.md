- 触发原因：今日连续新增多份审计意见、执行卡、交接文档，但未稳定挂载上位文档 / 下位文档关系，导致文档之间只有文件存在、没有结构关系，形成孤儿文档簇。

## 一、当前问题判断

今天这批文档的核心问题，不是“内容没写”，而是：

> **文档结构关系没有落进去。**

也就是说，现在很多文档虽然已经存在，但仍然缺：
- 上位文档链接
- 下位文档链接
- 所属主线标识
- 在哪条链上继续推进的导航信息

所以你说“全是孤儿文档”，这个判断成立。

而且要明确一条边界：

> **新建 README 去假装有结构，不等于真的完成挂载。**

如果 README 只是目录页，但不把：
- 谁是上位方案
- 谁是审计意见
- 谁是执行卡
- 谁是交接文档
- 谁是结果记录

这些关系写清楚，
那只是再造一层假入口，不是修结构。

## 三、从现在开始必须执行的挂载规则

### 规则 1：审计意见文档必须显式挂上位文档
每份审计意见至少应包含：
- **审计对象 / 上游输入文档路径**
- **所属主线 / 上位方案文档路径**

最少不能少这两类。

### 规则 2：执行卡必须显式挂主方案与前置判断
每份执行卡至少应包含：
- 它服务哪份主方案
- 它基于哪份审计意见 / 风险扫描 / 评估卡

### 规则 3：交接文档必须显式挂“来源 + 去向”
每份交接文档至少应包含：
- 来源文档
- 要交给谁
- 下游要产出什么

### 规则 4：结果记录文档必须显式挂执行卡
每份“执行完成 / 结果记录 / 验收记录”类文档至少应包含：
- 对应执行卡路径
- 对应方案路径
- 对应审计意见路径

---

## 四、推荐的最小挂载模板

建议后续所有正式文档头部统一补一个区块：

```md
## 文档挂载
- 上位文档：
- 同级关联：
- 下位文档：
- 所属主线：
```

如果当前没有下位文档，也要写：
- 下位文档：待补

而不是留空。

## 五、对“README 假链接”这件事的正式判断

### 当前不建议
不建议只通过新建 README 来掩盖结构断裂。

### 当前建议
README 可以建，但只能作为：
- 真实导航页
- 真实总索引
- 真实链路入口

必须满足：
1. 能指出上位方案
2. 能指出审计意见
3. 能指出执行卡
4. 能指出结果记录
5. 能指出当前做到哪一步

否则：
> **README 不是修复，是包装。**
