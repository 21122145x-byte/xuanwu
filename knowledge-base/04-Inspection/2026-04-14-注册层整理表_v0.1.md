## 一、当前定位

这张表不是完整注册层成品，
而是先把“注册层到底应包含什么”按当前口径展开列出来。

当前目标不是一步做完，
而是先把以下问题收清：
- 注册层里应该登记哪些东西
- 当前哪些已经存在
- 当前哪些还只是方向，尚未做实
- 主入口位默认读取时，实际会碰到哪些层级对象

---

## 二、注册层的最小目标

当前注册层至少要回答 6 件事：
1. 当前有哪些现役能力
2. 默认先读谁
3. 冲突时先调谁
4. 哪些入口已废弃或仅作辅助视图
5. 不同任务类型默认先读哪一层
6. 哪些稳定经验已达到可进入注册层候选的程度

## 三、注册层栏目展开（v0.1）

### A. 现役主入口层
#### 作用
回答：项目态任务默认从哪进入。

#### 当前口径
- **项目总表** = 默认入口
- 主文 = 机制主承载位
- README = 导航入口 / 回找入口

#### 当前已知登记
| 层级 | 对象 | 当前状态 | 当前口径 |
|---|---|---|---|
| 主入口 | 项目总表 | 已成立 | 项目态任务默认先回总表 |
| 主承载位 | 主文 | 已成立 | 正式机制承载位，已 accepted |
| 导航层 | README | 部分成立 | 已能回找关键文档，但尚未完全进化为分层导航中心 |

#### 当前缺口
- README 仍偏手动补挂式导航
- 还缺“入口层级标记表”来明确：默认入口 / 辅助视图 / 历史参考

### B. 现役规则件层
#### 作用

|---|---|---|---|
| 起手规则 | 系统级注册与默认起手顺序规则卡 | 已成立 | 承接注册层 / 默认起手顺序 / 项目总表入口 |
| 审计规则 | 审计插入规则 | 已成立 | 承接阶段送审与审计插入 |
| 停损规则 | 停损规则卡 | 已成立 | 承接停损与正式分流口 |

#### 当前缺口
- 还缺 revision-needed 回转卡
- 还缺更通用轮转规则件
- 还缺停损后恢复分流细则

### D. 现役试跑与补证层
#### 作用
回答：当前哪些试跑结果已经能作为注册层中的“已验证事实”使用。

#### 当前已知登记
| 试跑事实 | 当前状态 | 当前口径 |
|---|---|---|
| 先编排后生产 | 已验证 | 起手默认动作成立 |
| 送审前最小闭环 | 已验证 | 机制不是空话 |
| 审后回转最小验证 | 已验证 | 审计结论后主入口位可继续接链 |

#### 当前缺口

#### 作用

#### 当前缺口

- 还缺“什么时候只能回聊天参考，不能以聊天定口径”的显式规则表

---

### F. 冲突裁决层
#### 作用
回答：当不同对象口径不完全一致时，先信谁。

#### 当前已知原则
按当前已确认口径，至少可先立下以下顺序：
1. 项目态阶段定位：先信项目总表
2. 机制主线定义：先信主文
3. 具体执行口径：再看对应规则卡 / 模板件
4. 历史聊天与旧计划：仅作参考，不应反向覆盖总表与主文

#### 当前缺口
- 这层还没有正式成表
- 遇到“主文 vs 规则卡”“总表 vs 展开表”“新件 vs 旧件”时，仍缺更精细的优先级条目

---

### G. 按任务类型读取层
#### 作用
回答：不同任务到底默认该读哪一层。

#### 当前可先确认的最小口径
| 任务类型 | 默认先读 |
|---|---|
| 项目态续接 | 项目总表 → 主文 → 当前阶段规则件 |
| 编排任务 | 项目总表 → 主文 → 最小单元 / 阶段门相关件 |
| 审后回转任务 | 上一轮审计结论 → 项目总表 → 阶段门 / 回转相关件 |
| 复盘任务 | 复盘件 → 项目总表 → 相关问题 / 规则件 |
| 停损 / 升级任务 | 停损规则卡 → 项目总表 → 当前阶段判断 |
| 导航修订任务 | README → 主文头部 → 项目总表 |

#### 当前缺口
- 这层还没正式成表
- 仍缺更多任务类型与读取顺序的细化

---

#### 当前缺口
