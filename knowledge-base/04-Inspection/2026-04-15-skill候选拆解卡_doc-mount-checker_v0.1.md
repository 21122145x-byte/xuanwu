## 一、为什么第二张选它

`doc-mount-checker` 适合作为第二张 skill 候选，原因和 `handoff-checker` 类似：

1. **触发条件清楚**
   - 正式文档落盘时
   - 机制文档、规则卡、SOP、审计意见、研究卡、执行卡需要补挂载字段时

2. **输入输出稳定**
   - 输入：文档正文 / 文档路径 / 当前文档定位信息
   - 输出：
     - 挂载字段缺项检查
     - 挂载字段草稿
     - 结构规范化版本

3. **风险可控**
   - 主要做链路字段检查与补全
   - 不负责替代正文逻辑
   - 不负责最终决定文档真正应落哪一层

4. **收益明显**
   - 能减少孤儿文档
   - 能降低“写完正文但链路不明”的问题
   - 能让后续回找、协作、复用成本下降

所以它非常适合作为：
> **第二批模板化 / 检查化 / 链路规范化 skill 候选**

## 二、对应的上位机制

这个 skill 候选主要承接：

### 文档挂载强制字段规则
来源：
- 团队协作宪法与治理总纲

当前稳定口径：
所有正式文档、执行卡、审计意见、研究评估卡、SOP、机制文档，默认必须显式补齐：
- 上位文档
- 同级关联
- 下位文档
- 所属主线

目的不是增加格式负担，
而是强制文档暴露链路位置，避免：
- 文档孤立
- 只写正文不写上下游
- 后续协作时重新猜它属于哪条线

## 三、这个 skill 到底做什么

### 建议 skill 名称
- `doc-mount-checker`

### skill 的最小职责
只做下面三件事：

1. **检查**
   - 当前文档是否已有挂载字段
   - 是否缺：上位文档 / 同级关联 / 下位文档 / 所属主线
   - 字段是否明显空泛

2. **补全**
   - 基于当前文档上下文，生成一版挂载字段草稿

3. **规范化**
   - 若已有挂载字段但格式混乱，整理成统一结构

### 明确不做的事
这个 skill 不应：
- 替代作者判断真正的文档归属
- 自动伪造并不存在的上下游关系
- 把模糊关系写成已确认关系
- 替代主线负责人决定“是否需要起下位件”
- 替代 Anna / 粉雪做文档治理裁决

一句话：
> 它是挂载字段检查器与草稿补全器，不是文档治理裁判。

## 四、建议的输入 / 输出

### 输入（最小版）
至少支持：
1. 完整文档文本
2. 文档路径 + 文档摘要
3. 已有但不完整的挂载字段块
4. 当前主线 / 上位件 / 同位件的已知列表

### 输出（最小版）
建议支持三类：
1. 缺项检查结果
2. 标准挂载字段草稿
3. 整理后的统一挂载字段块

## 五、为什么适合做成 skill

因为它是典型的：
- 高频重复动作
- 结构相对固定
- 出错时能明确指出缺什么
- 结果好坏容易人工复核
- 不需要高强度最终裁决

与其每次人工提醒：
- 记得补上位文档
- 记得写所属主线
- 记得把下位件挂出来

不如直接有一个 skill 来做：
- 检查
- 补齐
- 整理

---

## 六、为什么不能全自动决定文档归属

即使这个 skill 很适合做，仍然有边界：

### 不能自动做的判断
- 这个文档到底应不应该立
- 它真正的上位文档是谁
- 某条“同级关联”是强关联还是弱关联
- 是否真的该起下位件
- 这份文档到底属于哪条主线

所以：
- **字段完整** ≠ **链路真实**
- **格式正确** ≠ **归位正确**

最合理定位是：
> **链路字段检查 + 草稿补全**
而不是：
> **自动归位器**

## 八、建议的触发语境

适合触发：
- “帮我补文档挂载字段”
- “检查这份文档是不是缺上位 / 同位 / 下位 / 主线”
- “把这份机制文档的挂载整理一下”
- “这份 SOP 的挂载是不是还不完整”

不适合触发：
- 用户只是要正文，不是要治理链路
- 当前文档定位还极不清楚
- 当前是在做最终治理裁决，而不是补挂载草稿

---

## 九、当前建议的最小试跑方式

### 试跑样本建议
选 3 类样本：
1. 机制文档
2. 审计意见 / 执行卡
3. 已有挂载但格式混乱或明显缺项的旧文档

### 试跑要看什么
- 能否稳定识别缺字段
- 补出的挂载是否贴当前主线
- 会不会伪造不存在的上下游
- 会不会过度补出“下位文档”
- 会不会把弱关联误写成强挂载关系

### 最小通过标准
- 至少 3 个样本不出现明显伪造链路
- 至少 2 个样本输出明显优于手工临场补挂载
- 不出现“把文档硬挂错主线”的重大问题
