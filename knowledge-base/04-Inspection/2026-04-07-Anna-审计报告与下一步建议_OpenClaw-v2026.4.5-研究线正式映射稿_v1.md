## 二、当前已成立的部分

### 1. 研究线 / 事实线边界是清楚的
这份稿子最重要的一点是边界没有再混：
- 明确写清 `目标研究版本 = 2026.4.5`
- 明确不成立为 `2026.4.2` 本地运行事实结论
- 明确不构成本地定因、已修复或放行依据

这条我认定成立。

### 2. 当前主攻线已经收住
当前没有继续泛化成“很多线都重要”，而是正式收成两条 G3 主线：
1. `active run / listener ownership / gateway crash`
2. `reply / delivery / context leak`

这条收口是对的，也足够聚焦。

### 3. 风险映射表等级仍然够克制
当前：
- `R1 / R2 / R3` 保持在 `L2`
- `R4 / R5 / R6` 保持在 `L1`

这说明当前没有把“高相关”偷偷写成“已同根”，边界守住了。

### 4. 问题簇强弱分级仍然合理
当前：
- 两条主线 = `G3｜值得专项验证`
- `memory-core / dreaming` = `G2`
- `channel / media regressions` = `G0`

这个层级关系我认可，主次没有失真。

### 2. 仍未成立“本地修复已完成”
release 修复项、issue 簇映射、G3 主线收口，
都不能直接推出：

- 升级就会自然解决

## 四、下一步建议

### 1. 这份稿现在可以作为 v2 的正式研究输入
这一点我同意。

即：
> **v2 的设计可以正式引用这份研究线映射稿。**

### 2. v2 必须显式回答五类问题
既然当前主攻是两条 G3 线，
那 v2 至少要显式回答：
1. 当前样本是 `E1｜未续跑`，还是 `E2｜已续跑未投递`，还是 `E3｜边界错位`？
2. 是否伴随 `session rotation`
3. 是否伴随 `ownership change`
4. 是否伴随 `active run mismatch`
5. 是否伴随 `reply buffering / final_answer 收口异常 / tool result 影响`

### 3. 双版本口径继续保持
后续仍必须按：
- `2026.4.5` = 研究线 / 背景输入 / 映射判断
- `2026.4.2` = 本地事实线 / 真实收证 / before-after

两条线并行推进，不能再混写。

### 4. `memory-core / dreaming` 与 `channel / media` 继续压住边界
当前不建议让这两条线抢占当前主资源。
它们可以继续保留为：
- `G2` 背景输入
- `G0` 监控保留项

但不应进入当前 v2 主攻验证。
