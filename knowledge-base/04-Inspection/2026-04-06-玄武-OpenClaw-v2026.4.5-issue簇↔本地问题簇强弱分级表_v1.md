## 1. 使用边界

## 2. 强弱分级口径

- **G0｜背景噪音**：同版本出现，但与当前本地主线距离较远
- **G1｜弱相关参考**：问题域相邻，可作背景输入，但不足以指导当前优先验证
- **G2｜高相关参考**：与本地问题簇在现象层/链路层明显同域，值得持续盯
- **G3｜值得专项验证**：与本地问题簇高度贴近，已足以进入专门 before/after 设计

> 当前表中允许给到 G3，但其含义是“值得专项验证”，不是“已完成定因”。

## 5. 当前不建议拉得过近的映射

### 1. channel / media regressions ↔ 当前主线
- 当前没有本地直接样本
- 不宜因为同版本就强行挂入主治理问题簇

### 2. memory-core issue ↔ Curie 续跑样本
- 虽然都可能受上下文/记忆影响放大
- 但当前证据不足，不宜把 memory 问题直接解释成续跑问题根因

### 3. startup/plugin reload ↔ 单次续跑失稳
- 可能有上位关联
- 但若没有 restart/reload 邻接证据，不宜直接并案

## 6. 下一步验证建议

### 第一优先：围绕 G3 两簇补专项验证
1. **active run / listener ownership / gateway crash**
   - 看本地是否存在 active run 边界错位、listener ownership 错位、delivery 邻接异常
2. **reply / delivery / context leak**
   - 看本地是“未续跑”、还是“已续跑但未投递”、还是“转录/渲染失真”

### 第二优先：围绕 G2 两簇补背景证据
1. gateway crash / startup / plugin reload
2. memory-core / dreaming

原则：
- 先做“是否与本地主线真接上”的证据补强
- 再决定是否升为更正式的并案/治理议题

### 第三优先：G0 保持监控，不抢主线
- channel / media regressions 当前不抢本轮主线资源
