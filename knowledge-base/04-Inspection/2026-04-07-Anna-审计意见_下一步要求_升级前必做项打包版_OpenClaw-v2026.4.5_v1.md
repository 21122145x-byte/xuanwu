## 五、升级前必做项目

在进入 `2026.4.5` 受控升级验证前，以下项目必须完成：

### 必做 1：冻结升级前事实样本
至少保留并固定：
- A 组：旧主会话失败硬样本
- B 组：新 session 正常收口样本
- 当前运行版本口径：`2026.4.2`

目的：保证升级后能做 before/after 对照。

### 必做 2：写死升级目标与边界
升级前文档中必须明确：
- 目标版本：`2026.4.5`
- 升级性质：**受控升级验证**
- 不得写成：升级放行 / 升级即修复 / 升级完成结论

### 必做 3：明确升级后优先复测序列
升级前必须提前写死升级后优先复测顺序，至少包括：
1. Curie 旧主会话链
2. reply / delivery / final_answer 收口
3. ownership change
4. active run mismatch
5. gateway timeout

### 必做 4：保持支线不抢主线
升级前继续明确：
- `memory-core / dreaming` 保持 G2 背景输入
- `channel / media regressions` 保持 G0 监控保留项

不得在本轮升级前把它们拉进主验证线。

### 必做 5：保留未成立边界
升级前文档必须继续写明：
- 未完成本地定因
- 未完成修复确认
- 未形成系统级放行依据

## 六、升级后优先复测项

升级完成后，第一优先不是“看版本号变了没”，而是复测这四类：

### 1. Curie 旧主会话链
检查：
- 是否仍出现 `Request timed out.`
- `user` 是否已入链但 assistant 仍不收口

### 2. reply / delivery / final_answer 收口
检查：
- 是否仍存在 reply 未正常收口
- 是否仍疑似已续跑未投递
- final_answer buffering 是否改善

### 3. ownership / active run 边界
检查：
- 是否仍有 ownership change
- 是否仍有 active run mismatch
- 是否仍有 listener ownership 相关异常

### 4. 旧主会话 vs 新 session 差异
检查：
- A / B 差异是否收敛
- 是否仍表现为“旧主会话链集中失稳”
