## 二、当前总判断

### 1. 必须退场的
- 默认 memory slot
- 默认 auto-recall 注入链
- memos plugin 的默认活动 entry 地位
- memos plugin 的默认 allow / 默认可信基础设施地位

### 2. 可以保留但必须降格的
- `memos.db`
- 插件源码目录
- viewer 残留 / debug logs / hold / backup
- RCA / 治理文档 / SOP / skill 材料

### 3. 第一版承接层只先接最小工作闭环
- 文件层工作记忆：`MEMORY.md`、`memory/YYYY-MM-DD.md`
- 正式知识入口：`knowledge_base/`
- 显式 memory tools：`memory_search / memory_get / memory_timeline`
- 第一版监控层：progress / no-progress / stuck signal / recovery entry

### 4. 当前明确不进入第一版施工的
- retrieval policy 自动化
- durable memory 治理
- viewer / UI 修理回主线
- 继续观察 MemOS 1.0.8 作为主解法

## 三、玄武施工时的执行原则

### 原则 1：先退默认绑定，再谈保留材料
固定顺序：
1. `slot unbind`
2. `entry disable`
3. `allow remove`

### 原则 2：保留材料不等于继续默认挂载
任何“研究保留”“对照保留”“源码保留”都不能被写回默认活动角色。

### 原则 3：监控层先做轻，不要顺手做厚
第一版监控层先只接：
- progress-aware monitoring
- no-progress / stuck signal
- 最小状态切换
- 最小 recovery entry

不要在这一版顺手把 retrieval policy / durable memory 一起做厚。

### 原则 4：施工时不再现场裁边界
边界已经由 Anna + Curie 收口，玄武在执行窗口内不再现场判断：
- 某组件是否还能默认保留
- auto-recall 是否还能先留着观察
- viewer 是否应该重新回主线

本轮不是要把“新 memory system”一次做完。

1. MemOS 默认角色退出
