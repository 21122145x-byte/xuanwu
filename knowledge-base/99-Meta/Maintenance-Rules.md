# Knowledge Base Maintenance Rules v1

> 团队知识库持续运维规则。目标不是一次性整理，而是把知识库维护成长期稳定的工作基础设施。

---

## 一、总原则

### 1. 入口优先，迁移次之
遇到结构混乱时，优先补入口页 / README / Index，再做大规模迁移。

### 2. 先判落点，再存文件
任何新文件在写入前，先判断它属于：
- `00-Inbox/`
- `01-Constitution/`
- `01-Projects/`
- `02-Areas/`
- `03-Notes/`
- `03-Resources/`
- `04-Archive/`
- `30-evergreen/`
- `99-Meta/`

### 3. 不让根目录承载具体内容文件
知识库根目录原则上只允许：
- 顶层结构目录
- 极少数明确约定的根层说明件

不允许长期挂：
- RTF / RTFD
- 单篇文章
- 项目执行材料
- 中间稿

### 4. Notes 是中间层，不是垃圾桶
`03-Notes/` 用来承接：
- 阶段底稿
- 研究分析稿
- 审计前准备卡
- 执行前准备卡
- 过渡性收口件

### 5. 知识库维护是持续任务，不是一次性整理
每轮新增、迁移、归档后，都要回写：
- 入口页
- 状态页
- 必要的索引

---

## 二、新文件落点规则

## 1. `00-Inbox/`
### 放什么
- 临时收件
- 尚未判断去向的原始输入
- 临时转存待处理材料

### 不放什么
- 已明确归属的正式文件
- 长期留存的 Daily
- 已经完成整理的项目件 / 资源件 / 制度件

### 规则
- Inbox 允许短暂停留
- 进入 Inbox 的文件，后续必须判断去向
- 不允许无说明长期停放

---

## 2. `01-Constitution/`
### 放什么
- 团队制度总纲
- 执行卡
- 审计意见
- 固定主任务制度件
- 跨 Agent 协作协议
- 已转正的团队机制主线（如 Spawn 主线）

### 不放什么
- 项目专属方案
- 临时研究稿
- 纯资源收藏

---

## 3. `01-Projects/`
### 放什么
- 具体项目页 / README
- 项目专属方案、流程、样例、数据源
- 项目推进记录
- 项目专属执行材料 / Subagent 材料

### 不放什么
- 公共制度
- 公共方法论
- 公共资源收藏

### 规则
- 新项目优先补入口页 / README
- 项目专属执行材料应收在项目目录中，不挂公共层根目录

---

## 4. `02-Areas/`
### 放什么
- 公共可复用方法论
- 公共系统设计
- 公共流程知识
- 技术边界与通用能力模型

### 不放什么
- 某个项目专属执行件
- 临时准备卡
- 原始资源收藏

### 规则
- 能明确归到 `Process / Systems / Technology / Business` 子层的，不挂根层
- Areas 根层尽量只保留入口型说明件

---

## 5. `03-Notes/`
### 放什么
- 阶段底稿
- 研究分析稿
- 审计前准备卡
- 执行前准备卡
- 过渡性收口件

### 不放什么
- 项目正式页
- 公共制度正式件
- 纯参考资料

### 规则
- Notes 中的文件后续应明确去向：Projects / Areas / Constitution / Archive

---

## 6. `03-Resources/`
### 放什么
- 参考资料
- 外部文章
- 经验案例
- SOP 参考
- 黄页、清单、风格研究

### 不放什么
- 当前运行状态结论
- 项目正式方案
- 公共制度总纲

### 规则
- Resources 是“可查阅、可引用”层，不是“正式结论层”

---

## 7. `04-Archive/`
### 放什么
- 历史 Daily
- 已不再活跃的旧材料
- 被新主线覆盖但仍保留轨迹价值的旧稿

### 规则
- 归档不是删除
- 归档件仍应可追溯

---

## 8. `30-evergreen/`
### 放什么
- 高复用、低时效衰减内容
- 长期有效方法与判断框架

### 不放什么
- Daily
- 临时排错记录
- 项目推进过程稿

---

## 9. `99-Meta/`
### 放什么
- 总入口
- 当前状态页
- 维护规则
- 元信息与盘点页

### 不放什么
- 具体项目执行件
- 公共制度正文
- 临时研究稿

### 规则
- Meta 页一页只承担一个角色
- 不要重复承担“索引 + 状态 + 规则”三种职责

---

## 三、索引更新规则

### 1. 新增结构性文件后，优先更新入口页
以下情况必须考虑更新入口页：
- 新增项目目录
- 新增 Systems 主线
- 新增重要资源簇
- 新增制度主线

### 2. 迁移文件后，要检查目标层是否接得住
迁移不是“换地方乱放”。迁移后至少确认：
- 目标层已有入口页，或
- 本轮同步补一个入口说明

### 3. `99-Meta/Index.md` 只做总导航
- 不再承担状态盘点
- 不再承担维护规则
- 不再承担重复索引功能

### 4. `Knowledge-Base-Current-State.md` 只写状态
- 当前主线
- 当前缺口
- 当前优先动作

### 5. 若某页功能重复，应降级或合并
例如：
- 重复索引页
- 旧版状态页
- 被新入口覆盖的旧导航稿

---

## 四、周期巡检规则

## 1. 巡检频率
建议至少做两层：

### A. 轻巡检（高频）
检查：
- 根目录是否出现新游离文件
- `00-Inbox/` 是否新增遗留件
- `02-Areas/` 根层是否出现挂件
- 新增文件是否进入索引

### B. 结构巡检（低频）
检查：
- 入口页是否仍有效
- Projects / Areas / Meta 边界是否再次模糊
- Archive 是否开始真正承接旧材料
- Notes 是否承担了中间层职责

---

## 2. 巡检命中后的动作级别

### A. 仅记录
适用于：
- 无明显断点
- 仅有少量正常新增

### B. 提醒处理
适用于：
- 根目录游离文件
- Inbox 长期遗留
- 大类根层挂件
- 新内容未进入索引

### C. 升级审阅
适用于：
- 大范围结构失控
- 多轮未处理
- 牵涉层级边界重构
- 需要 BOSS / Anna / 玄武拍板的治理问题

---

## 五、当前维护优先级口径

### 第一优先
- 入口页完整性
- 根目录纯度
- Projects / Systems / Meta 的结构稳定性

### 第二优先
- 游离文件回收
- Inbox 清理
- Areas 根层挂件回收

### 第三优先
- 历史稿去重
- 资源索引细化
- evergreen 内容蒸馏

---

## 六、一句话收口

团队知识库的维护目标，不是“越来越整齐”这么表面，而是：

> **让知识真正有落点、有入口、有边界、有回收机制，避免再次退化成聊天残留与临时堆放。**
