# 团队 HEARTBEAT 设计原则 v1

## 一、目标

这份文档用于回答三件事：
- HEARTBEAT 应该做什么
- HEARTBEAT 不该做什么
- 不同 Agent 的 HEARTBEAT 应该如何按岗配置

一句话：

> HEARTBEAT 不是摆设，也不是伪 cron；它应该是按角色配置的轻量行为维护器。

---

## 二、先定边界：HEARTBEAT 不是什么

### 1. 不是精准 cron
HEARTBEAT 是被固定频率轮询唤醒的行为入口，
不是“每天 18:00 准点执行”的精确调度器。

### 2. 不是系统巡检大脚本
不要把 HEARTBEAT 塞成：
- 大量系统指标检查
- 重型文件扫描
- 多步复杂维护任务

### 3. 不是第二套记忆写入系统
如果长期记忆和 memory 文件已由其他机制（如 MemOS / 定时脚本）接管，
HEARTBEAT 不应再无边界重复读写，以免产生冲突或竞态。

---

## 三、HEARTBEAT 应遵守的约束

### 原则 1：轻量
单次唤醒应优先做：
- 快速判断
- 轻量检查
- 小步动作

### 原则 2：按岗配置
不同 Agent 的 HEARTBEAT 不应一刀切。
它应该服务于该 Agent 的主责，而不是让所有人都跑同一张 checklist。

### 原则 3：避免与现有机制打架
如果某项工作已经有更合适的机制负责（如 cron、MemOS、专门脚本），
就不要再让 HEARTBEAT 重复承担。

### 原则 4：优先高价值、低摩擦动作
优先做：
- 发现异常
- 暴露问题
- 触发同步
- 提醒该提醒的人

不要优先做形式主义维护。

---

## 四、按岗配置建议

### 粉雪：调度 / 流量 / 协作维护
适合的 HEARTBEAT 类型：
- 检查是否有任务积压或状态失衡
- 检查多 Agent 协作是否卡在无人收束的状态
- 必要时发起轻量调度提醒

不适合：
- 深度技术巡检
- 替别人做专业判断

### Anna：验证 / 审计 / 规则防线
适合的 HEARTBEAT 类型：
- 检查最近是否出现 FAIL / TypeError / 明显异常日志
- 发现高风险问题时发出告警
- 检查知识同步或协议执行是否明显失守

不适合：
- 承担日常业务执行
- 替其他 Agent 做持续内容生产

### 富贵：交易时段异常监测
适合的 HEARTBEAT 类型：
- 仅在交易时段进行轻量异常观察
- 发现大盘 / 板块 / 波动超阈值时发盘中异动速报
- 非交易时段优先静默

不适合：
- 全天候无差别轮询
- 把 HEARTBEAT 变成连续盯盘主流程

### 旺财：协作经验与创作教训维护
适合的 HEARTBEAT 类型：
- 检查最近协作里是否出现值得 `[KNOWLEDGE_SYNC]` 的经验
- 检查是否有创作判断、结构教训、表达方法仍停留在局部上下文
- 必要时做轻量提醒或知识回写准备

不适合：
- 系统运维巡检
- 精准时点任务

---

## 五、推荐采用的落地策略

### 第一阶段
先不上“全员通用 checklist”。
先为每位 Agent 只定义 1 个最贴近主责的 HEARTBEAT 动作。

### 第二阶段
运行一段时间后，观察：
- 是否真的产生价值
- 是否产生误触发
- 是否和现有机制冲突

### 第三阶段
只有在真实验证有效后，再考虑扩展 HEARTBEAT 内容。

---

## 六、明确不推荐的做法

- 在 HEARTBEAT 里写伪 cron 表达（如 Every Day @ 18:00）却不加可靠时间判断
- 让所有 Agent 跑同一套 checklist
- 让 HEARTBEAT 重复承担已有 cron / MemOS / 专门脚本负责的工作
- 让 HEARTBEAT 变成高 token 消耗的维护入口

---

## 七、一句话结论

> HEARTBEAT 的正确方向，不是把它做成另一套定时调度系统，而是让它成为每位 Agent 按职责运行的轻量行为维护器：少而准，触发真正值得动作的事。
