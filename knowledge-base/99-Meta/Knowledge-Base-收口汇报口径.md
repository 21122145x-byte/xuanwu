# Knowledge Base 收口汇报口径 v1

- 日期：2026-04-02
- 性质：汇报口径
- 作用：统一知识库治理的收口汇报格式，避免只报动作、不报状态与下一步

---

# 一、汇报结构

每次知识库治理汇报，默认按三层输出：

## 1. 收口动作层
本轮实际做了什么：
- 新增了哪些入口页 / 索引页
- 回收了哪些游离文件
- 收口了哪些 Meta 职责
- 清理了哪些遗留件

## 2. 当前状态层
本轮之后知识库状态发生了什么变化：
- 哪些主线更稳了
- 哪些层更干净了
- 哪些风险还没解决
- 哪些目录已经从“乱放”进入“可维护”状态

## 3. 下一步建议层
接下来该做什么：
- 继续回收
- 补入口
- 补规则
- 做审阅
- 升级给谁判断

# 三、推荐口径要求

## 1. 不只报“做了什么”
必须同时报：
- 做完后状态有没有变化
- 还有什么没解决

## 2. 不夸大“已完成”
如果只是：
- 补了入口
- 做了第一轮回收
- 收口了部分主线

就明确写“第一轮完成 / 当前阶段完成”，不要写成“知识库已整理完成”。

## 3. 不把“迁移”当成“治理完成”
迁文件只是动作，真正收口还要看：
- 是否有入口
- 是否有规则
- 是否有状态页
- 是否有持续巡检口径

## 4. 汇报里默认包含下一步
知识库治理是连续任务，不应只停在“本轮做完”。

# 五、一句话收口

知识库治理汇报的目标，不是展示工作量，而是：

> **让人一眼看清：这轮做了什么、现在到哪了、下一步该怎么继续。**
