# 通用项目 Agent 编排架构审计与建议

本文档脱离单一的“小说写作”场景，从泛通用项目（如代码工程、深度研报、复杂数据处理）的视角，对三种主流的 Agent 编排模式进行重新评估，并给出团队复用的最终建议。

## 一、 技能审计（通用视角）

### 1. Agent Team Orchestration (10k ⭐)
- **核心逻辑**：重交接、重审批的双向漏斗模型。
- **适合的项目类型**：**高容错成本、多环节依赖的瀑布流项目**。例如：复杂的全栈软件开发（需求->前端->后端->测试）、跨领域的深度研报（框架->数据抓取->图表生成->终稿撰写）。
- **通用评价**：极佳。它的“5字段交接协议”和“Spec Review（防带病开工）”是放之四海而皆准的防呆神器。

### 2. Multi-Agent CN (2.4k ⭐)
- **核心逻辑**：主节点中枢调度（纯包工头），多子节点并发执行，任务分级。
- **适合的项目类型**：**横向拓展宽、低上下文依赖的“扇出型 (Fan-out)”任务**。例如：全网舆情监控（派5个Agent去不同平台抓数据）、海量简历/简历的并发初筛、多线程漏洞扫描。
- **不适合的项目**：强因果链的创作（如小说、连贯性强的代码架构），因为把核心上下文拆散再组装会造成巨大的语义损耗。
- **通用评价**：在中枢控制和批量任务处理上效率极高，但对主节点的“拆解和组装能力”要求苛刻。

### 3. Agent Orchestrator (9.1k ⭐ ⚠️安全可疑)
- **核心逻辑**：动态造人（运行时生成 `SKILL.md`）、Inbox/Outbox 异步邮箱通信。
- **适合的项目类型**：**极度探索性、且对宿主系统绝对隔离的沙盒实验**。例如：红蓝对抗测试、自动化渗透、或者完全不知道下一步需要什么工具的黑盒任务。
- **通用评价**：**不推荐在生产环境使用**。动态向系统写入技能是巨大的安全漏洞。此外，Inbox/Outbox 邮箱模式（Message Passing）在大部分场景下，都劣于直接对同一个文件进行锁操作的“环境沟通 (Stigmergy)”。

### 4. Agent Council (5.8k ⭐ ⚠️安全可疑)
- **核心逻辑**：通过脚本动态创建实体代理（生成 `SOUL.md`、环境配置），并强绑定 Discord 频道进行群组管理。
- **适合的项目类型**：**大规模的人机混合社区运营，或需要快速建立大量“虚拟角色”进行沙盒推演的场景**（如：斯坦福小镇实验、剧本杀NPC生成）。
- **通用评价（不可用）**：安全风险极高。它越权访问了底层的 Discord Token 和 Guild ID，且存在 Prompt Injection（提示词注入）指标。这意味着如果我们在生产环境中跑它，攻击者可以通过特定的对话输入，劫持这个 Agent 去创建恶意子代理，或者窃取服务器的高权限 Token。
- **结论**：**封杀。禁止安装。**

## 二、 现有团队（粉雪/旺财/Anna/富贵）的通用复用能力

我们的固定阵容虽然是为了写作工坊建立的，但其底层的“职能骨架”具备极强的泛用性，只需进行少量的“角色映射”即可复用到其他领域：

| 当前角色 | 底层职能 | 通用项目中的映射 | 示例（代码工程） | 示例（商业研报） |
|---|---|---|---|---|
| **富贵** (策划) | **架构师 / 需求方** | 拆解大局，设定约束条件，画蓝架 | 系统架构师 (Architect) | 课题主编 (Editor) |
| **粉雪** (研究) | **信息源 / 外部接口** | 处理并行杂活、数据拉取、竞品对标 | 资料搜寻 / 基础脚本工 | 数据挖掘员 (Data Miner) |
| **旺财** (主笔) | **核心生成器** | 强力执行，输出大块实质内容 | 全栈工程师 (Builder) | 执笔分析师 (Analyst) |
| **Anna** (审校) | **质控 / 规则守护者**| 逻辑验证、安全审核、沉淀SOP | 测试工程师 / 审计 (QA) | 事实核查员 (Fact Checker) |

**结论**：这个团队结构不需要因为换了项目就解散，它本身就是一个经典的 **"Plan -> Source -> Build -> Verify"** 闭环。

## 三、 最终整合与保留建议

基于以上审计，建议在知识库中提炼一套**“双轨制混合调度协议”**。我们应该在保留现有优势的前提下，将优质开源技能的骨架吸收进来。

**动作建议：**
1. **保留**我们在 `02-Process/Collaboration/Protocol.md` 中已经写好的 **“5字段交接 (Handoff Payload)”** 和 **“前置审核 (Spec Review)”** 机制（来自技能 1）。这必须成为所有通用项目的底线原则。
2. **保留**我们的 **“共享环境沟通 (Stigmergy)”** 原则。无论做代码还是研报，尽量用一个主文档加锁流转，坚决抵制技能 3 的 Inbox/Outbox 垃圾文件堆积法。
3. **吸收**技能 2 的 **“扇出调度 (Fan-out Dispatch)”**。如果是类似代码项目的初期，粉雪或主控节点可以退化成纯调度员，向子进程（如多个 `subagents`）并发分派搜集任务，但规定**子进程必须将结果写入文件**再回归，而不是在消息流中互喷。
4. **封杀**技能 3 的动态 `SKILL.md` 写入行为。任何新工具和新技能，必须由 Anna 静态审核并提交，禁止 Agent 在运行中自我繁衍。
